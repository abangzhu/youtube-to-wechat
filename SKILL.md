# 内容 → 微信公众号完整工作流

把任意来源的内容变成一篇可发布的微信公众号文章。支持 YouTube 视频、网页文章、本地文档（PDF / Markdown / TXT）。整套流程对访谈、演讲、播客录像、深度报道效果最好。

---

## Step 1：获取并整理内容

### 1.0 判断内容来源类型

在开始之前，先确认来源类型，选择对应路径：

| 来源类型 | 判断条件 | 执行路径 |
|----------|----------|----------|
| YouTube 视频 | URL 包含 `youtube.com` 或 `youtu.be` | → 路径 A |
| 网页文章 | 其他网页 URL（博客、新闻、论文页面等） | → 路径 B |
| 本地文件 | 文件路径（.pdf / .md / .txt / .docx 转存版） | → 路径 C |

---

### 路径 A：YouTube 视频

#### A.1 下载字幕

**方法一：yt-dlp（最快，优先尝试）**

```bash
# 优先下载中文字幕（zh-Hans）
yt-dlp --write-auto-sub --sub-lang zh-Hans --skip-download \
  --cookies-from-browser chrome \
  -o "%(title)s" \
  "VIDEO_URL"

# 如果没有中文字幕，下载英文字幕
yt-dlp --write-auto-sub --sub-lang en --skip-download \
  --cookies-from-browser chrome \
  -o "%(title)s" \
  "VIDEO_URL"
```

**常见错误**：如果报 "Sign in to confirm you're not a bot"，加 `--cookies-from-browser chrome` 参数（需本地已登录 Chrome）。

---

**方法二：Chrome DevTools MCP（yt-dlp 失败时的备用路径）**

遇到 SABR 限制（报 `Some web client https formats have been skipped` 或 `Requested format is not available`），切换到备用路径。完整步骤见 [→ references/youtube-transcript-fallback.md](references/youtube-transcript-fallback.md)。

---

#### A.2 解析 VTT 字幕文件

使用 `scripts/parse_vtt.py` 解析字幕：

```bash
python scripts/parse_vtt.py "字幕文件目录路径"
```

输出：`full_transcript_zh.txt`（去重后的纯文本转录）

---

### 路径 B：网页文章

#### B.1 抓取页面内容

用 WebFetch 工具读取页面，prompt 设为「提取文章全文，保留段落结构和小节标题，不要摘要，不要省略」。

如果文章有分页或 WebFetch 截断，分页多次调用，拼接成完整文本后保存为 `full_transcript_zh.txt`。

抓取后记录文章的标题、作者、发布日期、来源网站，这些信息后续写来源注记时需要用到。

---

### 路径 C：本地文件

#### C.1 读取文件

用 Read 工具直接读取文件路径。PDF 较长时分段读取（每次不超过 20 页），将所有内容拼接后保存为 `full_transcript_zh.txt`。

---

### 通用：整理结构化内容（含本地化处理）

三条路径都走完后，将内容整理为带主题分节的 Markdown 文件（`transcript.md`），标注：
- 作者 / 主播 / 嘉宾名字
- 每个话题段的主要论点
- 重要引语（保留原话）

**如果原始内容是英文，整理阶段同步完成本地化处理，不要留到写稿时再改：**

| 情形 | 处理方式 | 示例 |
|------|----------|------|
| 有通用中文译法的专有名词 | 译文（原文） | 产品经理（Product Manager） |
| 缩写首次出现 | 译文（全拼） | 首席产品官（Chief Product Officer，CPO） |
| 缩写再次出现 | 直接用缩写 | CPO |
| 无合适中文译法 / 中文比英文更陌生 | 保留英文 | vibe coding、Agent、Eval |
| 俚语和口语 | 意译，不直译 | "going upstream" → 逆流而上，不是「去上游」 |
| 英文句式痕迹 | 改写为自然汉语 | 被动句改主动，长定语从句拆成两句，避免名词堆砌 |

转录完成后做一次快速通读，确认没有「翻译腔」——读起来像中文，不像翻译。

**中文表达质量——五条高频陷阱（翻译完成后逐条检查）：**

| 陷阱 | 错误示范 | 正确做法 |
|------|----------|----------|
| 先果后因（数字裸奔，意义在后）| 最后的数字，广告制作时间从 2 小时压缩到 15 分钟 | 先交代意义（效率超越整支团队），再给出具体数字 |
| 时间量词省略（「一个周末」无数量感）| 在一个周末做出产品 | 用周末两天就做出一个产品 |
| 主语丢失（英文有主语，中文省略导致语义漂移）| 能做东西这件事，正在变成一种商品 | 保留主语指代，或用代词「这种能力」接上文 |
| 「thing」在产品语境下误译为「东西」| 让人看见你的东西、在意你的东西 | 在产品/用户语境中，"thing" = 产品，应译作「产品」 |
| 方位副词「here」误译为实体「这里」| 六个月内所有人都会到这里 | 六个月内所有人都能做到这一点（到达某个状态，不是某个地点）|

## Step 2：人物背景调查

**适用范围**：访谈、演讲、播客等有具体说话人的内容，调查主播和嘉宾。如果来源是署名文章或报告，调查作者。如果是匿名文档或机构报告（没有具体说话人），跳过此步，直接进入 Step 3。

在正式分析内容之前，先对核心人物做背景调查。这一步的目的不是写简介，而是理解他们的世界观是怎么来的——只有知道一个人经历了什么，才能理解他为什么说这些话、这些话对他意味着什么。

用 WebSearch 搜索以下维度，每个人物各做一次：

**传记与职业轨迹**
- 成长背景、教育经历（什么时期、在哪里、受谁影响）
- 职业关键节点：第一份重要工作、重大转型、成名作或成名事件
- 现在在做什么、为什么做这个

**思想形成与演变**
- 他最重要的几个观点是什么，这些观点是怎么形成的？
- 思想有没有发生过明显转变？什么事件或经历触发了转变？
- 他在哪些问题上改变了主意，为什么？

**个人得失与教训**
- 他公开谈过的失败或低谷是什么？他从中学到了什么？
- 他最引以为傲的成就背后有哪些不为人知的代价或取舍？
- 有没有他反复提到、对他影响深远的某段经历或某个人？

**价值观与立场**
- 他在乎什么？他反对什么？他选边站的地方在哪里？
- 他的观点在同行中处于什么位置（主流/异见/边缘）？

调查完成后，为每位主要人物写一份「人物小传」，通常 1000-2500 字，根据资料丰富程度决定。信息收集阶段不要吝啬，宁可多收集、写稿时按需取用，也不要因为篇幅限制而丢失有价值的背景。

小传不是简历罗列，而是叙述性文字：这个人是怎么一步步走到今天的，哪些经历是真正的转折点，他的核心观点是从什么土壤里长出来的。写完之后，这份材料将在 Step 3 分析和 Step 4 写稿阶段持续参考。

## Step 3：内容分析与提炼

分三步执行，顺序不能颠倒。

**提炼的目标是洞察和分析，不是压缩。** 一个被压缩掉细节的观点，读起来像在告诉读者「他说了什么」，而不是让读者真正理解「为什么这很重要」。保留具体的数据、案例、故事和原话，它们是观点成立的血肉。一段带着具体细节的内容，比十条提炼出来的要点更有价值。

---

### 3.1 主播视角——他想传递什么？

站在主播（host/嘉宾）的立场，理解他主动选择说这些话的意图：

- 他的核心论点是什么？整期内容想让观众走出去记住一件事，那件事是什么？
- 他用了哪些案例、数据、故事来支撑这个论点？把这些具体细节记录下来，不要只记结论。
- 他刻意强调或反复提到的观点有哪些？
- 他对行业/未来的判断是什么，对接下来 12-18 个月有没有明确预测？
- 他给出了哪些可操作的建议？
- 结合 Step 2 的人物背景，他今天说的这些话，和他过去的经历或立场有哪些关联？哪些观点是他从失败或转折中形成的？

---

### 3.2 观众视角——什么对读者真正有价值？

切换到目标读者（创业者/AI 从业者/产品经理）的视角，独立判断哪些内容值得带走：

- 最反直觉、最容易被忽视的发现是什么？
- 哪些观点会让读者停下来想"啊，原来是这样"？
- 哪些方法论可以直接指导行动？
- 有没有描述了但没有命名的新角色、新工作方式、新产品范式？
- 金句筛选：信息密度高 + 有画面感 + 可以脱离上下文独立理解，用原语言记录并标注说话人
- 结合人物背景：嘉宾说这句话的底气来自哪里？知道他经历过什么，读者对他的观点会有不同的理解。

---

### 3.3 对照转录原文查漏

两个视角总结完后，回到转录原文做一次遍历，逐段确认：

- 有没有重要论点在两个视角里都没有出现？
- 有没有具体的数字、时间线、案例细节被遗漏？这些细节是观点的根基，不能因为「显得繁琐」就丢掉。
- 有没有嘉宾对未来的判断没有被捕捉到？

把补充发现合并进 3.1 或 3.2 对应的位置，再进入 Step 4。

## Step 4：写微信公众号初稿

### 文章长度

文章字数不设固定上限，根据访谈时长和信息密度动态决定：
- 30 分钟以内的短访谈，内容集中，1000-1500 字足够
- 1 小时左右的标准播客，覆盖多个话题，1500-2500 字合理
- 超过 1.5 小时的长对谈，或信息密度极高的专业访谈，2500 字以上不必克制

判断标准只有一个：内容说清楚了没有。还有重要的观点、案例、数据没有放进去，就不是写完了。

### 文章结构规范

按以下逻辑组织，但**不要用标题或 Markdown 结构**：

1. **开头**：一个具体场景或反常识的事实切入，1-2 段，吸引继续读
2. **核心故事/案例**：把最有意思的那条叙事线展开讲清楚
3. **多个 Take-aways**：每个一个段落，用口语衔接（「说到」「再说」「顺便说说」）
4. **新角色/新趋势**：如果有，独立成段讲清楚
5. **结尾**：不要总结，留一个开放问题或引发思考的句子

### 交代来源与人物身份

文章要让读者在前三四段内清楚三件事：节目/频道是什么（名称、大概定位）、主播是谁（一句话背景，为什么他做这期内容）、嘉宾是谁（一句话背景，为什么他说的话值得听）。

这三件事不需要单独成段或逐一罗列，而是在叙述故事或论点时自然带出。最差的做法是等了三段以后才告诉读者「本期嘉宾 XX 是 YY 大学 ZZ 教授」，那时读者已经在没有背景的情况下读了一大段内容了。

好的做法是：第一次提到某人时，在语流里顺手交代一句他的身份和来头。不是完整简历，而是「这个人为什么有资格说这些话」的最短理由。对于主播，需要让读者知道他的视角和动机，而不只是他的职位。

如果演讲或访谈围绕一个技术产品、协议或框架展开（比如 MCP、某个平台、某个工具），在介绍人物的同时顺手给它一句定义。不需要教科书式解释，「X 是一套让 A 和 B 标准化对接的协议」这样一句就够——让不熟悉这个领域的读者不至于带着疑问往下读。不要假设读者已经知道这是什么。

### 让人物立体

人物背景调查（Step 2）的结论，应该自然地渗透进文章，而不是单独成段介绍。方式包括：
- 在引用某个观点时，带出这个观点是怎么来的（「他在 X 经历之后才形成了这个判断」）
- 用嘉宾的亲身经历或失败来支撑他的论点，而不只是陈述论点
- 让读者感受到说话的人是有来历的，不是凭空发论

### 保留具体性，不要为压缩牺牲细节

文章最常见的失败模式是：把一个有血有肉的论点压缩成一句话断言，把一个具体案例变成抽象原则，把原话的力度和语境全部抹去。

写每一段时问自己：这段有没有具体的人、具体的事、具体的数字或具体的原话？如果四个都没有，这段就是空洞的。至少要有其中一个。

原则：宁可写少一点观点、写透一点，也不要每个观点都提一下、没有一个说清楚。

### 写作规范

完整规范见 [→ references/wechat-writing-rules.md](references/wechat-writing-rules.md)。核心原则：禁用 Markdown 结构（标题/列表/表格）、禁用冒号和破折号和双引号、禁用 AI 腔词语。「」只用于概念名词，直接引语融入正文或加粗。衔接词用「先说」「再说」「说到」替代「首先/其次/最后」。全文加粗不超过 3-5 处。

## Step 5：检查遗漏的高价值内容

写完初稿后，回到转录原文做一次「漏网检查」：

检查清单：
- [ ] 有没有反常识的数字或数据？
- [ ] 有没有嘉宾描述的具体失败案例细节？
- [ ] 有没有对未来 18 个月的明确判断？
- [ ] 有没有提到但没有展开的新角色或新工作方式？
- [ ] 有没有值得记住的金句还没引用？
- [ ] 人物背景里有没有某个经历或转折，和文章里的某个观点特别契合，但还没有用上？

如果有遗漏，判断：是否足够重要到要加入正文？可以加一段还是加一小节？

## Step 6：用双视角评估

**评估报告格式要求**：评估结果本身要用标准 Markdown 写，使用 `##` 章节标题、`**加粗**` 标注问题点、列表和表格呈现对比。评估报告里不要用「」做强调，改用 **加粗**。评估报告是内部工作文档，和微信文章的写作规范无关。

---

### 读者视角

- 标题能不能让目标读者（创业者/AI 从业者）停下来点进来？
- 开头 3 段能不能让人继续读？
- 每个 Take-away 有没有让人"啊对"的感觉？
- 结尾有没有留下值得思考的东西？

**文字可读性**（逐段检查）：
- 句子断句是否合理？过长的句子是否需要拆开？
- 语句是否通顺，读出来不会别扭？
- 上下句语义是否连贯，有没有意思跳跃？
- 有没有倒装句——倒装在书面汉语里不自然，除非刻意营造语气，否则改成正常语序

### 编辑视角

- 逐条对照上方写作禁区，有无违规？
- 有没有**告诉**而不是**展示**（空洞断言 vs 具体细节）？
- 段落之间衔接是否自然，还是硬切换？
- 全文读完感觉是**文章**还是**报告摘要**？

**如果原始内容是英文，对照 Step 1.3 的本地化规范复查文章：**

- 专有名词、缩写是否按「译文（原文/全拼）」格式首次标注？
- 有没有遗漏本地化、直接留着英文句式的段落？
- 嘉宾金句的中文表达意思有没有跑偏？如有歧义，可在括号内附英文原话

**事实核查（对照原始转录）：**

逐段扫描文章中的事实性陈述，确认每一条都能在转录中找到依据：
- 数字、时间线、人名、产品名是否准确？
- 有没有把嘉宾的「可能」「我们在尝试」写成了已经发生的事实？
- 有没有把两件不同的事合并成一件？

区分两类内容，处理方式不同：
- **延伸解读**（作者从原话推导出的判断、补充背景、类比）——允许，但语气上要与嘉宾原话区分开，不要混在一起让读者误以为是嘉宾说的
- **无中生有**（原文没有提及、但文章中当作事实陈述的内容）——必须删除或改为明确标注是作者推测

如果评估发现问题，列出具体修改清单，进入 Step 7。

---

## Step 7：根据评估重写

重写原则：

**如果是结构问题**：把文章内容打散，用连续段落重新组织，删掉所有标题和列表。

**如果是语气问题**：重读最难改的段落，把第一句改成一个口语化的事实陈述或场景描述。

**如果是 AI 腔问题**：找到每一个触发词，用更具体的说法替换，或者把这句话删掉——大多数 AI 腔句子删掉后文章反而更好。

**重写完成后**：再做一次 Step 6 评估，直到两个视角都通过为止。

---

## Step 8：发布准备

文章定稿后，在正式发布前完成三项准备工作。

---

### 8.1 拟标题

为文章拟 3-5 个备选标题，覆盖不同风格，供选择或组合：

| 风格 | 核心角度 |
|------|----------|
| 观点型 | 说话人的核心判断，适合有强观点的演讲或访谈 |
| 信息型 | 文章带来的关键事实或结论，适合有具体数据的内容 |
| 反常识型 | 文章里最让人意外的发现，读前和读后判断不同 |
| 场景型 | 用一个具体细节切入，让读者想知道接下来发生了什么 |
| 读者价值型 | 读完这篇文章能带走什么，直接说明 |

**选题原则**

好标题不等于博眼球的标题。出完备选后，对照以下标准选定：

- 标题是否准确反映文章核心内容？读完不会觉得被骗
- 标题是否有具体的信息量，而不是泛泛描述
- 标题是否有一个清晰的角度，让读者知道这篇文章和其他同类文章有什么不同

避开的陷阱：制造悬念但文章没有揭晓（「你绝对想不到」）、夸大重要性（「改变一切」「颠覆认知」）、说了等于没说（「X 的现状与未来」）。

选定后写入 `article.md` 第一行：`# 标题`

---

### 8.2 章节标题（按需添加）

**何时需要**：文章超过 2000 字，且存在 3 个以上主题明显不同的段落群，读者在扫读时可能迷失。如果文章本身叙事流畅、段落衔接自然、不超过 2000 字，不需要加标题。

**格式**：在对应段落群的第一段之前，单独一行写加粗短标题，不用 Markdown `##` 语法，直接写 `**标题**`。标题控制在 6-12 字，是内容的最短概括，不是 AI 式的「探索……」「深入了解……」。

**示例**：
```
**独立来自正确的依赖**

在访谈里，Honneth 说的最反直觉的……
```

标题只加在内容有明显转折或主题切换的地方，不要每段都加，加多等于没加。

### 8.3 封面图截取

根据内容来源类型，选择对应方式：

**YouTube 视频 — 方法一：curl 直接下载（最稳定）**

YouTube 缩略图有固定 URL 格式，不需要 yt-dlp，直接 curl 下载：

```bash
# 替换 VIDEO_ID 为视频 URL 中 ?v= 后面的部分（如 v3Fr2JR47KA）
curl -L -o cover.jpg "https://i.ytimg.com/vi/VIDEO_ID/maxresdefault.jpg"
```

这个方法不受 SABR 流式传输影响，始终可用。`maxresdefault.jpg` 通常是 1280×720。

**YouTube 视频 — 方法二：yt-dlp 下载缩略图**

```bash
yt-dlp --write-thumbnail --skip-download \
  --cookies-from-browser chrome \
  -o "cover" \
  "VIDEO_URL"
```

输出 `cover.webp` 或 `cover.jpg`。如果报 SABR 相关错误（`Requested format is not available`），改用方法一。

**YouTube 视频 — 方法三：截取指定帧**（可选特定画面）

用 Chrome DevTools MCP 打开视频，拖动进度条到合适位置暂停，截图保存为 `cover.png`。选嘉宾正脸清晰、有表情或动作的帧，或者有文字画面感的关键时刻，避开片头片尾和黑屏。

**网页文章 — 抓取 OG 图片**

大多数文章页面有 Open Graph 图片，适合直接做封面：

```bash
# 用 curl 查看 og:image meta 标签
curl -s "ARTICLE_URL" | grep -i 'og:image'
```

如果 OG 图片质量不佳，用 Chrome DevTools MCP 截取文章首屏或文中核心图表。

**本地文件 — 手动选图**

PDF / Markdown 通常没有配套图片，建议从文档内容中选取有代表性的图表截图，或根据文章主题自行准备封面图。

封面图保存路径和文件名一并写入输出文件规范里记录。

## 输出文件规范

| 文件 | 内容 |
|------|------|
| `transcript.md` | 结构化转录（带分节） |
| `full_transcript_zh.txt` | 纯文本原始转录（备查） |
| `article.md` | 最终微信公众号文章，第一行必须是 `# 标题` |

**文件名冲突处理**：如果当前目录已存在 `article.md` 且内容与本次任务无关，使用 `<slug>-article.md`，其中 slug 取视频/文章标题前 4-6 个汉字或英文关键词（小写连字符，如 `mcp-future-article.md`）。

文章末尾必须注明来源，格式根据内容类型选用：

```
# YouTube 访谈 / 播客
内容来源，[频道名称]「[视频标题]」，YouTube 链接 [URL]，嘉宾为 [嘉宾名称]。

# 网页文章
内容来源，[作者]「[文章标题]」，原文链接 [URL]，发布于 [平台/媒体名称]。

# 本地文档
内容来源，[作者 / 机构]「[文档标题]」，[发布日期或版本号]。
```
