# 公众号风格与 AI 腔排查

读取时机：写稿和编辑自审时。

## 六、AI 腔禁区

以下词语和句式一律删除，替换为更具体的说法，或直接删掉这句话。

**形容词类：**

- 「深入」「深度」「深刻」「全面」「系统」「卓越」「显著」「丰富」

**连接词类：**

- 「不仅如此」「此外」「综上所述」「总的来说」「值得注意的是」

**视角切换类：**

- 「让我们」「我们来看看」「接下来让我们」

**结构词类：**

- 任何以「在……的背景下」开头的句子
- 任何以「总结」或「综上」开头的段落
- 「不是……而是……」这类公式化对比句。除非原文必须保留强对立，否则改成直接陈述、两句递进或具体场景说明
- 「不去 X，而是 Y」「没有 A，而是 B」「而不是 X」「并非 A，而是 B」这些同一家族的变体同样要避开
- **省略「而」字的隐性变体**：「不是 X，是 Y」「不是 X，就是 Y」。这一类是 grep `不是.*而是` 抓不到的盲点，必须用宽松 grep `不是[^，。]{0,40}[，,][^是]{0,8}是` 单独扫一遍。例：「这场分享最值钱的不是清单，是几个失败模式。」「修复方法不是把指令删掉，是给均衡视角。」改写时把对比拆成正向陈述：「这场分享最值钱的经验是几个失败模式」「修复方法是把指令改写成均衡视角」

**戏剧化修辞类：**

- 「开口就要」「拆到最细，发现」「硬是」「干脆」「没有直接 X，而是 Y」这些把中性商务事实写成茶馆段子或剧本旁白的修辞
- **不必要的比喻修辞**：「简单得像玩笑」「快得像闪电」「重得像一块砖」这类拿比喻代替直接陈述的句式。改法：去掉比喻，换中性描述——「简单得像玩笑」改「非常简单」
- 写作时每出现一次戏剧化动词就问一句：原文讲者有没有表达过这种情绪？没有就砍掉，改成白描动词

**文学化动词 → 大白话动词:**

公众号读者要的是大白话,不是散文。中文写作里有一类动词偏文学化、有空间感或动态感,但放在事实陈述里会让句子显得用力过猛——读起来像在写小说,不像在讲事实。改写时换成大白话动词。

| 文学化动词(错) | 大白话动词(对) | 触发场景 |
|---|---|---|
| 滑到、跌入、爬到、蹿到、攀升 | 变成、降到、涨到、是 | 数据、状态、阶段变化 |
| 蔓延、笼罩、席卷、弥漫、漫过 | 扩散、覆盖、影响到、出现在 | 现象传播 |
| 溢出、涌入、灌进 | 超过、进入、装到 | 容量、流量 |
| 凝结、沉淀、汇聚 | 形成、留下、集中到 | 概念形成 |
| 撕开、撬开、撕裂、撬动 | 打开、改变、推动 | 局面变化 |
| 倾斜、偏移、压向 | 偏向、靠近、转向 | 立场、资源 |
| 抛出、抛了、抛掷、丢出 | 给出、提出、说了 | 提观点 / 提问题 / 提判断 |

错例 → 改写:

- 「排序的依据已经从模型滑到了组合」 → 「排序的依据已经从模型变成了组合」
- 「焦虑情绪蔓延到整个团队」 → 「焦虑情绪扩散到整个团队」/「整个团队都在焦虑」
- 「数据涌入仪表盘」 → 「数据进入仪表盘」/「数据写到仪表盘」
- 「用户增长爬到了 1000 万」 → 「用户增长到 1000 万」
- 「20 分钟里,他们抛出了三个判断」 → 「20 分钟里,他们给出了三个判断」

**保留场景**:文章主题本身就是讲一个动态过程、讲者本人就用了富有空间感的表达、或者你确实需要那一刻的画面感时,文学化动词可以保留。判断标准:删掉文学化动词换成大白话之后,事实有没有损失?没损失就换;有损失(比如丢了关键的"过程感"),保留。

特殊说明:**「抛出」是公众号文章里最高频的轻度戏剧化动词之一**。「他抛出一个观点」「他抛出一个问题」「他抛出一个判断」读起来很顺,但在中性事实陈述里换成「给出 / 提出 / 说了」,文章会立刻显得稳重。在多人对谈、辩论、博弈语境里(讲者是真的在"抛出挑战"),保留;在讲者单方陈述的场合,几乎都该换成大白话。

grep 触发词:`滑到` `滑向` `滑入` `跌入` `蹿到` `攀升` `蔓延` `笼罩` `席卷` `弥漫` `漫过` `溢出` `涌入` `灌进` `凝结` `沉淀.{0,3}成` `汇聚` `撕开` `撬开` `撕裂` `撬动` `倾斜.{0,3}向` `偏移.{0,3}向` `压向` `抛出.{0,5}(观点|问题|判断|看法|疑问)` `抛了(一个|个).{0,5}(观点|问题|判断)` `丢出(一个|个).{0,5}(观点|问题|判断)`

**作者视角主观评价词类：**

- 「最精彩」「最反直觉」「被低估」「很聪明」「特别真诚」「观察很硬」「立场锋利」「措辞冷幽默」「值得抄」
- **对讲者语气的评价**：「他说得很冷」「他说得很尖」「他一针见血地」「他云淡风轻地」「他半开玩笑地」——这些都是作者替读者下结论，属于同一类评价词。改法：去掉评价词，中性陈述。「还有一层他说得很冷」→「他还描述了一个当下的现实」
- **非必要动作修饰**：「直接讲了一句」「接着讲了一句」「简单地说」「缓缓地说」「接着一口气讲完」——这些修饰并不增加信息量。改法：「Marc 接着直接讲了一句」→「Marc 接着讲」
- 展示而非告诉的原则：让读者自己觉得精彩，不要你告诉他这很精彩。写完文章全文 grep 这些词，命中即改或删
- 改法：删掉评价词直接给事实；或者把评价换成具体动作（「这个设计让 writer 和 reviewer 各跑一套独立上下文」比「这是最精彩的设计」更有说服力）

**对客户 / 同行的主观揶揄类：**

- 「AI 味越足越好」「玩 PR 套路」「收智商税」
- 触发情境：讲者在原视频里对客户 / 对手有轻度调侃，作者把这种调侃的语气放大成正文里的态度
- 改法：客观陈述对方的诉求或做法，不评判动机。原文是 "sounded really well for the grant because it was AI focused" 就写「他们需要一个 AI Native 的产品来匹配这笔资助」，不改成「AI 味越足越好」

**翻译腔词类：**

- 「理由是」（reason is）→ 改「原因是」
- 「事实上」（in effect / in fact）→ 改「实际上」或直接删
- 「你不希望」（you don't want）→ 改成陈述句「这种做法会 X」
- 「这也是为什么」（that's why）→ 改「这也解释了为什么」或直接拆成两句
- 「内核压成 N 句话」「浓缩成 N 句」（compress to N sentences / boil down to N points）→ 改「N 个 Take Away」「N 句话总结」或者直接给那几句不带 framing
- 「核心在于」「精髓是」「本质上是」「究其根本」（the core is / the essence is / fundamentally）→ 直接给具体内容，不要先给抽象框架词
- 「让我们看看」「我们来分析一下」「我们来聊聊」（let's look at / let's analyze）→ 直接进入分析，删掉招呼语
- 「不能不提」「不得不说」（we cannot but mention / it must be said）→ 直接说要讲的内容，删掉自述铺垫

grep 触发词：`内核.*压成` `浓缩成.{0,3}[一二三四五六七八九十0-9]` `精髓.{0,3}是` `核心.{0,3}在于` `本质上.{0,3}是` `究其根本` `让我们看看` `我们来分析` `不能不提` `不得不说`

**口语化归纳句要书面化**

定义：用偏俚的口语短语作为正文的归纳句。这些短语做引子（标题、段首）有节奏感，但作为正文里说"通常会怎样"的判断时，会让公众号文章显得草率或调侃过头。

典型错例：

- 「让 agent 跑得久，**卡在三个地方**。」（"卡在 X 个地方" 是口语化俚语化的归纳）
- 「这件事**绕不开** evaluator 的诚实度问题。」
- 「模型走到这一步是个**挺好的**信号。」（"挺好的" 是口语形容词）
- 「Ash 在台上专门停下来说了一段**不太好听的实话**。」（"不太好听" 是口语评价）
- 「**坑就在这**。」/「**难就难在**这里。」

改写方向：换成书面化表达。

- 「卡在三个地方」 → 「通常有三个问题」「难点有三个」「常见瓶颈是三类」
- 「绕不开」 → 「核心问题是」「无法回避」
- 「挺好的信号」 → 「值得记下的信号」「一个积极信号」
- 「不太好听的实话」 → 「一句直接判断」「一段不留情面的判断」
- 「坑就在这」/「难就难在」 → 「难点在 X」「问题出在 X」

grep 触发词：`卡在.{0,5}地方` `绕不开` `绕在 .{1,3} 上` `坑就在` `难就难在` `挺好的` `挺 .{0,3} 的` `不太好听` `不算 .{0,3} 的话`

### 6.x 强调副词稀释信任

慎用「真的」「确实」「果然」「真正」这类强调副词。书面文本里读者本就预期断言为真，加强调副词反而暴露作者底气不足，有 AI 自我说服的味道。

**反例**：evaluator 用 Playwright 真的打开浏览器、点来点去
**改写**：evaluator 用 Playwright 打开浏览器、点来点去

**反例**：Anthropic 团队真的做过一个练习
**改写**：Anthropic 团队做过一个练习

**保留场景**：仅当强调副词承担**反预期反差**（读者本来不信，作者要点出真有此事）时才用，且每千字不超过 1 处。

grep 触发词：`真的` `确实` `果然` `真正` `的确` `事实上`

### 6.x 主语态度修饰冗余

当一句话已经在后文用具体描述支撑某个判断，前置的态度修饰从句（「故意做得很高层」「刻意保持低调」「特意挑了简单方案」「有意识地」）往往无信息量，删掉不损失任何意义。

**反例**：Anthropic 又加了一个 planner，但它故意做得很高层。planner 接到一句话 prompt，只把整体工作流分解成一组 sprint，不规划颗粒度细的技术细节。
**改写**：Anthropic 又加了一个 planner。planner 接到一句话 prompt，只把整体工作流分解成一组 sprint，不规划颗粒度细的技术细节。

检查方法：删去态度从句后，如果读者从后文一样能读出「这是设计选择」，就该删。

grep 触发词：`故意` `刻意` `特意` `有意识地` `专门 .{0,3} 做`

## 六之二、给事实加戏与施事受事完整性

这两类问题不是 AI 腔词汇触发的，而是句子骨架层面的错误。一旦发生，读者能直观感到文章「在演」或者「漏了什么」，但往往说不出是哪里出了问题。编辑阶段必须显式检查。

**给事实加戏 / 无中生有一个态度**

定义：作者在文章里暗示了原视频 / 原文**没有**表达过的态度、反转或姿态。

典型错例：

- 原视频里 Louis 只是直接和客户沟通，没有表达「拒绝 / 犹豫 / 本来要接」的态度。但写作时用「Louis **没有直接接活**，而是……」——「没有直接接活」暗示了「本来要接但没接」的戏剧反转，读者会误读讲者的姿态
- 原视频里 Paul 只是选了方案 B，没有表达「曾经尝试 A 但失败」。但写作时用「Paul 放弃了这条路」——「放弃」暗示了曾经走过

判断方法：每次写完带有态度动词的句子（「没有 / 放弃 / 拒绝 / 选择 / 转向 / 反思」），回到原视频或转录，检查讲者是否真的表达过这层情绪。没有就改成白描。

这条规则和 SKILL.md 6.3 节的「延伸解读 vs 无中生有」不同。6.3 针对**事实**是否原文提过（补充背景、类比是允许的，纯事实虚构要删）；这里针对**态度 / 姿态**是否原文暗示过，即使事实层面没错，多加一层戏剧情绪也算无中生有。

**施事 / 受事关系完整性**

定义：动词描述了动作但漏掉关键主体。常见于「找上门 / 联系 / 接洽 / 提到 / 批评」这类关系动词。

典型错例：

- 「加拿大一家 CRM 平台找上门」——找上门找的是谁？必须补全：「加拿大一家 CRM 平台**找到 Towards AI**」
- 「客户提到了这个功能」——对谁提到？必须补全受事

检查方法：文章里每个关系动词前后都必须能回答「谁对谁做了这事」。编辑阶段用 grep 扫 `找 / 联系 / 接触 / 问 / 提到 / 批评 / 指出`，逐一确认施事和受事都在句子里出现。

**英文名词化 → 中文动词化**

定义：英文里常见的 "its own X call / the Y-ification of X / the processing of Z" 这类名词化结构，直译成中文读起来生硬。中文更自然的表达是换成动宾结构。

典型错例：

- 英文 "each tool can have its own LLM call" → 直译「每个 Tool 有自己的 LLM 调用」。改法：「每个 Tool 可以自己调用 LLM」
- 英文 "the generation of marketing content" → 直译「营销内容的生成」。改法：「生成营销内容」
- 英文 "the validation of results" → 直译「结果的验证」。改法：「校验结果」

触发特征：`X 的 Y 调用` / `X 的 Y 生成` / `X 的 Y 检测` / `Y 化的 X`。编辑阶段 grep 扫 `的.{1,8}调用 / 的.{1,8}生成 / 的.{1,8}处理 / 的.{1,8}验证`，凡是能改成动宾的都改。

**英文修饰前置 → 中文自然语序还原**

定义：英文里 "the X they built on top of Y" 这类后置定语 / 介词短语，直译成中文会变成把整段动作硬塞进定语，读起来不自然。中文倾向于把动作 / 状语前置，定语保持简短。另外，英文里 "X that the generalist AI doesn't have" 这种关系从句直译成中文会漏掉「是…所没有的」的句式骨架，变成倒装。

典型错例：

- 英文 "the automations they built on top of your product" → 直译「他们建在你产品之上的自动化」（把"建在你产品之上"硬塞进定语）。改法：「他们在你产品之上搭建的自动化」（把"在你产品之上"作前置状语）
- 英文 "specific logic that the generalist AI doesn't have" → 直译「specific logic 通用 AI 没有」（漏了"是…所没有的"的句式骨架）。改法：「specific logic 是通用 AI 所没有的」
- 英文 "build on top of your product" → 直译「在你产品之上建东西」。中文工程语境下"建"单字偏轻，改用「搭建 / 开发」：「在你产品之上搭建东西」

判断方法：每次写完一个有英文影子的句子，把它念一遍，问自己——这是不是中国人自然会用的语序？如果定语过长、动词单字、主谓宾被截断成定语，就调。

中文动词选词偏好：英文 "build" 在产品 / 工程语境里译中文，单字「建」偏轻，「搭建 / 开发」更自然。同理 "make / create" 在抽象内容生产里译「生成 / 制作 / 写出」更准。

**被动 / 容器隐喻 → 主动拥有关系**

定义：把抽象实体放在位置容器里（「X 留在 Y 里」「X 保存在 Y 中」），读者要多花一步才能 parse 出真正的拥有关系。中文更自然的是让真正的持有者做主语。

典型错例：

- 「全局上下文留在那个唯一的 Agent 里，它同时是 Planner 和决策者」——主语是"上下文"，行动者是"留在"，Agent 变成容器。改法：「唯一的 Agent 拥有全局上下文，这个 Agent 同时是 Planner 和决策者」——Agent 做主语，关系直接

判断边界：如果 Y 真的是一个**位置/存储**（工作目录、内存、磁盘），「X 保留在 Y 里」是合理的。如果 Y 是一个**实体角色**（Agent、Reviewer、Writer、系统），让 Y 做主语用「Y 拥有 / 持有 X」更好。

**弱代词 → 具体名词重复**

定义：跨句或句末的 `它 / 这个 / 那个 / 他 / 她` 作为唯一指代，中文读者跟不上。

典型错例：

- 「教别人怎么搭它」——「它」指代前面的"系统"，但读到句末时读者已经回溯一次。改法：「教别人怎么搭建这套系统」
- 「两个 Agent 放同一个项目，它能同时看懂两边的代码」——「它」指代模糊（是"项目"还是"放法"）。改法：把主语换成具体的东西（「这样的目录布局让 AI 辅助工具能同时看懂两边的代码」）

判断规则：段落内第一次指代可以用代词，但跨段、章节收尾句、定义句、总结句必须用具体名词。特别是句末的"搭它 / 做它 / 改它"这种含混收尾，一律换成具体名词。

**预告-展开结构（「只有一个，…」式）**

定义：前半句做悬念引导（「只有一个」「只有一句话」「原因只有一个」「答案很简单」），后半句才给真正内容。形式上制造节奏感，实际上多数是 filler。

典型错例：

- 「他们想回答的问题只有一个，AI 工程师的判断力到底应该放在哪里」——前半句"只有一个"是纯预告，没有信息量。改法：「他们想回答的是 AI 工程师的判断力到底应该放在哪里」

判断边界：如果前半句本身有独立信息量（比如「他们问过十几个客户，答案出奇地一致——X」），可以保留。纯节奏性的预告（「只有一个 / 只有一句」），删掉。

**拟人化 / 诗意动词用于工程语境**

定义：把文艺动词用在工程技术描述里，会把读者从技术上下文拉到散文上下文。

典型错例：

- 「判断力应该**长在**哪里」——"长"是植物隐喻，工程文里不用。改法：「判断力应该**放在**哪里」或「**体现在**哪里」
- 「产品**生根**在用户工作流里」——"生根"是植物动词。改法：「产品**嵌入**用户工作流」或「**贴着**用户工作流做」
- 「让飞轮**转起来**」——"飞轮"是商业金句式隐喻，过度使用

触发词：`长在 / 生根 / 扎根 / 落脚 / 落地（当做动作而非结果时） / 开花结果 / 飞轮转起来 / 土壤`。

判断边界：如果引用原作者的原话并且原话就是这么说的，可以保留（属于原文金句）。作者自己写时避免。

**引入新概念要给一个迷你说明**

定义：第一次出现某个专名、自造术语或比喻时，要让读者在当下就能猜到它是什么，而不是再读三段才能 infer。

典型错例：

- 「Louis 开场先画了一条**自主性滑杆**」——"滑杆"是比喻，读者不知道它是条什么线。改法：「Louis 开场先画了一条**关于系统自主性的滑杆**」
- 「Paul 采用了**评估器-优化器循环**」——直接给术语名，读者猜不到它干什么。改法：「Paul 采用了**评估器-优化器循环（写一轮、审一轮、改一轮的反复打磨流程）**」

检查方法（通读级，不是 grep 级）：通读时问每一个英文术语和比喻「如果读者不认识这个词，文章在这里或临近位置有没有给出说明」。如果要往下读三段才能拼出意思，前面必须加一句。

边界：业界通用且 AI 工程师熟知的术语（RAG / F1 / Pydantic / few-shot）不用解释；作者自造术语或比喻（滑杆 / 拳头产品 / 水坝）必须解释。

**段落间的逻辑衔接词不能省**

定义：上段和下段之间有延续、对比、转折、补充四种关系，字面上必须通过衔接词表达出来。省掉衔接词的时候，读者的脑子要自己补连接，读起来断。

典型错例：

- 上段讲了正例（Agent 合适的场景），下段开头「反例很具体」——缺「也」。改法：「反例**也**很具体」
- 上段讲 Agent 的优点，下段开头「这种克制带来一条反直觉的工程约束」——缺「但」。改法：「**但**这种克制带来一条反直觉的工程约束」

检查方法（通读级）：每段首句读一遍，问自己「这段和上段是什么关系？是**延续 / 对比 / 转折 / 补充**四种中的哪一种？这个关系在字面上表达了吗？」

衔接词参考：延续（还 / 同样 / 进一步）、对比（也 / 相反 / 反过来）、转折（但 / 不过 / 虽然）、补充（此外 / 另一个）。

**列举推论类**（用于多个独立观点的有序串联，常见于"主轴拆推论"段落）：首先 / 因此 / 同时 / 最后 / 接下来 / 进一步。

**语气转折类**（用于轻微转折、强调反预期、不破坏正向推进的回马枪）：其实 / 反而 / 但实际上 / 相反 / 事实上（注意：「事实上」属于强调副词，慎用，详见 §6.x 强调副词稀释信任）。

新增反例 / 改写对照（来自 IlqJqcl8ONE session）：

- 「主轴是这样的。模型在变强。两个推论:你越想约束模型,代价越大;Agent 跑得越久,核查必须前移。三层进阶就是把这两条推论分别落到 ...」 — 三个独立推论挤一段，缺列举标记。改写：「**首先**,模型在变强,Agent 能跑越来越长的任务。**因此**,你越想约束模型,代价越大;**同时**,Agent 跑得越久,核查必须前移。**最后**,需要将以上两条推论分别落到 ...」
- 「你也一样。Claude 比你更擅长把它挖出来。」 — 上一句「用户讲不清自己要什么」，下一句直接「你也一样」，缺语气转折标记。改写：「**其实**,你也一样。」
- 「信息密度高的 HTML spec 让后续迭代次数少很多」 — 缺「反而」表达「直觉认为费 token，实际相反」。改写：「信息密度高的 HTML spec **反而**让后续迭代次数少很多」
- 「这一条乍看像小技巧,实际是一个反复出现的工程瓶颈。」 — 缺「但」标转折。改写：「这一条乍看像小技巧,**但实际上**是一个反复出现的工程瓶颈。」
- 「烂 prompt 长这样,`make it better`」 — 上一段讲好 prompt，下一段是反例，缺「相反」。改写：「**相反**,烂 prompt 长这样,`make it better`」
- 「这条工作流明确不推荐 Sonnet。Sonnet 的视觉模型不够强。」 — 两句间是因果关系，缺「因为」。改写：「这个方法明确不推荐 Sonnet,**因为** Sonnet 的视觉模型不够强。」

为什么写稿时容易漏：列举推论类和语气转折类的衔接词在中文里常被当成"可以省略的副词"。但它们承担的不是修饰，是逻辑骨架——把它们删掉，读者不知道几条事实之间是顺序、并列还是因果。

判断方法（通读级）：写完每段，逐句读相邻两句，问"这两句的逻辑关系是什么？字面上有没有标出来？"。没标出来又不能从语序自然 infer，就补一个衔接词。

grep 不容易精确触发（否定型规则），靠通读。如果想做兜底扫描，找连续句号但缺衔接词开头的段落：把每段拆成句子列表，第二句开头是名词或主语而非衔接词时，人工核对一次。

**冗余「那个叫 X 的 Y」翻译腔**

定义：英文 "that X thing called Y" 或 "the X called Y" 直译，用「那个叫 / 所谓的 / 一种被称为」把术语包起来。中文里术语本身可以直接做定语，不用套这个框架。

典型错例：

- 「那个叫「Context Rot」的天花板」——改法：「「Context Rot」的天花板」
- 「所谓的 MCP 协议」——改法：「MCP 协议」

触发词扫描：`那个叫 / 那种叫 / 所谓的 / 一种被称为 / 一个叫做`

边界：如果这个概念在业界有多个名字，说明哪个是你采用的叫法时可以用「业界一般叫做 X / 原作者把它叫作 Y」——这时候「叫」有信息量。纯粹装饰性的「那个叫」一律删。

**抽象名词堆 → 问句或动作陈述**

定义：`X 的真正门槛 / Y 的核心挑战 / Z 的关键在于 / W 的本质是` 这种把判断或问题包成名词短语的写法，读者要 unpack 两层才能 get 到。中文里更直接的表达是问句或动作陈述。

典型错例：

- 「上多 Agent 的真正门槛也随之清晰」——三重抽象（"上多 Agent" + "真正门槛" + "随之清晰"）。改法：「什么时候应该上 Multi-Agent 也就明确起来」
- 「CTR 预估的核心挑战是数据稀疏」——改法：「CTR 预估最难的是数据稀疏」或「数据稀疏是 CTR 预估最难的地方」

触发词扫描：`的真正门槛 / 的核心挑战 / 的关键在于 / 的本质是 / 随之清晰 / 一目了然 / 跃然纸上`

改法方向：
- 「X 的真正门槛是 Y」→「什么时候才该上 X」或「Y 什么时候会让 X 失效」
- 「随之清晰 / 一目了然」→「也就明确 / 也就清楚」

**路径 / 旅程隐喻用作动作**

定义：`走这条路 / 走不通 / 绕开 / 踩过这条路 / 下山 / 转弯`，把方法选择描述成旅程。在工程文里换成白描动词更准。

典型错例：

- 「Paul 没走这条路」——改法：「Paul 没使用这个方法」
- 「这条技术路线走不通」——改法：「这个方法行不通」
- 「绕开了这个坑」——改法：「避开了这个问题」

触发词扫描：`走这条路 / 走不通 / 绕开 / 踩过 / 下山 / 转弯`

边界：原讲者原话里用了这个隐喻（加引号引用时），保留。作者自己写时不用。

**口语形容词 → 工程描述词**

定义：描述系统行为时用 `很吵 / 很糊 / 很乱 / 很死板 / 很硬 / 很油` 这种口语化、拟人化的形容词。生动但不精准，工程文里换成对应的工程词。

典型错例：

- 「这个 loop 会变得**很吵**」——"吵"是拟人。改法：「这个 loop **噪音很大**」
- 「这个模型输出**很糊**」——改法：「这个模型**精度很低**」或「这个模型**结果模糊**」
- 「架构**很乱**」——改法：「架构**结构混乱**」
- 「代码写得**很死板**」——改法：「代码**僵化**」或「代码**缺乏灵活性**」

触发词扫描：`很吵 / 很糊 / 很乱 / 很死板 / 很硬 / 很油 / 很脆`

边界：描述**人物行为**可以口语化（"他直接说"、"他一口气讲完"），描述**系统 / 流程 / 指标**不行。区别在于读者期待哪种词——对人，期待生动；对系统，期待精准。

**评价式名词结构 → 动作式主谓宾结构**

定义：以被评价对象做主语、把真正的施事"S"变成定语的句式。中文读起来主语是抽象概念，而动词是"非常+形容词"的软表达，力度被稀释。

典型错例：

- 「Marc 描述的场景非常具体」——主语是"场景"，动词是"非常具体"。改法：「Marc 描述了一个具体场景」，主语是 Marc，动作是"描述"
- 「他提出的论点很有意思」——改法：「他提出了一个有意思的论点」
- 「作者写的段落挺精彩」——改法：「作者写出了一段精彩的段落」

判断方法：每一句里问"谁是主语？谁在做动作？"。如果主语是抽象概念（场景/细节/论点/段落）而动词是"非常/很/挺+形容词"，就转回成主谓宾结构。

触发词扫描：`描述的.{2,10}非常 / 说的.{2,10}很 / 写的.{2,10}挺 / 提出的.{2,10}很 / 构造的.{2,10}很`

**概念引入要用完整主谓宾，不要用省略 / 口头表达**

定义：引入一个新名词 / 新角色 / 新职能时用"出口叫 X"、"这就是 X"、"结果就成了 X" 这类省略式的口头表达，读起来轻浮、没重量。

典型错例：

- 「于是出口叫 Builder」——"X 的出口叫 Y"是口头省略。改法：「于是被称为 Builder 的新角色诞生了」
- 「然后就是一个 Agent Harness」——改法：「接下来出现的是一种叫做 Agent Harness 的运行框架」

改法方向：用完整结构"被称为 X 的 Y 诞生了" / "出现了一种叫做 X 的 Y" / "X 定义为 Y" 来引入概念。

**段内句间的逻辑连接词要显式（模式 15 的段内版本）**

定义：这是第六之二节里「段落间逻辑衔接词不能省」的段内版本。模式 15 针对**段落之间的首句**，这条针对**同一段里相邻句子**。

典型错例：

- 「这个合伙人已经搭起了一整套 AI 系统来处理他日常工作的所有事情，这套系统成了他工作里的合作伙伴。」——两句之间是递进关系，缺衔接词
- 改法：「但是这个合伙人已经搭起了一整套 AI 系统来处理他日常工作的所有事情，并且这套系统已经成了他工作里的合作伙伴。」

检查方法：每段内每两个相邻句之间问"这两句是什么关系（延续 / 对比 / 转折 / 递进）？连接词写出来了吗？"

关于时间衔接的扩展：「10 到 20 年后我们回头看」这种表达，模糊可能指向过去也可能指向未来。改法：加"再过"或"到 20XX 年"让时间指向明确。

**同一概念在文章里用词前后要一致**

定义：前段用"场景"指代某个画面，后段换成"画面"或"细节"指代同一个东西。读者要在脑子里反复确认"这几个词是不是同一个东西"，认知负担上升。

典型错例：第 13 行「这个画面里还有一个更反直觉的细节」，第 15 行「Marc 用这个画面引出他的核心论点」——"画面 / 场景 / 细节"三个词混用却指同一个东西。

改法：全文锁定一个词。第 13 行改成「还有一个反直觉的场景」，第 15 行改成「Marc 用这个场景引出他的核心论点」。

检查方法（通读级，不是 grep 级）：文章里作为概念锚点反复出现的词（场景 / 画面 / 观察 / 论点 / 细节 / 画像 / 现象 / 时刻），在终稿阶段统计每个词的出现次数，只应活一个。辅助 grep：`场景.*画面 / 画面.*场景` 在同一篇文章里不应共存。

**中文行业术语精度**

定义：用通用词 / 口语词替代行业内约定俗成的术语。

典型对照表：

- 软件行业："造" → "开发"。"更多产品会被造出来" 应为 "被开发出来"。类似：`代码被造 / 软件被造 / 系统被造` 都要改
- 借鉴 vs 抄袭："抄" → "借鉴"。`这套体系最值得抄的是它的分层逻辑` 应为 `最值得借鉴的`。"抄" 在中文里偏负面 / 偏口语，工程文里换 "借鉴 / 参考"。例外：自嘲式 "抄作业" 可保留
- 改进 vs 修改："改" → "改进"（在描述持续优化、迭代时）。`KP 一直在改 Deep Research Agent` 应为 `在改进 Deep Research Agent`。边界：小修小补的 bugfix / refactor 用 "改" 没问题；和 "优化 / 迭代 / 完善" 在同一段并列时用 "改进"
- 解决方案的全称："解" → "解决方案"（单字 "解" 指方案时）。`绕开 context bloat 的一个反复出现的解` 应为 `一个反复出现的解决方案`。单字 "解" 在数学语境保留（"方程的解"），但在工程方案语境里要写全
- 演讲场景动名："讲的人 / 讲的两个人" → "演讲的人 / 演讲的两个人"。开场介绍讲者时，"讲的两个人是 X 和 Y" 偏口语，"演讲的两个人是 X 和 Y" 更书面
- 薪酬体系："报酬" → "薪酬"。"报酬数据 / 报酬曲线" 应为 "薪酬数据 / 薪酬曲线"。"报酬" 多用于一次性劳务
- 工程动作："搭" → "搭建"（正式语境）、"做出" → "开发出来 / 打造出来"、"跑" 描述代码正常，描述项目应为 "推进 / 运行"
- 工程推广："推到 X" → "应用到 X / 应用在 X"。`把模型能力推到具体产品和工作流` 应为 `把模型能力应用在具体产品和工作流`。"推到" 偏 push 口语，"应用到 / 应用在" 在描述把能力 / 工具放进某个场景时更书面。边界：`把代码推到 GitHub` 这种 git 语境的 "推到" 保留，只针对 "推到产品 / 推到流程 / 推到 X 中" 这种抽象推广。
- 人事："岗位 / 职位" 要看上下文选准一个
- 数据 / 仓库术语："ledger" → "明细"或"流水"，不是"账本"。"账本"是会计场景，进入开发/数据流/仓库语境（Git repo、写操作日志、事件流）时用"明细"。类似：`写进账本 / 回到账本 / 账本末尾` 都应该换成"明细"。同一篇里同一个 ledger 概念要统一成一个词

改法：写完扫一遍这些通用动词 / 名词，看有没有对应的行业专有说法。

判断边界：口语化是这套 skill 的特色，不是所有"造"都要改成"开发"——但在描述**产品开发 / 软件工程 / 薪酬体系**这种明确行业语境时，用行业内固定说法。

触发词扫描：`被造 / 制造.{0,3}软件 / 报酬.{0,3}数据 / 报酬.{0,3}曲线 / 值得抄 / 一直在改[^进良善] / 一个.{0,3}解[。，] / 讲的两个人 / 讲的人是 / 推到[^GitHub仓远端]{0,8}产品 / 推到.{0,8}流程 / 能力推到`

**同一概念在文章里同义词要择一**

定义：上面"作为概念锚点的词只活一个"主要针对叙事词（场景 / 画面 / 现象）。这条针对**意义重叠的描述性近义词**——同一篇文章里描述同一个事物用了不同中文译法或近义形容，读者得反复确认这是不是同一个东西。

常见易混对偶：

- **完成度 / 完整度 / 完整性** → 任选其一，全文统一。描述一个产品做出来"有多完整"时，"完成度"是更书面的中文表达
- **体感 / 感受 / 体验** → 任选其一
- **信号 / 迹象 / 苗头** → 任选其一
- **出乎预期 / 反直觉 / 意外** → 任选其一（注意"反直觉"是作者评价词，写公众号正文里慎用，详见第六章「作者视角主观评价词类」）
- **判断 / 结论 / 论断** → 任选其一
- **模式 / 套路 / 做法 / 玩法** → 选准用词，"玩法"偏口语

终稿操作：把每个候选词 grep 一遍，统计次数；如果两个近义词都出现，选定一个，把另一个全部换掉。多数派优先。

反例：

- 同一篇文章前段写"这个**完整度**是 harness 给的"，后段又写"两个版本**完成度**差不多"——同一概念两词混用
- 同一篇文章写了三处"反直觉"、一处"出乎预期"、一处"意外"——读者不知道作者是不是在指同一种感觉

触发词扫描：`完整度.*完成度 / 完成度.*完整度 / 反直觉.*出乎预期 / 出乎预期.*反直觉`（同一文档内两词共存即触发）

**英文数字习语 / 比喻习语要本地化**

定义：英文里约定俗成的数字习语或比喻习语直译成中文后读者看不懂。

典型错例：

- 「high nines」直译成「高 9」—— 中文读者看不懂。本地化为「90% 大几」/「99% 以上」
- 「first order of business」直译成「第一等大事」—— 应为「眼下最要紧的事」
- 「low-hanging fruit」直译成「低垂的果实」—— 业界有人用但非必需，可改「容易摘到的果实」或「显而易见的机会」

改法：翻译时对数字习语和固定习语多走一步，查一下中文对应的口语说法。

检查方法：回译自检时特别关注数字表达和固定短语。触发词扫描：`高 9 / 低垂的果实 / 第一等大事`

**引用同义冗余择一**

定义：同一段里连续引用多个说同一件事的短语，给读者的信息量是重复的。

典型错例：「底下回复都在说『我以前那家公司是 8 倍膨胀』『你这还太宽容了』」——两句是同义的。

改法：择一留下，一般选更具体 / 更有画面感的那个。这里保留「8 倍膨胀」更具体，删掉「你这还太宽容了」。

**主观评判补充句要删**

定义：客观陈述之后作者加一句主观价值判断。

典型错例：

- 「花 200 美元订 premium。**这点钱不多。**」——"这点钱不多"是作者的价值判断，对不同收入读者意义不同
- 「这个岗位年薪 50 万。**非常高。**」——"非常高"对不同行业的读者不同

改法：删掉这句。如果是原作者/讲者说的那句，在正文里明确标注是讲者原话再保留。

**段尾「可以借鉴的」桥接句换 Tips: 标签**

定义：写到段落最后想给读者一个"能拿走什么"时，常用「可以借鉴的判断:X」「可借鉴的是 X」「值得记下来的是 X」「值得带走的判断是 X」「最值得抄走的一条是 X」这种「作者评价 + 桥接」组合。这种句式有两个问题：作者下场做评价（违反 §六「作者视角主观评价词类」），同时把段落变成"事实段 + 评价桥 + 干货"的三段式，桥句本身没信息量。

中文公众号里更紧的写法是直接用 `Tips:` 这种结构标签，跳过桥句，直接给干货。读者扫到 `Tips:` 就知道接下来是可以拿走的东西，不需要作者亲自指认。

典型错例 → 改写：

- 「可以借鉴的判断:每次想跟 Claude 描述前端 UI 问题之前,先问自己一句。这个问题截图比文字更省力吗?」 → 「**Tips:** 每次想跟 Claude 描述前端 UI 问题之前,先问自己一句。这个问题截图比文字更省力吗?」
- 「可借鉴的是分层。底层 contract 用 data-attribute 暴露状态,中层用 schema...」 → 「**Tips:** 底层 contract 用 data-attribute 暴露状态,中层用 schema...」
- 「值得记下来的是,Skill 治理已经是一个工种。」 → 「**Tips:** Skill 治理已经是一个工种。」
- 「最值得抄走的一条是,把核查从日志变成证据。」 → 这一句要分两步处理:先把"抄走"换成中性表达，再把整条桥句改成 `Tips: 把核查从日志变成证据包`

判断方法：写完段落最后一句时，问自己——这一句是"事实"还是"评价 + 桥接"？是后者就换 `Tips:`。如果连 Tips 标签都没必要（段尾干货已经自然落在前一句的事实里），直接删桥句，什么都不补。

边界：`Tips:` 标签密度别太高。一篇 2000 字稿子最多 2-3 个 `Tips:`，再多读者会觉得在读 cheat sheet。如果一段段都想加 `Tips:`，回去检查段落结构——多半是把太多干货塞到一段了。

grep 触发词：`可以借鉴的判断` `可借鉴的是` `值得记下来的是` `值得带走的` `值得抄走` `最值得.{0,5}的一条`

**冗余默认属性限定词**

定义：给本来就默认具备某个属性的名词再加一个限定词，是废字。公开人物的公开发言默认是"公开的"，作者写的文章默认是"他写的"，巨头产品默认是"它旗下的"。

典型错例：

- 「Mitchell Hashimoto 的**公开**判断」——他作为前创始人在博客写的东西默认是公开的。改法：「Mitchell Hashimoto 的判断」
- 「他在 Twitter 上**公开**发的一条推」——Twitter 发推默认公开。改法：「他在 Twitter 上发的一条推」
- 「Google **旗下的** YouTube」——熟知关系不需要限定。改法：「Google 的 YouTube」或「YouTube」
- 「作者**亲自**写的」——作者署名的文章默认是本人写的，加"亲自"反而暗示"可能不是"。改法：「作者写的」

判断方法：把限定词抽掉，信息是否丢失？没丢就是默认属性冗余，删。

触发词扫描：`公开(判断|发言|说过|表示) / 旗下的 / 亲自(写|做|主持) / 本人(写|发|说)`

**时间推进衔接词用副词，不用定语**

定义：把"接过 / 站起 / 拿起话筒"这种动作做成人物前面的定语，再用"的"连到人物，是英文分词短语（participial phrase）直译。中文自然写法是把动作从定语里抽出来，用时间副词做段落衔接。

典型错例：

- 「**接过来的** Hugo 把视角从加速切到工作单元」——"接过来的 Hugo"是英文 "Taking over, Hugo..." 的分词结构直译。改法：「**接下来** Hugo 把视角从加速切到工作单元」
- 「**站起身的** Marc 说」→「Marc 站起身说」或「**接着** Marc 说」
- 「**拿起话筒的** 主持人问」→「**这时** 主持人拿起话筒问」

触发词扫描：`[来去起下住上]的 [A-Z]` / `的 Hugo / 的 Marc / 的 Madison`

**"那个 X 词 / 那个 X 东西"尾缀冗余**

定义：句尾用"那个 X 词""那种 X 做法"作为代词收尾，X 本身不承载新信息，属于装饰性回指。

典型错例：

- 「`continue` 大概是他们团队现在最常敲**的那个词**」——"最常敲"已足够指明对象。改法：「`continue` 大概是他们团队现在**最常敲**」
- 「这是他提出**的那种做法**」——"他提出"已经具体。改法：「这就是他提出的」或干脆删掉
- 「最反直觉**的那一条**」——可改成「最反直觉的一条」或去掉"那"

触发词扫描：`的那个[词句句子做法说法情况东西] / 的那一[条种个]`

**"让 X 快 / 让 X 好"里形容词要动词化**

定义：在"让 X Y"的使役结构里，Y 是形容词时，中文读起来像静态描述；改成"变 Y / 成为 Y"之后才是动作。

典型错例：

- 「要让这一切真的**快**」——"快"是状态形容词。改法：「要让这一切真的**变快**」
- 「这样做能让代码**整洁**」→「这样做能让代码**变整洁**」
- 「这个改动让循环**更稳**」→「这个改动让循环**变得更稳**」

判断边界：如果 Y 本身就是动词（"让它跑起来""让它停下"），不用变。只针对形容词 Y。

触发词扫描：`让[^，。]{1,15}真的[快好稳准稳严紧松] / 让[^，。]{1,15}更[快好稳准严紧松]`

**"不再有 X" 要补谓语**

定义："不再有 X"后面省略了谓语（X 在做什么），读者要自己补。加一个动词让陈述完整。

典型错例：

- 「External Validation **不再有人**」——不再有人什么？改法：「External Validation **不再有人参与**」
- 「系统里不再有人**在管**这件事」→对。
- 「前端页面不再有**手工录入**」→对（录入是名词也能成谓语）。

判断方法：每次写完"不再有 X"，问自己"X 在做什么/承担什么"，把动作补出来。

触发词扫描：`不再有[人团队产品]。 / 不再有[人团队产品]，`

**规范性判断要显式用"应该 / 应当"**

定义：陈述规范（事情应该怎么做）时如果只写"是 X 的地方"，中文读者会先读成事实描述；加"应该"才能把规范语气传出来。

典型错例：

- 「这个前置队列才是**人类介入的地方**」——读起来像描述事实"事实上人类就在这里介入"。改法：「这个前置队列才是**人类应该介入的地方**」
- 「这才是产品经理**做决策的场合**」→「这才是产品经理**应该做决策的场合**」

判断边界：如果上下文明确是描述（"今天在座的工程师大多在这里介入"），保留无"应该"版本。只在作者是在给规范建议时加"应该"。

**"由 X Y"显式归因**

定义：涉及"谁主导/谁负责"的关系时，"X Y"（动宾直连）容易被读成对等参与，加"由"才能明确主导关系。

典型错例：

- 「Namespace 今年 3 月宣布累计融资 2300 万美元，**NEA 领投**」——"NEA 领投"字面上像 NEA 自己做了个动作。改法：「Namespace 今年 3 月宣布累计融资 2300 万美元，**由 NEA 领投**」
- 「这次会议**张三主持**」→「这次会议**由张三主持**」
- 「产品**王五牵头**」→「产品**由王五牵头**」

判断边界：如果本来就是清楚的施事关系（"张三说"、"王五做了 X"），不用加。只在被动/主导关系需要明确时用。

**文艺/美术性名词比喻在工程文里换工程名词**

定义：规则「拟人化 / 诗意动词用于工程语境」处理了动词侧（长在 / 生根 / 扎根）；这条处理名词侧。「底色」「脉络」「骨架」「纹理」「色调」这些美术/文艺性名词比喻，在工程判断或商业判断里会把严肃陈述软化。

典型错例：

- 「也是人类评审崩不住这个论断**的底色**」——"底色"是美术词，讲的是背景色调。改法：「论断**的本质原因**」或「论断**成立的前提**」
- 「产品设计**的纹理**」→「产品设计**的细节层次**」
- 「团队的**底色**是工程文化」——作为团队/组织描述可以保留（常见表达），不算工程判断
- 「技术栈**的骨架**是 Rust」→「技术栈**的主干**是 Rust」

判断边界：描述**人、团队、文化**时这些美术词可以保留（"这家公司的底色""这位作者的纹理"属常用中文修辞）；描述**系统、指标、论断、判断**时必须换工程名词。

触发词扫描：`的底色 / 的脉络 / 的骨架 / 的纹理 / 的色调`

**作者对自己叙事行为的元评述要删**

定义：规则「判断前后的元评价」处理了作者对**内容品质**的评价（"这是最值得抄的一句"）；这条处理作者对**自己叙事行为**的元评述（"这层关系先摆出来""接下来进入第二个点""下面再展开"）。

典型错例：

- 「所以这对搭档同台，本身也是领投方合伙人和被投公司 CEO 的联合发声，**这层关系先摆出来**」——"这层关系先摆出来"是作者宣告自己要做披露，宣告动作本身没有信息量。改法：保留前半句披露，删掉"这层关系先摆出来"
- 「**先说一句**，下面两段有点技术细节」→ 删
- 「**话又说回来**，我们聊回正题」→ 删（除非用于真正的话题回切）

判断方法：作者说的是"内容"还是"我即将要做的叙事动作"？是后者的一律删。

触发词扫描：`这层关系先摆出来 / 先说一句 / 这点先放这里 / 下面展开说 / 接下来进入 / 这部分先过`

### 6.2.x 主语漂浮 / 主谓宾在一处

一句话的主语应能被读者**第一遍就读出来**，不要让主语藏在长名词短语里、要读到句末才反推。

**反例**：Retro Forge 那个 demo，这两个 agent 谈出来的 contract 总共 27 条。
（主语「contract」被「这两个 agent 谈出来的」修饰成长名词短语，读者读完才反推「27 条」对应什么）

**改写**：回到 Retro Forge 演示的那个 demo，两个 agent 总共谈出来 27 条 contract。
（主语「两个 agent」 + 谓语「谈出来」 + 宾语「27 条 contract」，标准主谓宾）

检查方法：把句子简化成「主-谓-宾」骨架，如果骨架里没有合法主语、或主语是修饰过的长名词短语，重写。

更隐蔽的失误形式：用并列从句堆叠遮蔽主语。

**反例**：custom sub-agents 已经是基础能力，你的 evaluator 和 QA 角色，给它一个严厉 system prompt 加详细 rubric 即可。
（"你的 evaluator 和 QA 角色"作为话题悬空，"给它"的主语是谁？读者得反推）

**改写**：custom sub-agents 已经是一个基础能力了，你只需要定义一个严格的 system prompt 加详细 rubric，即是一个 evaluator 和 QA 的角色。

更隐蔽的失误形式三：分句间主语切换缺承接词。

当一个长句里前半交代了动作主体，后半切换为被动接尾结构（「跑出来」「做出来」「写出来」「算出来」「弄出来」），需要在切换点用「结果」「最终」「这一来」「于是」等连接词显式承接，否则后半句的主语会"悬空"——理论上可以从前文推出来，但读者第一遍读会卡顿。

**反例**：「第一版用最简单的 prompt 加最便宜的模型跑出来烂得离谱。」（"跑出来"前缺承接，主语像漂浮在两个分句之间。读者第一遍会问"什么跑出来？"）

**改写**：「第一版用最简单的 prompt 加最便宜的模型，结果跑出来烂得离谱。」（"结果"承接前半的方法描述，把"跑出来"明确指向上一节的结果）

**反例**：「他拿了三家公司的财报花了一周时间分析得出来一个结论。」（"分析得出来"前缺承接，且整句没有清晰的句段断点）

**改写**：「他拿了三家公司的财报，花了一周时间分析，**最终**得出一个结论。」

扫描提示（很难精确 grep，扫描提示）：检查含「跑出来」「做出来」「写出来」「算出来」「弄出来」的句子，若前半是工具或方法描述、紧接被动结尾，多半缺一个连接词。`grep -nE "[^结于这]?跑出来|做出来|写出来|算出来|弄出来" article.md` 命中后人工核查。

### 6.2.x 省略指代要补回来

把口语化的省略指代（"前沿一动"、"老板一拍"、"市场一变"）显式补回主语，避免读者反推。

**反例**：前沿一动，他们就跑个简化版看看够不够用。
**改写**：前沿模型一改动，他们就跑个简化版看看够不够用。

**反例**：风口一来，所有人都开始抄。
**改写**：风口一来到，所有人都开始抄。/ 一旦风口出现，所有人都开始抄。

grep 触发词：`一动` / `一变` / `一拍` / `一来` / `一开`（后接逗号或句号且前面省略主语时命中）

### 6.2.x 抽象名词改动词短语

抽象名词作宾语或表语时，往往让句子变虚。能换成动词短语的尽量换。

**反例**：你需要这种对模型的同理心。
**改写**：你需要站在模型的视角去思考。

**反例**：团队需要更强的同理心。
**改写**：团队需要更愿意从对方角度想问题。

**反例**：这一招的本质是对约束的尊重。
**改写**：这一招的本质是不去突破已知约束。

判断方法：句尾是抽象名词（同理心、责任感、判断力、敏感度、同情心、敬畏心）时，问自己「这种能力具体表现为什么动作？」用动作描述替代名词。

### 6.2.x 副词修饰范围错位

定义：中文里副词的修饰范围由位置决定。把状态副词放到触发动词前面，会让修饰错位——读者读到副词时它的 anchor 还没出现，下意识把副词挂到错的动词上。

**反例**：「原本通过的用例突然零零散散开始失败。」（"零零散散"想修饰"失败"的状态，但放在"开始"前，第一遍读会让人以为它在修饰"开始"这个触发动作）

**改写**：「原本可以通过的用例突然开始零零散散跑不通了。」（"零零散散"挪到它实际修饰的"跑不通"前面；同时"开始失败"换成更自然的"跑不通了"，"开始失败"作为工程语境措辞偏书面）

**反例**：「服务**忽然慢慢地变卡**。」（"慢慢地"在"变卡"前是对的，但和"忽然"并置语义打架——一个在说骤变，一个在说渐变。读者要停一下消化）

**改写**：「服务**忽然变卡，而且越来越卡**。」或「**服务慢慢变卡了**。」（看作者实际意思取一个）

判断方法：找到每个状态/方式副词，问"它实际修饰哪个动词？"，再问"这个动词和副词之间是否被其他动词隔开？"。被隔开就把副词挪到 anchor 旁边，必要时把触发动词提到副词之前。

边界：当多个副词形成累积时间感（"先慢慢地，然后突然"），允许在前文铺垫，但每个副词的 anchor 都要在它之后紧邻。

触发词扫描（这一类很难精确 grep，扫描提示而非触发词）：
- 「开始 + 状态副词」组合：`开始[一-鿿]{0,3}地?[一-鿿]{0,3}`，命中后人工核查"开始"和后面动词之间是否插了状态副词
- 「突然 + 渐变副词」组合：`突然[一-鿿]{0,3}地?(慢慢|渐渐|逐渐|不断)`，命中后看是否需要拆成两句

## 六之三、来源元评论填充句

定义：文章在转述某份资料（YouTube 视频、播客、PDF、手册、博客）时，频繁用「视频里 / 手册里 / playbook 里 / 文档里 / 整本 / 这一章 / 这一段」加上一个评价或总结性形容词作为引介铺垫，但下文不依赖这个铺垫。读者从开头就知道全文基于这份资料，这种导游式 meta 句反而显得作者在替资料做导览，没有增加信息量。

典型错例：

- 「playbook 的核心立场写在最后一章。」(下一句紧接着给立场，这句是空铺垫)
- 「整本手册里最 actionable 的一条建议在 MVP 这章。」(下一句直接给建议)
- 「playbook 写得很细。」(下一句紧接着给细节)
- 「手册里有一句话直接把这件事讲透了，X」(直接进 X 即可)
- 「整本 playbook 把护城河来源讲得最透的一章在 Scale。它直接回答了…」(改成「这一章回答了…」)
- 「工作流锁定这一层逻辑也清楚。」(下一段就在解释逻辑，这句多余)
- 「手册里给了一个具体例子。X」(改成「一个具体例子，X」)
- 「playbook 的结尾把这一切收成一句。」(直接给那句)
- 「意思就清楚了。」/「这件事的含义比表面看起来更深。」(纯节奏性结尾)
- 「这是手册自己留下的金句，也是这一节标题的来源。」(读者自己能看出来)
- 「手册里有一段说得很狠」/「手册里给的预警信号挺锋利」(评价词嵌在 meta 引介里，把评价词去掉)

判断方法：删掉这句话后，下一句独立成立、信息没丢吗？没丢就删。

特殊情况：当 meta 评论本身有信息价值（指出资料这一章的位置、给读者导览读这本资料的次序、解释为什么作者特别提这一段、引出一个被命名的术语），可以保留。但作为节奏性 filler 时一律删。

grep 触发词：`playbook 写得` / `手册里` / `视频里` / `整本.*最` / `这一章.*讲得最` / `直接把.*讲透` / `意思就清楚了` / `把这一切收成一句` / `讲得很细` / `最 actionable` / `最透` / `说得很狠` / `挺锋利`

### 6.3.x 反复发言人元归属

同一发言人在文章中第二次及以后出现时，**不要**再用「X 自己说」「X 在台上专门停下来」「X 拿起话筒强调」「X 顿了顿才说」这类舞台动作描写。直接「X 认为/X 总结/X 给出」即可。这类描写第一次用一次设场景就够了，反复出现是元舞台描述，稀释主线。

**反例**：Ash 自己说，如果你仔细看，这就是个普通的 PM、IC、QA 三角组织。
**改写**：如果你仔细看，这就是个普通的 PM、IC、QA 三角组织。
（前文 Ash 已多次出现，第二次以后这层归属可省）

**反例**：Ash 在台上专门停下来给了一句直接判断，开箱即用的 Claude 是个糟糕的 QA agent。
**改写**：Ash 认为开箱即用的 Claude 是个糟糕的 QA agent。

例外：当发言人改变态度（从平和转激动、从笃定转犹豫）且这个变化对内容理解必要时，可以保留一次动作描写。

grep 触发词：`自己说` / `专门停下来` / `特别强调` / `拿起话筒` / `顿了顿` / `又补了一句` / `一字一句` / `提高了声音`

#### 归属式 / 评价式变体

上一段处理的是动作描写型（X 拿起话筒 / X 顿了顿）。还有一类更隐蔽的元归属，是把讲者当成"动作主体"去引介他的说法、判断、用词、风格。已确立讲者后，这类引介词没有信息量，应该删掉，让内容直接出场。

**反例 → 改写**：

- 「KP 用的词是「几乎是达尔文式的」」 → 「「几乎是达尔文式的」」（直接给词）
- 「KP 把痛点拆得清楚」 → 「关于 Eval，机械层面就够烦……」（直接进内容）
- 「KP 给的答案很坦白。全部自研。」 → 「关于 observability，全部自研。」
- 「KP 把下一步定义得很直白」 → 「他的解决方案是」 或直接进解决方案描述
- 「KP 描述的下一步是」 → 「下一步是」
- 「KP 想要的未来是 X」（标题或正文） → 「将 X 做成 Y 的内置能力」 / 直接命名工程方向
- 「panel 里 Eval 那一段不长，但 KP 把痛点拆得清楚」 → 「关于 Eval，……」
- 「KP 给的理由也没藏」 → 「理由也没藏」 或直接给理由

判断方法：每次写出「X 用的词是 / X 把 Y 拆得 Z / X 给的答案是 / X 想要的未来 / X 描述的下一步」时，问自己——下文会真的给出那个词 / 那个判断 / 那个方向吗？会。那直接写下文，不需要这层引介。

为什么写稿时容易写出来：作者写完一个观点引子之后，会本能用"X 把 Y 拆得很清楚"来给读者一个"接下来会有干货"的提示。但这个提示是节奏性 filler，读者不需要预告，他们直接看到干货就够了。

边界：第一次引入讲者时可以用一次「X 给出一段判断」「X 用了一个比喻」做归属。**第二次及以后**，直接让内容出场，不归属、不引介。

grep 触发词追加：`用的词是` / `把.{0,5}痛点拆` / `把.{0,5}定义得很直白` / `给的答案.{0,3}是` / `想要的未来` / `描述的下一步` / `给的理由.{0,3}是`

### 6.3.x 英文俚语 / 习语直译要还原中文白话

英文里的形象习语直译成中文常常违和，翻译时还原成中文里同义的白话。

| 英文 | 直译（避免） | 中文白话 |
|---|---|---|
| squint at it | 眯起眼看 | 仔细看 |
| move the needle | 动了指针 | 有显著影响 |
| low-hanging fruit | 低垂的果子 | 容易拿的好处 / 简单的优化点 |
| the elephant in the room | 房间里的大象 | 谁都不愿提的明显问题 |
| bite the bullet | 咬子弹 | 硬着头皮 / 认了 |
| skin in the game | 把皮放进游戏 | 利益绑定 / 自己也下场 |
| boil the ocean | 煮沸大海 | 想一口气解决所有问题 |

判断方法：直译出来的中文如果在中文语料里几乎找不到搭配，就是典型直译腔，换白话。

### 6.3.x 反向陈述补刀冗余

定义：用「不是 X」「没有 X」「不带 X」「绝非 X」给前句补刀强调，但反向陈述本身不带新信息——它说的是前句已经说清楚的内容的反面，删掉前句意思不变。

典型错例：

- 「案例全部来自她接触过的真实客户 prompt，**没有凭空构造的教学例题**。」（前半已经说清是真实客户 prompt，后半反向陈述零增量）
- 「这是来自工程现场的判断，**不是凭空的理论**。」
- 「他给出的是一个具体方法，**并非泛泛的口号**。」
- 「整套方案都是基于实测数据，**不是拍脑袋**。」

判断方法：删掉补刀分句，前句独立读是否信息丢失？没丢就删。

改法：直接保留正向陈述，删掉反向补刀。如果反向陈述本身有信息（揭示了一个具体的反面误解），则改成独立一句陈述事实，不要用补刀语气。

边界：当反向陈述指向**具体而非泛泛的反面**（"不是基于公开财报的猜测，而是基于他亲自参与的内部讨论"），可以保留，因为它在精确划清边界。但「不是凭空构造」「不是泛泛的口号」「不是拍脑袋」这种泛泛反面陈述，一律删。

grep 触发词：`，没有.{0,15}的[\s。]` `，不是.{0,20}[的句]` `，并非.{0,20}` `，绝非.{0,15}` `不是.{0,20}拍脑袋` `不是.{0,20}凭空`

### 6.3.x 空泛读者承诺

定义：开头钩子或段尾用「你下次会做错」「你也会有这个反应」「你一定会想到」等对读者的预测式承诺，但下文没有具体兑现这个承诺，是节奏 filler。

典型错例：

- 「你下次遇到的时候大概率第一反应会做错。」（钩子之后没有兑现"做错"是怎么个错法）
- 「你也会经历这一切。」
- 「相信你看完以后也会有同样的感受。」
- 「你大概率没意识到这件事的严重性。」（下文没有具体说严重在哪）

改法：

- 如果有具体内容兑现承诺，直接给具体内容，删掉这句承诺
- 如果没有具体内容兑现，整句删掉
- 如果想保留对读者的连接感，用具体场景替代抽象承诺：「你下次遇到的时候大概率第一反应会做错」→「下次你遇到一条 prompt 在新模型上行为变怪，大概率会先去改 prompt——但更可能的是上一代的补丁在反作用」

判断方法：这句承诺之后下文是否真给了"读者会做错的具体方式"或"读者会有的具体反应"？没给就删。

grep 触发词：`你下次` `你一定会` `你也会有` `相信你` `大概率你` `你大概率` `你看完.{0,5}就会`

### 6.3.x 章节标题与开篇句的 setup 重复

定义：当 H2 章节标题已经包含「总结/反直觉/Take Away/案例/警告/这里有 N 条」等 framing 词，开篇句不要再用一个句子重复说明这是什么。读者看到标题已经知道了，开篇再 setup 一次是节奏 filler 的另一种形态——它把读者已经接受的提示再 paraphrase 一遍。

典型错例：

- 标题「这场分享的 3 个 Take Away」+ 开篇「如果时间紧只能记三句，这场分享的内核压成这样。」（开篇在用别的话说一遍标题已说的 framing）
- 标题「失败模式一」+ 开篇「这一节我们讲第一个失败模式。」
- 标题「最关键的反直觉」+ 开篇「这一段最反直觉的是这一句。」
- 标题「三个值得记的判断」+ 开篇「下面三条判断特别值得记下来。」

判断方法：把标题和开篇句连读，问"开篇句的信息是否已经被标题覆盖？"。覆盖就删开篇句。

改法：删掉开篇 setup 句，直接进具体内容。具体内容自己会兑现标题的承诺。

边界：当章节内容确实需要一句过渡引子（比如标题是名词式但内容跨度大、需要一句话点出主线），可以保留，但引子句必须**带具体新信息**，不能只是 paraphrase 标题。

这一规则不依赖 grep（要靠语义自审），扫描方法：写完通读每个 H2 章节的开篇第一句，问"这一句的信息是否被标题已经覆盖？"。

### 6.3.x 引言评价词

定义：在引述讲者原话或观点之前，用一句包含评价形容词的引导句给读者预先定调（"她说了一句最值得划下来的话""她对这件事做了个干脆的概括""他的判断只用了一句话""他的点破很短，但很扎手"）。这种引导是作者替讲者下定论，没有信息量，反而稀释引语本身的力量——读者看到引语前的"很犀利""很到位"，会预先期待，反而读到引语时觉得没那么 striking。

与现有 §六的「作者视角主观评价词类」的关系：那一节讲正文中作者评价讲者语气的词散在哪里都不该用，本节专门处理「引语正前方的引导句」这一特定位置。同一家族但出现位置最常见。

典型错例：

- 「她说了一句最值得划下来的话。」+ 引语
- 「Margot 的点破很短，但很扎手。」+ 引语
- 「他给出了一段非常精准的判断。」+ 引语
- 「她在这一段做了一个干脆的概括。」+ 引语
- 「这是他全场最有重量的一句。」+ 引语
- 「她说了一句让我反复琢磨的话。」+ 引语
- 「他用一句话点破了关键。」+ 引语

改法：用中性引导直接接引语，把评价形容词全部删掉。

- 「Margot 对这一类失败的建议如下。」+ 引语
- 「她在这一段说。」+ 引语
- 「他对这件事的判断是。」+ 引语
- 「Margot 用原话说了出来。」+ 引语

让引语自己说话。如果引语真的犀利，读者会自己感受到；如果引语不够犀利，加一万个形容词也不会让它变犀利。

判断方法：读引导句，删掉所有形容词和情感副词后是否还能成立？能成立就该删形容词。

grep 触发词：`说了.{0,5}最[^一-鿿]?值得` `点破.{0,5}短` `干脆.{0,5}概括` `精准.{0,5}判断` `只用了.{0,3}一句` `一针见血` `云淡风轻` `特别有重量` `最值得划` `这一段最` `让我反复琢磨` `点破了关键`

## 七、标点、引号、加粗和术语书写

**标点禁区：**

- 冒号（`：`），尤其是「如下：」「包括：」「原因：」。标题可以用。
- 破折号（`——`），包括双破折号用作插入语的写法。该解释的地方另起一句。
- 双引号（`""`、`""`）。直接引语融入正文叙述，不包引号。

**删除禁用标点是重构，不是查找替换：**

很多时候初稿会带着破折号、冒号、双引号写出来。收稿时的第一反应往往是「查找 —— 替换成句号」，这是错的。标点在句子里承担的是语义连接功能，机械替换会把前后两句变成互相悬空的陈述，让读者自己去补中间那层逻辑关系。

破折号在中文里有四种定型功能，每种要用不同的重构方法处理：

- **引出具体说明**（「A —— 具体是 B」）。机械换句号会让前句变成悬空判断。正确做法是调换语序，让后句先讲具体内容，前句做收尾判断。
  - 坏的改法：「批评方向很清楚 —— 用『幻觉』替代『撒谎』」 → 「批评方向很清楚。用『幻觉』替代『撒谎』」（「方向很清楚」悬在那里）
  - 好的改法：「用『幻觉』替代『撒谎』。批评方向很清楚」（判断落在已说清的具体之后）

- **列举收尾的停顿**（「『X』『Y』『Z』 —— 这些词……」）。简单抽掉破折号会让带引号的列举紧贴下一句「这些词」，视觉挤、节奏碎。正确做法是把列举改成顿号，加一个收拢词自然衔接。
  - 坏：「『AGI』『Agent』『Hallucination』这些词每个人都在说」
  - 好：「AGI、Agent、Hallucination 这三个词每个人都在说」

- **泛指后的逐项展开**（「X 分成几层 —— 第一层……第二层……」）。直接换句号后，泛指句和展开句之间的对应关系被切断，读者要自己搭桥。正确做法是用一个引子词（「落在哪一层上」「分成三类」「具体到一个动作」）让结构关系显现。
  - 坏：「意识到它们在哪个阶段 —— 账单侧决定……训练侧决定……」 → 「意识到它们在哪个阶段。账单侧决定……训练侧决定……」（「哪个阶段」和「账单侧」「训练侧」的对应关系没说出来）
  - 好：「意识到它们落在哪一层上。账单侧……训练侧……」（「哪一层」和后面四层自然对位）

- **插入补充说明**（「A —— 顺便说 B —— 继续」）。直接删掉两道破折号会让补充和主句融成一句，语义乱。正确做法是补充信息独立成一句放相邻位置，或者做成括号。
  - 坏：「这套方案 —— 其实不算新 —— 本质就是 Y」 → 「这套方案其实不算新本质就是 Y」（读不通）
  - 好：「这套方案本质就是 Y。说起来不算新」（独立成句）

冒号、双引号的禁用同理：

- **冒号**（「A：B」）：如果作用是"提示下文"，把 A 改成带动词的完整句（「A 做的是以下几件事」），或者把 B 独立成一段开头；如果作用是"定义"，换成「指的是」「也就是」「其实就是」这类动词短语。
- **英文双引号**（`"..."`）：包裹概念的，换成中文「」；包裹直接引语的，删掉引号让话融入叙述（「他说他很担心这件事」，不用加引号）。

自检方法：删掉一个禁用标点之后，把修改过的两句独立读一遍。如果前句变成"悬空判断"（没有下文它自己说不完整）、或者指代变得含糊、或者和下句的关系要读者自己猜，就是没改好，回到语义层重写。

**「」只用于：**

- 概念名称
- 自造术语
- 被讨论的词语本身

示例：

- 对：「建设者」、「信息搬运工」、「vibe coding」
- 错：他说「我很担心这件事」

直接引语不用「」，自然融入正文叙述。如果这句话是全文最值得读者记住的金句或关键论点，用 **加粗** 标注。

**加粗规范（本节为权威定义，其他位置只引用本节）：**

章节标题改用 H2（`## 标题`），不再算入加粗预算（详见第一章）。下面这条规则只针对**段中加粗**——出现在段落中间、用于强调单个短语的 `**X**`。

**用在哪三类位置：**

- **硬数字判断**（「每年账单从 500 万美元砍到 7.3 万美元」「Day 7 留存 12%」）
- **反直觉断言**（「根本没人点那些反馈按钮」「越能写代码的工程师越早被取代」）
- **文章核心论点的关键词**（这一篇文章读完读者最该带走的那句话）

**密度上的软参考：每千字段中加粗不超过 2 处**。

不是硬上限。1500 字短文一两处足够；3000 字文章三四处合理；5000 字长文五六处不算多。判断标准是读者扫读时这些加粗能不能成为「锚点」。如果一段文字里两个加粗紧挨着，或者整段文章加粗多到读者扫一眼分不清重点，那就是过了——加粗多到稀释了锚点价值，等于没加粗。

**反过来也要警惕**：1500 字以下的文章如果硬塞 3 处加粗，往往是把不够硬的句子也加粗了。短文里一两处足矣。

**加粗前不要堆修饰链：**

加粗本身就是强调，加粗前再堆"X 的核心""X 的根因""最 X 的一点""真正 X 的"会形成铺垫 + 强调的双重冗余。读者读到加粗那一行已经知道这是重点，不需要前置铺垫；前置铺垫还会引入作者评价词，进一步稀释加粗的力度。

**触发情境**：加粗短语前 30 字以内出现这些前缀短语：

- 「最锋利的一点」「最反直觉的发现」「最有意思的设计」「最不容易看到的」
- 「真正值得 X 的」「真正 X 的关键」「真正的核心」
- 「X 的根因」「X 的核心」「X 的关键」（接加粗时）
- 「也是后面所有设计的根因」（典型双重铺垫）

**反例**：

```
第三个最不直觉，也是后面所有设计的根因，**模型很不擅长评估自己的输出**。
```

读者读完前置铺垫还没拿到信息，到加粗才出来；加粗强度被稀释一次。

**改写**：

```
第三个是，**模型很不擅长评估自己的输出**。
```

再举一个：

**反例**：「Ash 给的答案是这场分享里最锋利的一点，**X**」 → **改写**：「Ash 给的答案是，**X**」

允许的引语动词（中性，不算修饰链）：「X 说」「X 的答案是」「X 的判断是」「X 的总结是」「他给出的判断是」。

grep 触发词（在加粗符号 `**` 前的 30 字内）：`最锋利 / 最反直觉 / 最有意思 / 最不容易 / 真正值得 / 真正的核心 / 的根因 / 的核心 / 的关键`（如果触发词与紧跟其后的 `**` 之间字数 ≤ 30，就要重写）

**英文术语大小写：**

英文术语夹在中文段落里时，按下面三类分别处理。为什么分三类：首字母大写让英文术语在中文段落里成为视觉锚点，读者一眼识别出「这是一个专有概念」，而不是被夹带的普通英文短语；缩写保持原样保留技术背景身份；代码标识符一旦改大小写就不是那个符号了。

- **术语概念和词组：每个单词首字母大写。** 不论是单个词还是组合词组：Agent、Agentic Search、Agentic RAG、Semantic Search、Vector Search、Context Engineering、Context Window、Context Curation、Context Source、Context Compaction、Search Tool、Shell Tool、Bash Tool、Exec Tool、Query Tool、Database Tool、Memory Tool、Web Search、Agent Skill、System Prompt、Tool Description、File Store、Progressive Disclosure、Specialized Tool、General Purpose Tool、Low Floor、High Ceiling、Prompt、Evals、Embedding、Chunk、Token、Multi-hop。
- **缩写原样全大写**：RAG、LLM、ESQL、SQL、CLI、UX、ID、QR、API、SDK、PR、SSE、MCP。
- **产品、公司、框架名维持原有形态**：LangChain、Anthropic、OpenCode、Elastic、Elasticsearch、LlamaIndex、LightOn、ColGrep、Jina、Jina Grep、Vercel、DeepMind、Medium、TU Dresden、DevRel、ChatGPT、Claude Code。看上去像普通英文词的产品名（Gemma、GEPA）也算这一类，不要擅自改动。
- **代码标识符、命令、参数名、配置项一律原样**：用行内 code 包起来（反引号），包括 `@tool` `grep` `ls` `curl` `rm` `top K` `jina-grep` `%GEPA%` `*` `%` `plan.md` `FILTER date=...` `get_customer_by_id` `try/except` `try/catch`。命令里附带的符号（星号、百分号、斜线）也不改。

常见误区：

- `semantic search` / `shell tool` 全小写，阅读时会被当成普通英文短语，损失辨识度。中文段落里应该是 Semantic Search / Shell Tool。
- `Semantic search`（只大写第一个词）是英文句首大写的写法，不适合在中文段落的中间出现。
- `Rag` `Llm` `Esql` 这种首字母式大写是画蛇添足，缩写必须全大写。
- 把真正的代码标识符（如 `@tool`）写成 `@Tool`，就不是那个符号了。代码标识符的判断标准：它会不会出现在代码里？如果会，一律反引号加原样。

### 7.x 逻辑转折显式标

中文长句里前后两段语义构成转折时，加显式连接词「但」「不过」「然而」「可是」，不要省略让读者自己推。

**反例**：evaluator 会让 generator 推倒重来，这是好事。（读者第一感：推倒重来是坏事，后半句突兀）
**改写**：evaluator 会让 generator 推倒重来，但这是好事。

**反例**：Harness 不会随模型变强而消失，它会变形。
**改写**：Harness 不会随模型变强而消失，但它会变形。

判断方法：前后两个分句独立朗读，如果第二句读起来"反预期"，就加转折词。

### 7.x 单字结尾收句不顺口，改双字词

中文断句倾向偶数音节收尾，单字动词/形容词结尾常显得仓促。

| 单字结尾 | 双字替换 |
|---|---|
| 齐了 | 齐备了 / 凑齐了 |
| 够了 | 足够了 |
| 完了 | 完成了 / 收尾了 |
| 好了 | 做好了 / 准备好了 |
| 行了 | 可以了 / 行得通 |
| 成了 | 做成了 / 完成了 |
| 散了 | 散开了 / 解散了 |

例外：口语化对话引用、刻意制造干脆收尾节奏（每段最多一处）。

### 7.x 反复出现的英文产品 / 系统 / 框架名首字母大写

出现 3 次以上的英文专有名词（产品名、系统名、框架名、概念缩写）在中文段落中应统一首字母大写，作为视觉锚点便于扫读。

高频例：Harness、Agent（作为框架/系统名时）、Sprint（作为方法论术语时）、Eval、PM/IC/QA（角色缩写本就大写）。

### 7.x 英文术语括号释义的标准格式

定义：当一个英文术语在文章里第一次出现、且不是中文读者一看就懂的常用词时，在英文后面用半角括号给一句中文释义。这是工程类爆款样本里最高频的处理方式之一，能让非技术读者继续读下去，同时不剥夺英文术语作为视觉锚点的功能。

标准格式：`英文术语(中文释义)`——英文在前，中文在括号里，括号是半角。释义要短（5-15 字），用动作或具体作用描述，不要给定义辞典式的展开。

样本（来自爆款样本「CI/CD Is Dead, Continuous Compute」）：

- Continuous Compute(持续计算)
- Ingress Shaping(入口流量整形)
- Agentic Identity(给 Agent 一份类似服务账号的身份)
- Internal Validation(内部校验)
- External Validation(外部校验)
- Pre-Merge Queue(前置合并队列)
- Serialization(串行化)
- Invariant(不变量)
- Regression(回归问题)

注意 Agentic Identity 那条释义不是字面翻译"Agent 身份"，而是描述了它的作用("给 Agent 一份类似服务账号的身份")。这种"作用式释义"比"字面式释义"对读者更有用。

什么时候不用括号：

- 已经在公众号读者中流行的术语：Agent / RAG / LLM / API / SDK / Token / Prompt——这些不需要括号
- 同一术语在文章后续出现时：只在第一次出现给括号，之后裸用
- 代码标识符（反引号包裹的）：不需要括号，符号本身就在代码块里

判断方法：写完通读一遍，找每一个英文术语，问"如果读者不熟悉这个领域，他读到这里会不会卡住？"。会卡住就给一个 5-15 字的作用式释义，不会就裸用。

低频通用词不必大写：context、loop、prompt、token、tool、scope、feature 这类已成为日常技术词的概念名，中文段落里小写更自然。

判断界线：如果该词在文章里**承担专有名词角色**（指代特定系统/产品/方法论而不是泛指概念）且出现 ≥3 次，大写；否则小写。同一文章内部对同一术语保持大小写一致。

特殊例外：术语首次出现做定义/解释时（如「一个『harness』，指模型外面那一层…」），可保留小写以表示这是「这个词本身」而不是专有名词指称，从第二次起大写。
