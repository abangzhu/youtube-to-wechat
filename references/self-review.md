# 公众号文章自审规则

读取时机：`article.md` 初稿完成后。先做语义自审，再做 grep 兜底。

## 三层评估

### 读者视角

- 标题能不能让目标读者停下来点进来？尤其检查是否踩了“如实概括”短板（裸人名 / 复盘过去 / 结论写进冒号后），见 [title-and-publishing.md](title-and-publishing.md) §8.x「从如实概括到制造认知缺口」
- 开头 3 段能不能让人继续读？
- 每个 take-away 有没有让人产生明确收获？
- 结尾有没有留下值得思考的东西？

### 编辑视角

- 正文是否像中文作者写给读者看的文章，而不是来源摘要？
- 是否残留转述腔、元评论、AI 腔、翻译腔、空转场句？
- 新概念第一次出现时是否给了定义？
- 段落首句连读是否有清楚推进关系？
- 需要视觉锚点的地方是否已截图，是否还有 `[图:MM:SS - 描述]` 占位符？

### 事实视角

- 专有名词、数字、时间线、产品名是否准确？
- 因果、转折、限定语和不确定性是否保留？
- 是否把延伸解读误写成来源事实？
- 英文来源是否已按 `translation-quality.md` 做术语、代词、泛指词和回译抽查？
- **人物的多重身份头衔（founder / creator / contributor）是否逐一对回背景资料核实，而不是用一个笼统词打包？** background.md 或来源资料里，人物对不同项目的身份往往并不一致（对 A 项目是 founder，对 B 项目只是 creator，对 C/D 项目只是"列出的参与项目"）。写稿时为了省事把它们合并成一个模糊词（"建设者" / "参与打造了"），会丢掉背景资料本来给出的精确对应关系。核对方法：文中每提到一个人物身份头衔，回 `background.md` 逐个项目核对原词（founder/creator/维护者/贡献者），身份不同就分开表述，不要用一个词覆盖所有项目。错例：「T3 Stack 的创建者，也是 T3 Chat、UploadThing、Ping.gg 等项目的建设者」（background.md 明确写 T3 Chat 是 founder、T3 Stack 是 creator，被这句话的"建设者"抹平）→「T3 Chat 的创始人，也是 T3 Stack 的创建者，还参与打造了 UploadThing、Ping.gg 等项目」。
- **文中出现的倍数 / 百分比，读者能否用同段给出的数字算出来？锚到的基数对不对？** 这类错 grep 抓不到，只能动笔核算。实例：「100 亿→320 亿」被写成「24 个月翻了 10 倍」，但 320÷100 只有 3.2 倍——讲者说的「10 倍」锚的是另一个口径（算上 OpenAI、Anthropic 后到 9 月可能的 1000 亿）。倍数贴错基数，会被算账的读者当场抓住。报任何倍数 / 百分比前，先用文中数字回算一遍。
- **任何具体数字（分钟、行数、次数、金额、第 N 个）都要回 `bilingual-transcript.md` 核对：是来源真有，还是为生动而具象化编出来的？** 这类错 grep 抓不到、读起来也顺，只有对照原文能发现。最常见的是把一个区间或泛指坐实成精确值，让读者误以为是讲者原话。实例：原文只说 Agent「跑 5 分钟、15 分钟」是个区间，文章写成「第 12 分钟某一步失败」——这个「12」是凭空具象化，把举例坐实成了来源事实。写稿时为画面感具象化数字前先问一句：这个数字原文真给了吗？没给就用区间或模糊说法。
- **专有名词同样会被坐实——把原文的模糊指代写成一个具体公司 / 产品 / 人名。** 和数字坐实同源，grep 抓不到，读起来还更顺（因为补了个"看起来合理"的名字），只有对照原文能发现。实例：原文是「we tried everything, built a model for us, openai finetuning or another, none worked」（有人给我们做了模型），文章写成「**Anthropic** 给他们做过模型」——原文根本没出现 Anthropic，是把"有人做了个模型"脑补成了具体公司。核对方法：文中每个公司 / 产品 / 人名，回 `full_transcript_en.txt` 确认这个名字原文真出现过，且归因对象没错；原文用 "someone / a model / another" 这类泛指时，中文也保持泛指，不要替读者补名字。
- **对谈 / 双人访谈 / 播客类来源，每段话归给了谁（speaker diarization）必须独立核查——自动字幕和初稿的发言人分配会系统性出错，且 grep 抓不到、读起来还顺。** 三个高命中信号：
  1. **内部代词矛盾**：同一个第一人称「I」在文章不同段落被标成不同的人 → 必有一处错。实例（Hth_tLaC2j8 session）：原文 "I pinged him" 和下一段 "Claude tells me this all the time" 是同一个「I」，文章却把前段写成「她去问那位工程师」、后段写成「Boris 说……Claude 经常告诉他」——「她 / 他」指同一人却性别不一致，自相矛盾。
  2. **提问 ≠ 回答**：访谈里 "What are your tips? / What are your go-to features?" 这种提问句的人，和紧接着的回答者是两个人。整段回答（含 "I've been mainly hacking on the desktop app"）要归回答者，别归提问者。本次错例就是把 Boris 回答的桌面应用例子误归成提问的 Cat。
  3. **跨时间戳的连续句 + 自指线索**：一句话被时间戳切断（"one of my favorite things" → 下一时间戳 "everyone codes"）说明两段是同一人；"my PM would code, your code is really good" 这种自指（说话人有「我的 PM」、且对在场另一人说「你的代码很好」）能直接锁定身份是 Boris 对 Cat 说。
  核对方法：把文章里每一处「X 说 / X 认为 / 他 / 她」回 `bilingual-transcript.md` 沿对话顺序核对；先锚定有铁证的几段（自指句、被时间戳切断的连续句），再用「提问-回答交替」推其余；最后通读全文，确认同一个人的代词性别和名字前后一致——文章自身的代词矛盾往往是最快的破绽。

## 机械检查 checklist(写完后跑一遍)

这一节是**写完整稿后立刻跑的兜底扫描**。每条都给可直接执行的 grep 命令,命中后回到对应 reference 修正,再跑一遍直到全部干净。

```bash
F=article.md   # 把 article.md 换成你的文件路径

# 0. H1 存在性(批改已有 / AI 生成稿高发缺失:title-options 写好了却没写进正文第一行)
head -1 "$F" | grep -q "^# " && echo "✓ 有 H1" || echo "✗ 缺 H1,定稿前需把选定标题写进第一行"

# 1. 破折号密度(每千字 ≤ 5)
DASH=$(grep -c "——" "$F"); CHARS=$(wc -m < "$F"); \
  echo "破折号 $DASH 个 / 字符 $CHARS 个 ≈ $(awk "BEGIN{printf \"%.1f\", $DASH*1000/$CHARS}")/千字"

# 2. H2 字数(每个 ≤ 20 字。awk length 在中文下算字节,用 wc -m 取字符数)
grep "^## " "$F" | while IFS= read -r line; do
  text="${line#\#\# }"
  printf '%2d 字 → %s\n' "$(printf %s "$text" | wc -m | tr -d ' ')" "$line"
done

# 3. "不是 X 而是 Y" 隐性变体(对比家族,含换介词变体 不在...而在)
grep -nE "不是.{1,40}而是|而不是|不是[^,。]{0,40}[,,][^是]{0,8}是|不在.{1,30}(而在|也不在)" "$F"

# 4. 戏剧化动词("扔/砸/甩/抛出" + 文学化动词族)
grep -nE "扔了|砸了|甩了|抛出.{0,5}(观点|问题|判断)|滑到|蔓延|笼罩|席卷|凝结|涌入|撕开" "$F"

# 5. 强调副词稀释信任
grep -nE "真的|确实|果然|真正|的确|事实上" "$F"

# 6. 翻译腔(术语 + 非技术英文口语)
grep -nE "理由是|niche take|hot take|no-brainer|game changer|内核压成|精髓.{0,3}是|核心.{0,3}在于|本质上.{0,3}是|让我们看看|不能不提|不得不说" "$F"

# 7. 元过渡 + 冒号铺垫
grep -nE "这件事(在|说明|背后).{0,15}(问题|事|判断|启示)[::]|这背后.{0,10}是.{0,5}(问题|逻辑|趋势)|这给我们.{0,5}启示" "$F"

# 8. 来源转述腔
grep -nE "想清楚的(事|是)|提了(一个|个)?问题|[一-龥]{2,4}在(演讲|台上|视频|文章|访谈)(里|上)?" "$F"

# 8b. 元叙述命名包装句(起了个说法 / 把它归到一种 / 用了一个类比 + 评价词)
grep -nE "起了一个.{0,6}的(说法|叫法|名字|类比|比方)|把它归到一种|用了一个.{0,6}的(说法|类比|比方|概念)|给.{0,6}起了" "$F"

# 9. 段尾泛话兜底
grep -nE "得另.{0,3}办法|还有.{0,3}空间|还需要.{0,3}时间|值得.{0,3}思考|未来.{0,3}可期|这只是.{0,3}开始" "$F"

# 10. 末段综合升华(通读级,无 grep)
tail -n 40 "$F"   # 抽出最后两段,问"整段拿掉,文章的论证是否受损?"

# 11. 半角标点混入中文正文(整篇 AI 生成稿高发,命中数应为 0)
#     检测"中文字符紧邻半角 , ; ? !",URL 与英文标题内的半角标点不会命中
grep -nE "[一-龥][,;?!]|[,;?!][一-龥]" "$F"
#     中文后的半角冒号(英文标题"X: Y"、URL"https:"不会命中)
grep -nE "[一-龥]:" "$F"
#     命中后批量修正(冒号需排除来源行 URL / 英文标题,先确认再替):
#       sed -i '' 's/,/，/g; s/;/；/g; s/?/？/g; s/!/！/g' "$F"
```

## 段级精修必查清单(无 grep 触发词,只能通读)

机械 checklist 抓不到的句法 / 词序问题,只能在段级精修时逐段核查:

- **多重定语顺序**:看每个名词短语,问"时间 > 主体 > 中心词"是否成立。错例:"昨天的人类能力" → 改:"昨天人类的能力"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 多重定语顺序中文化
- **数量 / 范围副词位置**:看"绝大部分 / 大多数 / 一半"是否放在所修饰短语最前。错例:"你日常工作的绝大部分" → 改:"你绝大部分的日常工作"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 数量 / 范围副词应放在所修饰短语最前
- **表语连接动词翻译腔**:看"是说 / 是去做 / 是要做",换成"变成 / 在于 / 做的是"。错例:"人类的工作是说" → 改:"人类的工作变成"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 表语连接动词的英文翻译腔
- **被动 / 容器隐喻 → 主动拥有关系**:看"X 留在 Y 里 / X 保存在 Y 中",检查 Y 是不是位置容器还是实体角色,后者改成"Y 拥有 X"。详见 [sentence-level-errors.md](sentence-level-errors.md) §六之二
- **元叙述命名包装句**:看"他给这个动作起了一个说法,X / 把它归到一种 X / 用了一个很容易懂的类比",砍掉命名 / 归类的叙述动作和评价词,让讲者的主张直接出场。错例:"他给这个动作起了一个很直接的说法,别向机器做认知投降" → 改:"他建议我们不要向机器做认知投降"。这是访谈 / 演讲转述里高频的别扭表达。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 元叙述命名包装句要砍掉
- **触感形容词评价定义 / 判断**:看"X 讲的 Y 更硬一些 / 更软一些","硬 / 软 / 轻 / 重"不是定义的合法形容词,删掉、用"X 认为 / X 指的是"直接给定义。错例:"Fadell 讲的 taste 更硬一些,指的是 Z" → 改:"Fadell 认为 taste 指的是 Z"。详见 [sentence-level-errors.md](sentence-level-errors.md) §度量 / 触感形容词
- **空间方位隐喻代转折**:看"A 是 X,B 另一边看到的是 Y","另一边 / 另一头"含糊代替"但",换成显式转折词。错例:"Apple 另一边看到的是 98% 普通用户" → 改:"但 Apple 看到的是 98% 普通用户"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 空间方位隐喻替代转折连词
- **去转述腔别走偏成"说得很直接"**:把"X 说"改成"X 说得很直接 / 很清楚"不算去转述腔(只是叠了评价词)。内容是判断时,换成"X 的判断是 / X 认为"。错例:"在代码上,Fadell 说得很直接" → 改:"在代码上,Fadell 的判断是"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 元叙述命名包装句(特别警告)
- **否定判断别用"很不像"软化**:实际意思是"不是 X"却写"很不像 X / 不太像 X",直接"不是"。错例:"看法很不像「发布前包装」" → 改:"看法不是「发布前包装」"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 否定判断不要用"很不像"软化
- **形容词与抽象名词搭配(配价)**:"提醒"配"重要"不配"要紧","判断"配"准确"不配"快";"放到 X 时代更 Y"容器框架简化成"在 X 时代更 Y"。错例:"这层提醒放到 AI 时代更要紧" → 改:"这个提醒在 AI 时代更重要"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 形容词与抽象名词的搭配
- **"那种角色的人"冗余**:看"X 那种角色的人 / X 这种类型的人",改成"X 的角色"。详见 [sentence-level-errors.md](sentence-level-errors.md) §六之二
- **主语被"处所化 / 容器化"的直译费解句**:英文 "put X in a bigger question / frame X within Y / situate X in Z" 直译成中文会变成"把 X 放进一个更大的问题里问 / 把 X 放进 Y 框架里"——读者完全 parse 不出动作。这类句子单独读一遍若需要停顿反推"到底谁对谁做了什么",就改成最朴素的主谓宾。错例:"Brian 把 Ivan 放进一个更大的问题里问" → 改:"Brian 问了 Ivan 一个更大的问题"。判断方法:把英文动词还原(put / frame / situate / place + 抽象名词容器),中文几乎都该拆成"谁 + 动词 + 谁 + 什么"。
- **末段综合升华段**:整篇结尾段是否只是把前文 thesis 再说一遍?整段拿掉论证是否受损?见 [article-structure.md](article-structure.md) §3.x.收尾2
- **开头两段的连接 / 主语生硬**:前两段最高发三种——"先否定窄断言 + 句号 + 短句补正面动作"(没有只 X。它 Y → 不仅 X,还 Y)、存现句"这件事发生在 X 身上"(→ 这是 X 的一个实践)、对比项主语不一致硬并到"是 X,不是 Y"(当 Y 在说另一主体的状态时拆成两句)。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 开头段的"先否定后断句 / 存现句"连接生硬

- **比喻 / 动作的物理可行性**：检查文章里含主体执行动作的比喻（"坐在 X 上" / "骑在 X 上" / "站在 X 上面"），这个动作对这个主体物理上是否成立？不成立就换成描述实际管理关系的动词（"同时管理 / 负责监控 / 跟踪进度"）。错例："他没法坐在 150 个 agent 上面，逐一判断质量" → 改："他没法同时管理 150 个 agent 干活，逐一判断质量"。

- **标题/判断句里的施受方向**：看含"服务 / 支撑 / 驾驭 / 扛住"这类及物动词的句子，A、B 谁是施事谁是受事有没有搞反。错例："2000 万行代码能服务 Agent"（读成代码库服务 Agent，但正文讲的是 Agent 能不能扛住这个规模）→ 改："Agent 扛得住 2000 万行代码库"。详见 [title-and-publishing.md](title-and-publishing.md) §8.x 标题施受关系核查

- **"先 V"式标题有没有呼应的"后"**：标题里出现"先"字时检查有没有对应的"再/后来"呼应，没有就删"先"，把动词短语改成定语修饰主体。错例："Honk 先从维护工具长出来" → 改："从维护工具长出来的 Honk"。详见 [title-and-publishing.md](title-and-publishing.md) §8.x 标题里『V + 先 + 动词』读起来悬空

- **"从 A 走向 B"里 A、B 是否同类**：A、B 不能互相转化时（"PR 数"变不成"用户价值"）要补一个共同的评估动词（看/盯/衡量）。错例："ROI 会从 PR 数走向用户价值" → 改："ROI 会从看 PR 数走向看用户价值"。详见 [title-and-publishing.md](title-and-publishing.md) §8.x 『从 A 走向 B』标题里 A、B 不同类时要补评估动词

- **条件句省略"如果"**："[动词短语]，X 只会 Y"这种压缩条件句，读者容易先读成陈述句。检查是不是隐含"如果……"，是就补上。错例："承受不了，Agent 只会把……暴露出来" → 改："如果承受不了，Agent 只会……"。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 段内句间的逻辑连接词要显式（条件句的段内版本）

- **"进入 X" 有没有漏"到…中"**："进入"后面直接接抽象系统/流程名词收尾时，检查补上"到…中"是否更完整。错例："这些判断要进入测试、CI……和内部工具" → 改："……要进入到测试、CI……和内部工具中"。详见 [sentence-level-errors.md](sentence-level-errors.md) §"进入 X" 要补全"到…中"的介宾结构

- **抽象名词的动词搭配（配价）**：除形容词外，动词与抽象名词的搭配也要成立。"重心"配"转移"，不配"换位置"（后者是物理位置/职位调动的搭配）。详见 [sentence-level-errors.md](sentence-level-errors.md) §形容词与抽象名词的搭配

- **首次出现的术语，即使只在比较句/插入语里一带而过，也要给最短括注**：不能因为后面章节会专门介绍就推迟。错例："判断比 Honk 这类具体功能更朴素" → 补注："判断比 Honk（Spotify 内部的代码维护 Agent）这类具体功能更朴素"。详见 [sentence-level-errors.md](sentence-level-errors.md) §引入新概念要给一个迷你说明

- **复合术语全文统一**：文中自创或多字符复合的专用词（"验证门禁" / "信号层" / "注意力边界"），从第一次出现起全文字形必须完全一致。混用"验证门" / "验证门禁"这类近义变体会让读者以为是两个不同概念。定稿前：把术语第一次出现的完整形式存下来，全文 `grep` 核对一遍。grep 触发：在正文里找出第一次出现的完整词形，再搜截短版（去掉最后 1-2 字），命中即修。

- **结论 / 标题里的限定词宽窄**：章节标题或全文结尾的判断句里含限定词（"早期工程师 / 小团队 / 特定场景"）时，问一遍：把限定词删掉，结论是否仍然成立且更有力？成立就删。错例："早期工程师不能把学习外包" → 改："工程师不能把学习外包"（原则对所有工程师成立，「早期」只是问题的引子，不是结论的必要条件）。

判断方法:把每段单独抽出来读一遍。这一段如果读者只读到这里就退出,他能带走什么?如果带走的东西其实在前面段落里已经讲过了,这段就是冗余,删或重写。

## 十二、自查清单

**第一条心法：读者视角审计**

写完每段话后，问自己——这一句话对一个不熟悉来源的读者，直接读这篇文章的人，有什么帮助？

没有帮助就删。这条比任何 grep 都重要——grep 只能抓词汇级问题，读者视角审计抓的是整段冗余。

「没有帮助」的常见特征：

- 元评论铺垫（「playbook 写得很细」「手册里讲得很透」）→ 见六之三
- 通用承诺（「这件事很重要」「这一点不容忽视」）
- 节奏性总结（「意思就清楚了」「这件事的含义比表面看起来更深」「在结果上差很多」）
- 过度通用的过渡（「整本 X 把 Y 讲得最透的一章在 Z」）
- 自述结构（「这是手册自己留下的金句，也是这一节标题的来源」）
- 给作者自己评论的对资料的形容词（「说得很狠」「挺锋利」「挺细」）

操作方法：文章定稿前，从头到尾把每段第一句和每段最后一句单独抽出来读。如果一句抽出来后，你说不清它给读者带来什么（哪怕是「过渡」「framing」也行），那就是删掉的候选。

**第二条心法：章节级读者价值审计**

第一条心法做的是句子级抽查，逐句问「读者带走什么」。但句子级会漏掉一类更隐蔽的问题：当一段轶事**整段占了一个章节**、戏剧性强（演讲事故、讲者人设彩蛋、内部段子）、句句读起来又都没毛病时，整个章节本身可能就是 anecdotal padding。这种情况下每句话都看得过去，但章节存在的理由站不住。

操作方法：写完初稿后，**在章节级别再过一遍**。每个 H2 章节问一句：「读者读完这一章具体要带走什么？」答案必须是一条具体可复用的工程判断、方法、反常识发现或新概念命名。如果只能答出下面几类，这一章就要整个删，或合并到别的章节里压成一两句：

- 「讲者台上 demo 出了问题」「Wi-Fi 不稳」「设备故障」这种演讲事故彩蛋
- 「现场互动很热闹」「观众举手」「主持人调侃」这种现场氛围描写
- 「讲者顺势讲了段题外话」「为缓解尴尬展开的话题」这种延伸闲聊
- 「这是讲者的人设 / 风格 / 段子」这种纯人物色彩
- 「Anthropic 也管 Claude 了」这种行业新闻级别的引用，读者多半在你发文章前已经在 Twitter 上看过

**关键判断**：轶事可不可以保留要看可复用洞察是否被它**独占**。如果同一条洞察在另一个章节里已经独立支撑得起，原始轶事章节就该删，不要为了「钩子戏剧感」让它独立成章。

反例（来自 7gujZrJ9L5I session）：曾写过一个章节「演示一开场就在客户后面排队」，把「讲者在台上点了一百次继续，揭晓是 Google 内部 token 配额比客户紧，Anthropic 管 Claude，SRE 找你停 job」全写成一个章节。现场戏剧感强，但读者真正能带走的洞察「Agent 经济学难解」已经被后面的「模型自动 fallback」章节独立承载，这一章被整章删除。开场段里和 demo 强绑定的钩子句也一并删掉，开场重写为「直接列出本场 panel 实际兑现的几堵墙」。

**第三条心法：段后行业对照式 / 内外推测式补刀句要删**

事实陈述完毕后，作者再补一句「行业里 X 也都在做」「内外一样」「只是公司不会公开讲」「这个方向也会带动 Y」「这证实了 Z 也朝同一方向」「这件事和外界感受到的 Y 是同一种焦虑」「对外面写 X 的人来说是个安慰」。这种段后总括 / 行业对照 / 内外推测式补刀句几乎无信息量，基本都是作者站出来替读者总结，读者要么早就知道、要么不需要这层抽象，删。

为什么写稿时容易写出来：钩子段落讲完事实之后，作者会本能想给一个「升华」或「行业坐标」，让段落收得"完整"。但读者要的不是收尾仪式，是事实本身和它带来的判断。读者已经能自己定位行业坐标，作者多说一遍只是稀释信息密度。

典型错例：

- 「这件事在 Anthropic 的 Skill 和 Sub-agent 设计里也能看到影子」
- 「KP 这段证实了 Google 这边也朝同一个方向」
- 「Anthropic、OpenAI、外面做 Agent 平台的初创也都在解，只是不会公开讲得这么直白」
- 「Gemini 那边的 Agent 形态也会跟着变」
- 「这一点对外面写 X 的人来说是个安慰」
- 「真要看到一个开源生态长出来，DeepMind 大概率会跟」
- 「这件事和外界感受到的 Y 是同一种焦虑」
- 「这些事在 X 一场 panel 里被 Y 直接讲了出来」（同时是元归属，见 §6.3.x）

判断方法：把这句删掉，前一段 / 前一句的事实独立读，是否信息丢失？没丢就删。不要给读者写「行业坐标」，他们能自己定位。

边界：当补刀句**带具体信息差**（不是简单比较，而是给出"对方做法的具体点"）时可以保留。比如「Anthropic 在 Claude Code 里也做了 fallback，但他们的实现是 X，Google 这边走的是 Y」，这种比较带新信息。但泛泛的「外面也都在做」「这件事不只是 Google 的问题」一律删。

grep 触发词：`也都在解` `也朝同一` `不会公开讲` `跟外面没什么不同` `证实了` `跟着变` `也都在做` `是同一种焦虑` `内外一样` `这一点对.{0,15}是个安慰` `也朝同一个方向` `也能看到影子` `直接讲了出来` `大概率会跟`

反例（来自 7gujZrJ9L5I session #2）：第一轮自审通过的文章里，仍有 7 处段后行业对照补刀句没被删——开场第一段末句、Skill 章节末段、Deep Research 章节末段、observability 章节末段、fallback 章节末段、Eval 章节末段、文末段。这些位置都是"段尾习惯性补总括"的高发区。

写完初稿后逐条过一遍：

**结构与开头**
- [ ] 正文有没有 Markdown 标题、列表、表格？
- [ ] 开头第一段是不是直接给判断 / 场景 / 数字，而不是先写人物简历？
- [ ] **来源里有没有可验证的反直觉数字,却没有放进开头?** 例如删掉 95%、10,000 行到 553 行、68 分钟到 6 分钟、77% vs 97%。有这类硬数字时,优先用"动作 + 数字结果"开场,不要先写抽象观点。详见 article-structure.md §2.x「可验证的反直觉数字优先于观点」
- [ ] 前三四段是否自然交代了来源、人物身份和核心问题？
- [ ] 章节标题（如果有）是不是完整短句带动词，不是冷名词？
- [ ] **每条方法论原则前有没有具体事故 / 失败 / 数字支撑?** 如果正文先写「强制,别指示」「引导,别规定」这类原则,但前面没有 `.case-tested`、`start.ts`、10,000 行 Skill 变差这类事故,读者会觉得像口号。先讲事故,再抽原则。详见 article-structure.md §2.x「先讲事故,再抽原则」
- [ ] **章节标题是不是可复用判断,不是素材目录?** 「内部工具 case」「WorkOS CLI」「三条原则」这种目录标题要改成「让老实干活,比撒谎更省事」「删掉 95% 的 Skill 更准」这类读者能带走的判断。详见 title-and-publishing.md §8.x「H2 要压成可复用判断,不是素材目录」
- [ ] **钩子段涉及多方主体(两家公司、几个团队、当事人 + 对手)时,每个动作的施事方是否都显式点明,而不是依赖「我们 / 他们 / 这家 / 那家」让读者反推?** 把钩子段读给没看过原视频的人,他能不能一遍复述出"谁干了什么导致了什么"?需要回扫第二遍就改。这条是开头最致命的翻车点,见 article-structure.md §2.x「钩子段涉及多方主体时,每个动作的施事方必须显式」
- [ ] **第二段(来源交代段)有没有踩五条高频陷阱?** 来源转述句(「这件事是 X 在 Y 大会的开场」)、演讲标题塞正文、空头衔(「产品经理」「工程师」)、Kaggle 简介像官网关于页(brag 列表)、人物简历集中堆砌——五条任意命中一条都要重写。详见 article-structure.md §3.x「来源交代段最常踩的五条陷阱」
- [ ] **「你在写新文章,不是转述讲者」**——把所有含「X 说 / X 提到 / X 想清楚 / X 在演讲里 / ——X 的原话 / X 在台上 / X 抛出 / X 提了一个问题」的句子标出来,逐条问"这一句相比上一句带来了什么新东西"。带不来就删。详见 article-structure.md §3.w「核心判断:你在写一篇新文章,不是在转述讲者」。grep 触发词:`想清楚的(事|是)` `——.{0,4}(原话|的话|说的|讲的|的判断)` `[一-龥]{2,4}在(演讲|台上|视频|文章|访谈)(里|上)?` `提了(一个|个)?问题`
- [ ] **文学化动词换成大白话动词?**「滑到 / 蔓延 / 笼罩 / 涌入 / 凝结 / 撕开 / 倾斜向」这类有空间感的动词放在事实陈述里会让句子显得用力过猛。改成「变成 / 扩散 / 覆盖 / 进入 / 形成 / 打开 / 偏向」。详见 ai-voice-prohibitions.md §六「文学化动词 → 大白话动词」
- [ ] **章节标题读者一眼看懂吗?** 标题里有没有非周知缩写(PVP / HRT / ELT / MoE 这种圈外人不立刻认识的)、专业领域隐喻(解饱和 / 击穿 / 跌停 / 爬坡 / 破防 / 拉满)?把标题给一个不读 AI/工程圈日常 newsletter 的中文读者看,他能不能立刻 infer 出意思?不能就换大白话动作描述。RAG / API / SDK / LLM / Agent 等周知缩写不在此列。详见 title-and-publishing.md §8.x「依赖正文解释才能看懂的标题」
- [ ] **章节末段最后一句承担收尾力度吗?** 把每一章末段最后一句单独抽出来读,有没有用「得另想办法 / 还有空间 / 值得思考 / 未来可期 / 还需要时间 / 这只是开始」这类泛话兜底?有就删,让段落停在前一句更具体、更有名字的判断上。详见 article-structure.md §3.x.收尾「章节末段最后一句话必须承担收尾力度」
- [ ] **结尾段的行动指引是不是读者现在能用的动作?** 「GitHub 链接在演讲幻灯片里」「中场会在 DeepMind 展台」「Q&A 环节有人问到」「现场举手过半」——这种是演讲台下听众有效的指引,转录到文章里对读者零价值。改成「在 Kaggle 上搜 X」「项目地址 github.com/Y」这种读者打开手机能照做的形式,或删除。详见 title-and-publishing.md「结尾的行动指引必须是读者现在能用的动作」。grep 触发词:`幻灯片里` `PPT 上` `展台` `Q&A.{0,3}问到` `中场` `会后` `现场.{0,5}(举手|笑|鼓掌|看到)` `演讲幻灯片`
- [ ] **元过渡 + 冒号铺垫的判断句?**「这件事在 X 上回答了一个问题:」「这背后其实是一个更大的问题:」「这给我们的启示是:」这类作者站出来做导游 + 冒号 + 一个判断的结构,删掉元过渡,让判断独立成句。详见 article-structure.md §3.w 错例 4「元过渡 + 冒号铺垫式」。grep 触发词:`这件事(在|说明|背后).{0,15}(问题|事|判断|启示)[:：]` `这背后.{0,10}是.{0,5}(问题|逻辑|趋势)[:：]` `这给我们.{0,5}启示`
- [ ] **轻度戏剧化动词「抛出」「丢出」?**「他抛出一个观点 / 他抛出一个判断 / 他抛了一个问题」在公众号文章里特别高频但偏文学化,在中性陈述里换成「给出 / 提出 / 说了」。详见 ai-voice-prohibitions.md §六「文学化动词 → 大白话动词」清单
- [ ] **整稿自审通过后,有没有再做一遍段级精修?** 把每一段单独抽出来读,问"读者只读到这里就退出他带走了什么"。整稿审视会放过段级深层问题,段级精修能挖出 3-5 个隐藏卡顿点。具体逐项见本文上方「段级精修必查清单」(多重定语 / 范围副词 / 表语翻译腔 / 末段综合升华)。详见 article-structure.md §3.y「段级互动重写比一次性整稿改更准」

**标点、加粗与术语**
- [ ] 有没有冒号（`：`）、破折号（`——`）、双引号？**这条容易漏。写完后必须 grep `——` 全文扫一遍,命中数量必须为 0**(章节标题、引用块和文末来源行除外)。每出现一处破折号都要回到语义层重写为完整句号断句或显性连接词,不能查找替换
- [ ] 删掉破折号/冒号的地方，把前后两句独立读一遍，前句是否变成悬空判断、对应关系是否要读者自己猜？是就回到语义层重写，不是查找替换
- [ ] 「」是否只用于概念名词，而非直接引语？
- [ ] 章节标题用 H2（`## 标题`），段中加粗只用于硬数字 / 反直觉判断 / 核心论点，密度在每千字 ≤ 2 处之内？
- [ ] 情绪加粗是否只用在硬数字 / 反直觉判断上？
- [ ] 英文术语词组是否每个单词首字母大写（Agent / Semantic Search / Context Window）？
- [ ] 反复出现的英文产品 / 系统 / 框架名（Harness / Agent / Sprint / Eval 等）首次定义可保留小写引号形式（「一个『harness』，指……」），但后续 ≥3 次出现要统一首字母大写
- [ ] 缩写是否全大写（RAG / LLM / ESQL）而不是 Rag / Llm / Esql？
- [ ] 段落收尾是不是「齐了」「够了」「完了」「好了」「行了」这种单字动词收？改成「齐备了 / 足够了 / 完成了 / 做好了 / 可以了」之类双字词收口
- [ ] 代码标识符、命令、参数名是否全部用反引号包起来且保持原形（`@tool` / `top K` / `jina-grep`）？

**语气与内容**
- [ ] 有没有 AI 腔词语（包括「不是 A 而是 B」及其变体）？
- [ ] 每段是否至少有一个具体的人、事、数字、场景或原话？
- [ ] 是否避免了「某某说」「在演讲中」这类转场腔？
- [ ] 是否避免了「X 对 Y 评价挺复杂」这种空转场句？
- [ ] 有没有把模糊形容词换成可量化的数字？
- [ ] 有没有残留对读者零价值的转录细节（讲者念不准某词、现场笑了、让观众举手、时间关系跳过、未展开的人名）？抽掉后文章是否更干净？
- [ ] 涉及「这张图」「这根箭头」「这段代码」「屏幕上」等 deixis 指示语的段落都截图插入了吗？有没有 `[图:MM:SS - 描述]` 占位符忘了替换？
- [ ] 有没有「X 做了一件事。他……」这种先卖关子再揭晓的结构？有就砍铺垫直接写动作
- [ ] 有没有「那句『』」「这条路」这种指代嵌套或抽象指代？能换成具体名词就换
- [ ] 讲者跑了几分钟、调了哪个参数、用了哪个版本 —— 这些执行细节是判断依据还是凑字数？不是前者就删
- [ ] 「直接」「其实」「就是」「真的」「确实」「果然」「真正」「的确」「事实上」这类强调副词在同一段出现两次以上没？后面那个大概率能删；「真的」单独出现绝大多数能直接删
- [ ] 「故意」「刻意」「特意」「有意识地」这类主语态度修饰词出现时，后半句是不是已经把"为什么这么做"写出来了？写出来了就删态度词，没写出来就把态度词换成"为了 X 才……"补出动机
- [ ] 删完冗余后，相邻分句的关系是否还清楚？不清楚就加显式连接词（「还有一个」「同样」「反过来说」「但」「不过」「然而」）—— 转折关系尤其要显式标，否则读者要回扫才能看出是同向递进还是反向制衡
- [ ] 有没有「X 问、Y 答」「主持人追问，嘉宾回应」这类对谈元信息？抽掉这句后观点推进是否损失？不损失就删，直接分点或直接写判断
- [ ] 同一发言人短段内反复出现「X 自己说」「X 专门停下来」「X 特别强调」「X 拿起话筒」「X 顿了顿」「X 又补了一句」这类发言元归属？在已经确立讲者的段落里，第二次以后改成「X 认为」「X 给的答案是」或直接陈述判断
- [ ] 有没有英文俚语 / 习语直译没还原中文白话？常见有：眯起眼看（squint at it →仔细看 / 凑近看）、推动指针（move the needle →带来实质变化）、低垂的果实（low-hanging fruit →最容易拿下的部分）、屋里的大象（elephant in the room →没人愿意说但都看见的问题）、咬子弹（bite the bullet →咬牙做了）、煮海（boil the ocean →一口吃成胖子）
- [ ] 判断出现之前或之后有没有「这是最值得抄的一句」「讲者原话是」「作者的批评很克制，方向却很清楚」「这个观点挺犀利」这类元评价（铺垫或收尾）？基本全删，让判断自己显重要性 —— 测试：抽掉这句话读者对前文的感受会不会变？不会就删
- [ ] 有没有「评价式导语句」——H2 下第一句或段首先给后文下一句评价 / 定性（比外界想象的更具体 / 也很直接 / 很有意思），再展开内容？删评价导语，内容直接出场，能和下句合并就合并。grep `比外界.{0,6}(更|想象)` `也很直接` `更具体。` `更直接。`。详见 [sentence-level-errors.md](sentence-level-errors.md) §6.2.x 评价式导语句要删
- [ ] 有没有「做了两件事」「还有三件事」「几样动作」「几个方面」这种不带语义的笼统量词？按上下文换成「两处调整/三条建议/三个手段/三种方法/三个路径/三次失败/三类做法」等携带内容性质的量词
- [ ] 有没有替原视频/原讲者做宣传（课程、订阅、newsletter、social handles）？默认删。例外是「读者当场就能用到的具体免费资源」，保留时压到一句话
- [ ] 有没有冗余默认属性限定词？公开人物的公开发言默认是公开的，作者写的文章默认是他写的，巨头旗下产品的归属默认熟知。grep `公开(判断|发言|说过) / 旗下的 / 亲自(写|做) / 本人(写|发)`，命中即删
- [ ] 有没有「接过来的 Hugo / 站起身的 Marc」这种分词短语做人物定语？改成时间副词衔接（接下来 / 这时 / 接着）
- [ ] 有没有「的那个词 / 的那种做法」这种尾缀冗余？改成直接收尾或改定语
- [ ] 「让 X 快」「让 X 好」里 Y 是形容词的使役结构，有没有把 Y 动词化为「变 Y / 成为 Y」？
- [ ] 「不再有 X」之类的省略陈述，后面是否补了谓语（不再有 X 参与 / 做 Y / 承担 Z）？
- [ ] 主谓宾是否在同一句里完整出现？有没有"主语漂浮"——动作没明确施事者，让读者自己脑补？
- [ ] 「一动 / 一变 / 一拍 / 一来 / 一开 / 一改」这类省略指代的紧凑结构，有没有补回到底是谁在动 / 变什么 / 拍哪里？省略指代的副作用是读者必须停下来回扫上文
- [ ] 「同理心 / 共情力 / 反思能力 / 决断力」这类抽象名词独立成句，能不能改成动词短语（「站在用户视角思考」「停下来想清楚再做」「该决定的时候不拖」）？抽象名词读者得自己脑补内涵，动词短语直接把动作画出来
- [ ] 行业术语 / 学术名词 / 英文直译是否换成了白话候选？参见 §五·学术 / 直译术语的白话化候选清单（如 leniency bias→容易放水、sycophancy→讨好倾向、出乎预期→出乎意料、临门一脚→还有缺口、朴素→简单）
- [ ] 规范性建议有没有用「应该 / 应当」显式化？否则会被读成事实描述
- [ ] 涉及主导关系（融资领投、会议主持、项目牵头）的归因有没有用「由 X Y」？动宾直连会读成对等参与
- [ ] 描述系统、指标、论断时有没有用美术/文艺名词比喻（底色/脉络/骨架/纹理/色调）？工程判断换工程名词（本质原因 / 主干 / 前提）。描述人/团队/文化时可以保留
- [ ] 有没有作者对自己叙事行为的元评述（这层关系先摆出来 / 这点先放这里 / 下面展开说）？默认删
- [ ] ledger/账本用词一致性：开发/数据流/Git 语境里统一用「明细」或「流水」，不用「账本」（会计语境）

**标题与结尾**
- [ ] 标题是否包含具体主体（不是纯抽象概念）？
- [ ] 标题长度是否在 15–25 字之间？
- [ ] 如果是问题句标题，文章里是否真的给出了答案？
- [ ] 结尾是启发清单型还是收束画面型？有没有以「总结」「综上」开头的空段落？
- [ ] 如果材料能对应实操场景，是否做了一次桥接把判断落到具体团队 / 岗位上？
- [ ] 桥接里有没有出现「国内团队」「中国 AI 圈」「海外做法」这类地域二分？如果有，抽掉地域限定词，经验是否还成立？成立就不要保留地域词

**整体阅读感**
- [ ] 正文读起来是在分享观点，还是替原视频做广告 / 逐段复述演讲者说了什么？
- [ ] 文章抽掉主讲人名字，读起来是作者自己的判断，还是 transcript 的缩写？

**高阶手法使用检查（可选，第十章内容）**
- [ ] 如果用了两件事合流结构，合流点是自然浮现还是被点破了？
- [ ] 如果用了跨领域 reframing，这个框架是帮助读者看清事件，还是只是在炫耀？
- [ ] 如果文章典故密度偏高，有没有一两处自嘲式切换器做节奏调节？
