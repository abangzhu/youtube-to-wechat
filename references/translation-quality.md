# 英译中翻译与本地化手册

这份文件负责英文来源、机器中文字幕和本地化改写后的中文稿。目标有两层：先保证原文没有走样，再把中文改到自然可读。它适用于 `bilingual-transcript.md`、`transcript.md` 和最终文章的本地化复查。

微信公众号成文规则（结构、开头、来源交代、标点、引号、加粗、标题）统一看 [wechat-writing-rules.md](wechat-writing-rules.md)。

---

## 处理层级

### 一、忠实翻译层

用于 `bilingual-transcript.md`。目标是让用户能逐段核查原文含义，不能为了顺口而压缩、删改或重组内容。

| 情形 | 处理方式 | 示例 |
|------|----------|------|
| 技术术语 | 保留英文 | microVM、PR、Agent、Session、plan mode、diff、merge conflict、live preview、terminal、commit message |
| 产品名 / 工具名 | 保留英文 | ACE、GitHub Next、Copilot、Slack、Jira、Linear、VS Code |
| 首次出现的缩写 | 展开全称后括号注英文 | 拉取请求（Pull Request，PR） |
| 口语化俚语 / 脏话 | 意译，保留语气，不直译 | "department of f*** around and find out" → 「边试边学、大胆探索部门」 |
| 新造词 / 演讲者自创表达 | 中文意译 + 括号保留英文 | 「社交信息织物（social information fabric）」 |
| 普通口语停顿（"uh"、"um"）| 不必译出，但内容不能删 | — |

保持原文的时间戳拆分方式，多段不合并。忠实原意，不美化、不压缩、不省略实质性内容。

### 二、本地化整理层

用于 `transcript.md` 和 `article.md`。目标是让中文读者读起来不费劲，但仍能追溯原文意思。

| 情形 | 处理方式 | 示例 |
|------|----------|------|
| 有通用中文译法的专有名词 | 译文（原文） | 产品经理（Product Manager） |
| 缩写首次出现 | 译文（全拼） | 首席产品官（Chief Product Officer，CPO） |
| 缩写再次出现 | 直接用缩写 | CPO |
| 无合适中文译法 / 中文比英文更陌生 | 保留英文 | vibe coding、Agent、Eval |
| 俚语和口语 | 意译，不直译 | "going upstream" → 逆流而上，不是「去上游」 |
| 英文句式痕迹 | 改写为自然汉语 | 被动句改主动，长定语从句拆成两句，避免名词堆砌 |

### 三、指南和官方文档层

官方文档、产品指南、技术规范、操作手册不能压缩成摘要。这类内容的价值在具体说明、参数细节和示例里。原文有几条建议就翻几条，原文有示例就保留并翻译。

英文 prompt 示例、代码块或官方模板必须中英文对照，放在同一个代码块内。英文在上，空行分隔，中文在下。

```
Ask for clarification only when the missing information would materially change the answer.

只在缺失信息会实质性地改变答案时才请求澄清。
```

## 翻译腔排查方法

转录整理（Step 1）和文章评估（Step 6）各做一次全文通读排查。

**举一反三的方法**：每个问题都有一个「触发特征」——在原文或译文里看到它，就该停下来想想。多数错误不需要背例子，认清根源模式，新句子自然会判断。

---

## 一、形合 vs 意合：英中句法的根本差异

**根源**：英文用显性连接词（because, so, although, which, that...）把句子黏在一起，叫「形合」。中文靠语序和语境传递逻辑，不说出来读者也懂，叫「意合」。把英文的胶水词硬搬进来，就是最常见的翻译腔。

**触发特征**：译文里出现「因为…所以」「虽然…但是」两者都有，或者每句开头都有逻辑连接词。

**举一反三模式**：

| 英文连接词 | 直译（错） | 中文习惯写法 |
|-----------|-----------|------------|
| Because A, so B | 因为下雨，所以我没出门 | 下雨，我没出门（只用一个） |
| Although A, but B | 虽然很难，但他还是做到了 | 很难，他还是做到了 |
| This is why... | 这就是为什么我们需要… | 于是 / 所以才有了… |
| In other words... | 换句话说，也就是说… | 直接说，不用引桥 |
| Therefore / Thus | 因此，我们可以得出… | 所以 / 直接陈述结论 |
| Additionally / Furthermore | 此外，另外，再者 | 说到 / 还有一点 / 顺带 |

**举一反三**：看到「因此」「此外」「另外」密集出现，想的问题是：这些词删掉之后句意还完整吗？完整就删。

**特别警惕“不是……而是……”**：英文里常用 not A but B 搭强对比，中文公众号里频繁使用会有明显 AI 味。除非真的存在必须强调的二元对立，否则改成正面陈述或递进解释。

| 翻译腔/AI腔 | 中文作者写法 |
|------------|--------------|
| 它不是一个测试工具，而是一个数据平台 | 做到后面，团队会发现自己在维护一套数据平台 |
| 这不是上线前检查，而是生命周期里的飞轮 | 这个循环会贯穿 Agent 的整个生命周期，远远不只是发布前的一道检查 |
| 重点不是工具选择，而是一个判断 | 最值得带走的是一个判断，具体工具反而没那么重要 |

## 二、被动语态：「被」字的语义陷阱

**根源**：英文被动是中性语态，可以表达任何完成状态。中文「被」字有受害、被迫的意涵（「被打」「被迫」），用在中性场合听起来像出了什么事。

**触发特征**：译文里出现「被+动词」。

**举一反三模式**：

| 英文被动 | 直译（错，有受害感） | 正确（主动或无主） |
|---------|-----------------|-----------------|
| The problem was solved | 问题被解决了 | 问题解决了 |
| The report was submitted | 报告被提交了 | 报告交上去了 |
| Users are given access | 用户被给予访问权限 | 用户可以访问 |
| It was decided that... | 被决定… | 决定… |
| The feature was removed | 功能被移除了 | 这个功能去掉了 |
| Time was saved | 时间被节省了 | 省了时间 |

**例外**：真正有受害感或被迫感时用「被」是对的。「他被开除了」「数据被泄露了」——这里用「被」没有问题。

**举一反三**：看到「被」，问自己：这件事对主语来说是被动接受的坏事，还是只是一个中性结果？如果是中性结果，改成主动句或无主句。

## 三、名词化：把动词变成名词

**根源**：英文学术/商业文体喜欢把动词变成名词（nominalization），显得「正式」。中文相反——动词主导，过多名词化反而生硬。

**触发特征**：「…的+名词」结构密集出现，大量「之」「性」「度」「化」结尾的名词，或「进行…」「实现…」「给予…」等动词替代词。

**举一反三模式**：

| 英文名词化 | 直译（错） | 恢复动词（对） |
|-----------|----------|-------------|
| the improvement of efficiency | 效率的提高 | 效率提高了 |
| the implementation of the plan | 计划的实施 | 把计划落地 / 执行计划 |
| the analysis of data | 数据的分析 | 分析数据 |
| the growth of the user base | 用户基础的增长 | 用户数在增长 |
| to make a decision | 做出一个决定 | 决定 |
| to give support | 给予支持 | 支持 |
| to have an understanding of | 拥有对…的理解 | 理解 |
| a quantifiable cost source | 可以量化的成本来源 | 实实在在的成本 |

**举一反三**：看到「进行」「给予」「实现」「做出」后面跟名词，几乎可以直接删掉，换成动词本身。「进行讨论」→「讨论」；「给予帮助」→「帮助」。

## 四、形式主语和空洞强调结构

**根源**：英文用「It is... that...」「There is...」「It seems that...」搭出形式主语框架，实义在后。中文不需要这个框架，直接说。

**触发特征**：「重要的是」「有趣的是」「值得注意的是」「显而易见的是」这类开头。

**举一反三模式**：

| 英文结构 | 直译（冗余） | 精简版 |
|---------|-----------|-------|
| It is important that | 重要的是 | 关键在于 / 有一点要说 / 直接说那件事 |
| It is worth noting that | 值得注意的是 | 顺带一提 / 这里要说一下 |
| It seems that | 看起来似乎是 | 看来 / 感觉 |
| It is clear that | 很明显的是 | 显然 / 明显 |
| There is a tendency to | 存在一种倾向 | 往往会 / 常常 |
| It is interesting that | 有趣的是 | 有意思的是 / 直接讲那个有趣的事 |

**举一反三**：「值得一提的是」「不得不说的是」这类套话通常可以直接删掉，把后面的内容直接开始。删掉后读一遍，如果反而更流畅，就说明原来那几个字是废话。

## 五、副词和程度词堆叠

**根源**：英文用 truly, genuinely, very, highly, particularly 等副词叠加强调，用中文一字直译变成副词堆砌，读起来像翻译。中文强调靠语序、句式、停顿，不靠副词叠加。

**触发特征**：两个及以上程度副词出现在同一个动词或形容词前。

**举一反三模式**：

| 英文叠加副词 | 直译（错） | 中文惯用法 |
|-----------|----------|----------|
| truly and genuinely understand | 真正且真诚地理解 | 真的理解 / 真正理解 |
| very significantly improve | 非常显著地改善 | 大幅改善 |
| actively and proactively engage | 积极主动地参与 | 积极参与 |
| deeply and carefully consider | 深入而仔细地考虑 | 认真想清楚 |
| 需要人主动早早地、自然地把上下文带进来 | （已是直译错误） | 需要人从一开始就把这些东西带进来 |

**举一反三**：看到两个以上副词修饰同一个词，留一个最核心的，其余删掉。如果语气感觉弱了，用句式本身来强调，不用副词堆砌。

## 六、逻辑连接和句式倒置

**根源**：英文「topic-comment」顺序有时和中文相反。英文也习惯把结论放后面（建悬念），中文更常先说话题再说评价。

**触发特征**：句子主干在最后出现，前面一大段修饰语。或者主谓宾被拆散，评论在前，事实在后。

**举一反三模式**：

| 直译的倒装/倒置（错） | 恢复自然语序（对） |
|-----------------|----------------|
| 还有一件事让问题变得更难，Maggie 点出来了 | Maggie 还点出了一个让问题更难解的原因 |
| 这听起来像废话，但后面的逻辑很有力量 | 听起来像废话，但往下推有点意思 |
| 关于这件事，他说了一段值得单独记下的话 | 他说了一段话，值得单独记下来 |
| 在那个领域，他是最具影响力的声音之一 | 他是那个领域最有影响力的声音之一 |
| AI 编程圈子里有一种做法在流行 | AI 编程圈子里流行一种做法 |

**「有一种X在V」存现倒装**：英文「There is a thing that's trending / happening / spreading」的直译落成「有一种X在V」，把动词压在句末做定语，读起来像英文语序。中文自然写法是把动词提到前面——「流行一种做法」「兴起一股风气」「蔓延一种焦虑」。触发特征：「有一种/一个…在+动词」结构。看到就把动词前移。

**举一反三**：读完一句话，问：「这句话最重要的信息是什么？」如果最重要的信息在后半句，考虑是否可以把它提到前面。中文读者习惯在句子开头得到定向信息。

## 七、冒号 + 分号列举结构

**根源**：英文用冒号+分号把并列内容串成正式的列举。中文文章（尤其公众号风格）更习惯散文式叙述，硬保留这种结构会让文章读起来像 PPT 而不是文章。

**触发特征**：「包括：……；……；……」这种带冒号分号的列举。

**举一反三模式**：

| 列举结构（错） | 散文化（对） |
|------------|-----------|
| 没对齐的代价包括：花时间做了没人要的功能；做完了收到推翻一切的反馈；产生了解不开的 merge conflict | 没对齐的代价，现在一次全来了。做了没人要的功能，花时间做完才收到推翻一切的反馈，merge conflict 解不开…… |
| 他的优势包括：丰富的经验；广泛的人脉；强大的执行力 | 他有经验，有人脉，也有执行力 |

**举一反三**：看到列举，不是一定要改散文，但要问：这几条之间有没有递进关系、因果关系、时间顺序？有的话，用这些关系串联，比分号更有力量。三条以下的列举通常值得改散文。

## 八、人称代词密度

**根源**：英文必须有主语（I, we, it, they 省不掉），中文是零代词语言，反复出现「他」「她」「它」「他们」读起来像机翻。

**触发特征**：段落里「他」「她」「它」「我们」「他们」高密度出现，尤其是指物的「它」。

**举一反三模式**：

| 高密度代词（错） | 省略/替换（对） |
|--------------|-------------|
| 它是一个新的系统，它可以做到很多事，它让团队… | 这是一个新系统，能做很多事，让团队… |
| 他说他认为他的团队需要… | 他认为团队需要… |
| 我们需要我们的系统变得更好，我们打算… | 系统要更好，打算… |

**举一反三**：指物的「它」在中文里几乎不出现，遇到直接省略或用「这」替代。指人的「他/她」在同一段内已经明确主语后，后续句子通常可以省略主语。省略之后读一遍，语义还清楚就说明省掉是对的。

## 九、数量感和时间表达

**触发特征**：时间量词模糊（「一个周末」「一段时间」），或数字没有配单位感。

**举一反三模式**：

| 模糊（错） | 有量感（对） |
|----------|-----------|
| 在一个周末做出产品 | 用周末两天做出一个产品 |
| 花了一些时间 | 花了三个月 / 花了好几周 |
| 一段时间之后 | 几个月后 / 两年后 |
| 10 million users | 一千万用户（不是「十百万」） |
| 1 billion | 十亿（不是「一亿」，也不是「百万 ×1000」） |

**中英数字分位差异**：英文以 thousand（千）为单位，中文以万为单位。
- 100,000 → 十万（不是「一百千」）
- 1,000,000 → 一百万
- 10,000,000 → 一千万
- 1,000,000,000 → 十亿

**举一反三**：英文 million 不是「百万」那么简单，要换算成中文习惯的万/亿表达。「50 million」是五千万，不是「五十百万」。

## 十、长句拆分

**根源**：英文一个复合句可以很长（有从句支撑），中文读者习惯短句流水。一般经验：英文一个复合句，中文可以拆成 2-3 个简单句，反而更有力。

**触发特征**：超过 40 字的单句；「的」字结构叠加超过 3 次；同一句里有两个以上从句。

**举一反三模式**：

| 长句直译（费力） | 拆句（有力） |
|--------------|-----------|
| 尽管项目面临了包括预算限制和技术困难在内的众多挑战，团队还是设法在最后期限前按时交付了成果。 | 项目一路麻烦不断——预算不够，技术也卡壳。团队还是按时交出来了。 |
| 这是一个由公司去年成立的、专注于探索前沿 AI 技术的研究团队。 | 这个研究团队是公司去年成立的，专门研究前沿 AI。 |

**举一反三**：「的」是中文「定语+名词」的标志。超过三层「的」叠加（「…的…的…的+名词」）几乎总是可以拆成两句。

## 十一、信息排列顺序（先果后因）

**根源**：英文常用 topic sentence 结构，先给结论，再给理由和数据。中文文章也可以这样写，但有时候先讲情境再讲数字会更有冲击力。

**触发特征**：数字或结果孤立出现，没有先交代为什么这个数字值得记住。

**举一反三**：

问自己：「如果读者不知道这个数字的背景，这个数字会让他们觉得震撼吗？」如果不会，先交代背景再给数字。

- 错：广告制作时间从 2 小时压缩到 15 分钟。
- 对：一件以前要一个团队做两个小时的事，现在他一个人 15 分钟搞定。

## 十二、套话和空洞词

这类词从英文里来，在中文里是翻译腔的典型标志，删掉之后文章反而更好。

**高频套话**：

| 套话 | 原始英文 | 处理方式 |
|-----|---------|---------|
| 值得一提的是 | It is worth noting that | 直接说那件事 |
| 值得关注的是 | It is notable that | 同上 |
| 不可否认的是 | It is undeniable that | 同上 |
| 毫无疑问 | Without a doubt | 删掉 |
| 不言而喻 | It goes without saying | 删掉 |
| 显而易见 | It is obvious that | 删掉，或换成「明显」「显然」 |
| 我们可以看到 | We can see that | 删掉 |
| 在…方面 | In terms of... | 通常可删，直接说那个方面 |
| 基于上述 | Based on the above | 删掉，直接说结论 |

**举一反三**：看到这类开头，先试试直接删掉。删掉之后如果意思还完整，原来那几个字就是废话。

---

## 十三、「thing」「stuff」等泛指词的语境处理

**根源**：英文用 thing/stuff/it/something 作泛指，中文里需要根据语境具体化。

**触发特征**：「东西」「事情」在产品、技术语境里高密度出现。

**举一反三模式**：

| 英文 | 直译（错） | 语境具体化（对） |
|-----|---------|--------------|
| your thing | 你的东西 | 你的产品 / 你做的东西（创作语境）|
| this thing | 这个东西 | 这个功能 / 这件事 / 这套系统 |
| build something | 做出一些东西 | 做出一个产品 / 造出东西 |
| here（方位）| 这里 | 这一点 / 这种状态 / 这个位置 |
| here（比喻）| 六个月内所有人都到这里 | 六个月内所有人都能做到这一点 |

## 十四、科技/产品类专项

**术语一致性**：同一个概念，全文只用一种中文译法。常见混用：智能代理/智能体/Agent，机器学习/ML，模型/大模型/LLM。

**UI 字符串过度翻译**：
- Save → 保存（不是「进行保存操作」）
- Settings → 设置（不是「设置选项」）
- Submit → 提交（不是「点击提交按钮」）

**缩写处理原则**：API/SDK/UI 这类通用缩写不必翻。第一次出现可加括号注中文全称，之后直接用缩写。

## 快速诊断问题

读完文章或转录稿，用这几个问题快速定位翻译腔：

1. **大声读一遍**——读起来像在背书、需要停下来重读的地方，基本都有问题
2. **数「的」字**——一句话里超过 3 个「的」，几乎可以断定需要拆句
3. **数「被」字**——每一个「被」都检查是否有受害感，没有就改成主动
4. **数连接词**——「因此」「此外」「另外」密集出现，检查能否删掉
5. **找套话**——「值得一提的是」「不言而喻」这类词，先试试删掉

## 翻译理论背景（为什么这些原则成立）

- **严复「信达雅」**：信（忠于原意）> 达（表达流畅）> 雅（文字优美）。顺序重要——最常见的错误是过分追求「雅」而牺牲「达」
- **奈达「动态对等」**：译文在目标语读者身上产生的效果，应等同于原文在源语读者身上产生的效果。意味着翻译的判断标准是「读者感受」，不是「是否和原文结构一样」
- **形合 vs 意合（吕叔湘）**：中文靠语序和语境传递逻辑（意合），英文靠连接词（形合）。这是几乎所有翻译腔的根本来源
