# 英译中翻译与本地化手册

这份文件负责英文来源、机器中文字幕和本地化改写后的中文稿。目标有两层：先保证原文没有走样，再把中文改到自然可读。它适用于 `bilingual-transcript.md`、`transcript.md` 和最终文章的本地化复查。

微信公众号成文规则（结构、开头、来源交代、标点、引号、加粗、标题）统一看 [wechat-writing-rules.md](wechat-writing-rules.md)。

## 处理层级

### 翻译前的上下文标记

翻译前先读完整段，不要逐句硬翻。英文常把真正的主语、指代对象和回指关系藏在前后文里，如果没先标记，下游的 `transcript.md` 和 `article.md` 会建立在错误理解上。

每段翻译前快速标记：

- 说话人和真实主语是谁，是个人、公司、团队、产品还是系统
- `it`、`they`、`this`、`that`、`here`、`there` 指向什么
- `thing`、`stuff`、`component`、`something` 这类泛指词在当前语境里具体是什么
- 这句话有没有 call back 到前文概念、比喻或术语
- 主语是否发生切换，例如从人切到公司，从产品切到系统

这一步对 `bilingual-transcript.md` 尤其重要。双语对照是核查原意的底稿，一旦这里把指代和逻辑译错，后面的提炼和写作很难补救。

### 一、忠实翻译层

用于 `bilingual-transcript.md`。目标是让用户能逐段核查原文含义，不能为了顺口而压缩、删改或重组内容。

| 情形 | 处理方式 | 示例 |
|------|----------|------|
| 技术术语 | 优先保原意可核查，有成熟中文译法时用中文锚点 + 英文 | 拉取请求（Pull Request，PR）、上下文窗口（context window）、推理链条（chain of thought） |
| 产品名 / 工具名 / UI 标签 | 保留英文 | ACE、GitHub Next、Copilot、Slack、Jira、Linear、VS Code、terminal、commit message |
| 首次出现的缩写 | 展开全称后括号注英文 | 拉取请求（Pull Request，PR） |
| 口语化俚语 / 脏话 | 意译，保留语气，不直译 | "department of f*** around and find out" → 「边试边学、大胆探索部门」 |
| 新造词 / 演讲者自创表达 | 中文意译 + 括号保留英文 | 「社交信息织物（social information fabric）」 |
| 普通口语停顿（"uh"、"um"）| 不必译出，但内容不能删 | — |

保持原文的时间戳拆分方式，多段不合并。忠实原意，不美化、不压缩、不省略实质性内容。

### 二、本地化整理层

用于 `transcript.md` 和 `article.md`。目标是让中文读者读起来不费劲，但仍能追溯原文意思。

| 情形 | 处理方式 | 示例 |
|------|----------|------|
| 有通用中文译法的专有名词 | 译文（原文） | 产品经理（Product Manager） |
| 缩写首次出现 | 译文（全拼） | 首席产品官（Chief Product Officer，CPO） |
| 缩写再次出现 | 直接用缩写 | CPO |
| 无合适中文译法 / 中文比英文更陌生 | 保留英文 | vibe coding、Agent、Eval |
| 俚语和口语 | 意译，不直译 | "going upstream" → 逆流而上，不是「去上游」 |
| 英文句式痕迹 | 改写为自然汉语 | 被动句改主动，长定语从句拆成两句，避免名词堆砌 |

### 三、指南和官方文档层

官方文档、产品指南、技术规范、操作手册不能压缩成摘要。这类内容的价值在具体说明、参数细节和示例里。原文有几条建议就翻几条，原文有示例就保留并翻译。

英文 prompt 示例、代码块或官方模板必须中英文对照，放在同一个代码块内。英文在上，空行分隔，中文在下。

```
Ask for clarification only when the missing information would materially change the answer.

只在缺失信息会实质性地改变答案时才请求澄清。
```

## 翻译腔排查方法

转录整理（Step 1）和文章评估（Step 6）各做一次全文通读排查。

**举一反三的方法**：每个问题都有一个「触发特征」——在原文或译文里看到它，就该停下来想想。多数错误不需要背例子，认清根源模式，新句子自然会判断。

## 一、形合 vs 意合：英中句法的根本差异

**根源**：英文用显性连接词（because, so, although, which, that...）把句子黏在一起，叫「形合」。中文靠语序和语境传递逻辑，不说出来读者也懂，叫「意合」。把英文的胶水词硬搬进来，就是最常见的翻译腔。

**触发特征**：译文里出现「因为…所以」「虽然…但是」两者都有，或者每句开头都有逻辑连接词。

**举一反三模式**：

| 英文连接词 | 直译（错） | 中文习惯写法 |
|-----------|-----------|------------|
| Because A, so B | 因为下雨，所以我没出门 | 下雨，我没出门（只用一个） |
| Although A, but B | 虽然很难，但他还是做到了 | 很难，他还是做到了 |
| This is why... | 这就是为什么我们需要… | 于是 / 所以才有了… |
| In other words... | 换句话说，也就是说… | 直接说，不用引桥 |
| Therefore / Thus | 因此，我们可以得出… | 所以 / 直接陈述结论 |
| Additionally / Furthermore | 此外，另外，再者 | 说到 / 还有一点 / 顺带 |

**举一反三**：看到「因此」「此外」「另外」密集出现，想的问题是：这些词删掉之后句意还完整吗？完整就删。

**反向风险：逻辑不能被删没。** 中文意合不是永远省略连接词。关键因果、转折、类比、前文回指如果删掉连接词会变模糊，就用轻一点的连接方式补清楚，例如「之所以」「正是因为」「同样」「也就是说」「换到这个语境里」。要删的是机械的连接词链条，不是原文真正的逻辑关系。

**特别警惕“不是……而是……”**：英文里常用 not A but B 搭强对比，中文公众号里频繁使用会有明显 AI 味。除非真的存在必须强调的二元对立，否则改成正面陈述或递进解释。

| 翻译腔/AI腔 | 中文作者写法 |
|------------|--------------|
| 它不是一个测试工具，而是一个数据平台 | 做到后面，团队会发现自己在维护一套数据平台 |
| 这不是上线前检查，而是生命周期里的飞轮 | 这个循环会贯穿 Agent 的整个生命周期，远远不只是发布前的一道检查 |
| 重点不是工具选择，而是一个判断 | 最值得带走的是一个判断，具体工具反而没那么重要 |

## 二、被动语态：「被」字的语义陷阱

**根源**：英文被动是中性语态，可以表达任何完成状态。中文「被」字有受害、被迫的意涵（「被打」「被迫」），用在中性场合听起来像出了什么事。

**触发特征**：译文里出现「被+动词」。

**举一反三模式**：

| 英文被动 | 直译（错，有受害感） | 正确（主动或无主） |
|---------|-----------------|-----------------|
| The problem was solved | 问题被解决了 | 问题解决了 |
| The report was submitted | 报告被提交了 | 报告交上去了 |
| Users are given access | 用户被给予访问权限 | 用户可以访问 |
| It was decided that... | 被决定… | 决定… |
| The feature was removed | 功能被移除了 | 这个功能去掉了 |
| Time was saved | 时间被节省了 | 省了时间 |

**例外**：真正有受害感或被迫感时用「被」是对的。「他被开除了」「数据被泄露了」——这里用「被」没有问题。

**举一反三**：看到「被」，问自己：这件事对主语来说是被动接受的坏事，还是只是一个中性结果？如果是中性结果，改成主动句或无主句。

## 三、名词化：把动词变成名词

**根源**：英文学术/商业文体喜欢把动词变成名词（nominalization），显得「正式」。中文相反——动词主导，过多名词化反而生硬。

**触发特征**：「…的+名词」结构密集出现，大量「之」「性」「度」「化」结尾的名词，或「进行…」「实现…」「给予…」等动词替代词。

**举一反三模式**：

| 英文名词化 | 直译（错） | 恢复动词（对） |
|-----------|----------|-------------|
| the improvement of efficiency | 效率的提高 | 效率提高了 |
| the implementation of the plan | 计划的实施 | 把计划落地 / 执行计划 |
| the analysis of data | 数据的分析 | 分析数据 |
| the growth of the user base | 用户基础的增长 | 用户数在增长 |
| to make a decision | 做出一个决定 | 决定 |
| to give support | 给予支持 | 支持 |
| to have an understanding of | 拥有对…的理解 | 理解 |
| a quantifiable cost source | 可以量化的成本来源 | 实实在在的成本 |

**举一反三**：看到「进行」「给予」「实现」「做出」后面跟名词，几乎可以直接删掉，换成动词本身。「进行讨论」→「讨论」；「给予帮助」→「帮助」。

## 四、形式主语和空洞强调结构

**根源**：英文用「It is... that...」「There is...」「It seems that...」搭出形式主语框架，实义在后。中文不需要这个框架，直接说。

**触发特征**：「重要的是」「有趣的是」「值得注意的是」「显而易见的是」这类开头。

**举一反三模式**：

| 英文结构 | 直译（冗余） | 精简版 |
|---------|-----------|-------|
| It is important that | 重要的是 | 关键在于 / 有一点要说 / 直接说那件事 |
| It is worth noting that | 值得注意的是 | 顺带一提 / 这里要说一下 |
| It seems that | 看起来似乎是 | 看来 / 感觉 |
| It is clear that | 很明显的是 | 显然 / 明显 |
| There is a tendency to | 存在一种倾向 | 往往会 / 常常 |
| It is interesting that | 有趣的是 | 有意思的是 / 直接讲那个有趣的事 |

**举一反三**：「值得一提的是」「不得不说的是」这类套话通常可以直接删掉，把后面的内容直接开始。删掉后读一遍，如果反而更流畅，就说明原来那几个字是废话。

## 五、副词和程度词堆叠

**根源**：英文用 truly, genuinely, very, highly, particularly 等副词叠加强调，用中文一字直译变成副词堆砌，读起来像翻译。中文强调靠语序、句式、停顿，不靠副词叠加。

**触发特征**：两个及以上程度副词出现在同一个动词或形容词前。

**举一反三模式**：

| 英文叠加副词 | 直译（错） | 中文惯用法 |
|-----------|----------|----------|
| truly and genuinely understand | 真正且真诚地理解 | 真的理解 / 真正理解 |
| very significantly improve | 非常显著地改善 | 大幅改善 |
| actively and proactively engage | 积极主动地参与 | 积极参与 |
| deeply and carefully consider | 深入而仔细地考虑 | 认真想清楚 |
| 需要人主动早早地、自然地把上下文带进来 | （已是直译错误） | 需要人从一开始就把这些东西带进来 |

**举一反三**：看到两个以上副词修饰同一个词，留一个最核心的，其余删掉。如果语气感觉弱了，用句式本身来强调，不用副词堆砌。

## 六、逻辑连接、主语主干和句式倒置

**根源**：英文「topic-comment」顺序有时和中文相反。英文也习惯把结论放后面（建悬念），中文更常先说话题再说评价。另一个常见问题是英文可以靠形式主语、被动结构、长定语和上下文省略主语，中文读者却需要尽早知道「谁做了什么」。

**触发特征**：句子主干在最后出现，前面一大段修饰语。或者主谓宾被拆散，评论在前，事实在后。主语只在上一句出现、当前句换了主语却继续用「它/这/他们」，或者类比两端省略了比较对象。

**举一反三模式**：

| 直译的倒装/倒置（错） | 恢复自然语序（对） |
|-----------------|----------------|
| 还有一件事让问题变得更难，Maggie 点出来了 | Maggie 还点出了一个让问题更难解的原因 |
| 这听起来像废话，但后面的逻辑很有力量 | 听起来像废话，但往下推有点意思 |
| 关于这件事，他说了一段值得单独记下的话 | 他说了一段话，值得单独记下来 |
| 在那个领域，他是最具影响力的声音之一 | 他是那个领域最有影响力的声音之一 |
| AI 编程圈子里有一种做法在流行 | AI 编程圈子里流行一种做法 |

**「有一种X在V」存现倒装**：英文「There is a thing that's trending / happening / spreading」的直译落成「有一种X在V」，把动词压在句末做定语，读起来像英文语序。中文自然写法是把动词提到前面——「流行一种做法」「兴起一股风气」「蔓延一种焦虑」。触发特征：「有一种/一个…在+动词」结构。看到就把动词前移。

**主语和谓语主干**：翻译完一句话，必须能一眼看出谁做了什么。主语从人切到公司、产品、团队或系统时，要显性补出新主语。类比句尤其要补清两端对象，不能让读者猜是「机器像人」还是「DQN 像大脑」。如果一句话翻完只剩抽象判断，没有动作主体，通常要重排语序。

**举一反三**：读完一句话，问：「这句话最重要的信息是什么？」如果最重要的信息在后半句，考虑是否可以把它提到前面。中文读者习惯在句子开头得到定向信息。

## 七、冒号 + 分号列举结构

**根源**：英文用冒号+分号把并列内容串成正式的列举。中文文章（尤其公众号风格）更习惯散文式叙述，硬保留这种结构会让文章读起来像 PPT 而不是文章。

**触发特征**：「包括：……；……；……」这种带冒号分号的列举。

**举一反三模式**：

| 列举结构（错） | 散文化（对） |
|------------|-----------|
| 没对齐的代价包括：花时间做了没人要的功能；做完了收到推翻一切的反馈；产生了解不开的 merge conflict | 没对齐的代价，现在一次全来了。做了没人要的功能，花时间做完才收到推翻一切的反馈，merge conflict 解不开…… |
| 他的优势包括：丰富的经验；广泛的人脉；强大的执行力 | 他有经验，有人脉，也有执行力 |

**举一反三**：看到列举，不是一定要改散文，但要问：这几条之间有没有递进关系、因果关系、时间顺序？有的话，用这些关系串联，比分号更有力量。三条以下的列举通常值得改散文。

## 八、人称代词密度

**根源**：英文必须有主语（I, we, it, they 省不掉），中文是零代词语言，反复出现「他」「她」「它」「他们」读起来像机翻。

**触发特征**：段落里「他」「她」「它」「我们」「他们」高密度出现，尤其是指物的「它」。

**举一反三模式**：

| 高密度代词（错） | 省略/替换（对） |
|--------------|-------------|
| 它是一个新的系统，它可以做到很多事，它让团队… | 这是一个新系统，能做很多事，让团队… |
| 他说他认为他的团队需要… | 他认为团队需要… |
| 我们需要我们的系统变得更好，我们打算… | 系统要更好，打算… |

**举一反三**：指物的「它」在中文里几乎不出现，遇到直接省略或用「这」替代。指人的「他/她」在同一段内已经明确主语后，后续句子通常可以省略主语。省略之后读一遍，语义还清楚就说明省掉是对的。

**不要把省略变成歧义**：中文可以省代词，但前提是指代对象清楚。如果同一段里人物、公司、产品、系统交替出现，或者主语从说话人切到团队/工具/模型，就补出具体主语。翻译校验时优先保清楚，最终写作时再按上下文删掉机械重复。

## 九、数量感和时间表达

**触发特征**：时间量词模糊（「一个周末」「一段时间」），或数字没有配单位感。

**举一反三模式**：

| 模糊（错） | 有量感（对） |
|----------|-----------|
| 在一个周末做出产品 | 用周末两天做出一个产品 |
| 花了一些时间 | 花了三个月 / 花了好几周 |
| 一段时间之后 | 几个月后 / 两年后 |
| 10 million users | 一千万用户（不是「十百万」） |
| 1 billion | 十亿（不是「一亿」，也不是「百万 ×1000」） |

**中英数字分位差异**：英文以 thousand（千）为单位，中文以万为单位。
- 100,000 → 十万（不是「一百千」）
- 1,000,000 → 一百万
- 10,000,000 → 一千万
- 1,000,000,000 → 十亿

**举一反三**：英文 million 不是「百万」那么简单，要换算成中文习惯的万/亿表达。「50 million」是五千万，不是「五十百万」。

## 十、长句拆分

**根源**：英文一个复合句可以很长（有从句支撑），中文读者习惯短句流水。一般经验：英文一个复合句，中文可以拆成 2-3 个简单句，反而更有力。

**触发特征**：超过 40 字的单句；「的」字结构叠加超过 3 次；同一句里有两个以上从句。

**举一反三模式**：

| 长句直译（费力） | 拆句（有力） |
|--------------|-----------|
| 尽管项目面临了包括预算限制和技术困难在内的众多挑战，团队还是设法在最后期限前按时交付了成果。 | 项目一路麻烦不断——预算不够，技术也卡壳。团队还是按时交出来了。 |
| 这是一个由公司去年成立的、专注于探索前沿 AI 技术的研究团队。 | 这个研究团队是公司去年成立的，专门研究前沿 AI。 |

**举一反三**：「的」是中文「定语+名词」的标志。超过三层「的」叠加（「…的…的…的+名词」）几乎总是可以拆成两句。

**英文同位语拆分**：英文常把背景塞进一个同位语或长名词短语里，中文不必硬塞回一个名词结构。可以拆成「逗号 + 这个/这类/这件事」或独立短句。比如 “AlphaFold, a system that solved the 50-year protein folding problem” 不要译成一串长定语，可以写成「AlphaFold 破解了蛋白质折叠问题，这个难题困扰了生物学界 50 年」。

## 十一、信息排列顺序（先果后因）

**根源**：英文常用 topic sentence 结构，先给结论，再给理由和数据。中文文章也可以这样写，但有时候先讲情境再讲数字会更有冲击力。

**触发特征**：数字或结果孤立出现，没有先交代为什么这个数字值得记住。

**举一反三**：

问自己：「如果读者不知道这个数字的背景，这个数字会让他们觉得震撼吗？」如果不会，先交代背景再给数字。

- 错：广告制作时间从 2 小时压缩到 15 分钟。
- 对：一件以前要一个团队做两个小时的事，现在他一个人 15 分钟搞定。

## 十二、套话和空洞词

这类词从英文里来，在中文里是翻译腔的典型标志，删掉之后文章反而更好。

**高频套话**：

| 套话 | 原始英文 | 处理方式 |
|-----|---------|---------|
| 值得一提的是 | It is worth noting that | 直接说那件事 |
| 值得关注的是 | It is notable that | 同上 |
| 不可否认的是 | It is undeniable that | 同上 |
| 毫无疑问 | Without a doubt | 删掉 |
| 不言而喻 | It goes without saying | 删掉 |
| 显而易见 | It is obvious that | 删掉，或换成「明显」「显然」 |
| 我们可以看到 | We can see that | 删掉 |
| 在…方面 | In terms of... | 通常可删，直接说那个方面 |
| 基于上述 | Based on the above | 删掉，直接说结论 |

**举一反三**：看到这类开头，先试试直接删掉。删掉之后如果意思还完整，原来那几个字就是废话。

## 十三、「thing」「stuff」等泛指词的语境处理

**根源**：英文用 thing/stuff/it/something 作泛指，中文里需要根据语境具体化。

**触发特征**：「东西」「事情」在产品、技术语境里高密度出现。

**举一反三模式**：

| 英文 | 直译（错） | 语境具体化（对） |
|-----|---------|--------------|
| your thing | 你的东西 | 你的产品 / 你做的东西（创作语境）|
| this thing | 这个东西 | 这个功能 / 这件事 / 这套系统 |
| build something | 做出一些东西 | 做出一个产品 / 造出东西 |
| here（方位）| 这里 | 这一点 / 这种状态 / 这个位置 |
| here（比喻）| 六个月内所有人都到这里 | 六个月内所有人都能做到这一点 |
| there | 到那里 / 在那里 | 按上下文还原成目标状态、终点、AGI 位置或前文提到的能力边界 |
| today | 今天 | 多数技术/行业语境里是「如今」「现在」「现阶段」 |
| I can't see a world in which... | 我无法看到一个……的世界 | 我不认为…… / 很难想象…… |
| components | 组件 | 可能是关键要素、组成部分、能力模块，按上下文判断 |

**call back 处理**：如果一个词是在回指前文概念，不要翻表面词，要把它指回的概念翻出来。比如前文一直在讲模型的长期记忆，后面一句里的 `that capability` 就不是「那个能力」这么简单，而是「长期记忆能力」。

## 十四、科技/产品类专项

**术语一致性**：同一个概念，全文只用一种中文译法。常见混用：智能代理/智能体/Agent，机器学习/ML，模型/大模型/LLM。

**术语保留边界**：产品名、工具名、模型名、API/SDK/UI 标签、无成熟中文译法的概念，或中文译法比英文更陌生的词，可以保留英文。有成熟中文译法的普通技术表达要翻出来，不要为了显得专业而保留英文。

| 英文 | 推荐中文 |
|-----|----------|
| experience replay | 经验回放 |
| brute force | 粗暴 / 原始粗暴（按语境） |
| context window | 上下文窗口 |
| working memory | 工作记忆 |
| chain of thought | 推理链条 |
| thinking modes | 思考模式 |

**术语优先大白话译法**：

定义：英文术语进入中文文章时，首先尝试用大白话或既有中文常用译法，只有当大白话明显失真或行业里就用直译时，才保留字面直译。

判断标准：把这个译法读给一个不懂英文的中国读者，他能不能立刻 infer 出术语背后的动作或角色？能 → 用这个译法；不能 → 换。

固定译法对照表（出现时优先用左侧）：

| 英文 | 推荐中文（优先） | 不推荐（看起来直译标本） |
|------|----------------|------------------------|
| devil's advocate | 唱反调的人 | 魔鬼代言人 |
| testable hypothesis | 可验证的假设 | 可测试假设 |
| zero-friction scope creep | 零成本的功能膨胀 | 零摩擦的范围蔓延 |
| product-market fit | PMF | 产品市场契合 |
| problem-solution fit | 问题-方案匹配 / 保留英文 | 问题方案契合 |
| agentic technical debt | agentic coding 留下的技术债 / 保留英文 | agentic 技术债务 |

边界：行业里早已固定的英文直译（「上下文」「生产环境」「沙箱」）不要为了大白话再换。这条规则只针对那些字面直译让中文读者卡顿的术语。

工程动词选词：英文 "build" 在产品 / 工程语境里译中文，单字「建」偏轻，「搭建 / 开发」更自然（「在你产品之上搭建」而非「在你产品之上建」）。

**UI 字符串过度翻译**：
- Save → 保存（不是「进行保存操作」）
- Settings → 设置（不是「设置选项」）
- Submit → 提交（不是「点击提交按钮」）

**缩写处理原则**：API/SDK/UI 这类通用缩写不必翻。第一次出现可加括号注中文全称，之后直接用缩写。

## 中文搭配检查

翻译后单看每个词都对，不等于整句话自然。中文搭配要检查动宾关系、修饰语、量词和所有格，尤其是英文抽象名词、形容词最高级和 `of` 结构直译过来的地方。

**触发特征**：动词和宾语不搭，形容词修饰对象别扭，量词缺失，或者「他的/它的/我们的」这类所有格密集出现。

| 机械搭配（错） | 中文搭配（对） |
|--------------|--------------|
| 攻克一个想法 | 想清楚一个想法 / 解决一个问题 |
| 最大的强项 | 最强的地方 / 最大优势 |
| 必须服务这个目标 | 需要服务于这个目标 / 要为这个目标让路 |
| 他的团队的工作方式 | 团队的工作方式 |
| 这个模型有很多能力 | 这个模型能做很多事 / 具备多种能力 |

**举一反三**：看到抽象动词配抽象名词，先问中文里平时会不会这么说。不会，就换成具体动作。看到英文所有格，先试着删掉「的」前面的代词；中文上下文能接住，就不要补。

## 快速诊断问题

读完文章或转录稿，用这几个问题快速定位翻译腔：

1. **大声读一遍**——读起来像在背书、需要停下来重读的地方，基本都有问题
2. **数「的」字**——一句话里超过 3 个「的」，几乎可以断定需要拆句
3. **数「被」字**——每一个「被」都检查是否有受害感，没有就改成主动
4. **数连接词**——「因此」「此外」「另外」密集出现，检查能否删掉
5. **找套话**——「值得一提的是」「不言而喻」这类词，先试试删掉
6. **做回译测试**——抽查关键论点、因果关系、技术术语、数字和直接引语，把中文倒回英文后，主语、逻辑、术语和程度不能变

## 真实案例回归表：10 个错译 / 问题 / 对译

下面这些真实 case 是回归测试样本，不能只吸收方法论而丢掉原始错译、问题和对译。英文来源进入 `bilingual-transcript.md`、`transcript.md` 和最终文章前，都要用这些 case 反查一次。

### 六类高频问题和处理公式

**问题 1：主语缺失 / 主语倒置**

症状：译文照搬英文语序，主语要么掉了，要么被埋在句末，中文读者要读两遍才知道谁在做什么。

根源：英文可以用 `it`、`there is`、逻辑倒装来延迟主语；中文读者习惯先给主语，再讲事情。

处理公式：英文主语在后，中文提到句首。英文省略主语时，中文要显性补出，尤其是主语切换时。类比句涉及机器 vs 人类、DQN vs 大脑时，两端主语都要补齐，不能只写一边。

case 举例：`these are still unsolved` 不能只译成「这些问题仍然没有解决」，要按上下文补成「我认为这些问题仍然没有解决」。`co-founded DeepMind with one mission` 不能译成「使命只有一个」，主语从人切到公司时要写「公司的使命只有一个」。`The brain does that... replaying back episodes` 后一句主语仍是大脑，必须显性重复，不能省略。

**问题 2：脱离上下文的字面直译**

症状：词对词翻译，丢掉了原文在整段里的指代关系和 call back。

| 原文 | 错译 | 对译 | 判断依据 |
|---|---|---|---|
| solve intelligence | 解决智能 | 解决人工智能 | 上文「认知神经科学博士」和下文 AGI call back |
| get to there | 达到那里 | 达到 AGI / 直接省略 | there 指 AGI 研发终点 |
| today | 今天 | 如今 / 现阶段 | 讲的是时代，不是日期 |
| I can't see a world in which | 我无法想象 | 我不认为 | 英文惯用表态语，非字面含义 |
| components | 组件 | 关键要素 | 语境是 AGI 架构的组成部分 |

处理原则：翻译前先通读整段，把代词、泛指词、call back 全标记出来，再逐句翻译。

**问题 3：该翻的英文硬留着**

症状：`experience replay`、`brute force`、`context window`、`working memory`、`thinking modes`、`chain of thought` 全留英文。

根源：混淆了技术术语保留与普通英文偷懒的边界。

判断标准：API、SDK、Gemini、DeepMind、PR、Agent 这类产品名、无通用译法或中文更陌生的词可以留英文。`brute force` 要译成「粗暴」或「原始粗暴」，`experience replay` 要译成「经验回放」，`working memory` 要译成「工作记忆」，`thinking modes` 要译成「思考模式」，`chain of thought` 要译成「推理链条」。这些都有成熟中文译法，留英文等于偷懒。

**问题 4：机械直译导致中文搭配不通**

症状：出现「攻克想法」「最大的强项」「必须服务」「自己的第一款游戏」这类中文里不存在的搭配。

| 错误搭配 | 问题 | 修复 |
|---|---|---|
| 攻克想法 | 「攻克」只搭「难题/难关」 | 攻克难题 |
| 最大的强项 | 语义重复，强项本身就是突出的 | 强项之一 |
| 必须服务 | must/have to 中文没这么强迫 | 需要服务 |
| 自己的第一款游戏 | 冗余代词 | 第一款游戏 |
| 他小时候是国际象棋神童 | 缺少量词 | 他小时候是个国际象棋神童 |

检验法：译完后读一遍，凡是平时写公众号不会用的搭配，就是机翻味。

**问题 5：标点切分不够，信息密度过高**

症状：一个长定语把补充说明压进主句，读者喘不过气。

case：AlphaFold 破解了蛋白质结构预测这个生物学中持续 50 年的重大难题。应改成：AlphaFold 破解了蛋白质结构预测，这个困扰了生物学界 50 年的重大难题。

处理：原文 appositive（同位语）结构，中文用逗号加指代词拆开，信息节奏立刻对。

**问题 6：连接词缺失导致逻辑断裂**

症状：英文靠语序连的两句话，中文需要显性连接词才能把逻辑接上。

case：DQN 能掌握 Atari 游戏，其中一种方式就是 experience replay。应改成：DQN 之所以能掌握游戏，正是通过经验回放，和大脑的 REM 阶段一样。

中文意合不等于不要连词，关键转折、因果、类比一定要显性化。

**问题 7：动作名词化冗余和直译句式骨架**

这是一类在转录出稿时最容易漏掉的翻译腔。英文习惯先用一个动作名词把事件「打包」再交代内容，中文更直接，一个动词就把事情讲掉。同时英文讲者口语里常见的句式骨架（She gave a brief history / He said one thing that surprised me / What she really wanted to answer / The interesting thing here is…）如果按字面语序照搬，中文读起来会立刻出戏。

| 翻译腔写法 | 症状 | 对译 |
|---|---|---|
| 对方说了一句让他意外的话 | 「说了一句 + 让他意外的话」，把动作包成名词短语 | 对方的回答让他很意外 |
| 她给了一个三年简史 | 直译 "gave a three-year history"，「给 + 一个 + 简史」不是中文说法 | 她先回顾了过去三年 |
| 她真正想回答的问题只有一个 | "What she really wanted to answer was one thing" 的顺序照搬，主语不自然 | 这场演讲真正要回答的只有一个问题 |
| 她没有直接给答案，而是先做了一个 demo | 「没有 A，而是 B」是英文 instead 的直译，中文更常说「她没有立刻给答案，先做了一个 demo」 | 她没有立刻给答案，先搭了一个最常见的 demo |
| 所以行业走到了 Agentic RAG | 直译 "so we moved to"，「走到了」是英文方向动词的硬搬 | 于是行业顺势走到了 Agentic RAG |
| 开场先警告一句 | 动词名词化「先 + 警告一句」，中文更常用具体动作词 | 她开场先打了一剂预防针 |
| Q&A 环节有一个问题特别值得记录 | 「值得记录」是英文 worth noting 的直译 | Q&A 里有一个问题特别值得留意 |
| 所有 demo 讲完，她给出实操框架 | 主谓靠近英文语序，读起来像会议纪要 | demo 全部讲完后，她抛出了一套实操框架 |
| 真正有意思的是下一个 case | "The interesting thing is…" 的照搬 | 真正让人意外的是下一个 case |
| 另一个值得抄的是 X 补充的 Y | 从句结构硬翻，「值得抄的 + 是 + 补充 + 的」多层嵌套 | Elastic 同事 X 还补了一段关于 Y 的思路，也值得记下来 |
| 她由此引出一个新兴生态 | "From this she introduced…" 式直译 | 顺着这个观察，她引出了一个新兴生态 |

处理原则：

- **动词优先于动作名词**。看到「说了一句 X 的话 / 做了一个 Y 的动作 / 给了一个 Z 的简史」这种结构，直接找一个动词替掉整个短语。
- **不要照搬英文主谓顺序**。"What X really is" / "The thing here is" / "What she wanted to answer" 这种强调句型，中文不靠句式强调，靠语序（把强调的部分放前面）或副词（「真的」「其实」「只」）。
- **同一个动词别连续用四次**。转录出稿时，「演示」「提到」「说」「讲」这类动词很容易反复出现。写完一段扫一眼，如果同一动词出现 3 次以上，把其中 1-2 个换成具体动作（演示 → 当场把 demo 打破 / 把数据迁到本地 / 现场跑了一遍）。
- **把「她说 / 她讲 / 她提到」换成她真正做的事**。讲者不只是「说」，她在演示、在反问、在打比方、在切换模型、在写一段 ESQL。让动词携带具体的动作信息。

### 译后一段一检

| # | 检查点 | 不合格特征 | 修复动作 |
|---|---|---|---|
| 1 | 主语完整？ | 「这些还没解决」「使命只有一个」 | 补主语，主语切换处必须显性 |
| 2 | 代词 / 泛指还原了？ | 出现「这里」「那里」「intelligence」「thing」 | 回到上文找具体所指 |
| 3 | 英文该不该留？ | `brute force`、`experience replay` 还在 | 有通用中译必须翻，只留产品名 |
| 4 | 搭配成立吗？ | 攻克想法 / 最大的强项 / 必须服务 | 换词或删重复 |
| 5 | 逻辑连词够吗？ | 两句并列读起来像各说各的 | 补「正因如此 / 也就是说 / 同样」 |
| 6 | 长句要不要拆？ | 一个定语塞 20 字 | 用逗号 + 指代词拆成两句 |
| 7 | 句式骨架 / 动词 | 「说了一句 X 的话」「给了一个 Y 简史」「所有 demo 讲完」 / 同一动词连续 3+ 次 | 动词替掉动作名词；同义词替掉重复动词；按问题 7 的对译表修 |
| 8 | 回译自检 | 译文倒回英文和原文差很远 | 重写 |

### 回译测试

把翻好的中文再翻回英文，表达的还是原文吗？所有机翻味的句子，一旦回译都会暴露：要么主语没了，要么逻辑断了，要么术语换了。回译测试是质检的最终门槛。

### 10 个 case 对照

**Case 1 · AGI 时间线**

错译：持续学习、长期推理、记忆的某些方面，这些问题仍然没有解决。

问题：缺主语，谁认为没解决不明。

对译：我认为持续学习、长期推理、以及记忆的某些方面，这些问题目前仍然没有解决。

**Case 2 · Demis 生平**

错译：2010 年，他共同创立 DeepMind，使命只有一个，解决智能。

问题：主语从「他」切到「公司」没显性；solve intelligence 脱离语境直译。

对译：2010 年他共同创立 DeepMind，公司的使命只有一个，解决人工智能。

**Case 3 · AlphaFold**

错译：AlphaFold 破解了蛋白质结构预测这个生物学中持续 50 年的重大难题。

问题：同位语未拆，长定语压句。

对译：AlphaFold 破解了蛋白质结构预测，这个困扰了生物学界 50 年的重大难题。

**Case 4 · AGI 架构**

错译：你刚才提到的那些组件，我很确定会成为 AGI 最终架构的一部分。

问题：components 直译为「组件」；主语倒置；I can't see 错译为「无法想象」。

对译：我非常确信你刚才提到的那些关键要素，未来会成为 AGI 最终架构的一部分。我不认为几年后我们会发现研发 AGI 这条路是死路，但我们可能会发现现有可行的架构上还缺一两个关键要素。

**Case 5 · 攻克想法**

错译：也许现有技术只需要继续规模化就能达到那里。但也可能还剩下一两个需要攻克的大想法。

问题：there 直译为「那里」；「攻克想法」搭配错。

对译：也许现有技术只需要继续规模化，再加上一些创新就够了。但到那时也可能还剩下一两个需要攻克的难题。

**Case 6 · REM 与 DQN**

错译：大脑在睡眠中把这件事做得非常好……DQN 能掌握 Atari 游戏，其中一种方式就是 experience replay。

问题：主语不连续；experience replay 未翻；两句类比关系未显性化。

对译：大脑在睡眠过程中将这件事做得非常好，尤其是在 REM 睡眠阶段，大脑会回放当天重要的经历，才能更好地从中学习记忆。事实上，我们最早的 Atari 程序 DQN 之所以能掌握游戏，正是通过经验回放，和大脑的 REM 阶段是一样的。

**Case 7 · 胶带拼方案**

错译：查找信息、找到当前决策真正相关的东西，仍然有成本。

问题：整段逻辑未理清，翻译成碎句。

对译：我们现在的做法多少有点像用胶带拼方案，把所有东西都塞进上下文窗口。这并不令人满意。虽然我们面对的是机器而不是生物大脑，理论上可以拥有几百万甚至几千万 token 的上下文窗口或记忆，记忆也可以保存得很完美，但每次都要去这些海量上下文里查找、定位到和当前决策真正相关的那一条，本身就是有成本的。

**Case 8 · brute force**

错译：现在的方法很 brute force。

问题：brute force 未翻；人机对比未显性化。

对译：如果把上下文窗口类比为人类的工作记忆，人类工作记忆大概只能装十几位数字，而我们的模型已经有一百万甚至一千万 token 的上下文窗口。但问题是，现在我们的解决方案是试图把所有东西都存进机器，包括那些不重要的东西和错误的经验教训，这种训练方式非常原始且粗暴。

**Case 9 · today ≠ 今天**

错译：今天很多事情……都可以看作 AlphaGo 当年开创的一些思想又回来了。

问题：today 直译；thinking modes / chain of thought 未翻。

对译：你可以把现阶段我们所做的很多事情，包括领先大模型中的思考模式和推理链条，都看作 AlphaGo 当年开创的一些想法又回来了。

**Case 10 · 最大的强项**

错译：我们最大的强项之一……我们必须服务可能是世界上最大的 AI 使用场景。

问题：「最大」+「强项之一」语义重复；have to 错译为「必须」。

对译：我们的强项之一，就是把这种能力快速蒸馏并压缩进越来越小的模型里。我们需要服务可能是世界上最大的 AI 使用界面，包括 Search 里的 AI Overviews 和 AI Mode、Gemini app，以及 Google 每一款产品中某种 Gemini 或 Gemini 相关技术。

## 翻译理论背景（为什么这些原则成立）

- **严复「信达雅」**：信（忠于原意）> 达（表达流畅）> 雅（文字优美）。顺序重要——最常见的错误是过分追求「雅」而牺牲「达」
- **奈达「动态对等」**：译文在目标语读者身上产生的效果，应等同于原文在源语读者身上产生的效果。意味着翻译的判断标准是「读者感受」，不是「是否和原文结构一样」
- **形合 vs 意合（吕叔湘）**：中文靠语序和语境传递逻辑（意合），英文靠连接词（形合）。这是几乎所有翻译腔的根本来源
