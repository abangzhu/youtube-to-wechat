# 微信公众号写作规范

这份文件只负责最终 `article.md` 的成文规则：文章结构、开头、来源交代、叙事视角、公众号语气、标点、引号、加粗、标题和自查。

英译中、本地化、术语、翻译腔排查、英文示例中英对照，统一看 [translation-quality.md](translation-quality.md)。

---

## 一、正文结构

以下任何一项出现在正文中都要立刻修改：

- Markdown 标题（`#`、`##`、`###`）。`article.md` 第一行的 `# 标题` 是唯一例外
- 无序列表（`-`、`•`）和有序列表（`1.`、`2.`）
- 斜体（`*`）
- 表格

公众号正文默认用连续段落推进。只有文章超过 2000 字、主题切换明显、读者扫读会迷失时，才用 `**短标题**` 做章节标题，不用 Markdown `##`。

**章节标题要是"完整短句带动词"，不是冷名词**。真实发在公众号表现好的小标题大多长这样：

- 「Validation Contract 才是这场演讲真正要讲的东西」
- 「串行比并行好，这个结论违反所有人的第一反应」
- 「前 10% 免费了」
- 「可塑软件会回来」
- 「品味是一台虚拟机」
- 「GEPA 的机制朴素到有点可爱」
- 「Shopify 那一刀，什么时候做优化才值回来」

长度控制在 5–20 字之间。理由是公众号读者在扫读时，冷名词标题（比如「架构」「验证合同」）提供不了信息增量，读者还是得读正文才知道这段在讲什么；带动词的完整短句本身就是一个判断，读者扫过小标题就能决定要不要读下面这段。

**章节标题数量的参考值**：8 段以内的文章基本不需要，8–15 段的文章加 3–5 个，超过 15 段的长文可以多到 7 个左右。不是每个段落群都要加；只在主题明显切换的地方加。

## 二、开头策略

开头的任务不是复述来源，而是让读者迅速知道「谁在说」「为什么值得听」「他在回答什么问题」。

**开头三段的推荐节奏**（观察高表现样本得出的共性模式）：

1. **第一段先扔一个具体场景、数字或判断**，不要先做人物介绍。把读者直接丢进现场。人物是谁放到第二段去交代。
2. **第二段建立可信度**。交代他是谁、公司背景、关键量化锚点（融资、估值、下载量、GitHub star 数、用户量）。这一段是说服读者"继续读下去"的那一段。
3. **第三段锁定文章要回答的问题或核心张力**。用一两句话把读者的阅读目标钉死。

**样本开头示范：**

```
Luke Alvoeiro 在 AI Engineer 的一场 18 分钟演讲里抛了一个判断，今天做软件工程的
瓶颈不再是智能，而是人的注意力。他给这句话附上了具体的场景，一个工程师手头可能
积压 50 个 feature，但每天真正能往前推的只有两三件。
```

```
Pydantic 的创始人走上 AI Engineer 大会 2026 的 workshop 舞台，开场第一件事就是
亲手拆了自家招牌。他说自己根本不相信 AI observability 是一个能长期存在的独立品类。
```

```
Anthropic 是 Claude 背后的公司，估值超过 3800 亿美元。这样规模的公司，整个增长
营销部门曾经只有一个人，独自运营了整整 10 个月。
```

共同点：第一段里要么是一个反常识判断，要么是一个具象到能在脑子里成画面的场景 / 数字。主体立刻出现，但它是为判断服务的，不是为自己服务的。

**偶遇式开头（可选分支）**：有些文章不是要给读者传递一条外部信息，而是作者自己正在想一件事，顺手把思考路径分享出来。这类文章的起笔可以用「正巧」「无独有偶」「源自巧合」这种词做锚，把两件看似无关的事串起来。为什么可用：这种开头的姿态是「我在想 → 邀请你一起想」而不是「我来告诉你 →」，姿态会决定读者的预期，也会决定你的结尾能不能不做总结。

偶遇式开头的示范：

```
陀思妥耶夫斯基早就预言过个人主义和现代性的危机。顺着个人主义推演，就可以推演出
《罪与罚》中拉斯科尔尼科夫对杀人的动机同样的阐述。
```

```
说起为什么要读这本书，也是源自巧合。其一，正巧最近在读汉娜·阿伦特的《人的境况》。
```

只在文章本身更贴近个人思考、而非外部素材转述时才用这条分支。纯访谈 / 演讲 / 新闻类材料，仍然走三段节奏。

**开头禁区：**

- 不能以提问开头，例如「你有没有想过」「为什么……？」
- 不能以「在 AI 时代」「在这个快速变化的时代」开头
- 不能以「本文」「本期」「今天我们来聊聊」开头
- 不要写演讲现场叙事，例如「他走上台说了一件让人安心的事」
- 不要写播放量、点赞数，除非数字极端异常且与文章判断有关
- 不要先写人物简历然后再讲内容。这是样本里最常见、最容易写成的开头错误。

## 三、来源交代与作者视角

文章要自然交代来源，但不能让来源成为正文主角。

**需要交代的内容：**

- 节目/频道/原文是什么，一句话说明定位
- 主播、嘉宾、作者是谁，一句话说明为什么他说这件事值得听
- 技术产品、协议或框架是什么，一句话给非熟悉读者一个中文锚点

这些信息要在前三四段内自然带出，不要单独罗列人物简历，也不要等读者读了一大段后才补背景。

**来源转述腔禁区：**

- 不要频繁用「某某说」「某某提到」「某某认为」「在演讲中」「这期视频里」推进段落
- 不要逐段复述视频或原文的展开顺序
- 不要把文章写成节目推荐、视频摘要或读后感
- 不要加「X 对这件事的评价挺复杂」「X 对 Y 非常赞赏」这种只承担转场功能的空句，样本里一句都没有

**更好的写法：**

先写读者能感知的问题、场景和判断，再在需要建立可信度的地方轻轻归因。人物身份、关键原话、独特经历、具体数据和争议性判断需要归因；已经被作者消化成场景、方法论或分析判断的内容，不必每句都挂回演讲者。

## 四、公众号语气

文章应该像中文作者读完材料后写给中文读者的文章，而不是英文 transcript 的中文转述，也不是内部研究报告。

**具体性优先：**

每段尽量有具体的人、事、数字、场景或原话。四个都没有，这段通常就是空洞判断。宁可少写几个观点、把一个观点写透，也不要每个观点都提一下、没有一个说清楚。

**量词和分类名词要携带语义：**

常见偷懒写法是「做了两件事」「有三件事要说」「还有几样动作」「几个方面」—— 读者读到这里不知道到底是"两个操作"还是"两条建议"还是"两次失败"，要读完后半句才能反推。准确的写法是让量词本身说出内容性质，读者一眼知道自己即将读到什么。按上下文选：

- **方法论场景**：三条建议 / 三个手段 / 三种方法 / 三个路径 / 三条经验 / 四个动作
- **案例场景**：三次失败 / 两个反例 / 三个教训 / 四个案例
- **分类场景**：三类做法 / 三种模式 / 四个典型 / 五种情况
- **观点场景**：三条反常识判断 / 两个反问 / 三条警告 / 四个判断
- **调整和改动**：两处调整 / 两个改动 / 三轮迭代 / 两次重写

坏：「下面这份中文版做了两件事」 → 好：「下面这份中文版做了两处调整」
坏：「还有三件事可以直接拿去用」 → 好：「还有三条可以直接抄的做法」或「三个具体动作」

同一个问题的反面：「一些」「一点」「几种」这种不带具体数字的笼统限定也要尽量换掉。知道数字就用数字，「几点建议」不如「三条建议」；不知道就重写成不需要量词的表达。

**数字量化原则：** 只要能量化就量化，不用「大幅提升」「显著下降」「提升了很多」这种模糊形容词。样本里所有数字都不是装饰，都在支撑具体判断：

- 下载量 / star 数 / 估值 / 融资轮次
- 具体成本差：500 万美元 → 7.3 万美元
- 时间压缩：2 小时 → 15 分钟
- 百分比：60% 时间、90% 覆盖率、96.7% 准确率
- 具体约束：30 字符标题、90 字符描述、700 行编排文本

**命名角色而非匿名化：**

样本里的 Boris Cherny、Luke Alvoeiro、Max Schoening、David Soria Parra、GRITCULT 全部被点名。不要写「一位工程师」「某位 CEO」。能具名就具名。

**中文表达优先：**

先给中文读者能想象的场景、动作或问题，再自然带出术语和判断。不要沿用英文演讲的抽象判断顺序，不要连续堆英文术语。

**衔接词推荐：**

取代「首先/其次/最后」，改用更口语化的衔接：

> 「先说……」「再说……」「说到……」「顺便说说……」「回到……」「说到未来……」「但有一件……」

**指代容忍度：**

代词指代不需要每次都显式补全名字，只要上下文连贯读者能跟住就够了。频繁重复名字会显得机械。真正需要补名字的情况是话题转换了，或者两个人交替出现时。

**句式节奏：**

- 单句长度平均 20–40 字，尽量不超过 60 字。长句超过 60 字先看能不能断成两句。
- 并列结构用句号分隔，不要塞进同一句。「这不是因为资源紧张，也不是因为他是某种天才。这是匿名 X 账号 GRITCULT 在 4 月 15 日写的一篇长文里的核心案例。」
- 动词要强。样本里的表达方式：「抹平编码壁垒」「踩改动」「吃掉速度收益」「哗哗地烧」「亲手拆了自家招牌」「一刀砍到」。弱动词（进行、实现、达成、展开）能换就换。

**干涩幽默：**

样本里不时出现轻度吐槽 / 自嘲 / 小结论，让文章有人味但不油腻。不做情感煽动，不用感叹号，不反问煽情。例：

- 「大多数人并不想去打猎，只想去 Costco 买一块已经放在泡沫托盘里的牛排。」
- 「一个拿对工具的人，胜过十个用老方法的人。」
- 「这种说法也许有些夸张，但 Anthropic 那个只有一个人的增长部门，多少给这句话做了一点背书。」

**筛掉对读者无信息量的转录细节：**

转录逐字稿里有大量演讲现场才有意义、进入公众号就成为噪音的细节。出稿前逐段扫一遍，下面几类主动删掉：

- **讲者本人的口头小插曲**：「她自己都念不准」「他说自己 English 不好」「她现场笑了出声」「他拍了一下桌子」这类对演讲者形象有贡献、对读者判断为零的花絮。读者不认识讲者本人，不关心讲者有没有念对一个词。
- **现场互动**：「有多少人做过 X 的请举个手」「大概一半举手」「有人在笑」「观众发出一阵笑声」「掌声」。转录里是有生命的，文章里是废字。
- **主持人 / 讲者对流程的元评论**：「我们先快速过一下历史」「这部分我跳过」「时间关系不展开」「Sorry 今天太紧张」「由于时间限制」。读者不需要知道演讲内部节奏管理，直接把那部分内容按逻辑讲出来就好。
- **讲者对自己 slide 的吐槽**：「这张幻灯片我自己都不喜欢」「这图做得太丑」。如果吐槽本身不承载观点（比如不是"这图我讨厌它，是因为……"这种带论点的），就删掉。
- **没有展开的碎片引用**：「有一次我朋友 Sam 跟我说……」后面没下文，删掉这个人名。
- **反复的自我修正**：「我是说 A，不，其实是 B」。只保留 B。

判断标准：这一句抽掉，文章的信息量和论点会不会受损？不会就删。保留一两处能让文章有人味的现场细节可以（参考干涩幽默那条），但前提是它承载了观点或者画面，不是单纯"讲者那一刻做了什么"。

反例对照：

- 坏：「GEPA 是一种优化算法的缩写，她自己都念不准。」（「她自己都念不准」对读者零价值，直接删）
- 好：「GEPA 是一种优化算法的缩写。」
- 坏：「她问现场有没有人做过 agentic rag，大概一半人举手。」（互动过程删掉，如果"一半的人都做过"这件事本身是判断依据，改成下面这样）
- 好：「agentic RAG 在这届大会上已经是默认选项，现场一半人都动手做过。」

**同信息最精简表达优先：**

「筛掉无信息量的转录细节」讲的是哪些素材根本不该写进来；这条讲的是写进来的素材怎么用最少的字表达。把整段写完后，逐句反问自己：「同样的意思，有没有更短的说法？」有就改。四类高频可砍：

- **先卖关子再揭晓**。「X 做了一件事。他……」「X 是全场最不起眼但最关键的……」这种先铺垫再揭晓的结构，是英文叙事的悬念节奏，中文直接把动作写出来更自然。
  - 坏：「Ravi 把数据切成独立文件之后做了一件事。他回到 Claude……」
  - 好：「Ravi 把数据切成独立文件之后，回到 Claude……」
  - 坏：「「Full-Stack Prompt 里那句『把数据保存成 data.json』」是全场最不起眼但最关键的一个动作。」
  - 好：「Full-Stack Prompt 里把数据保存成 data.json 是最关键的一个动作。」

- **指代嵌套和重复主语**。引号里再套引号、主语没换还写一次、用抽象名词替代具体动作，都会让读者多做一次解析。
  - 坏：「那句『把数据保存成 data.json』是最关键的一个动作」
  - 好：「把数据保存成 data.json 是最关键的一个动作」
  - 坏：「这条路的实用价值是……」（读者要回头想"哪条路"）
  - 好：「这样做还有一个实用价值是……」

- **对读者无关的执行细节**。跑了几分钟、调了哪个参数、用了哪个版本、中间卡在哪一步 —— 除非这些数字本身就是论点，否则删。讲者的时间成本不是读者的时间成本。
  - 坏：「回到 Claude，用同样格式生成 JSON（带 MCP Server 拉封面，跑了 15 分钟），然后……」
  - 好：「回到 Claude 生成 JSON，然后……」

- **强调副词成瘾**。「直接」「一下子」「真的」「其实」「就是」—— 一段里出现两次以上，通常后面那个可以删。
  - 坏：「Prototype 页面直接变成 Psychedelic Rock」
  - 好：「Prototype 页面就变成了 Psychedelic Rock」

还有一个和简洁相关的反面动作：**段与段、分句与分句之间的连接词要显式**。当你删掉了一堆冗余，相邻段落的关系可能会从"理所当然接得上"变成"读者需要猜关系"。此时加一个显式连接词，读起来反而更顺。这不是增加字数，是减少读者的二次解析成本。

- 并列新增一类：「还有一个」「另一条」「同样值得」
- 递进解释：「背后原因是」「之所以这样，是因为」
- 对比反转：「反过来说」「但」「不过」
- 条件限定：「只有在……时才」「前提是」

**正文主角是观点，不是素材和讲者：**

「筛掉无信息量的转录细节」处理的是哪些素材不该进来；「同信息最精简表达优先」处理的是进来的素材如何用最少字说清。这条处理的是第三类情况：素材干净、表达也紧，但段落里仍然有一层「围绕观点的外壳」—— 讲者是谁说的、这段多重要、去哪能看更多 —— 读者需要的是肉，不是壳。三类常见外壳和各自的判断标准：

- **对谈元信息**。默认删。「X 问得直接」「主持人提到」「嘉宾回应说」这类对谈形式句，通常只作为叙事外壳，不推进观点。章节标题已经是「三个 hot take」的情况下，直接「第一个……第二个……第三个……」分点就好，不需要先用一句「X 问 Y 答」铺垫。
  - 坏：「X 问得直接，Y 的回答可以直接拿去打几个在圈内流行的错误判断。第一个……」
  - 好：「第一个……」（直接进入内容）
  - 例外：对谈信息本身承载叙事张力或推进观点时保留。判断方式——抽掉「X 问、Y 答」这一句，观点推进或叙事张力是否损失？
    - 保留的例子：「主持人追问这个数字是怎么来的，讲者被逼出一个更具体的答案：……」（「被逼出」改变了读者对后面答案的感知，说明讲者原本没打算讲这么细）

- **判断前后的元评价**。基本全删，没有例外。作者对内容本身做评价，不论放在判断前（铺垫）还是放在判断后（收尾），都是读者不需要的外壳。判断的分量应该由判断本身显出，不靠作者做导游。
  - 铺垫型（判断前）：「这是最容易被忽略但最值得抄走的一句」「讲者原话是」「接下来这段特别关键」。
    - 坏：「**判断自己是不是 AI native 的标志** / 这场对谈最容易被忽略但最值得抄走的一句话，来自结尾。/ 讲者的原话是，过去那些真正以产品为中心、做得最好的公司……」（章节标题下两层元评价 + 出处铺垫，三道门才到真正的话）
    - 好：「**判断自己是不是 AI native 的标志** / 过去那些真正以产品为中心、做得最好的公司……」（章节标题下直接进判断）
  - 收尾型（判断后）：「作者的批评很克制，方向却很清楚」「这个观点挺犀利」「讲得真好」「这是一个很有力的论断」「这段尤其值得记住」。
    - 坏：「用『幻觉』这样一个带艺术气的词，把『模型会自信地撒谎』这个严重的准确性问题做了一次语义打包。作者的批评很克制，方向却很清楚。」（读者读完前一句已经能自己感受到克制和方向，再加一句等于替读者做判断）
    - 好：「用『幻觉』这样一个带艺术气的词，把『模型会自信地撒谎』这个严重的准确性问题做了一次语义打包。」（到此为止，读者自己感受）
  - 判断标准：这句话抽掉，读者对前文的感受会不会变？不会就是外壳，删。

- **给原素材做宣传/导流**。默认删。讲者末尾推自己的课、订阅、书、newsletter、social handles，这些信息是服务原视频观众的，对公众号读者学习观点没有贡献。文章的目的是把值得思考的观点传递给读者，不是替原视频拉流。
  - 坏：「讲者在最后顺便宣传了他和 X 合作的免费课，第一期刚开，春天还有一期，配套视频系列也在做。但这一整场访谈本身已经是一份相当完整的入门教材。」
  - 好：（整段删除，文章在前一段自然结束）
  - 例外：讲者提到一个读者当场就能用到的具体、免费资源（公开 GitHub repo、可下载的 prompt 模板、开源 eval 集）。保留但压到一句话，不要完整推广文案。
    - 保留的例子：「讲者把这次演示用的 MCP Server 代码放在 GitHub 上，链接在视频简介里。」（读者打开就能用）
    - 判断分界线：读到这句话，读者能不能当场用到一个具体东西？不能就是外壳，能就是正文。

三类外壳合起来的精神：文章主角是观点本身。每次下笔交代「谁说、说得多好、去哪看更多」之前，先过一遍「读者抽掉这句话会不会损失」。不会损失就是外壳，砍掉。

## 五、概念命名与定义

如果文章里引入了一个值得读者带走的新概念 / 术语 / 方法论，用「命名 + 破题动作」的方式一次讲清楚，之后再出现就裸写、不再加定义。

**样本里的范例：**

- validation contract：在代码写下去之前就把正确性定义清楚。
- droid whispering：给每个角色挑合适模型。
- progressive discovery：给模型一个工具搜索工具，按需加载。
- programmatic tool calling：让模型写脚本组合工具调用，不再一步一步推理。
- tiny core superpower：一个产品核心的"超能力"，比如 iPhone 的 multi-touch、GitHub 的 Pull Request。
- 分发工程师：把分发当作工程问题来处理的人。
- agency：相信世界可以被改变并愿意亲手去改的那个劲头。

**模式**：一行名字 + 一行动作 / 效果，不做抽象哲学化定义。首次出现的时候用「」锚一次，后面再提到就裸写，不要反复戴引号。

## 六、AI 腔禁区

以下词语和句式一律删除，替换为更具体的说法，或直接删掉这句话。

**形容词类：**

- 「深入」「深度」「深刻」「全面」「系统」「卓越」「显著」「丰富」

**连接词类：**

- 「不仅如此」「此外」「综上所述」「总的来说」「值得注意的是」

**视角切换类：**

- 「让我们」「我们来看看」「接下来让我们」

**结构词类：**

- 任何以「在……的背景下」开头的句子
- 任何以「总结」或「综上」开头的段落
- 「不是……而是……」这类公式化对比句。除非原文必须保留强对立，否则改成直接陈述、两句递进或具体场景说明
- 「不去 X，而是 Y」「没有 A，而是 B」「而不是 X」「并非 A，而是 B」这些同一家族的变体同样要避开

**戏剧化修辞类：**

- 「开口就要」「拆到最细，发现」「硬是」「干脆」「没有直接 X，而是 Y」这些把中性商务事实写成茶馆段子或剧本旁白的修辞
- **不必要的比喻修辞**：「简单得像玩笑」「快得像闪电」「重得像一块砖」这类拿比喻代替直接陈述的句式。改法：去掉比喻，换中性描述——「简单得像玩笑」改「非常简单」
- 写作时每出现一次戏剧化动词就问一句：原文讲者有没有表达过这种情绪？没有就砍掉，改成白描动词

**作者视角主观评价词类：**

- 「最精彩」「最反直觉」「被低估」「很聪明」「特别真诚」「观察很硬」「立场锋利」「措辞冷幽默」「值得抄」
- **对讲者语气的评价**：「他说得很冷」「他说得很尖」「他一针见血地」「他云淡风轻地」「他半开玩笑地」——这些都是作者替读者下结论，属于同一类评价词。改法：去掉评价词，中性陈述。「还有一层他说得很冷」→「他还描述了一个当下的现实」
- **非必要动作修饰**：「直接讲了一句」「接着讲了一句」「简单地说」「缓缓地说」「接着一口气讲完」——这些修饰并不增加信息量。改法：「Marc 接着直接讲了一句」→「Marc 接着讲」
- 展示而非告诉的原则：让读者自己觉得精彩，不要你告诉他这很精彩。写完文章全文 grep 这些词，命中即改或删
- 改法：删掉评价词直接给事实；或者把评价换成具体动作（「这个设计让 writer 和 reviewer 各跑一套独立上下文」比「这是最精彩的设计」更有说服力）

**对客户 / 同行的主观揶揄类：**

- 「AI 味越足越好」「玩 PR 套路」「收智商税」
- 触发情境：讲者在原视频里对客户 / 对手有轻度调侃，作者把这种调侃的语气放大成正文里的态度
- 改法：客观陈述对方的诉求或做法，不评判动机。原文是 "sounded really well for the grant because it was AI focused" 就写「他们需要一个 AI Native 的产品来匹配这笔资助」，不改成「AI 味越足越好」

**翻译腔词类：**

- 「理由是」（reason is）→ 改「原因是」
- 「事实上」（in effect / in fact）→ 改「实际上」或直接删
- 「你不希望」（you don't want）→ 改成陈述句「这种做法会 X」
- 「这也是为什么」（that's why）→ 改「这也解释了为什么」或直接拆成两句

## 六之二、给事实加戏与施事受事完整性

这两类问题不是 AI 腔词汇触发的，而是句子骨架层面的错误。一旦发生，读者能直观感到文章「在演」或者「漏了什么」，但往往说不出是哪里出了问题。编辑阶段必须显式检查。

**给事实加戏 / 无中生有一个态度**

定义：作者在文章里暗示了原视频 / 原文**没有**表达过的态度、反转或姿态。

典型错例：

- 原视频里 Louis 只是直接和客户沟通，没有表达「拒绝 / 犹豫 / 本来要接」的态度。但写作时用「Louis **没有直接接活**，而是……」——「没有直接接活」暗示了「本来要接但没接」的戏剧反转，读者会误读讲者的姿态
- 原视频里 Paul 只是选了方案 B，没有表达「曾经尝试 A 但失败」。但写作时用「Paul 放弃了这条路」——「放弃」暗示了曾经走过

判断方法：每次写完带有态度动词的句子（「没有 / 放弃 / 拒绝 / 选择 / 转向 / 反思」），回到原视频或转录，检查讲者是否真的表达过这层情绪。没有就改成白描。

这条规则和 SKILL.md 6.3 节的「延伸解读 vs 无中生有」不同。6.3 针对**事实**是否原文提过（补充背景、类比是允许的，纯事实虚构要删）；这里针对**态度 / 姿态**是否原文暗示过，即使事实层面没错，多加一层戏剧情绪也算无中生有。

**施事 / 受事关系完整性**

定义：动词描述了动作但漏掉关键主体。常见于「找上门 / 联系 / 接洽 / 提到 / 批评」这类关系动词。

典型错例：

- 「加拿大一家 CRM 平台找上门」——找上门找的是谁？必须补全：「加拿大一家 CRM 平台**找到 Towards AI**」
- 「客户提到了这个功能」——对谁提到？必须补全受事

检查方法：文章里每个关系动词前后都必须能回答「谁对谁做了这事」。编辑阶段用 grep 扫 `找 / 联系 / 接触 / 问 / 提到 / 批评 / 指出`，逐一确认施事和受事都在句子里出现。

**英文名词化 → 中文动词化**

定义：英文里常见的 "its own X call / the Y-ification of X / the processing of Z" 这类名词化结构，直译成中文读起来生硬。中文更自然的表达是换成动宾结构。

典型错例：

- 英文 "each tool can have its own LLM call" → 直译「每个 Tool 有自己的 LLM 调用」。改法：「每个 Tool 可以自己调用 LLM」
- 英文 "the generation of marketing content" → 直译「营销内容的生成」。改法：「生成营销内容」
- 英文 "the validation of results" → 直译「结果的验证」。改法：「校验结果」

触发特征：`X 的 Y 调用` / `X 的 Y 生成` / `X 的 Y 检测` / `Y 化的 X`。编辑阶段 grep 扫 `的.{1,8}调用 / 的.{1,8}生成 / 的.{1,8}处理 / 的.{1,8}验证`，凡是能改成动宾的都改。

**被动 / 容器隐喻 → 主动拥有关系**

定义：把抽象实体放在位置容器里（「X 留在 Y 里」「X 保存在 Y 中」），读者要多花一步才能 parse 出真正的拥有关系。中文更自然的是让真正的持有者做主语。

典型错例：

- 「全局上下文留在那个唯一的 Agent 里，它同时是 Planner 和决策者」——主语是"上下文"，行动者是"留在"，Agent 变成容器。改法：「唯一的 Agent 拥有全局上下文，这个 Agent 同时是 Planner 和决策者」——Agent 做主语，关系直接

判断边界：如果 Y 真的是一个**位置/存储**（工作目录、内存、磁盘），「X 保留在 Y 里」是合理的。如果 Y 是一个**实体角色**（Agent、Reviewer、Writer、系统），让 Y 做主语用「Y 拥有 / 持有 X」更好。

**弱代词 → 具体名词重复**

定义：跨句或句末的 `它 / 这个 / 那个 / 他 / 她` 作为唯一指代，中文读者跟不上。

典型错例：

- 「教别人怎么搭它」——「它」指代前面的"系统"，但读到句末时读者已经回溯一次。改法：「教别人怎么搭建这套系统」
- 「两个 Agent 放同一个项目，它能同时看懂两边的代码」——「它」指代模糊（是"项目"还是"放法"）。改法：把主语换成具体的东西（「这样的目录布局让 AI 辅助工具能同时看懂两边的代码」）

判断规则：段落内第一次指代可以用代词，但跨段、章节收尾句、定义句、总结句必须用具体名词。特别是句末的"搭它 / 做它 / 改它"这种含混收尾，一律换成具体名词。

**预告-展开结构（「只有一个，…」式）**

定义：前半句做悬念引导（「只有一个」「只有一句话」「原因只有一个」「答案很简单」），后半句才给真正内容。形式上制造节奏感，实际上多数是 filler。

典型错例：

- 「他们想回答的问题只有一个，AI 工程师的判断力到底应该放在哪里」——前半句"只有一个"是纯预告，没有信息量。改法：「他们想回答的是 AI 工程师的判断力到底应该放在哪里」

判断边界：如果前半句本身有独立信息量（比如「他们问过十几个客户，答案出奇地一致——X」），可以保留。纯节奏性的预告（「只有一个 / 只有一句」），删掉。

**拟人化 / 诗意动词用于工程语境**

定义：把文艺动词用在工程技术描述里，会把读者从技术上下文拉到散文上下文。

典型错例：

- 「判断力应该**长在**哪里」——"长"是植物隐喻，工程文里不用。改法：「判断力应该**放在**哪里」或「**体现在**哪里」
- 「产品**生根**在用户工作流里」——"生根"是植物动词。改法：「产品**嵌入**用户工作流」或「**贴着**用户工作流做」
- 「让飞轮**转起来**」——"飞轮"是商业金句式隐喻，过度使用

触发词：`长在 / 生根 / 扎根 / 落脚 / 落地（当做动作而非结果时） / 开花结果 / 飞轮转起来 / 土壤`。

判断边界：如果引用原作者的原话并且原话就是这么说的，可以保留（属于原文金句）。作者自己写时避免。

**引入新概念要给一个迷你说明**

定义：第一次出现某个专名、自造术语或比喻时，要让读者在当下就能猜到它是什么，而不是再读三段才能 infer。

典型错例：

- 「Louis 开场先画了一条**自主性滑杆**」——"滑杆"是比喻，读者不知道它是条什么线。改法：「Louis 开场先画了一条**关于系统自主性的滑杆**」
- 「Paul 采用了**评估器-优化器循环**」——直接给术语名，读者猜不到它干什么。改法：「Paul 采用了**评估器-优化器循环（写一轮、审一轮、改一轮的反复打磨流程）**」

检查方法（通读级，不是 grep 级）：通读时问每一个英文术语和比喻「如果读者不认识这个词，文章在这里或临近位置有没有给出说明」。如果要往下读三段才能拼出意思，前面必须加一句。

边界：业界通用且 AI 工程师熟知的术语（RAG / F1 / Pydantic / few-shot）不用解释；作者自造术语或比喻（滑杆 / 拳头产品 / 水坝）必须解释。

**段落间的逻辑衔接词不能省**

定义：上段和下段之间有延续、对比、转折、补充四种关系，字面上必须通过衔接词表达出来。省掉衔接词的时候，读者的脑子要自己补连接，读起来断。

典型错例：

- 上段讲了正例（Agent 合适的场景），下段开头「反例很具体」——缺「也」。改法：「反例**也**很具体」
- 上段讲 Agent 的优点，下段开头「这种克制带来一条反直觉的工程约束」——缺「但」。改法：「**但**这种克制带来一条反直觉的工程约束」

检查方法（通读级）：每段首句读一遍，问自己「这段和上段是什么关系？是**延续 / 对比 / 转折 / 补充**四种中的哪一种？这个关系在字面上表达了吗？」

衔接词参考：延续（还 / 同样 / 进一步）、对比（也 / 相反 / 反过来）、转折（但 / 不过 / 虽然）、补充（此外 / 另一个）。

**冗余「那个叫 X 的 Y」翻译腔**

定义：英文 "that X thing called Y" 或 "the X called Y" 直译，用「那个叫 / 所谓的 / 一种被称为」把术语包起来。中文里术语本身可以直接做定语，不用套这个框架。

典型错例：

- 「那个叫「Context Rot」的天花板」——改法：「「Context Rot」的天花板」
- 「所谓的 MCP 协议」——改法：「MCP 协议」

触发词扫描：`那个叫 / 那种叫 / 所谓的 / 一种被称为 / 一个叫做`

边界：如果这个概念在业界有多个名字，说明哪个是你采用的叫法时可以用「业界一般叫做 X / 原作者把它叫作 Y」——这时候「叫」有信息量。纯粹装饰性的「那个叫」一律删。

**抽象名词堆 → 问句或动作陈述**

定义：`X 的真正门槛 / Y 的核心挑战 / Z 的关键在于 / W 的本质是` 这种把判断或问题包成名词短语的写法，读者要 unpack 两层才能 get 到。中文里更直接的表达是问句或动作陈述。

典型错例：

- 「上多 Agent 的真正门槛也随之清晰」——三重抽象（"上多 Agent" + "真正门槛" + "随之清晰"）。改法：「什么时候应该上 Multi-Agent 也就明确起来」
- 「CTR 预估的核心挑战是数据稀疏」——改法：「CTR 预估最难的是数据稀疏」或「数据稀疏是 CTR 预估最难的地方」

触发词扫描：`的真正门槛 / 的核心挑战 / 的关键在于 / 的本质是 / 随之清晰 / 一目了然 / 跃然纸上`

改法方向：
- 「X 的真正门槛是 Y」→「什么时候才该上 X」或「Y 什么时候会让 X 失效」
- 「随之清晰 / 一目了然」→「也就明确 / 也就清楚」

**路径 / 旅程隐喻用作动作**

定义：`走这条路 / 走不通 / 绕开 / 踩过这条路 / 下山 / 转弯`，把方法选择描述成旅程。在工程文里换成白描动词更准。

典型错例：

- 「Paul 没走这条路」——改法：「Paul 没使用这个方法」
- 「这条技术路线走不通」——改法：「这个方法行不通」
- 「绕开了这个坑」——改法：「避开了这个问题」

触发词扫描：`走这条路 / 走不通 / 绕开 / 踩过 / 下山 / 转弯`

边界：原讲者原话里用了这个隐喻（加引号引用时），保留。作者自己写时不用。

**口语形容词 → 工程描述词**

定义：描述系统行为时用 `很吵 / 很糊 / 很乱 / 很死板 / 很硬 / 很油` 这种口语化、拟人化的形容词。生动但不精准，工程文里换成对应的工程词。

典型错例：

- 「这个 loop 会变得**很吵**」——"吵"是拟人。改法：「这个 loop **噪音很大**」
- 「这个模型输出**很糊**」——改法：「这个模型**精度很低**」或「这个模型**结果模糊**」
- 「架构**很乱**」——改法：「架构**结构混乱**」
- 「代码写得**很死板**」——改法：「代码**僵化**」或「代码**缺乏灵活性**」

触发词扫描：`很吵 / 很糊 / 很乱 / 很死板 / 很硬 / 很油 / 很脆`

边界：描述**人物行为**可以口语化（"他直接说"、"他一口气讲完"），描述**系统 / 流程 / 指标**不行。区别在于读者期待哪种词——对人，期待生动；对系统，期待精准。

**评价式名词结构 → 动作式主谓宾结构**

定义：以被评价对象做主语、把真正的施事"S"变成定语的句式。中文读起来主语是抽象概念，而动词是"非常+形容词"的软表达，力度被稀释。

典型错例：

- 「Marc 描述的场景非常具体」——主语是"场景"，动词是"非常具体"。改法：「Marc 描述了一个具体场景」，主语是 Marc，动作是"描述"
- 「他提出的论点很有意思」——改法：「他提出了一个有意思的论点」
- 「作者写的段落挺精彩」——改法：「作者写出了一段精彩的段落」

判断方法：每一句里问"谁是主语？谁在做动作？"。如果主语是抽象概念（场景/细节/论点/段落）而动词是"非常/很/挺+形容词"，就转回成主谓宾结构。

触发词扫描：`描述的.{2,10}非常 / 说的.{2,10}很 / 写的.{2,10}挺 / 提出的.{2,10}很 / 构造的.{2,10}很`

**概念引入要用完整主谓宾，不要用省略 / 口头表达**

定义：引入一个新名词 / 新角色 / 新职能时用"出口叫 X"、"这就是 X"、"结果就成了 X" 这类省略式的口头表达，读起来轻浮、没重量。

典型错例：

- 「于是出口叫 Builder」——"X 的出口叫 Y"是口头省略。改法：「于是被称为 Builder 的新角色诞生了」
- 「然后就是一个 Agent Harness」——改法：「接下来出现的是一种叫做 Agent Harness 的运行框架」

改法方向：用完整结构"被称为 X 的 Y 诞生了" / "出现了一种叫做 X 的 Y" / "X 定义为 Y" 来引入概念。

**段内句间的逻辑连接词要显式（模式 15 的段内版本）**

定义：这是第六之二节里「段落间逻辑衔接词不能省」的段内版本。模式 15 针对**段落之间的首句**，这条针对**同一段里相邻句子**。

典型错例：

- 「这个合伙人已经搭起了一整套 AI 系统来处理他日常工作的所有事情，这套系统成了他工作里的合作伙伴。」——两句之间是递进关系，缺衔接词
- 改法：「但是这个合伙人已经搭起了一整套 AI 系统来处理他日常工作的所有事情，并且这套系统已经成了他工作里的合作伙伴。」

检查方法：每段内每两个相邻句之间问"这两句是什么关系（延续 / 对比 / 转折 / 递进）？连接词写出来了吗？"

关于时间衔接的扩展：「10 到 20 年后我们回头看」这种表达，模糊可能指向过去也可能指向未来。改法：加"再过"或"到 20XX 年"让时间指向明确。

**同一概念在文章里用词前后要一致**

定义：前段用"场景"指代某个画面，后段换成"画面"或"细节"指代同一个东西。读者要在脑子里反复确认"这几个词是不是同一个东西"，认知负担上升。

典型错例：第 13 行「这个画面里还有一个更反直觉的细节」，第 15 行「Marc 用这个画面引出他的核心论点」——"画面 / 场景 / 细节"三个词混用却指同一个东西。

改法：全文锁定一个词。第 13 行改成「还有一个反直觉的场景」，第 15 行改成「Marc 用这个场景引出他的核心论点」。

检查方法（通读级，不是 grep 级）：文章里作为概念锚点反复出现的词（场景 / 画面 / 观察 / 论点 / 细节 / 画像 / 现象 / 时刻），在终稿阶段统计每个词的出现次数，只应活一个。辅助 grep：`场景.*画面 / 画面.*场景` 在同一篇文章里不应共存。

**中文行业术语精度**

定义：用通用词 / 口语词替代行业内约定俗成的术语。

典型对照表：

- 软件行业："造" → "开发"。"更多产品会被造出来" 应为 "被开发出来"。类似：`代码被造 / 软件被造 / 系统被造` 都要改
- 薪酬体系："报酬" → "薪酬"。"报酬数据 / 报酬曲线" 应为 "薪酬数据 / 薪酬曲线"。"报酬" 多用于一次性劳务
- 工程动作："搭" → "搭建"（正式语境）、"做出" → "开发出来 / 打造出来"、"跑" 描述代码正常，描述项目应为 "推进 / 运行"
- 人事："岗位 / 职位" 要看上下文选准一个
- 数据 / 仓库术语："ledger" → "明细"或"流水"，不是"账本"。"账本"是会计场景，进入开发/数据流/仓库语境（Git repo、写操作日志、事件流）时用"明细"。类似：`写进账本 / 回到账本 / 账本末尾` 都应该换成"明细"。同一篇里同一个 ledger 概念要统一成一个词

改法：写完扫一遍这些通用动词 / 名词，看有没有对应的行业专有说法。

判断边界：口语化是这套 skill 的特色，不是所有"造"都要改成"开发"——但在描述**产品开发 / 软件工程 / 薪酬体系**这种明确行业语境时，用行业内固定说法。

触发词扫描：`被造 / 制造.{0,3}软件 / 报酬.{0,3}数据 / 报酬.{0,3}曲线`

**英文数字习语 / 比喻习语要本地化**

定义：英文里约定俗成的数字习语或比喻习语直译成中文后读者看不懂。

典型错例：

- 「high nines」直译成「高 9」—— 中文读者看不懂。本地化为「90% 大几」/「99% 以上」
- 「first order of business」直译成「第一等大事」—— 应为「眼下最要紧的事」
- 「low-hanging fruit」直译成「低垂的果实」—— 业界有人用但非必需，可改「容易摘到的果实」或「显而易见的机会」

改法：翻译时对数字习语和固定习语多走一步，查一下中文对应的口语说法。

检查方法：回译自检时特别关注数字表达和固定短语。触发词扫描：`高 9 / 低垂的果实 / 第一等大事`

**引用同义冗余择一**

定义：同一段里连续引用多个说同一件事的短语，给读者的信息量是重复的。

典型错例：「底下回复都在说『我以前那家公司是 8 倍膨胀』『你这还太宽容了』」——两句是同义的。

改法：择一留下，一般选更具体 / 更有画面感的那个。这里保留「8 倍膨胀」更具体，删掉「你这还太宽容了」。

**主观评判补充句要删**

定义：客观陈述之后作者加一句主观价值判断。

典型错例：

- 「花 200 美元订 premium。**这点钱不多。**」——"这点钱不多"是作者的价值判断，对不同收入读者意义不同
- 「这个岗位年薪 50 万。**非常高。**」——"非常高"对不同行业的读者不同

改法：删掉这句。如果是原作者/讲者说的那句，在正文里明确标注是讲者原话再保留。

**冗余默认属性限定词**

定义：给本来就默认具备某个属性的名词再加一个限定词，是废字。公开人物的公开发言默认是"公开的"，作者写的文章默认是"他写的"，巨头产品默认是"它旗下的"。

典型错例：

- 「Mitchell Hashimoto 的**公开**判断」——他作为前创始人在博客写的东西默认是公开的。改法：「Mitchell Hashimoto 的判断」
- 「他在 Twitter 上**公开**发的一条推」——Twitter 发推默认公开。改法：「他在 Twitter 上发的一条推」
- 「Google **旗下的** YouTube」——熟知关系不需要限定。改法：「Google 的 YouTube」或「YouTube」
- 「作者**亲自**写的」——作者署名的文章默认是本人写的，加"亲自"反而暗示"可能不是"。改法：「作者写的」

判断方法：把限定词抽掉，信息是否丢失？没丢就是默认属性冗余，删。

触发词扫描：`公开(判断|发言|说过|表示) / 旗下的 / 亲自(写|做|主持) / 本人(写|发|说)`

**时间推进衔接词用副词，不用定语**

定义：把"接过 / 站起 / 拿起话筒"这种动作做成人物前面的定语，再用"的"连到人物，是英文分词短语（participial phrase）直译。中文自然写法是把动作从定语里抽出来，用时间副词做段落衔接。

典型错例：

- 「**接过来的** Hugo 把视角从加速切到工作单元」——"接过来的 Hugo"是英文 "Taking over, Hugo..." 的分词结构直译。改法：「**接下来** Hugo 把视角从加速切到工作单元」
- 「**站起身的** Marc 说」→「Marc 站起身说」或「**接着** Marc 说」
- 「**拿起话筒的** 主持人问」→「**这时** 主持人拿起话筒问」

触发词扫描：`[来去起下住上]的 [A-Z]` / `的 Hugo / 的 Marc / 的 Madison`

**"那个 X 词 / 那个 X 东西"尾缀冗余**

定义：句尾用"那个 X 词""那种 X 做法"作为代词收尾，X 本身不承载新信息，属于装饰性回指。

典型错例：

- 「`continue` 大概是他们团队现在最常敲**的那个词**」——"最常敲"已足够指明对象。改法：「`continue` 大概是他们团队现在**最常敲**」
- 「这是他提出**的那种做法**」——"他提出"已经具体。改法：「这就是他提出的」或干脆删掉
- 「最反直觉**的那一条**」——可改成「最反直觉的一条」或去掉"那"

触发词扫描：`的那个[词句句子做法说法情况东西] / 的那一[条种个]`

**"让 X 快 / 让 X 好"里形容词要动词化**

定义：在"让 X Y"的使役结构里，Y 是形容词时，中文读起来像静态描述；改成"变 Y / 成为 Y"之后才是动作。

典型错例：

- 「要让这一切真的**快**」——"快"是状态形容词。改法：「要让这一切真的**变快**」
- 「这样做能让代码**整洁**」→「这样做能让代码**变整洁**」
- 「这个改动让循环**更稳**」→「这个改动让循环**变得更稳**」

判断边界：如果 Y 本身就是动词（"让它跑起来""让它停下"），不用变。只针对形容词 Y。

触发词扫描：`让[^，。]{1,15}真的[快好稳准稳严紧松] / 让[^，。]{1,15}更[快好稳准严紧松]`

**"不再有 X" 要补谓语**

定义："不再有 X"后面省略了谓语（X 在做什么），读者要自己补。加一个动词让陈述完整。

典型错例：

- 「External Validation **不再有人**」——不再有人什么？改法：「External Validation **不再有人参与**」
- 「系统里不再有人**在管**这件事」→对。
- 「前端页面不再有**手工录入**」→对（录入是名词也能成谓语）。

判断方法：每次写完"不再有 X"，问自己"X 在做什么/承担什么"，把动作补出来。

触发词扫描：`不再有[人团队产品]。 / 不再有[人团队产品]，`

**规范性判断要显式用"应该 / 应当"**

定义：陈述规范（事情应该怎么做）时如果只写"是 X 的地方"，中文读者会先读成事实描述；加"应该"才能把规范语气传出来。

典型错例：

- 「这个前置队列才是**人类介入的地方**」——读起来像描述事实"事实上人类就在这里介入"。改法：「这个前置队列才是**人类应该介入的地方**」
- 「这才是产品经理**做决策的场合**」→「这才是产品经理**应该做决策的场合**」

判断边界：如果上下文明确是描述（"今天在座的工程师大多在这里介入"），保留无"应该"版本。只在作者是在给规范建议时加"应该"。

**"由 X Y"显式归因**

定义：涉及"谁主导/谁负责"的关系时，"X Y"（动宾直连）容易被读成对等参与，加"由"才能明确主导关系。

典型错例：

- 「Namespace 今年 3 月宣布累计融资 2300 万美元，**NEA 领投**」——"NEA 领投"字面上像 NEA 自己做了个动作。改法：「Namespace 今年 3 月宣布累计融资 2300 万美元，**由 NEA 领投**」
- 「这次会议**张三主持**」→「这次会议**由张三主持**」
- 「产品**王五牵头**」→「产品**由王五牵头**」

判断边界：如果本来就是清楚的施事关系（"张三说"、"王五做了 X"），不用加。只在被动/主导关系需要明确时用。

**文艺/美术性名词比喻在工程文里换工程名词**

定义：规则「拟人化 / 诗意动词用于工程语境」处理了动词侧（长在 / 生根 / 扎根）；这条处理名词侧。「底色」「脉络」「骨架」「纹理」「色调」这些美术/文艺性名词比喻，在工程判断或商业判断里会把严肃陈述软化。

典型错例：

- 「也是人类评审崩不住这个论断**的底色**」——"底色"是美术词，讲的是背景色调。改法：「论断**的本质原因**」或「论断**成立的前提**」
- 「产品设计**的纹理**」→「产品设计**的细节层次**」
- 「团队的**底色**是工程文化」——作为团队/组织描述可以保留（常见表达），不算工程判断
- 「技术栈**的骨架**是 Rust」→「技术栈**的主干**是 Rust」

判断边界：描述**人、团队、文化**时这些美术词可以保留（"这家公司的底色""这位作者的纹理"属常用中文修辞）；描述**系统、指标、论断、判断**时必须换工程名词。

触发词扫描：`的底色 / 的脉络 / 的骨架 / 的纹理 / 的色调`

**作者对自己叙事行为的元评述要删**

定义：规则「判断前后的元评价」处理了作者对**内容品质**的评价（"这是最值得抄的一句"）；这条处理作者对**自己叙事行为**的元评述（"这层关系先摆出来""接下来进入第二个点""下面再展开"）。

典型错例：

- 「所以这对搭档同台，本身也是领投方合伙人和被投公司 CEO 的联合发声，**这层关系先摆出来**」——"这层关系先摆出来"是作者宣告自己要做披露，宣告动作本身没有信息量。改法：保留前半句披露，删掉"这层关系先摆出来"
- 「**先说一句**，下面两段有点技术细节」→ 删
- 「**话又说回来**，我们聊回正题」→ 删（除非用于真正的话题回切）

判断方法：作者说的是"内容"还是"我即将要做的叙事动作"？是后者的一律删。

触发词扫描：`这层关系先摆出来 / 先说一句 / 这点先放这里 / 下面展开说 / 接下来进入 / 这部分先过`

## 七、标点、引号、加粗和术语书写

**标点禁区：**

- 冒号（`：`），尤其是「如下：」「包括：」「原因：」。标题可以用。
- 破折号（`——`），包括双破折号用作插入语的写法。该解释的地方另起一句。
- 双引号（`""`、`""`）。直接引语融入正文叙述，不包引号。

**删除禁用标点是重构，不是查找替换：**

很多时候初稿会带着破折号、冒号、双引号写出来。收稿时的第一反应往往是「查找 —— 替换成句号」，这是错的。标点在句子里承担的是语义连接功能，机械替换会把前后两句变成互相悬空的陈述，让读者自己去补中间那层逻辑关系。

破折号在中文里有四种定型功能，每种要用不同的重构方法处理：

- **引出具体说明**（「A —— 具体是 B」）。机械换句号会让前句变成悬空判断。正确做法是调换语序，让后句先讲具体内容，前句做收尾判断。
  - 坏的改法：「批评方向很清楚 —— 用『幻觉』替代『撒谎』」 → 「批评方向很清楚。用『幻觉』替代『撒谎』」（「方向很清楚」悬在那里）
  - 好的改法：「用『幻觉』替代『撒谎』。批评方向很清楚」（判断落在已说清的具体之后）

- **列举收尾的停顿**（「『X』『Y』『Z』 —— 这些词……」）。简单抽掉破折号会让带引号的列举紧贴下一句「这些词」，视觉挤、节奏碎。正确做法是把列举改成顿号，加一个收拢词自然衔接。
  - 坏：「『AGI』『Agent』『Hallucination』这些词每个人都在说」
  - 好：「AGI、Agent、Hallucination 这三个词每个人都在说」

- **泛指后的逐项展开**（「X 分成几层 —— 第一层……第二层……」）。直接换句号后，泛指句和展开句之间的对应关系被切断，读者要自己搭桥。正确做法是用一个引子词（「落在哪一层上」「分成三类」「具体到一个动作」）让结构关系显现。
  - 坏：「意识到它们在哪个阶段 —— 账单侧决定……训练侧决定……」 → 「意识到它们在哪个阶段。账单侧决定……训练侧决定……」（「哪个阶段」和「账单侧」「训练侧」的对应关系没说出来）
  - 好：「意识到它们落在哪一层上。账单侧……训练侧……」（「哪一层」和后面四层自然对位）

- **插入补充说明**（「A —— 顺便说 B —— 继续」）。直接删掉两道破折号会让补充和主句融成一句，语义乱。正确做法是补充信息独立成一句放相邻位置，或者做成括号。
  - 坏：「这套方案 —— 其实不算新 —— 本质就是 Y」 → 「这套方案其实不算新本质就是 Y」（读不通）
  - 好：「这套方案本质就是 Y。说起来不算新」（独立成句）

冒号、双引号的禁用同理：

- **冒号**（「A：B」）：如果作用是"提示下文"，把 A 改成带动词的完整句（「A 做的是以下几件事」），或者把 B 独立成一段开头；如果作用是"定义"，换成「指的是」「也就是」「其实就是」这类动词短语。
- **英文双引号**（`"..."`）：包裹概念的，换成中文「」；包裹直接引语的，删掉引号让话融入叙述（「他说他很担心这件事」，不用加引号）。

自检方法：删掉一个禁用标点之后，把修改过的两句独立读一遍。如果前句变成"悬空判断"（没有下文它自己说不完整）、或者指代变得含糊、或者和下句的关系要读者自己猜，就是没改好，回到语义层重写。

**「」只用于：**

- 概念名称
- 自造术语
- 被讨论的词语本身

示例：

- 对：「建设者」、「信息搬运工」、「vibe coding」
- 错：他说「我很担心这件事」

直接引语不用「」，自然融入正文叙述。如果这句话是全文最值得读者记住的金句或关键论点，用 **加粗** 标注。

**加粗规范：**

- 全文加粗不超过 3–5 处，只给真正值得单独记住的句子。加粗过多等于没有加粗。
- 章节标题的 `**短句**` 也算加粗。5 个章节标题 + 1 个情绪加粗，总量刚好；5 个章节标题 + 3 个情绪加粗就过了。
- 情绪加粗优先给硬数字类判断（「每年账单从 500 万美元砍到 7.3 万美元」）或最坦诚的反直觉断言（「根本没人点那些反馈按钮」），而不是抒情句。

**英文术语大小写：**

英文术语夹在中文段落里时，按下面三类分别处理。为什么分三类：首字母大写让英文术语在中文段落里成为视觉锚点，读者一眼识别出「这是一个专有概念」，而不是被夹带的普通英文短语；缩写保持原样保留技术背景身份；代码标识符一旦改大小写就不是那个符号了。

- **术语概念和词组：每个单词首字母大写。** 不论是单个词还是组合词组：Agent、Agentic Search、Agentic RAG、Semantic Search、Vector Search、Context Engineering、Context Window、Context Curation、Context Source、Context Compaction、Search Tool、Shell Tool、Bash Tool、Exec Tool、Query Tool、Database Tool、Memory Tool、Web Search、Agent Skill、System Prompt、Tool Description、File Store、Progressive Disclosure、Specialized Tool、General Purpose Tool、Low Floor、High Ceiling、Prompt、Evals、Embedding、Chunk、Token、Multi-hop。
- **缩写原样全大写**：RAG、LLM、ESQL、SQL、CLI、UX、ID、QR、API、SDK、PR、SSE、MCP。
- **产品、公司、框架名维持原有形态**：LangChain、Anthropic、OpenCode、Elastic、Elasticsearch、LlamaIndex、LightOn、ColGrep、Jina、Jina Grep、Vercel、DeepMind、Medium、TU Dresden、DevRel、ChatGPT、Claude Code。看上去像普通英文词的产品名（Gemma、GEPA）也算这一类，不要擅自改动。
- **代码标识符、命令、参数名、配置项一律原样**：用行内 code 包起来（反引号），包括 `@tool` `grep` `ls` `curl` `rm` `top K` `jina-grep` `%GEPA%` `*` `%` `plan.md` `FILTER date=...` `get_customer_by_id` `try/except` `try/catch`。命令里附带的符号（星号、百分号、斜线）也不改。

常见误区：

- `semantic search` / `shell tool` 全小写，阅读时会被当成普通英文短语，损失辨识度。中文段落里应该是 Semantic Search / Shell Tool。
- `Semantic search`（只大写第一个词）是英文句首大写的写法，不适合在中文段落的中间出现。
- `Rag` `Llm` `Esql` 这种首字母式大写是画蛇添足，缩写必须全大写。
- 把真正的代码标识符（如 `@tool`）写成 `@Tool`，就不是那个符号了。代码标识符的判断标准：它会不会出现在代码里？如果会，一律反引号加原样。

## 八、标题策略

**核心结构**：具体主体 + 可验证张力 + 来源锚点（可选）。三个元素中至少两个。

- **具体主体**：读者一眼就能在脑子里锚定的对象。Mission / Anthropic 增长部门 / MCP / Pydantic 创始人 这种，不是「Agent」「方法论」这种抽象概念。
- **可验证张力**：暗示一个判断或悬念，但文章正文真的要给出答案。"如何打造"、"什么才是稀缺的"、"一人跑赢整支团队"、"要退出历史舞台了" 都属于这一类。
- **来源锚点**：「From Lenny's Podcast」「MCP 联合创始人谈」这种出处标签，给判断加权威。非必需，但在访谈类内容里很好用。

**长度**：15–25 字最舒服，最长不要超过 30 字。

**样本标题参照：**

- 「Mission 是如何打造 Multi-Agent 系统的」
- 「专业技能平权后什么才是稀缺的？From Lenny's Podcast」
- 「Anthropic 增长部门如何一人跑赢整支市场团队」
- 「MCP 要退出历史舞台了？MCP 联合创始人谈 MCP 的未来」

**问题句 vs 陈述句：** 两种都能用。问题句必须真问真答，文章里要明确给出答案，不能光问不答。演讲类、访谈类内容问题句常常更好，因为描述性标题预设结论，问题句留给读者自己确认。

**避免：**

- 制造悬念但文章没有揭晓，例如「你绝对想不到」
- 夸大重要性，例如「改变一切」「颠覆认知」
- 泛泛描述，例如「X 的现状与未来」
- 把复杂观点压成口号的对比框架，例如「A 变贵了，B 变便宜了」
- 纯抽象判断做标题，例如「AI 可观测性不是一个独立品类」——缺少具体主体，读者点不进来

## 九、结尾策略

样本里有两种常见的收尾套路，对应不同的文章类型：

**第一种：启发清单型**。适合方法论密集、可迁移动作多的文章。最后一段不是 Markdown 列表，而是用自然段落把 3–5 条读者能直接拿走的建议说清楚。Mission 篇以及「值得抄作业的三件事」这种收尾都是这种。写的时候每条建议要带一个具体动作或场景，不要只写抽象原则。

**第二种：收束画面型**。适合偏观点、偏视野的文章。关键动作是"抽象爬到顶之后降回一个具体画面"——如果文章前面在讨论尼采、戴尔菲神谕、AI 模拟论这种抽象度很高的内容，结尾不要再加一条抽象结论，而是让镜头切回一个具体的人、一个动作、一个场景。这种收尾放弃了"替读者得出结论"的姿态，用画面替代判断。

Max Schoening 那篇的"去你所在的城市或森林走一走……对于身边大多数东西，只要给你 6 到 9 个月，你都可以弄清楚怎么从零做出来"是一个样本。另一个样本是在一段讨论虚无和永恒回归的文章里，作者把结尾落在一位朋友照顾小狗、在日常里找到安静的画面上。抽象的问题不会被这句画面"回答"，但它让读者在自己的情绪里自然找到落点。

**如果原材料能对应具体的实操场景**，在结尾（或接近结尾处）自然做一次桥接，把抽象判断落到一个具体岗位、具体决策、具体团队日常的动作上。不硬凑。示范：

- 「这个判断对正在做企业流程自动化的团队有直接参考意义。」
- 「很多产品团队习惯把流程逻辑写进代码和状态机，等到模型升级就得重写一轮。」
- 「做企业数据 Agent 的团队，至少要有一个通用 Query Tool 兜底。」

不是每篇都要有。桥接的价值在于把判断具象化到读者能辨识的场景里，不是做「海外 vs 国内」「别人 vs 我们」「成熟市场 vs 本土」这种地域/阵营二分。

**地域二分是桥接最常见的翻车姿势**。出稿时如果下笔写出「国内团队」「中国 AI 圈」「海外做法」「外国团队」这类词，停下来问一句：这个实操经验是否真的只适用于某一地域，还是本来就是通用的？样本材料大多数时候讲的是通用工程判断（「Tool Description 要写三层」「错误要返回给 Agent 自修复」），它们不是进口商品，不需要「搬回」某个地域。把地域限定词拿掉，让经验自己站住。

只有当材料里确实讨论了明显的中外差异（监管、供应链、特定市场数据、本地产品生态），才保留地域指向，且要写得具体：不是「国内团队更倾向于」，而是「做跨境支付合规的团队在中国境内执行时……」。

**结尾禁区：**

- 以「总结一下」「综上所述」「让我们一起」开头的段落
- 空洞的升华，比如「这是属于每一个从业者的时代」
- 纯口号，比如「让我们一起迎接 AI 时代」
- 廉价反问，比如「你准备好了吗？」

## 十、高阶手法（可选）

以下是几条从"作者自己的高表现个人写作"里提炼出来的手法。它们不是必须，也不适合每一篇文章。当素材允许时用一两条，可以让文章的气质明显和千篇一律的 AI 生成稿拉开距离。

### 10.1 两件事 + 合流结尾

有些文章不必只讲一件事。把两件看似无关的事并列放进一篇里，各自推进几段，末尾让它们在一个判断或画面处悄悄合流。读者不会被你提前告知合流点，他们是在结尾自己感觉到两条线合在一起的。

为什么可用：比"论点 + 支撑证据"的线性结构更给读者参与感。他们不是被灌输结论，而是自己在脑子里把这两条线接起来了。

什么时候用：当访谈 / 演讲 / 原材料里有一个核心论点，同时你手头还有一件相关但出身不同的事（一个项目、一个书摘、一段个人经历），两者合起来能让观点更立体。纯信息密度型文章（比如纯行业新闻）不用这个结构。

一个示范骨架：
- A 线：素材里某位主讲人的一个反常识判断
- B 线：一个能把这个判断落地的案例，或一个来自另一个领域的对照
- 合流：在倒数第二段让 A 线的判断和 B 线的事实在一个具体动作上交汇，不点破

### 10.2 用另一个领域的框架重讲

把眼前的具体事件用另一个领域的解释框架重新描述一遍。比如把"引体向上"重新描述成"和时空曲率的短暂对抗"，把"给 AI 装手指"重新描述成哲学家说的"视觉是主动的探索性的"。

这不是比喻。比喻是「A 像 B」，这种手法是「用 B 的解释框架去重新映射 A 的事实」，结果是让日常动作获得意外的重量。

什么时候用：当你在写的事件本身容易被读者当成"又一条产品新闻"或"又一次模型更新"时，用一个不相关领域的框架把它重讲一次，能让事件显得陌生又合理。这是让文章具备"智识杠杆感"最容易学的一招。

训练方法：每写到一件具体的事，先问一遍"有没有另一个领域的框架可以把这件事重新讲一遍"。物理、生物、哲学、建筑、运动、音乐都可以是候选。不强求，合适的时候用。

### 10.3 自嘲切换器

如果文章的典故密度或理论密度高，中间要穿插一两句口语化的自嘲做节奏调节。比如正经讨论了半页哲学之后突然写一句「装杯完毕，说说第二件」，或者在展示完一个技术成果之后写「说好的我来，怎么还抢活呢」。

为什么重要：没有这个调节器，严肃段落之间缺少松弛点，读者会觉得作者在端着。自嘲不是破坏严肃，是保护严肃——让读者知道作者自己也意识到刚才那段有点端，愿意自己先笑一下。

操作建议：每当发现自己连续写了两段以上理论或典故，就检查一下要不要插一句松的。自嘲的句式不重要，字眼要俏皮但不油滑，节奏要对。这条没法机械套，得靠感觉。

### 10.4 不要硬抄的东西

有一种风格看起来特别有辨识度，就是**把古文、哲学家名字、物理定律直接混排进正文，不加引号不加解释**。这个看似是技巧，其实不是。它是长期阅读带出来的底层结构，作者能这么写是因为他真的读过，读过的东西在写的时候会自己浮上来。

如果你不是正在消化同类素材、也没有这些人物作为日常思维背景，硬抄这种密度会反效果。读者会觉得"这人好像很有文化但我没跟上"，而且容易翻车——引用错、语境错、轻重错。

规则：**不要为了显风格而增加典故密度**。典故出现的触发条件应该是"这个判断本来就可以由这句话更精准地表达"，不是"我希望这篇文章看起来更有文人气"。如果素材本身不带这种密度，不加。

## 十一、来源交代格式

文章最后必须注明来源。格式固定：

```
内容来源，[渠道 / 频道 / 平台]「[原标题]」，[类型] 链接 [URL]
```

样本里的实际用法：

- 「内容来源，AI Engineer 频道「The Multi-Agent Architecture That Actually Ships — Luke Alvoeiro, Factory」，YouTube 链接 https://...」
- 「内容来源，Lenny's Podcast「Why cultivating agency matters more than cultivating skills in the AI era | Max Schoening (Notion)」，YouTube 链接 https://...」
- 「内容来源，GRITCULT「Marketing is dead. Long live The Distribution Engineer.」，原文链接 https://...」

如果主讲人身份对读者有帮助，可以在来源行后补一句 "演讲者为 X（Y 公司，身份）"。

## 十二、自查清单

写完初稿后逐条过一遍：

**结构与开头**
- [ ] 正文有没有 Markdown 标题、列表、表格？
- [ ] 开头第一段是不是直接给判断 / 场景 / 数字，而不是先写人物简历？
- [ ] 前三四段是否自然交代了来源、人物身份和核心问题？
- [ ] 章节标题（如果有）是不是完整短句带动词，不是冷名词？

**标点、加粗与术语**
- [ ] 有没有冒号（`：`）、破折号（`——`）、双引号？
- [ ] 删掉破折号/冒号的地方，把前后两句独立读一遍，前句是否变成悬空判断、对应关系是否要读者自己猜？是就回到语义层重写，不是查找替换
- [ ] 「」是否只用于概念名词，而非直接引语？
- [ ] 加粗是否不超过 5 处（含章节标题）？
- [ ] 情绪加粗是否只用在硬数字 / 反直觉判断上？
- [ ] 英文术语词组是否每个单词首字母大写（Agent / Semantic Search / Context Window）？
- [ ] 缩写是否全大写（RAG / LLM / ESQL）而不是 Rag / Llm / Esql？
- [ ] 代码标识符、命令、参数名是否全部用反引号包起来且保持原形（`@tool` / `top K` / `jina-grep`）？

**语气与内容**
- [ ] 有没有 AI 腔词语（包括「不是 A 而是 B」及其变体）？
- [ ] 每段是否至少有一个具体的人、事、数字、场景或原话？
- [ ] 是否避免了「某某说」「在演讲中」这类转场腔？
- [ ] 是否避免了「X 对 Y 评价挺复杂」这种空转场句？
- [ ] 有没有把模糊形容词换成可量化的数字？
- [ ] 有没有残留对读者零价值的转录细节（讲者念不准某词、现场笑了、让观众举手、时间关系跳过、未展开的人名）？抽掉后文章是否更干净？
- [ ] 涉及「这张图」「这根箭头」「这段代码」「屏幕上」等 deixis 指示语的段落都截图插入了吗？有没有 `[图:MM:SS - 描述]` 占位符忘了替换？
- [ ] 有没有「X 做了一件事。他……」这种先卖关子再揭晓的结构？有就砍铺垫直接写动作
- [ ] 有没有「那句『』」「这条路」这种指代嵌套或抽象指代？能换成具体名词就换
- [ ] 讲者跑了几分钟、调了哪个参数、用了哪个版本 —— 这些执行细节是判断依据还是凑字数？不是前者就删
- [ ] 「直接」「其实」「就是」「真的」这类强调副词在同一段出现两次以上没？后面那个大概率能删
- [ ] 删完冗余后，相邻分句的关系是否还清楚？不清楚就加显式连接词（「还有一个」「同样」「反过来说」）
- [ ] 有没有「X 问、Y 答」「主持人追问，嘉宾回应」这类对谈元信息？抽掉这句后观点推进是否损失？不损失就删，直接分点或直接写判断
- [ ] 判断出现之前或之后有没有「这是最值得抄的一句」「讲者原话是」「作者的批评很克制，方向却很清楚」「这个观点挺犀利」这类元评价（铺垫或收尾）？基本全删，让判断自己显重要性 —— 测试：抽掉这句话读者对前文的感受会不会变？不会就删
- [ ] 有没有「做了两件事」「还有三件事」「几样动作」「几个方面」这种不带语义的笼统量词？按上下文换成「两处调整/三条建议/三个手段/三种方法/三个路径/三次失败/三类做法」等携带内容性质的量词
- [ ] 有没有替原视频/原讲者做宣传（课程、订阅、newsletter、social handles）？默认删。例外是「读者当场就能用到的具体免费资源」，保留时压到一句话
- [ ] 有没有冗余默认属性限定词？公开人物的公开发言默认是公开的，作者写的文章默认是他写的，巨头旗下产品的归属默认熟知。grep `公开(判断|发言|说过) / 旗下的 / 亲自(写|做) / 本人(写|发)`，命中即删
- [ ] 有没有「接过来的 Hugo / 站起身的 Marc」这种分词短语做人物定语？改成时间副词衔接（接下来 / 这时 / 接着）
- [ ] 有没有「的那个词 / 的那种做法」这种尾缀冗余？改成直接收尾或改定语
- [ ] 「让 X 快」「让 X 好」里 Y 是形容词的使役结构，有没有把 Y 动词化为「变 Y / 成为 Y」？
- [ ] 「不再有 X」之类的省略陈述，后面是否补了谓语（不再有 X 参与 / 做 Y / 承担 Z）？
- [ ] 规范性建议有没有用「应该 / 应当」显式化？否则会被读成事实描述
- [ ] 涉及主导关系（融资领投、会议主持、项目牵头）的归因有没有用「由 X Y」？动宾直连会读成对等参与
- [ ] 描述系统、指标、论断时有没有用美术/文艺名词比喻（底色/脉络/骨架/纹理/色调）？工程判断换工程名词（本质原因 / 主干 / 前提）。描述人/团队/文化时可以保留
- [ ] 有没有作者对自己叙事行为的元评述（这层关系先摆出来 / 这点先放这里 / 下面展开说）？默认删
- [ ] ledger/账本用词一致性：开发/数据流/Git 语境里统一用「明细」或「流水」，不用「账本」（会计语境）

**标题与结尾**
- [ ] 标题是否包含具体主体（不是纯抽象概念）？
- [ ] 标题长度是否在 15–25 字之间？
- [ ] 如果是问题句标题，文章里是否真的给出了答案？
- [ ] 结尾是启发清单型还是收束画面型？有没有以「总结」「综上」开头的空段落？
