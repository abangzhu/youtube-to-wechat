# 微信公众号写作规范

这份文件只负责最终 `article.md` 的成文规则：文章结构、开头、来源交代、叙事视角、公众号语气、标点、引号、加粗、标题和自查。

英译中、本地化、术语、翻译腔排查、英文示例中英对照，统一看 [translation-quality.md](translation-quality.md)。

---

## 一、正文结构

以下任何一项出现在正文中都要立刻修改：

- Markdown 标题（`#`、`##`、`###`）。`article.md` 第一行的 `# 标题` 是唯一例外
- 无序列表（`-`、`•`）和有序列表（`1.`、`2.`）
- 斜体（`*`）
- 表格

公众号正文默认用连续段落推进。只有文章超过 2000 字、主题切换明显、读者扫读会迷失时，才用 `**短标题**` 做章节标题，不用 Markdown `##`。

**章节标题要是"完整短句带动词"，不是冷名词**。真实发在公众号表现好的小标题大多长这样：

- 「Validation Contract 才是这场演讲真正要讲的东西」
- 「串行比并行好，这个结论违反所有人的第一反应」
- 「前 10% 免费了」
- 「可塑软件会回来」
- 「品味是一台虚拟机」
- 「GEPA 的机制朴素到有点可爱」
- 「Shopify 那一刀，什么时候做优化才值回来」

长度控制在 5–20 字之间。理由是公众号读者在扫读时，冷名词标题（比如「架构」「验证合同」）提供不了信息增量，读者还是得读正文才知道这段在讲什么；带动词的完整短句本身就是一个判断，读者扫过小标题就能决定要不要读下面这段。

**章节标题数量的参考值**：8 段以内的文章基本不需要，8–15 段的文章加 3–5 个，超过 15 段的长文可以多到 7 个左右。不是每个段落群都要加；只在主题明显切换的地方加。

## 二、开头策略

开头的任务不是复述来源，而是让读者迅速知道「谁在说」「为什么值得听」「他在回答什么问题」。

**开头三段的推荐节奏**（观察高表现样本得出的共性模式）：

1. **第一段先扔一个具体场景、数字或判断**，不要先做人物介绍。把读者直接丢进现场。人物是谁放到第二段去交代。
2. **第二段建立可信度**。交代他是谁、公司背景、关键量化锚点（融资、估值、下载量、GitHub star 数、用户量）。这一段是说服读者"继续读下去"的那一段。
3. **第三段锁定文章要回答的问题或核心张力**。用一两句话把读者的阅读目标钉死。

**样本开头示范：**

```
Luke Alvoeiro 在 AI Engineer 的一场 18 分钟演讲里抛了一个判断，今天做软件工程的
瓶颈不再是智能，而是人的注意力。他给这句话附上了具体的场景，一个工程师手头可能
积压 50 个 feature，但每天真正能往前推的只有两三件。
```

```
Pydantic 的创始人走上 AI Engineer 大会 2026 的 workshop 舞台，开场第一件事就是
亲手拆了自家招牌。他说自己根本不相信 AI observability 是一个能长期存在的独立品类。
```

```
Anthropic 是 Claude 背后的公司，估值超过 3800 亿美元。这样规模的公司，整个增长
营销部门曾经只有一个人，独自运营了整整 10 个月。
```

共同点：第一段里要么是一个反常识判断，要么是一个具象到能在脑子里成画面的场景 / 数字。主体立刻出现，但它是为判断服务的，不是为自己服务的。

**偶遇式开头（可选分支）**：有些文章不是要给读者传递一条外部信息，而是作者自己正在想一件事，顺手把思考路径分享出来。这类文章的起笔可以用「正巧」「无独有偶」「源自巧合」这种词做锚，把两件看似无关的事串起来。为什么可用：这种开头的姿态是「我在想 → 邀请你一起想」而不是「我来告诉你 →」，姿态会决定读者的预期，也会决定你的结尾能不能不做总结。

偶遇式开头的示范：

```
陀思妥耶夫斯基早就预言过个人主义和现代性的危机。顺着个人主义推演，就可以推演出
《罪与罚》中拉斯科尔尼科夫对杀人的动机同样的阐述。
```

```
说起为什么要读这本书，也是源自巧合。其一，正巧最近在读汉娜·阿伦特的《人的境况》。
```

只在文章本身更贴近个人思考、而非外部素材转述时才用这条分支。纯访谈 / 演讲 / 新闻类材料，仍然走三段节奏。

**开头禁区：**

- 不能以提问开头，例如「你有没有想过」「为什么……？」
- 不能以「在 AI 时代」「在这个快速变化的时代」开头
- 不能以「本文」「本期」「今天我们来聊聊」开头
- 不要写演讲现场叙事，例如「他走上台说了一件让人安心的事」
- 不要写播放量、点赞数，除非数字极端异常且与文章判断有关
- 不要先写人物简历然后再讲内容。这是样本里最常见、最容易写成的开头错误。

## 三、来源交代与作者视角

文章要自然交代来源，但不能让来源成为正文主角。

**需要交代的内容：**

- 节目/频道/原文是什么，一句话说明定位
- 主播、嘉宾、作者是谁，一句话说明为什么他说这件事值得听
- 技术产品、协议或框架是什么，一句话给非熟悉读者一个中文锚点

这些信息要在前三四段内自然带出，不要单独罗列人物简历，也不要等读者读了一大段后才补背景。

**来源转述腔禁区：**

- 不要频繁用「某某说」「某某提到」「某某认为」「在演讲中」「这期视频里」推进段落
- 不要逐段复述视频或原文的展开顺序
- 不要把文章写成节目推荐、视频摘要或读后感
- 不要加「X 对这件事的评价挺复杂」「X 对 Y 非常赞赏」这种只承担转场功能的空句，样本里一句都没有

**更好的写法：**

先写读者能感知的问题、场景和判断，再在需要建立可信度的地方轻轻归因。人物身份、关键原话、独特经历、具体数据和争议性判断需要归因；已经被作者消化成场景、方法论或分析判断的内容，不必每句都挂回演讲者。

## 四、公众号语气

文章应该像中文作者读完材料后写给中文读者的文章，而不是英文 transcript 的中文转述，也不是内部研究报告。

**具体性优先：**

每段尽量有具体的人、事、数字、场景或原话。四个都没有，这段通常就是空洞判断。宁可少写几个观点、把一个观点写透，也不要每个观点都提一下、没有一个说清楚。

**数字量化原则：** 只要能量化就量化，不用「大幅提升」「显著下降」「提升了很多」这种模糊形容词。样本里所有数字都不是装饰，都在支撑具体判断：

- 下载量 / star 数 / 估值 / 融资轮次
- 具体成本差：500 万美元 → 7.3 万美元
- 时间压缩：2 小时 → 15 分钟
- 百分比：60% 时间、90% 覆盖率、96.7% 准确率
- 具体约束：30 字符标题、90 字符描述、700 行编排文本

**命名角色而非匿名化：**

样本里的 Boris Cherny、Luke Alvoeiro、Max Schoening、David Soria Parra、GRITCULT 全部被点名。不要写「一位工程师」「某位 CEO」。能具名就具名。

**中文表达优先：**

先给中文读者能想象的场景、动作或问题，再自然带出术语和判断。不要沿用英文演讲的抽象判断顺序，不要连续堆英文术语。

**衔接词推荐：**

取代「首先/其次/最后」，改用更口语化的衔接：

> 「先说……」「再说……」「说到……」「顺便说说……」「回到……」「说到未来……」「但有一件……」

**指代容忍度：**

代词指代不需要每次都显式补全名字，只要上下文连贯读者能跟住就够了。频繁重复名字会显得机械。真正需要补名字的情况是话题转换了，或者两个人交替出现时。

**句式节奏：**

- 单句长度平均 20–40 字，尽量不超过 60 字。长句超过 60 字先看能不能断成两句。
- 并列结构用句号分隔，不要塞进同一句。「这不是因为资源紧张，也不是因为他是某种天才。这是匿名 X 账号 GRITCULT 在 4 月 15 日写的一篇长文里的核心案例。」
- 动词要强。样本里的表达方式：「抹平编码壁垒」「踩改动」「吃掉速度收益」「哗哗地烧」「亲手拆了自家招牌」「一刀砍到」。弱动词（进行、实现、达成、展开）能换就换。

**干涩幽默：**

样本里不时出现轻度吐槽 / 自嘲 / 小结论，让文章有人味但不油腻。不做情感煽动，不用感叹号，不反问煽情。例：

- 「大多数人并不想去打猎，只想去 Costco 买一块已经放在泡沫托盘里的牛排。」
- 「一个拿对工具的人，胜过十个用老方法的人。」
- 「这种说法也许有些夸张，但 Anthropic 那个只有一个人的增长部门，多少给这句话做了一点背书。」

## 五、概念命名与定义

如果文章里引入了一个值得读者带走的新概念 / 术语 / 方法论，用「命名 + 破题动作」的方式一次讲清楚，之后再出现就裸写、不再加定义。

**样本里的范例：**

- validation contract：在代码写下去之前就把正确性定义清楚。
- droid whispering：给每个角色挑合适模型。
- progressive discovery：给模型一个工具搜索工具，按需加载。
- programmatic tool calling：让模型写脚本组合工具调用，不再一步一步推理。
- tiny core superpower：一个产品核心的"超能力"，比如 iPhone 的 multi-touch、GitHub 的 Pull Request。
- 分发工程师：把分发当作工程问题来处理的人。
- agency：相信世界可以被改变并愿意亲手去改的那个劲头。

**模式**：一行名字 + 一行动作 / 效果，不做抽象哲学化定义。首次出现的时候用「」锚一次，后面再提到就裸写，不要反复戴引号。

## 六、AI 腔禁区

以下词语和句式一律删除，替换为更具体的说法，或直接删掉这句话。

**形容词类：**

- 「深入」「深度」「深刻」「全面」「系统」「卓越」「显著」「丰富」

**连接词类：**

- 「不仅如此」「此外」「综上所述」「总的来说」「值得注意的是」

**视角切换类：**

- 「让我们」「我们来看看」「接下来让我们」

**结构词类：**

- 任何以「在……的背景下」开头的句子
- 任何以「总结」或「综上」开头的段落
- 「不是……而是……」这类公式化对比句。除非原文必须保留强对立，否则改成直接陈述、两句递进或具体场景说明
- 「不去 X，而是 Y」「没有 A，而是 B」这些同一家族的变体同样要避开

## 七、标点、引号和加粗

**标点禁区：**

- 冒号（`：`），尤其是「如下：」「包括：」「原因：」。标题可以用。
- 破折号（`——`），包括双破折号用作插入语的写法。该解释的地方另起一句。
- 双引号（`""`、`""`）。直接引语融入正文叙述，不包引号。

**「」只用于：**

- 概念名称
- 自造术语
- 被讨论的词语本身

示例：

- 对：「建设者」、「信息搬运工」、「vibe coding」
- 错：他说「我很担心这件事」

直接引语不用「」，自然融入正文叙述。如果这句话是全文最值得读者记住的金句或关键论点，用 **加粗** 标注。

**加粗规范：**

- 全文加粗不超过 3–5 处，只给真正值得单独记住的句子。加粗过多等于没有加粗。
- 章节标题的 `**短句**` 也算加粗。5 个章节标题 + 1 个情绪加粗，总量刚好；5 个章节标题 + 3 个情绪加粗就过了。
- 情绪加粗优先给硬数字类判断（「每年账单从 500 万美元砍到 7.3 万美元」）或最坦诚的反直觉断言（「根本没人点那些反馈按钮」），而不是抒情句。

## 八、标题策略

**核心结构**：具体主体 + 可验证张力 + 来源锚点（可选）。三个元素中至少两个。

- **具体主体**：读者一眼就能在脑子里锚定的对象。Mission / Anthropic 增长部门 / MCP / Pydantic 创始人 这种，不是「Agent」「方法论」这种抽象概念。
- **可验证张力**：暗示一个判断或悬念，但文章正文真的要给出答案。"如何打造"、"什么才是稀缺的"、"一人跑赢整支团队"、"要退出历史舞台了" 都属于这一类。
- **来源锚点**：「From Lenny's Podcast」「MCP 联合创始人谈」这种出处标签，给判断加权威。非必需，但在访谈类内容里很好用。

**长度**：15–25 字最舒服，最长不要超过 30 字。

**样本标题参照：**

- 「Mission 是如何打造 Multi-Agent 系统的」
- 「专业技能平权后什么才是稀缺的？From Lenny's Podcast」
- 「Anthropic 增长部门如何一人跑赢整支市场团队」
- 「MCP 要退出历史舞台了？MCP 联合创始人谈 MCP 的未来」

**问题句 vs 陈述句：** 两种都能用。问题句必须真问真答，文章里要明确给出答案，不能光问不答。演讲类、访谈类内容问题句常常更好，因为描述性标题预设结论，问题句留给读者自己确认。

**避免：**

- 制造悬念但文章没有揭晓，例如「你绝对想不到」
- 夸大重要性，例如「改变一切」「颠覆认知」
- 泛泛描述，例如「X 的现状与未来」
- 把复杂观点压成口号的对比框架，例如「A 变贵了，B 变便宜了」
- 纯抽象判断做标题，例如「AI 可观测性不是一个独立品类」——缺少具体主体，读者点不进来

## 九、结尾策略

样本里有两种常见的收尾套路，对应不同的文章类型：

**第一种：启发清单型**。适合方法论密集、可迁移动作多的文章。最后一段不是 Markdown 列表，而是用自然段落把 3–5 条读者能直接拿走的建议说清楚。Mission 篇和本文的"国内团队能直接抄的几点"都是这种。写的时候每条建议要带一个具体动作或场景，不要只写抽象原则。

**第二种：收束画面型**。适合偏观点、偏视野的文章。关键动作是"抽象爬到顶之后降回一个具体画面"——如果文章前面在讨论尼采、戴尔菲神谕、AI 模拟论这种抽象度很高的内容，结尾不要再加一条抽象结论，而是让镜头切回一个具体的人、一个动作、一个场景。这种收尾放弃了"替读者得出结论"的姿态，用画面替代判断。

Max Schoening 那篇的"去你所在的城市或森林走一走……对于身边大多数东西，只要给你 6 到 9 个月，你都可以弄清楚怎么从零做出来"是一个样本。另一个样本是在一段讨论虚无和永恒回归的文章里，作者把结尾落在一位朋友照顾小狗、在日常里找到安静的画面上。抽象的问题不会被这句画面"回答"，但它让读者在自己的情绪里自然找到落点。

**如果原材料能对应中国实操场景**，在结尾（或接近结尾处）显式做一次桥接，向国内团队说一句对应的判断。不硬凑。示范：

- 「这个判断对国内正在做企业流程自动化的团队有直接参考意义。」
- 「大多数国内产品团队更习惯把流程逻辑写进代码和状态机，等到模型升级就得重写一轮。」

不是每篇都要有。当样本材料里真的有一条能对应中国实操的判断时，这种嫁接会显著提升文章在国内读者那里的切身感。

**结尾禁区：**

- 以「总结一下」「综上所述」「让我们一起」开头的段落
- 空洞的升华，比如「这是属于每一个从业者的时代」
- 纯口号，比如「让我们一起迎接 AI 时代」
- 廉价反问，比如「你准备好了吗？」

## 十、高阶手法（可选）

以下是几条从"作者自己的高表现个人写作"里提炼出来的手法。它们不是必须，也不适合每一篇文章。当素材允许时用一两条，可以让文章的气质明显和千篇一律的 AI 生成稿拉开距离。

### 10.1 两件事 + 合流结尾

有些文章不必只讲一件事。把两件看似无关的事并列放进一篇里，各自推进几段，末尾让它们在一个判断或画面处悄悄合流。读者不会被你提前告知合流点，他们是在结尾自己感觉到两条线合在一起的。

为什么可用：比"论点 + 支撑证据"的线性结构更给读者参与感。他们不是被灌输结论，而是自己在脑子里把这两条线接起来了。

什么时候用：当访谈 / 演讲 / 原材料里有一个核心论点，同时你手头还有一件相关但出身不同的事（一个项目、一个书摘、一段个人经历），两者合起来能让观点更立体。纯信息密度型文章（比如纯行业新闻）不用这个结构。

一个示范骨架：
- A 线：素材里某位主讲人的一个反常识判断
- B 线：一个能把这个判断落地的案例，或一个来自另一个领域的对照
- 合流：在倒数第二段让 A 线的判断和 B 线的事实在一个具体动作上交汇，不点破

### 10.2 用另一个领域的框架重讲

把眼前的具体事件用另一个领域的解释框架重新描述一遍。比如把"引体向上"重新描述成"和时空曲率的短暂对抗"，把"给 AI 装手指"重新描述成哲学家说的"视觉是主动的探索性的"。

这不是比喻。比喻是「A 像 B」，这种手法是「用 B 的解释框架去重新映射 A 的事实」，结果是让日常动作获得意外的重量。

什么时候用：当你在写的事件本身容易被读者当成"又一条产品新闻"或"又一次模型更新"时，用一个不相关领域的框架把它重讲一次，能让事件显得陌生又合理。这是让文章具备"智识杠杆感"最容易学的一招。

训练方法：每写到一件具体的事，先问一遍"有没有另一个领域的框架可以把这件事重新讲一遍"。物理、生物、哲学、建筑、运动、音乐都可以是候选。不强求，合适的时候用。

### 10.3 自嘲切换器

如果文章的典故密度或理论密度高，中间要穿插一两句口语化的自嘲做节奏调节。比如正经讨论了半页哲学之后突然写一句「装杯完毕，说说第二件」，或者在展示完一个技术成果之后写「说好的我来，怎么还抢活呢」。

为什么重要：没有这个调节器，严肃段落之间缺少松弛点，读者会觉得作者在端着。自嘲不是破坏严肃，是保护严肃——让读者知道作者自己也意识到刚才那段有点端，愿意自己先笑一下。

操作建议：每当发现自己连续写了两段以上理论或典故，就检查一下要不要插一句松的。自嘲的句式不重要，字眼要俏皮但不油滑，节奏要对。这条没法机械套，得靠感觉。

### 10.4 不要硬抄的东西

有一种风格看起来特别有辨识度，就是**把古文、哲学家名字、物理定律直接混排进正文，不加引号不加解释**。这个看似是技巧，其实不是。它是长期阅读带出来的底层结构，作者能这么写是因为他真的读过，读过的东西在写的时候会自己浮上来。

如果你不是正在消化同类素材、也没有这些人物作为日常思维背景，硬抄这种密度会反效果。读者会觉得"这人好像很有文化但我没跟上"，而且容易翻车——引用错、语境错、轻重错。

规则：**不要为了显风格而增加典故密度**。典故出现的触发条件应该是"这个判断本来就可以由这句话更精准地表达"，不是"我希望这篇文章看起来更有文人气"。如果素材本身不带这种密度，不加。

## 十一、来源交代格式

文章最后必须注明来源。格式固定：

```
内容来源，[渠道 / 频道 / 平台]「[原标题]」，[类型] 链接 [URL]
```

样本里的实际用法：

- 「内容来源，AI Engineer 频道「The Multi-Agent Architecture That Actually Ships — Luke Alvoeiro, Factory」，YouTube 链接 https://...」
- 「内容来源，Lenny's Podcast「Why cultivating agency matters more than cultivating skills in the AI era | Max Schoening (Notion)」，YouTube 链接 https://...」
- 「内容来源，GRITCULT「Marketing is dead. Long live The Distribution Engineer.」，原文链接 https://...」

如果主讲人身份对读者有帮助，可以在来源行后补一句 "演讲者为 X（Y 公司，身份）"。

## 十二、自查清单

写完初稿后逐条过一遍：

**结构与开头**
- [ ] 正文有没有 Markdown 标题、列表、表格？
- [ ] 开头第一段是不是直接给判断 / 场景 / 数字，而不是先写人物简历？
- [ ] 前三四段是否自然交代了来源、人物身份和核心问题？
- [ ] 章节标题（如果有）是不是完整短句带动词，不是冷名词？

**标点与加粗**
- [ ] 有没有冒号（`：`）、破折号（`——`）、双引号？
- [ ] 「」是否只用于概念名词，而非直接引语？
- [ ] 加粗是否不超过 5 处（含章节标题）？
- [ ] 情绪加粗是否只用在硬数字 / 反直觉判断上？

**语气与内容**
- [ ] 有没有 AI 腔词语（包括「不是 A 而是 B」及其变体）？
- [ ] 每段是否至少有一个具体的人、事、数字、场景或原话？
- [ ] 是否避免了「某某说」「在演讲中」这类转场腔？
- [ ] 是否避免了「X 对 Y 评价挺复杂」这种空转场句？
- [ ] 有没有把模糊形容词换成可量化的数字？

**标题与结尾**
- [ ] 标题是否包含具体主体（不是纯抽象概念）？
- [ ] 标题长度是否在 15–25 字之间？
- [ ] 如果是问题句标题，文章里是否真的给出了答案？
- [ ] 结尾是启发清单型还是收束画面型？有没有以「总结」「综上」开头的空段落？
- [ ] 如果材料能对应中国实操场景，是否做了一次国内桥接？

**整体阅读感**
- [ ] 正文读起来是在分享观点，还是替原视频做广告 / 逐段复述演讲者说了什么？
- [ ] 文章抽掉主讲人名字，读起来是作者自己的判断，还是 transcript 的缩写？

**高阶手法使用检查（可选，第十章内容）**
- [ ] 如果用了两件事合流结构，合流点是自然浮现还是被点破了？
- [ ] 如果用了跨领域 reframing，这个框架是帮助读者看清事件，还是只是在炫耀？
- [ ] 如果文章典故密度偏高，有没有一两处自嘲式切换器做节奏调节？
- [ ] 典故和引文是"本来就能更精准表达这个判断"才出现的，还是只为了让文章显得有文人气？
