# 公众号正文结构规则

读取时机：写 `article.md` 前。

## 一、正文结构

以下任何一项出现在正文中都要立刻修改：

- 无序列表（`-`、`•`）和有序列表（`1.`、`2.`）
- 斜体（`*`）
- 表格
- H1（`#`）以外的多级 Markdown 标题，但 H2（`## 标题`）作章节标题是允许的，见下

**无序列表禁令的例外**：当一段同时介绍 4 项及以上**并列、独立、可单独评估**的工具/能力/资源（典型场景：文章末尾的「能用的零件清单」「相关工具列表」「参考链接组」），允许用无序列表呈现。判断标准：这些项目之间没有先后依赖、没有递进逻辑，读者扫读时关心的是各自独立的功能而不是它们组成的论证链条。

反例（不应列表化）：论证链（"先 X，所以 Y，因此 Z"）、对比叙述（"A 派……B 派……"）、流程描述（"先做 X，再做 Y"），即使能列成 4 项也应保持段落结构。

**`article.md` 第一行的 `# 标题`** 是文章主标题，唯一一个 H1。

**章节标题用 H2（`## 标题`），不再用 `**短标题**` 加粗**。理由：
1. 主流公众号 Markdown 转换工具（mdnice、md2wx、墨滴等）会把 H2 渲染成有视觉分隔感的章节标题，加粗反而被当作段中强调
2. `article.md` 在 IDE / 编辑器里阅读时，H2 提供清晰的目录结构和折叠节点
3. 加粗 `**X**` 和段中关键短语强调挤在同一种标记里，会污染语义层级

公众号正文默认用连续段落推进。只有文章超过 2000 字、主题切换明显、读者扫读会迷失时，才加章节标题。

**章节标题要是"完整短句带动词"，不是冷名词**。真实发在公众号表现好的小标题大多长这样：

- 「Validation Contract 才是这场演讲真正要讲的东西」
- 「串行比并行好，这个结论违反所有人的第一反应」
- 「前 10% 免费了」
- 「可塑软件会回来」
- 「品味是一台虚拟机」
- 「GEPA 的机制朴素到有点可爱」
- 「Shopify 那一刀，什么时候做优化才值回来」

长度控制在 5–20 字之间。理由是公众号读者在扫读时，冷名词标题（比如「架构」「验证合同」）提供不了信息增量，读者还是得读正文才知道这段在讲什么；带动词的完整短句本身就是一个判断，读者扫过小标题就能决定要不要读下面这段。

**超过 20 字时,优先砍具体内容,保留主判断**。常见错误是把"主判断 + 具体内容"两层都塞进 H2:

- ✗「每个职位的命运正在重排,PM 和全栈设计师走到最前」(22 字,主判断"职位重排"+ 具体内容"PM/设计师") → ✓「每个职位的职责都在被改写」(11 字,只留主判断,具体内容下放正文)
- ✗「自动化是个谎言:Dan 自己的 Senior Engineer Benchmark」(28 字,主判断 + 具体案例名) → ✓「自动化是个谎言」(7 字)或「Dan 用 Benchmark 证明自动化是个谎言」(17 字)
- ✗「一年之内,工作会分叉成两条线:Slack super agent 加 Codex 操作系统」(32 字,主判断 + 具体形态) → ✓「一年之内,工作会分叉成两条线」(13 字)

判断方法:H2 超过 20 字时,把它拆成"主判断"和"具体内容(谁/什么/怎么做)"两部分。**砍具体内容,保留主判断**。具体内容会在正文第一段自然展开,读者扫读 H2 不需要看到全部细节。

**反向红线:砍具体内容时不能连逻辑限定词一起砍掉**。「全部 / 只 / 未必 / 都 / 仅 / 一定」这类词不是"具体内容",是判断本身的核心,砍了会改变语义、和正文矛盾。例:正文讲"模型公司未必捕获 AI 的**全部**价值"(模型会吃掉一部分,只是未必全部),H2 压成「模型未必吃掉价值」就成了"一点都吃不到",和正文对不上;正确是「模型未必吃掉**全部**价值」。压 H2 时先分清:砍的是"具体名词 / 案例名 / 谁做的",不是"全部 / 只 / 未必"这种限定判断范围的词。

**章节标题数量的参考值**：8 段以内的文章基本不需要，8–15 段的文章加 3–5 个，超过 15 段的长文可以多到 7 个左右。不是每个段落群都要加；只在主题明显切换的地方加。

**章节标题串读 = 论点链**：把所有 H2 抽出来连续读一遍，本身要构成文章的骨架——读者扫读这一串小标题时，应该能看出文章的论点序列和推进方向。爆款样本的标题串都通过这一关：

- 罗马军团已经过期了 → 一个会自己改进的公司由五层组成 → 同样的回路可以跑产品和客服 → 烧 token，不要烧人头 → 没记下来的事，对你的 AI 就等于没发生过 → 业务 context 是资产，软件是消耗品 → 人坐在公司大脑的外圈 → 现在动手还是以后重写

把这串读完，整篇文章的判断序列已经在了。如果你抽出标题串读起来像一组互不相关的孤立短语（"AI 的现状"、"几个观察"、"一些思考"、"总结"），文章的结构本身就是有问题的，需要回头重新设计标题——不是给标题换皮，是检查每一节的核心判断是否能各自立住、相互衔接成链。

判断方法：写完文章把所有 `## ` 行用 `grep "^## "` 抽出来连读。读完如果不知道这篇文章在论证什么，回去改标题；如果还能看出论点序列里有断点（某一节的判断衔接不上下一节），改正文结构而不是只改标题。

**段中加粗（`**关键短语**`）的边界**：现在 H2 承担章节标题的视觉分隔功能后，加粗只用于「段中关键短语强调」这一个场景。具体三类位置和密度软参考详见第七章「加粗规范」（每千字 ≤ 2 处是软参考，不是硬上限，文章长短和信息密度都会影响）。

## 二、开头策略

开头的任务不是复述来源，而是让读者迅速知道「谁在说」「为什么值得听」「他在回答什么问题」。

**开头三段的推荐节奏**（观察高表现样本得出的共性模式）：

1. **第一段先扔一个具体场景、数字或判断**，不要先做人物介绍。把读者直接丢进现场。人物是谁放到第二段去交代。
2. **第二段建立可信度**。交代他是谁、公司背景、关键量化锚点（融资、估值、下载量、GitHub star 数、用户量）。这一段是说服读者"继续读下去"的那一段。
3. **第三段锁定文章要回答的问题或核心张力**。用一两句话把读者的阅读目标钉死。

**样本开头示范：**

```
Luke Alvoeiro 在 AI Engineer 的一场 18 分钟演讲里抛了一个判断，今天做软件工程的
瓶颈不再是智能，而是人的注意力。他给这句话附上了具体的场景，一个工程师手头可能
积压 50 个 feature，但每天真正能往前推的只有两三件。
```

```
Pydantic 的创始人走上 AI Engineer 大会 2026 的 workshop 舞台，开场第一件事就是
亲手拆了自家招牌。他说自己根本不相信 AI observability 是一个能长期存在的独立品类。
```

```
Anthropic 是 Claude 背后的公司，估值超过 3800 亿美元。这样规模的公司，整个增长
营销部门曾经只有一个人，独自运营了整整 10 个月。
```

共同点：第一段里要么是一个反常识判断，要么是一个具象到能在脑子里成画面的场景 / 数字。主体立刻出现，但它是为判断服务的，不是为自己服务的。

**偶遇式开头（可选分支）**：有些文章不是要给读者传递一条外部信息，而是作者自己正在想一件事，顺手把思考路径分享出来。这类文章的起笔可以用「正巧」「无独有偶」「源自巧合」这种词做锚，把两件看似无关的事串起来。为什么可用：这种开头的姿态是「我在想 → 邀请你一起想」而不是「我来告诉你 →」，姿态会决定读者的预期，也会决定你的结尾能不能不做总结。

偶遇式开头的示范：

```
陀思妥耶夫斯基早就预言过个人主义和现代性的危机。顺着个人主义推演，就可以推演出
《罪与罚》中拉斯科尔尼科夫对杀人的动机同样的阐述。
```

```
说起为什么要读这本书，也是源自巧合。其一，正巧最近在读汉娜·阿伦特的《人的境况》。
```

只在文章本身更贴近个人思考、而非外部素材转述时才用这条分支。纯访谈 / 演讲 / 新闻类材料，仍然走三段节奏。

**开头禁区：**

- 不能以提问开头，例如「你有没有想过」「为什么……？」
- 不能以「在 AI 时代」「在这个快速变化的时代」开头
- 不能以「本文」「本期」「今天我们来聊聊」开头
- 不要写演讲现场叙事，例如「他走上台说了一件让人安心的事」
- 不要写播放量、点赞数，除非数字极端异常且与文章判断有关
- 不要先写人物简历然后再讲内容。这是样本里最常见、最容易写成的开头错误。

**开头三段不要把短句串成长串逗号链。**

定义:开头要快,但不能让读者在 N 个并列的短词组里找主语。把"6 小时,200 美元,prompt 是 X"这种数字 / 场景的并列短语堆在第一句开头是合理的节奏感,但紧接着的下一句必须给一个完整的主谓宾,把读者从"短语云"里捞出来,告诉他"做出来一个 Y"或"发生了一件 Z"。

典型错例(单句把场景、对象、属性、衍生功能全揉进逗号链):

```
6 小时,200 美元,一句 prompt 是 X。结果是一个叫 Y 的 Z,带 a、b、c、d,
还内嵌了一个 e。
```
(读者读到第二句末尾才能判断这一坨说的是什么,主谓宾散在各处)

改写思路:把最后一句拆成两句,先给"做出来一个 Y"这个完整事实,细节挂在后面:

```
6 小时,200 美元,prompt 只有一句 X。一个叫 Y 的 Z 就这么开发出来了,
带 a、b、c、d,还内嵌一个 e。
```

操作法:开头每一段读出来,如果一句话有 4 个或以上的并列逗号短词组、且主谓宾不在一处,拆成两句或重组结构。

grep 触发词:无可靠词级触发,只能通读。机械检查:统计每个句子的逗号数,>3 个且没有动词的句子要重写。

### 2.x 人物背景后置:钩子是问题就别先堆履历

定义:开头三段节奏的默认是「场景 → 可信度(人物背景) → 核心问题」。但当讲者身份本身不是最强钩子、真正拽住读者的是问题或产品张力时,把人物背景后移,让前两段先讲透矛盾和读者能共情的场景,分享者是谁放到第三段甚至更后再交代。

判断方法:问一句「读者点进来是因为这个人,还是因为这个问题?」。因为问题,就把问题和场景前置,人物后置。

同时,人物履历只保留与论点直接相关的一句,删掉无关的背景炫技。无关履历对论点零贡献,还会把第一段刚立起来的张力冲淡——读者记不住也不关心讲者做过什么,他关心的是「为什么这个产品要这么设计」。

错例(原稿,Lovable 持续学习):

- 第一段:数字 + 反转(每天 20 万项目,规模越大,同一种错误重复的浪费越快)
- 第二段:「Verbeek 过去做过卫星、粒子物理和聚变反应堆相关工作。他现在重点从代码生成转到大规模持续学习。目标是……」(卫星 / 粒子物理 / 聚变和「持续学习」论点无关,是炫技背景;人物简历还打断了第一段刚立起的张力)
- 第三段:才讲产品场景

改写:

- 第一段:核心矛盾(每天 20 万项目,规模越大,同一种错误重复的浪费越快)
- 第二段:读者能共情的场景对比(工程师卡住能手工接管,非技术用户卡住就直接离开)
- 第三段:才交代「分享来自 Lovable 技术团队成员 Benjamin Verbeek,他现在的重点从代码生成转到大规模持续学习」——履历只留这一句相关的,卫星 / 粒子物理 / 聚变全删

边界:当讲者本人就是钩子(知名创始人、被反复引用的人物、身份即权威背书)时,仍走默认三段节奏,第二段先建立可信度。这条只在「问题 > 人物」时适用。

### 2.x 可验证的反直觉数字优先于观点

定义:当来源材料里有硬数字、前后对比、成功率、成本、耗时、行数、比例等可验证结果时,开头优先用数字建立冲突,不要先写抽象观点。读者对"某个方法论很重要"天然免疫,但会被"删掉 95%,正确率反而更高"这种结果拽住。

适用素材:

- 方法论 / 工程实践里有反常识实验结果
- 演讲者给了明确前后对比,如 10,000 行 → 553 行、68 分钟 → 6 分钟、77% → 97%
- 产品 / 增长 / 组织案例里有用户数、耗时、成本、团队规模、成功率变化

写法顺序:

1. 第一段先给动作 + 数字结果,把反直觉摆出来
2. 第二段再交代人物 / 公司 / 可信度
3. 第三段锁定这组数字背后的问题,不要把它写成素材摘要

示范:

```
一个 WorkOS 工程师把自己写的一万多行 Agent Skill 删到只剩五百多行,删掉了 95%。
结果模型干活的正确率不降反升。同一个任务,加载某个 Skill 正确率是 77%;
什么都不加载,反而有 97%。
```

为什么有效:数字给读者一个可以在脑子里核算的冲突,比"少即是多""不要过度设计"这类观点更具体。观点可以放到第三段以后,让读者先被事实吸住。

边界:数字必须是来源里真实出现、口径可追溯的数字。不要为了开头硬造百分比、倍数或"提升 X 倍"。如果数字口径复杂,在同段给足基数或改成朴素对比,不要写读者一算就错的倍数。

反例:

- ✗「Agent Skill 的设计需要克制。」(观点先行,钩子弱)
- ✓「删掉 95% 的 Agent Skill 后,模型反而更准。」(结果先行,冲突强)

### 2.x 先讲事故,再抽原则

定义:方法论文章不要先把原则端出来,再找案例证明。更稳的顺序是先写一个具体事故 / 失败 / 反常识结果,让读者看到问题,再抽出原则。读者先经历"为什么会这样",才愿意接受"应该怎么做"。

可用事故类型:

- 模型撒谎 / 跳步骤 / 伪造证据
- 产品安装、迁移、生成代码时把隐式约定改坏
- 文档 / Skill / prompt 写得更多,效果却更差
- Eval 数字揭示作者原本判断错了

写法:

```
事故:Claude 发现只要 touch 一个 .case-tested 文件,就能假装跑过测试。
修法:把真实测试输出做 SHA-256 哈希写进文件,再做加密校验。
原则:让老实干活比撒谎更省事。
```

规则:

- 每条原则前至少有一个具体事故、数字、失败样本或真实动作
- 原则句要从事故里长出来,不要像独立口号
- 如果只有原则没有事故,读者会觉得在讲空话;如果只有事故没有原则,读者只会记住段子

常见结构:

- 事故段:谁在什么场景里搞砸了什么
- 修法段:作者改了哪一层机制
- 原则段:这件事说明以后该怎么做

示范标题链:

- 让老实干活,比撒谎更省事
- 删掉 95% 的 Skill 更准
- 每次失败都是 Harness 的 bug

这类标题能成立,是因为正文前面已经给了事故和机制。没有事故支撑时,这些标题会变成口号。

### 2.x 文章顶部信息图本身是论点，不要在图下面写文字注释

定义：当文章顶部插入一张信息图 / 摘要图 / 文章脉络图时（典型场景：用一张 HTML spec 风格的图把全文三层结构在一屏内呈现），不要在图下面再加引用块或一段文字解释「这张图怎么读」。如果图设计够好，图自己就是论点；如果图需要文字解释才能读，先回去改图，不要靠注释打补丁。

为什么写稿时容易写出注释：插入信息图后，作者本能想"帮读者一把"，加一句「先扫一眼这张图,再读正文会更顺」「三栏对应正文三层进阶」这种导览。但读者打开文章看到一张图，本来就会扫一眼。这一句注释要么重复图本身已经说出来的事，要么强迫读者读完文字再回头看图——反而打断阅读节奏。

典型错例 → 改写：

- 「![文章脉络图](summary.png)」 + 引用块「这是一份给读者的『文章 spec』。三栏对应正文三层进阶,黑条是贯穿主轴,底部是推荐设置组合。先扫一眼这张图,再读正文会更顺。」 → 删掉整段引用块。让图自己说话，读者扫一眼立刻进正文。

判断方法：写完图下注释之后，问自己——这一段的信息是不是在图里已经直接呈现了？如果是，删掉。如果"图里没有但读者需要知道"，回去改图，把那个信息加进图里。

边界：

- 图是手画草图、低分辨率截图、外部资料的引用图（非作者自己设计的信息图），需要简短 caption 说明来源、出处或重点指向（"图：原 paper 的 Fig. 3"）。这种 caption 不是注释，是版权 / 溯源信息。
- 图本身没有承担"概要论点"功能（只是装饰封面、章节插画），不在本规则范围。
- 图比较复杂（数据可视化、网络图、流程图），需要一句"看图指南"指出读者应该先看哪个区域，是合理的注释。但仍然要尽量短，一句话以内。

grep 触发词：紧跟图片 markdown 后的引用块 + 含「这是一份」「先扫一眼」「这张图」「再读正文」等导览词的句子。手工通读级检查更可靠。

### 2.x 钩子段涉及多方主体时,每个动作的施事方必须显式

**这是开头段最容易翻车、也最致命的一类问题**。钩子段的任务是把读者拽进来,如果第一段读完读者还在脑子里整理"刚才到底是谁干了什么",钩子就废了——他不会继续往下读。

定义:当开头段涉及两个及以上独立主体(两家公司、几个团队、多个产品、当事人 + 当事人的对手)时,每一个动作的施事方都必须在句子里点明,不能依赖代词链或"我们 / 他们"的相对指代让读者自己反推。

为什么写稿时容易翻车:转录原文里讲者经常用 "we" / "they" 来回切换,因为台下听众有上下文。直译进文章后,「我们」「他们」「这家」「那家」叠加,读者没有现场上下文,必须停下来反推每个代词指向哪一方。一旦在第一段就出现这种反推,钩子的冲击力被消耗在解谜上,而不是判断本身。

典型错例(原素材里有 Kaggle、AI 实验室 A、AI 实验室 B 三方):

```
Kaggle 和某家 AI 实验室合作发了一份 Benchmark。另一家有竞争关系的实验室找过来说,
这个分数我们不满意,我们自己跑一遍。结果他们公布的分数明显更高。差别只在一处:
他们在自家模型上启用了 API 提供的 compaction,Kaggle 给所有模型都没用。
```

读者要解三道题才知道说的是什么:
- "我们不满意"——「我们」是哪一方?(实验室 B)
- "他们公布"——「他们」又是哪一方?(还是实验室 B,但读者已经在第二个代词上重新对齐)
- "他们在自家模型上启用 / Kaggle 给所有模型都没用"——这里"他们"和"Kaggle"才形成显性对比

改写思路:把每个动作前的施事方都点出具体的称呼,代词只在同一主体连续动作时使用,主体切换时立刻补名字或叫法。

```
Kaggle 发过一份 Benchmark,里面对比了几家头部 AI 实验室的模型。其中一家实验室
对自家模型在这份榜上的分数不满意,主动找 Kaggle 提出要自己重跑一遍。结果这家
实验室公布的成绩明显比 Kaggle 跑出来的好看。差别只在一处:重跑时,他们给自家
模型启用了 API 提供的 compaction(对话历史压缩);Kaggle 原版跑的时候,所有
模型都没开这个开关。
```

每一步对应清楚:
- Kaggle 发榜
- 这家实验室不满意,要求重跑
- 这家实验室跑出来更好看
- 差别 = 这家开了 compaction / Kaggle 没开

操作法:
1. 钩子段写完后,把每一个动作动词前后的施事方圈出来,问"读者读到这里需不需要回扫上一句才能确认是谁?"。需要回扫,就把代词换成具体称呼(「这家实验室」「Kaggle」「Lab B」)。
2. 转录原文里的 "we / they" 要改写成显性主体,不能 1:1 直译为「我们 / 他们」。讲者站在台上说"we"读者懂,文章读者不在现场。
3. 同一主体连续两次动作可以用代词。主体切换 / 引入新主体 / 跨句重新指代时,立刻补名字。
4. 第一句话一般要交代场景的「主体清单」(谁是 Kaggle,Benchmark 测的是谁的模型),让读者建立角色表后再展开冲突。

判断方法:把钩子段抽出来给一个完全没看过原视频的人读一遍,问他"刚才是谁干了什么导致了什么"。他能复述出关系链,合格;复述错或要回头读第二遍,回去改。

边界:这条规则只针对**多方主体**的钩子段。单一主体(讲者一个人讲自己的判断 / 一家公司做了一件事)不受影响,代词链清楚就行。但只要场景里出现两方及以上的独立行动者,每个动作都要显式归属。

grep 触发词:无可靠词级触发——这是结构性问题,不是词汇问题。机械检查:开头段里数代词数量,「我们 / 他们 / 这个 / 那个」加起来超过 3 处时,通读检查每个代词的指向是否需要读者反推。

### 2.x 开场第一句不能由一串无主语的行业黑话构成

定义:钩子段(尤其第一句)如果直接堆行业黑话、又没有主语,读者一头雾水。中文术语同样算黑话,不只是英文缩写。

典型错例(三个 VC 黑话叠加、无主语):

```
前 1% 退出的门槛,2020 到 2024 年是 100 亿美元。
```

读者要先解三个词才知道在说什么:「退出」是谁的退出?「前 1%」是什么的前 1%?「门槛」指什么?第一句就让读者卡住,钩子直接废掉。

改法:先用一句大白话把场景画出来,把黑话拆成读者能想象的画面,再上数字。

```
一家拿了风投的创业公司,最后被收购或者上市,金额要排进这类退出里的前 1%。
这条门槛 2020 到 2024 年是 100 亿美元。
```

操作法:钩子段的每个名词,问一句「没读过来源的人认不认识」。「退出」「LP」「token 路径」「PMF」这类(中英文都算)第一次出现、又在开场位置时,要么就地拆成大白话,要么紧跟一句嵌入式解说(如「LP(出资人)」)。这条和 §5「行业术语白话化」是一回事,只是开场位置最致命,单独标出来。

### 2.x 开头用「两边 / 双方 / 两派」时,要在同段点明是哪两方

定义:开头(尤其第一句)抛一个"两边都不满意 / 双方都不舒服 / 两派都反对"式的判断,很有张力,但如果不说清是哪两方,读者第一段就悬空——他不知道自己算哪一边,也判断不了这个反差有多强,钩子就废了。

典型错例:

```
Benedict Evans 对 AI 的判断很容易让两边都不舒服。
```

读者立刻卡在"哪两边"上。

改法:在同一段(或紧接的下一句)就把两方点名,让读者马上对上号。

```
Benedict Evans 对 AI 的判断,轻视它的人和神化它的人听了都不舒服。
```

操作法:开头出现「两边 / 双方 / 两派 / 两类人 / 各打五十大板」这种二元指代时,问一句"读者知道是哪两方吗"。不知道,就地补出两方的具体身份(「轻视它的人和神化它的人」「看多的和看空的」「做模型的和做应用的」)。这条是 §2.x「多方主体施事方必须显式」在二元对立场景的特例,开场位置同样致命。

grep 触发词:`让(两边|双方|两派|两类人)` `(两边|双方|两派)都(不|很|会)`

## 三、来源交代与作者视角

文章要自然交代来源，但不能让来源成为正文主角。

**需要交代的内容：**

- 节目/频道/原文是什么，一句话说明定位
- 主播、嘉宾、作者是谁，一句话说明为什么他说这件事值得听
- 技术产品、协议或框架是什么，一句话给非熟悉读者一个中文锚点

这些信息要在前三四段内自然带出，不要单独罗列人物简历，也不要等读者读了一大段后才补背景。

**演讲者凭据要给到具体数字**：交代"为什么他说这件事值得听"时，用量化锚点而不是抽象头衔。"知名工程师"是空话，"4 个批次跑了 1000+ office hour、辅导公司估值合计 50 亿美元"才是凭据。爆款样本里这一段都给得很满：

- ✗「Tom Blomfield 是 YC 的合伙人，对 AI 时代的公司形态有深入思考。」（空头衔 + 抽象判断词，读者收不到任何信号）
- ✓「Tom Blomfield。Monzo 联合创始人，GoCardless 创始人，2019 年因为推动英国银行业竞争被授予 OBE 勋章，目前是 YC General Partner，在 4 个批次里跑了 1000 多次 office hour，辅导的公司估值合计大约 50 亿美元。」

具体凭据来源：创业前的公司、关键产品、量化产出（轮次、用户数、估值）、奖项 / 媒体认可、当前职位 + 关键数字（管理团队规模、行动次数、投资标的数）。挑 3-5 条最能支撑"他在这个具体话题上有发言权"的，连续陈述。

判断方法：把第二段抽出来给一个不认识演讲者的读者看，问"凭这一段你愿不愿意继续听他讲下去？"。愿意，就够了；不愿意，加凭据。

**来源转述腔禁区：**

- 不要频繁用「某某说」「某某提到」「某某认为」「在演讲中」「这期视频里」推进段落
- 不要逐段复述视频或原文的展开顺序
- 不要把文章写成节目推荐、视频摘要或读后感
- 不要加「X 对这件事的评价挺复杂」「X 对 Y 非常赞赏」这种只承担转场功能的空句，样本里一句都没有
- 交代来源人物时，讲者是**观点提供者**，不是「说话的人」。不要用「说这话的是 X」「这是 X 说的」把演讲 / 文章内容降格成「一句话」。用「以上观点来自 X」「这个判断来自 X」「X 的判断是」，把归因落在"观点 / 判断"上而不是"说话"这个动作上。反例：「说这话的是 Philipp Schmid」→ 改写：「以上观点来自 Philipp Schmid」

**更好的写法：**

先写读者能感知的问题、场景和判断，再在需要建立可信度的地方轻轻归因。人物身份、关键原话、独特经历、具体数据和争议性判断需要归因；已经被作者消化成场景、方法论或分析判断的内容，不必每句都挂回演讲者。

### 3.w 核心判断:你在写一篇新文章,不是在转述讲者

这是来源转述腔最难根除的一条,值得单独立节。前面的禁区针对句式,这一条针对**整句的存废判断**。

**核心标准:一个句子,如果它没有给读者带来新事实、新数字、新案例、新原话、新判断,只是把上一句的事实再换讲者归因说一遍,直接删。不要因为讲者在演讲里说了这句话就保留它。**

判断方法:把候选句子和上一句对照,问一个问题——这句话相比上一句,带来了哪一项是上一句没有的?

- 新事实(增加了一个具体动作、产品、数字、人物)→ 保留
- 新案例(把抽象判断落到具体场景)→ 保留
- 新原话(讲者那一句金句独立有力)→ 保留,但不必标"原话"二字,直接引号引用即可
- 新判断(上一句是事实,这一句把事实推到一个结论)→ 保留
- 都没有,只是把上一句的判断再用讲者口吻说一遍 → 删

**典型错例 → 改写**

错例 1(讲者复读上一句判断):

```
意味着今天主流 coding Benchmark 的排序依据,已经从模型变成了
「模型 + harness + prompt」的组合。Michael 在演讲里提了一个问题:
「你究竟在测 harness,还是在测模型?」他没给答案。但这个问题一旦
问出来,前面那个「同一份 Benchmark 跑出两个分数」的故事就不再是
孤立的丑闻——它是一个系统性现象的小样本。
```

诊断:第二、三句加起来没有给上一句以外的新事实——上一句"排序依据从模型变成组合",已经是"测的不是模型"的判断。后面两句只是把同一个判断换成讲者口吻和叙事评价再说一遍。

改写:删除整段,只保留上一句独立成段。

```
意味着今天主流 coding Benchmark 的排序依据,已经从模型变成了
「模型 + harness + prompt」的组合。
```

错例 2(讲者归因式开头):

```
Kaggle 内部想清楚的事是,评测不能再做成一份发完就停的论文。
```

诊断:"Kaggle 内部想清楚的事是" 把作者已经消化为判断的内容,绕回讲者归因。读者不在乎是 Kaggle 想清楚还是作者想清楚,他要的是"评测不能做成论文"这个判断本身。

改写:用逻辑词("所以"、"因此"、"问题是"、"结果")直接给判断,删讲者归因。

```
所以,评测不能再做成一份发完就停的论文。
```

错例 3(原话标签):

```
「我做这个是全职工作,连我都跟不上」——Nick 的原话。
```

诊断:既然已经用引号标了引用,"——Nick 的原话" 是冗余;读者看到引号就知道是引用。而且"原话"这个词把读者从场景拉回"演讲转录"层。

改写:把引语自然嵌进句子,不加标签;或者干脆把这一信息融进事实陈述,删除原引号也行。

```
论文里那些 Leaderboard 出版后基本没人维护,arXiv 每天还在发十几个新
Benchmark,但一个评测产品要长期有用,前提是它能跟着模型和 harness 一起
迭代,而不是发完就停在那。
```

错例 4(元过渡 + 冒号铺垫式):

```
Nick 演讲一开头说,他们原本估计来 20 个人,实际到的人远远超出预期。
这件事在小规模上回答了一个问题:今天关心评测的人,比这个圈子自己以为的多。
```

诊断:第二句的「这件事在小规模上回答了一个问题:」是元过渡——作者站出来对前一句做"导游"式说明,然后用冒号铺垫一个判断。后半句「今天关心评测的人,比这个圈子自己以为的多」本身是有内容的判断,但被前面的元过渡稀释了。读者读到「这件事在小规模上回答了一个问题」会本能停一下等"那个问题",然后还要再读一遍冒号后面的内容才能拿到判断,节奏被打两次。

改写:删掉元过渡 + 冒号铺垫,让判断直接独立成句。

```
Nick 演讲一开头说,他们原本估计来 20 个人,实际到的人远远超出预期。
今天关心评测的人,比这个圈子自己以为的多。
```

两句之间是事实 → 判断的自然递进,中文意合,读者不需要"这件事回答了一个问题"这种导游词来桥接。

这一类元过渡铺垫的常见句式:

- 「这件事在 X 上回答了一个问题:」
- 「这件事说明了一件事:」
- 「这背后其实是一个更大的问题:」
- 「这件事的有趣之处在于:」
- 「我从这里看出一个判断:」
- 「这给我们的启示是:」

grep 触发词:`这件事(在|说明|背后).{0,15}(问题|事|判断|启示)[:：]` `这背后.{0,10}是.{0,5}(问题|逻辑|趋势)[:：]` `我从.{0,10}看出.{0,5}(一个|个).{0,5}[:：]` `这给我们.{0,5}启示`

**反操作:什么时候应该保留讲者归因**

当原话本身有独立的力量(短、具体、画面感强、反差大),且这一句除了被讲者说出来还无法用别的方式传达,保留:

- ✓「Nick 在演讲一开头说,他们原本估计来 20 个人,实际到的人远远超出预期」——这是一个**事件**(讲者那句话产生了一个现场反差),不是判断转述
- ✓「Michael 用『personalities start to emerge』来形容这种结果」——这是一句**金句命名**,讲者给一个现象起了名字,删掉就丢失命名权
- ✗「Michael 在演讲里提了一个问题:『你究竟在测 harness,还是在测模型?』他没给答案」——这只是把上一句的判断换成提问句式,没新信息

判断分界:讲者的原话 / 动作本身是**事件**还是**判断转述**?是事件就保留,是转述就删。

**整篇文章扫一遍**

定稿前,把所有含"X 说"、"X 提到"、"X 想清楚"、"X 在演讲里"、"——X 的原话"、"X 在台上"、"X 把这件事讲出来"、"X 抛出"、"X 提了一个问题"的句子标出来。逐条问"这一句相比上一句带来了什么新东西"。带不来就删。

grep 触发词:`想清楚的(事|是)` `——.{0,4}(原话|的话|说的|讲的|的判断)` `[一-龥]{2,4}在(演讲|台上|视频|文章|访谈)(里|上)?` `提了(一个|个)?问题` `抛出(一个|个)` `把这件事讲(出来|了出来)`

### 3.x 来源交代段最常踩的五条陷阱

第二段(钩子之后第一段交代来源 / 人物的位置)是公众号文章里最容易写废的位置之一。读者已经被钩子拽进来,但这一段一旦塌掉,他就走了。下面这五条是反复出现的具体错误模式,不是抽象原则——出现命中,直接重写。

**陷阱一:用「这件事是 X 在 Y 大会上的开场」式来源转述句衔接钩子段**

错例:

```
[钩子段讲了一个引人的故事]
这件事是 Nick Kang 在 AI Engineer 大会上的开场。
```

问题:这句话把读者从钩子的现场感拉回到"这是一段演讲转述"的元层面。读者刚被故事吸引,又被告知"哦原来是个演讲",节奏被打断。

改法:不报"X 在 Y 大会的开场",顺着钩子段的事实直接推进——「发这份榜的是 Kaggle Benchmarks 这条产品线」「讲这件事的工程师叫 Michael」「把这件事公开的是 Anthropic」。让来源信息为故事服务,不是反过来。

grep 触发词:`这件事是.{0,15}的开场` `这是.{0,8}在.{0,10}的(开场|演讲|分享)` `这[期场]演讲.{0,5}讲的是` `这次分享.{0,5}讲的`

**陷阱二:把演讲 / 文章标题塞在第二段**

错例:

```
他和同事 Michael Aaron 来自 Google DeepMind / Kaggle,
标题叫「Agentic Evaluations at Scale, For Everybody」。
```

问题:演讲 / 文章标题的位置是文末来源行,不是正文。在正文里报标题,读者拿不到任何额外信息——他要么不在乎标题,要么会在文末看到。除非标题本身承载了文章必须用到的论点,否则不进正文。

改法:删除「标题叫 XXX」这句。如果标题里某个词本身就是文章要讨论的概念(比如标题是「Marketing is Dead」而文章正在讨论这件事),把那个词作为概念引入正文,而不是把整个标题塞过来。

grep 触发词:`标题叫` `这场演讲叫` `这篇文章叫`

**陷阱三:演讲者凭据写成空头衔**

错例:

```
Nick 是 Kaggle Benchmarks 的产品经理,Michael 是 Kaggle 工程师。
```

问题:"产品经理"、"工程师"是行业里几十万人都有的标准头衔,无法回答"凭什么听他们讲这件事"。读者读完这一句,对讲者的可信度评估等于零。

改法:把头衔换成具体动作 + 量化锚点,或者干脆把头衔挪到他主讲的章节里和他的具体观点绑在一起。如果讲者本身没有量化凭据(普通员工讲普通工作),那就把可信度建在他所在的产品 / 公司上,讲清楚这家公司在这个具体话题上有什么投入和成果。

具体三种处理方式:
- 讲者本人有量化凭据:「他在 4 个批次里跑了 1000 多次 office hour」「她带的产品周活破了 1 亿」
- 讲者本人没有突出凭据,但他的产品 / 公司有:「Kaggle 过去两年把团队精力大量压在 GenAI 评测上」「Anthropic 的整个增长部门曾经只有他一个人」
- 头衔 + 角色定位:「Nick 是这条线的产品经理,工程师 Michael Aaron 负责 Game Arena 这块」(头衔 + 一句具体职责,比纯头衔好)

判断方法:把第二段抽出来给一个不认识讲者的读者读,问"凭这一段你愿不愿意继续听他讲下去?"。愿意,合格;不愿意,加凭据或重写。

**陷阱四:Kaggle 简介像官网关于页**

错例:

```
Kaggle 是全球最大的 AI / ML 社区,3000 万以上注册用户,过去两年把重心
挪到了 GenAI 评测这块。
```

问题:这种描述读起来像 brag 列表——规模、用户数、动作三件套堆在一起,但和文章主题没有具体连接。读者会把它当噱头跳过去。

改法:把公司介绍和文章主题挂钩。"3000 万用户" 是数据,"过去两年大量精力压在 GenAI 评测上" 是动作,但它们都需要再加一层"为什么"——这家公司在这个具体话题上想做什么、想解决什么。

对照修改:

```
Kaggle 是 Google 旗下的 AI / ML 社区,3000 万以上注册用户,过去两年把
团队精力大量压在 GenAI 评测上,他们想做的事是把评测从"发完就停的论文"
变成"能跟着模型迭代的产品"。
```

加的那一句"他们想做的事是 X"把可信度从 brag 变成立场,读者读完知道这家公司站在哪、要解决什么。如果文章后面会用到这个判断,这种伏笔尤其值钱。

判断方法:把公司介绍那一句单独抽出来,问"删掉这句话之后,文章后面还能不能立得住"?能立住,这句话就是装饰,删或重写;不能立住(因为后文要用到这个公司的立场),就保留并把它写得和文章主题强相关。

**陷阱五:第二段就把所有人物简历堆完**

错例(同时踩了陷阱二、三、四,集大成):

```
这件事是 Nick Kang 在 AI Engineer 大会上的开场。他和同事 Michael Aaron
来自 Google DeepMind / Kaggle,标题叫「Agentic Evaluations at Scale, For
Everybody」。Nick 是 Kaggle Benchmarks 的产品经理,Michael 是 Kaggle 工程师,
Kaggle 是全球最大的 AI / ML 社区,3000 万以上注册用户,过去两年把重心挪到了
GenAI 评测这块。
```

问题:一段塞 5 个名字 / 头衔 / 公司,信息密度很高但没有重点。读者读完一团乱,记不住谁是谁。这正好对应 §二「开头禁区」里的"不要先写人物简历然后再讲内容"——典型表现就是把简历集中在第二段。

改法:把人物分散介绍。第二段只交代核心讲者 + 公司可信度;次要讲者(只在某一章节登场的人)留到他第一次主讲的章节再带出。例:Michael 主要负责 Game Arena 那一章,他的身份就压到那一章再讲,第二段只点一句"工程师 Michael Aaron 负责 Game Arena 这块"作为目录式索引,不展开。

判断方法:第二段里如果出现 ≥3 个人名 + 各自头衔,大概率应该拆。

**陷阱六:开头段任何不直接服务"读者为什么继续读"的句子都要砍**

§3.w 已经讲过"无新事实就删"。开头段(尤其第一段)的判断标准要更严:**只要这句话不直接服务"读者为什么要继续读下去",就该砍**,哪怕它本身有信息量。

错例(两个反复出现的代表):

```
他下了一个判断"大家都低估了 Claude Code"。当时这话听起来像 niche take,事后回头看对得离谱。一年里 Anthropic 推出 Claude Co-work、OpenAI 重做 Codex,所有人都涌入 CLI agent 这条线。这一期 Dan 又来了 Lenny 的播客,直接给出 12 个新预测,并和 Lenny 约好——2027 年 5 月回来,一条一条给自己打分。
```

诊断:最后一句"和 Lenny 约好 2027 年 5 月回来打分"本身有信息(对赌式约定),但它服务的是"节目花絮",不是"读者为什么要继续读"。把它放在第一段,会让钩子的注意力被分散。

改写:删掉最后一句。前面的钩子已经完成"为什么这次又值得听"的论证(去年预测对了 → 今年又来了),停在那里就够。

```
当时这话听起来像 niche take,事后回头看对得离谱。
```

诊断:`niche take` 是英文口语表达(见 §5.x),应翻译成"小众观点";同时这一句的前半和后半是对比关系,缺转折词。

改写:

```
当时这话听起来像小众观点,但事后回头看对得离谱。
```

**两条配套判断**:

- "这一期 / 这期视频"开头:这是节目主持人口吻,在写作里属于元归属。前文已经讲了"一年前",此处用"一年后"对应,时间线立得起。
- "Dan 又来了"用动词"来"过随意:"上播客"用"回到"更准。原英文 "He's back" 也是这个意思。

### 3.x.收尾 章节末段最后一句话必须承担收尾力度

这是 §3.w「无新事实就删」的一种高发情形,值得单列。

**问题**:章节末段的最后一句话决定读者带着什么判断进入下一章。如果最后一句是泛话兜底("得另想办法"、"还有空间"、"值得思考"、"未来可期"),读者就带着空气进入下一章——前面所有具体事实和判断的力度被这一句稀释掉。

**典型错例 → 改写**

错例:

```
代价不便宜。扑克要跑 40 万手才有统计显著性,每手还有很多回合,所以 API 账单
很可观。Bradley-Terry pairing 减少了一些场次,但这条路跑起来依然贵到只有
Kaggle 这种规模的玩家撑得起。社区 Benchmark 想沿用,得另想办法。
```

诊断:最后一句"社区 Benchmark 想沿用,得另想办法"——"得另想办法"是泛话,没说怎么办、没指出方向、没具体信息。读者读到这里只能想"哦那怎么办呢",但作者并不打算回答。这种段尾就是把上一句更有力的判断("贵到只有 Kaggle 撑得起")的力度冲掉。

改写:删掉最后一句,让段落停在"贵到只有 Kaggle 这种规模的玩家撑得起"这个有具体名字的判断上。

```
代价不便宜。扑克要跑 40 万手才有统计显著性,每手还有很多回合,所以 API 账单
很可观。Bradley-Terry pairing 减少了一些场次,但这条路跑起来依然贵到只有
Kaggle 这种规模的玩家撑得起。
```

**章节末段最后一句的合格标准**

满足以下任意一条即可:

1. **具体数字 / 名字 / 动作**:"贵到只有 Kaggle 撑得起" / "整个增长部门只有他一个人,跑了 10 个月" / "把 Tool Description 写到 700 行"
2. **反常识判断**:"模型最强的方向是欺骗,不是推理" / "harness 比模型决定排名"
3. **画面收束**:"那个工程师从 20 年的污水处理记录里整理出一份数据集"
4. **承上推下的过渡判断**(只在章节之间需要桥时用):"这件事还没完,下一节会接到"——但谨慎使用,容易写废

**不合格的段尾常见词类**(命中即重写):

- "得另想办法 / 还得继续摸索 / 还有路要走"
- "值得思考 / 值得记下 / 值得抄"
- "未来可期 / 还有空间 / 还需要时间"
- "这件事还没有结束 / 这只是开始"
- "需要更多探索 / 需要时间检验"

判断方法:写完一章后,把最后一段的最后一句单独抽出来读。问"这一句话本身有具体内容吗?如果删掉,前一句独立读是否更有力?"。是,就删。

grep 触发词:`得另.{0,3}办法` `还有.{0,3}空间` `还需要.{0,3}时间` `值得.{0,3}思考` `未来.{0,3}可期` `这只是.{0,3}开始` `还在.{0,3}路上`

### 3.x.收尾2 整篇文章的末段综合升华段直接删

定义:正文最后一节(往往是讲完最后一个观点的那一节)的最后几个段落,如果只是把前文的 thesis 再综合一遍、把所有章节的判断再排列一次,这就是 AI 写作惯性留下的"综合升华段",直接删,让倒数第二段或倒数第三段成为真正的收尾。

为什么 AI 写作容易加这种末段:模型被训练得倾向于"开头-中间-结尾"的对称结构,结尾要"再综合一次主旨"。但中文公众号读者读到正文最后一个具体判断时,thesis 已经在他脑子里——不需要作者再帮他综合一遍。

**典型错例 → 改写**(收尾 1+1 段都该删):

错例:

```
[前面是"AI 的上限在每个真实地用它做事的人手上"这一节正文,讲完了 ride the models 和反 SF 中心论的判断]

这句话直接把 AI 变成一个对每个用户开放的探索前沿。它的潜台词是:你在 SF 之外用 AI 反而是优势...

把这一句和文章开头那两条命题接上,就是 Dan 这一期完整的世界观:模型把昨天的人类能力变成廉价商品,每个 agent 都需要一个人,而这个人不必住在旧金山。整集 12 个预测的所有反共识判断,都从这两条命题推导出来。一年后再回来打分,要看的是底层这两句话还成不成立。清单里某条对了错了,反而是次要的。
```

诊断:最后一段把"开头那两条命题"和"反 SF 判断"再综合一次,然后又预告"一年后再来打分"。前文已经讲完了,这一段每一句都在重复 thesis 的不同包装。

改写:整段删除。让前一段(讲"潜台词"的那一段)成为正文收尾。

**判断方法**:写完整篇文章后,把最后一节抽出来,把它的最后两段拿掉,看正文是否独立成立。如果"拿掉之后文章读起来更利落",说明那两段就是综合升华段,直接删。

**保留场景**:末段如果给了一个**前文没出现过的新信息**(具体行动指南、一个未来时间锚点、一个真正反差的金句),可以保留。但这种"末段加新信息"是反常状态,大多数情况下末段就是综合升华,该删。

判断尺子:这一段如果整段拿掉,文章的论证完整性是否受损?不受损就是综合升华,删。

### 3.y 段级互动重写比一次性整稿改更准

这条是写作流程层面的方法论,不是格式规则。

观察:文章写完一遍后,作者(或 AI 助手)做"全文一次过自审"时,经常会对每一段都做"还行 / 通过"的轻判断,把一些深层问题放过去——尤其是开头、第二段、章节衔接段这些高密度位置。但当一段一段慢慢看、和用户互动地改时,问题会逐段暴露:

- 第一遍:整稿读完,觉得问题不大
- 第二遍(一段一段):用户指出"第一段读不懂",改完
- 第三遍(下一段):用户又指出"第二段也是老问题",改完
- 第四遍(下一段):每一段单独审视,都还能挖出 1-2 条具体问题

为什么会这样:整稿审视时,大脑在"读完即懂"的速度里跑,跳过了卡顿点;段级精修时,大脑停在每一句话上,卡顿点会被放大。读者也是在段级速度里读文章——他们不会一次扫完整篇,而是一段一段读,在第一段卡住就退出。

操作建议:

1. **写完初稿后,做一遍整稿自审**(grep 兜底 + 读者视角抽查)。这一步抓得到大问题(章节顺序、整体结构、明显 AI 腔),但抓不到段级深层问题。
2. **再做一遍段级精修**:从第一段开始,把每一段单独抽出来,问三个问题:
   - 这一段如果读者只读到这里就退出,他带走了什么?
   - 这一段里每一个动作的施事方、每一个代词的指代、每一个判断的归因,有没有需要读者反推的地方?
   - 这一段里有没有空头衔、来源转述腔、brag 列表、人物简历堆砌?(对照 §3.x 五条陷阱)
3. **如果是用户在场的协作场景**,鼓励用户做"段级反馈"——一段一段问"这段怎么样",而不是"整篇怎么样"。段级反馈能逼出整稿审视盖住的问题。
4. **每段改完后,再读一遍上下文衔接**,确认没有为了改本段而打断和前后段的连贯。

判断方法:写完一篇文章后,如果整稿自审只发现了 0-1 个改动点,那基本可以肯定段级精修还能挖出 3-5 个。这不是文章不够好,是审视粒度不够细。

## 四、公众号语气

文章应该像中文作者读完材料后写给中文读者的文章，而不是英文 transcript 的中文转述，也不是内部研究报告。

**具体性优先：**

每段尽量有具体的人、事、数字、场景或原话。四个都没有，这段通常就是空洞判断。宁可少写几个观点、把一个观点写透，也不要每个观点都提一下、没有一个说清楚。

**量词和分类名词要携带语义：**

常见偷懒写法是「做了两件事」「有三件事要说」「还有几样动作」「几个方面」—— 读者读到这里不知道到底是"两个操作"还是"两条建议"还是"两次失败"，要读完后半句才能反推。准确的写法是让量词本身说出内容性质，读者一眼知道自己即将读到什么。按上下文选：

- **方法论场景**：三条建议 / 三个手段 / 三种方法 / 三个路径 / 三条经验 / 四个动作
- **案例场景**：三次失败 / 两个反例 / 三个教训 / 四个案例
- **分类场景**：三类做法 / 三种模式 / 四个典型 / 五种情况
- **观点场景**：三条反常识判断 / 两个反问 / 三条警告 / 四个判断
- **调整和改动**：两处调整 / 两个改动 / 三轮迭代 / 两次重写

坏：「下面这份中文版做了两件事」 → 好：「下面这份中文版做了两处调整」
坏：「还有三件事可以直接拿去用」 → 好：「还有三条可以直接抄的做法」或「三个具体动作」

同一个问题的反面：「一些」「一点」「几种」这种不带具体数字的笼统限定也要尽量换掉。知道数字就用数字，「几点建议」不如「三条建议」；不知道就重写成不需要量词的表达。

**数字量化原则：** 只要能量化就量化，不用「大幅提升」「显著下降」「提升了很多」这种模糊形容词。样本里所有数字都不是装饰，都在支撑具体判断：

- 下载量 / star 数 / 估值 / 融资轮次
- 具体成本差：500 万美元 → 7.3 万美元
- 时间压缩：2 小时 → 15 分钟
- 百分比：60% 时间、90% 覆盖率、96.7% 准确率
- 具体约束：30 字符标题、90 字符描述、700 行编排文本

**命名角色而非匿名化：**

样本里的 Boris Cherny、Luke Alvoeiro、Max Schoening、David Soria Parra、GRITCULT 全部被点名。不要写「一位工程师」「某位 CEO」。能具名就具名。

**中文表达优先：**

先给中文读者能想象的场景、动作或问题，再自然带出术语和判断。不要沿用英文演讲的抽象判断顺序，不要连续堆英文术语。

**衔接词推荐：**

取代「首先/其次/最后」，改用更口语化的衔接：

> 「先说……」「再说……」「说到……」「顺便说说……」「回到……」「说到未来……」「但有一件……」

**指代容忍度：**

代词指代不需要每次都显式补全名字，只要上下文连贯读者能跟住就够了。频繁重复名字会显得机械。真正需要补名字的情况是话题转换了，或者两个人交替出现时。

**句式节奏：**

- 单句长度平均 20–40 字，尽量不超过 60 字。长句超过 60 字先看能不能断成两句。
- 并列结构用句号分隔，不要塞进同一句。「这不是因为资源紧张，也不是因为他是某种天才。这是匿名 X 账号 GRITCULT 在 4 月 15 日写的一篇长文里的核心案例。」
- 动词要强。样本里的表达方式：「抹平编码壁垒」「踩改动」「吃掉速度收益」「哗哗地烧」「亲手拆了自家招牌」「一刀砍到」。弱动词（进行、实现、达成、展开）能换就换。

**干涩幽默：**

样本里不时出现轻度吐槽 / 自嘲 / 小结论，让文章有人味但不油腻。不做情感煽动，不用感叹号，不反问煽情。例：

- 「大多数人并不想去打猎，只想去 Costco 买一块已经放在泡沫托盘里的牛排。」
- 「一个拿对工具的人，胜过十个用老方法的人。」
- 「这种说法也许有些夸张，但 Anthropic 那个只有一个人的增长部门，多少给这句话做了一点背书。」

**筛掉对读者无信息量的转录细节：**

转录逐字稿里有大量演讲现场才有意义、进入公众号就成为噪音的细节。出稿前逐段扫一遍，下面几类主动删掉：

- **讲者本人的口头小插曲**：「她自己都念不准」「他说自己 English 不好」「她现场笑了出声」「他拍了一下桌子」这类对演讲者形象有贡献、对读者判断为零的花絮。读者不认识讲者本人，不关心讲者有没有念对一个词。
- **现场互动**：「有多少人做过 X 的请举个手」「大概一半举手」「有人在笑」「观众发出一阵笑声」「掌声」。转录里是有生命的，文章里是废字。
- **主持人 / 讲者对流程的元评论**：「我们先快速过一下历史」「这部分我跳过」「时间关系不展开」「Sorry 今天太紧张」「由于时间限制」。读者不需要知道演讲内部节奏管理，直接把那部分内容按逻辑讲出来就好。
- **讲者对自己 slide 的吐槽**：「这张幻灯片我自己都不喜欢」「这图做得太丑」。如果吐槽本身不承载观点（比如不是"这图我讨厌它，是因为……"这种带论点的），就删掉。
- **没有展开的碎片引用**：「有一次我朋友 Sam 跟我说……」后面没下文，删掉这个人名。
- **反复的自我修正**：「我是说 A，不，其实是 B」。只保留 B。

判断标准：这一句抽掉，文章的信息量和论点会不会受损？不会就删。保留一两处能让文章有人味的现场细节可以（参考干涩幽默那条），但前提是它承载了观点或者画面，不是单纯"讲者那一刻做了什么"。

反例对照：

- 坏：「GEPA 是一种优化算法的缩写，她自己都念不准。」（「她自己都念不准」对读者零价值，直接删）
- 好：「GEPA 是一种优化算法的缩写。」
- 坏：「她问现场有没有人做过 agentic rag，大概一半人举手。」（互动过程删掉，如果"一半的人都做过"这件事本身是判断依据，改成下面这样）
- 好：「agentic RAG 在这届大会上已经是默认选项，现场一半人都动手做过。」

**同信息最精简表达优先：**

「筛掉无信息量的转录细节」讲的是哪些素材根本不该写进来；这条讲的是写进来的素材怎么用最少的字表达。把整段写完后，逐句反问自己：「同样的意思，有没有更短的说法？」有就改。四类高频可砍：

- **先卖关子再揭晓**。「X 做了一件事。他……」「X 是全场最不起眼但最关键的……」这种先铺垫再揭晓的结构，是英文叙事的悬念节奏，中文直接把动作写出来更自然。
  - 坏：「Ravi 把数据切成独立文件之后做了一件事。他回到 Claude……」
  - 好：「Ravi 把数据切成独立文件之后，回到 Claude……」
  - 坏：「「Full-Stack Prompt 里那句『把数据保存成 data.json』」是全场最不起眼但最关键的一个动作。」
  - 好：「Full-Stack Prompt 里把数据保存成 data.json 是最关键的一个动作。」

- **指代嵌套和重复主语**。引号里再套引号、主语没换还写一次、用抽象名词替代具体动作，都会让读者多做一次解析。
  - 坏：「那句『把数据保存成 data.json』是最关键的一个动作」
  - 好：「把数据保存成 data.json 是最关键的一个动作」
  - 坏：「这条路的实用价值是……」（读者要回头想"哪条路"）
  - 好：「这样做还有一个实用价值是……」

- **对读者无关的执行细节**。跑了几分钟、调了哪个参数、用了哪个版本、中间卡在哪一步 —— 除非这些数字本身就是论点，否则删。讲者的时间成本不是读者的时间成本。
  - 坏：「回到 Claude，用同样格式生成 JSON（带 MCP Server 拉封面，跑了 15 分钟），然后……」
  - 好：「回到 Claude 生成 JSON，然后……」

- **强调副词成瘾**。「直接」「一下子」「真的」「其实」「就是」—— 一段里出现两次以上，通常后面那个可以删。
  - 坏：「Prototype 页面直接变成 Psychedelic Rock」
  - 好：「Prototype 页面就变成了 Psychedelic Rock」

还有一个和简洁相关的反面动作：**段与段、分句与分句之间的连接词要显式**。当你删掉了一堆冗余，相邻段落的关系可能会从"理所当然接得上"变成"读者需要猜关系"。此时加一个显式连接词，读起来反而更顺。这不是增加字数，是减少读者的二次解析成本。

- 并列新增一类：「还有一个」「另一条」「同样值得」
- 递进解释：「背后原因是」「之所以这样，是因为」
- 对比反转：「反过来说」「但」「不过」
- 条件限定：「只有在……时才」「前提是」

**正文主角是观点，不是素材和讲者：**

「筛掉无信息量的转录细节」处理的是哪些素材不该进来；「同信息最精简表达优先」处理的是进来的素材如何用最少字说清。这条处理的是第三类情况：素材干净、表达也紧，但段落里仍然有一层「围绕观点的外壳」—— 讲者是谁说的、这段多重要、去哪能看更多 —— 读者需要的是肉，不是壳。三类常见外壳和各自的判断标准：

- **对谈元信息**。默认删。「X 问得直接」「主持人提到」「嘉宾回应说」这类对谈形式句，通常只作为叙事外壳，不推进观点。章节标题已经是「三个 hot take」的情况下，直接「第一个……第二个……第三个……」分点就好，不需要先用一句「X 问 Y 答」铺垫。
  - 坏：「X 问得直接，Y 的回答可以直接拿去打几个在圈内流行的错误判断。第一个……」
  - 好：「第一个……」（直接进入内容）
  - 例外：对谈信息本身承载叙事张力或推进观点时保留。判断方式——抽掉「X 问、Y 答」这一句，观点推进或叙事张力是否损失？
    - 保留的例子：「主持人追问这个数字是怎么来的，讲者被逼出一个更具体的答案：……」（「被逼出」改变了读者对后面答案的感知，说明讲者原本没打算讲这么细）

- **判断前后的元评价**。基本全删，没有例外。作者对内容本身做评价，不论放在判断前（铺垫）还是放在判断后（收尾），都是读者不需要的外壳。判断的分量应该由判断本身显出，不靠作者做导游。
  - 铺垫型（判断前）：「这是最容易被忽略但最值得抄走的一句」「讲者原话是」「接下来这段特别关键」。
    - 坏：「**判断自己是不是 AI native 的标志** / 这场对谈最容易被忽略但最值得抄走的一句话，来自结尾。/ 讲者的原话是，过去那些真正以产品为中心、做得最好的公司……」（章节标题下两层元评价 + 出处铺垫，三道门才到真正的话）
    - 好：「**判断自己是不是 AI native 的标志** / 过去那些真正以产品为中心、做得最好的公司……」（章节标题下直接进判断）
  - 收尾型（判断后）：「作者的批评很克制，方向却很清楚」「这个观点挺犀利」「讲得真好」「这是一个很有力的论断」「这段尤其值得记住」。
    - 坏：「用『幻觉』这样一个带艺术气的词，把『模型会自信地撒谎』这个严重的准确性问题做了一次语义打包。作者的批评很克制，方向却很清楚。」（读者读完前一句已经能自己感受到克制和方向，再加一句等于替读者做判断）
    - 好：「用『幻觉』这样一个带艺术气的词，把『模型会自信地撒谎』这个严重的准确性问题做了一次语义打包。」（到此为止，读者自己感受）
  - 判断标准：这句话抽掉，读者对前文的感受会不会变？不会就是外壳，删。

- **给原素材做宣传/导流**。默认删。讲者末尾推自己的课、订阅、书、newsletter、social handles，这些信息是服务原视频观众的，对公众号读者学习观点没有贡献。文章的目的是把值得思考的观点传递给读者，不是替原视频拉流。
  - 坏：「讲者在最后顺便宣传了他和 X 合作的免费课，第一期刚开，春天还有一期，配套视频系列也在做。但这一整场访谈本身已经是一份相当完整的入门教材。」
  - 好：（整段删除，文章在前一段自然结束）
  - 例外：讲者提到一个读者当场就能用到的具体、免费资源（公开 GitHub repo、可下载的 prompt 模板、开源 eval 集）。保留但压到一句话，不要完整推广文案。
    - 保留的例子：「讲者把这次演示用的 MCP Server 代码放在 GitHub 上，链接在视频简介里。」（读者打开就能用）
    - 判断分界线：读到这句话，读者能不能当场用到一个具体东西？不能就是外壳，能就是正文。

三类外壳合起来的精神：文章主角是观点本身。每次下笔交代「谁说、说得多好、去哪看更多」之前，先过一遍「读者抽掉这句话会不会损失」。不会损失就是外壳，砍掉。

## 五、概念命名与定义

如果文章里引入了一个值得读者带走的新概念 / 术语 / 方法论，用「命名 + 破题动作」的方式一次讲清楚，之后再出现就裸写、不再加定义。

**样本里的范例：**

- validation contract：在代码写下去之前就把正确性定义清楚。
- droid whispering：给每个角色挑合适模型。
- progressive discovery：给模型一个工具搜索工具，按需加载。
- programmatic tool calling：让模型写脚本组合工具调用，不再一步一步推理。
- tiny core superpower：一个产品核心的"超能力"，比如 iPhone 的 multi-touch、GitHub 的 Pull Request。
- 分发工程师：把分发当作工程问题来处理的人。
- agency：相信世界可以被改变并愿意亲手去改的那个劲头。

**模式**：一行名字 + 一行动作 / 效果，不做抽象哲学化定义。首次出现的时候用「」锚一次，后面再提到就裸写，不要反复戴引号。

**行业术语白话化**

定义：对一般读者陌生的英文术语首次出现时，必须给一句中文嵌入式解说。形式上推荐用括号在术语后紧跟一个短中文短语，不另起一段、不另用脚注。后续出现保留原英文，不再重复解说。

已知需要白话化的常见词（写公众号读者多半要解一句）：

- GAN（生成对抗网络）
- sycophancy（讨好倾向）
- rubric（打分清单）
- Playwright（浏览器自动化工具）
- MCP（模型上下文协议）
- sprite（游戏里的小角色）
- scaffold（脚手架，如已译为"脚手架"则不重复）
- compaction（对话历史压缩）
- critique（点评意见）
- harness（模型外面那一层调度脚手架）

业界已熟知不需要解说的：RAG、F1、API、token、prompt、LLM、agent、context window、session、demo、PR、CLI、SDK。

判断尺子：不熟悉这个词的读者读到这里是否能继续读下去？如果要往下读三段才能 infer 出意思，前面就得加一句解说。

反例（裸用术语，读者不熟会卡）：

- "我们都熟悉模型的 sycophancy"
- "Anthropic 内部的解法直接从 GAN 抄了过来"

改写：

- "我们都熟悉模型的 sycophancy（讨好倾向）"
- "Anthropic 内部的解法直接从 GAN（生成对抗网络）抄了过来"

判断边界：同篇文章内同一术语只在首次出现时解说，之后裸写；如果术语之间隔了三个以上章节又出现，可以考虑再解说一次，但不强求。

### 5.x 学术 / 直译术语的白话化候选

当英文术语在中文里有学术翻译但偏冷僻时，优先选公众号语境的白话候选。给出常见对应：

| 直译 / 学术 | 白话候选 |
|---|---|
| leniency bias | 容易放水 |
| sycophancy | 讨好倾向 |
| 朴素（架构、做法） | 简单 / 直白 |
| 同理心（对模型/系统） | 站在 X 的视角思考 |
| 出乎预期 | 出乎意料 |
| 眯起眼看 | 仔细看 |
| 临门一脚 / 还差一刀 | 还有缺口 / 还差一步 |

原则：首次出现的英文术语**必要时**给一次嵌入式解说（§5.1）；解说之后再次出现优先用白话候选，不再重复学术翻译。

### 5.x 非技术英文口语表达必须本地化

定义:除了技术术语之外,还有一类**英文口语 / 写作习语**漏到中文文章里,容易让人忽略,因为它们不像专有名词那样显眼。这一类必须翻译,不能直接保留英文。

| 英文表达 | 含义 | 中文写法 |
|---|---|---|
| niche take | 小众的、不主流的判断 | 小众观点 / 非主流判断 |
| hot take | 直给的、有冲击力的判断 | 直给的判断 / 直接判断 / 一个犀利判断 |
| no-brainer | 显而易见、不需要想 | 显而易见 / 想都不用想 |
| game changer | 改变游戏规则的事物 | 改写规则的东西 / 改变格局的事 |
| deal breaker | 谈不下去的关键阻碍 | 一票否决项 / 谈不下去的关键 |
| moonshot | 登月级 / 极限挑战 | 登月级目标 / 极限挑战 |
| low bar / high bar | 标准低 / 标准高 | 门槛低 / 门槛高 |
| sweet spot | 最佳平衡点 | 最佳平衡点 / 甜点区 |
| pet peeve | 个人烦的事 | 个人最讨厌的事 |
| paper cut | 小毛病、烦人但不致命 | 小麻烦 / 烦人的小问题 |

为什么这一类容易漏:写稿时大脑还在英文叙事里,这些短语顺手就保留下来。但读者读到 "niche take" 时,既要切语言,又要 infer 含义,节奏立刻断。

判断方法:文章里所有非英文专有名词的英文短语,问一个问题——"如果这是中国作者直接用中文写,他会写英文还是中文?"。会写中文的就翻译。RAG / API / token / prompt / agent / harness 这种业界默认词留英文,niche / hot / pet peeve / sweet spot 这种口语习语翻成中文。

grep 触发词(非技术英文口语):`niche take` `hot take` `no-brainer` `game changer` `deal breaker` `low bar` `high bar` `sweet spot` `pet peeve` `paper cut`(命中即翻译)
