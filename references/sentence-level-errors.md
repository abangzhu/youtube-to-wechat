# 句法骨架与词序错误

读取时机:**段级精修阶段**(写完整稿后,逐段精修时查阅)。本文集中处理"句子骨架层面的错误"——读者能感觉到"在演"或"漏了什么",但说不出哪里出问题的那一类。

覆盖范围:

- **施事 / 受事完整性**:多方主体动作必须显式归属、给事实加戏的态度词
- **主语 / 主谓宾**:主语漂浮、省略指代、抽象名词改动词、副词修饰范围错位
- **多重定语 / 词序**:时间 > 主体 > 中心词、范围副词放最前、表语翻译腔(是说 / 是去做)
- **英文翻译腔**:名词化、修饰前置、被动 / 容器隐喻、弱代词、预告-展开结构、拟人化、Marginalia 类直译

每条规则都附**典型错例 → 改写**对照,以及通读级判断方法或 grep 触发词。

写稿前的词汇级禁区见 [ai-voice-prohibitions.md](ai-voice-prohibitions.md);转述腔与格式约定见 [narration-and-formatting.md](narration-and-formatting.md)。

## 六之二、给事实加戏与施事受事完整性

这两类问题不是 AI 腔词汇触发的，而是句子骨架层面的错误。一旦发生，读者能直观感到文章「在演」或者「漏了什么」，但往往说不出是哪里出了问题。编辑阶段必须显式检查。


**给事实加戏 / 无中生有一个态度**

定义：作者在文章里暗示了原视频 / 原文**没有**表达过的态度、反转或姿态。

典型错例：

- 原视频里 Louis 只是直接和客户沟通，没有表达「拒绝 / 犹豫 / 本来要接」的态度。但写作时用「Louis **没有直接接活**，而是……」——「没有直接接活」暗示了「本来要接但没接」的戏剧反转，读者会误读讲者的姿态
- 原视频里 Paul 只是选了方案 B，没有表达「曾经尝试 A 但失败」。但写作时用「Paul 放弃了这条路」——「放弃」暗示了曾经走过

判断方法：每次写完带有态度动词的句子（「没有 / 放弃 / 拒绝 / 选择 / 转向 / 反思」），回到原视频或转录，检查讲者是否真的表达过这层情绪。没有就改成白描。

这条规则和 SKILL.md 6.3 节的「延伸解读 vs 无中生有」不同。6.3 针对**事实**是否原文提过（补充背景、类比是允许的，纯事实虚构要删）；这里针对**态度 / 姿态**是否原文暗示过，即使事实层面没错，多加一层戏剧情绪也算无中生有。

**施事 / 受事关系完整性**

定义：动词描述了动作但漏掉关键主体。常见于「找上门 / 联系 / 接洽 / 提到 / 批评」这类关系动词。

典型错例：

- 「加拿大一家 CRM 平台找上门」——找上门找的是谁？必须补全：「加拿大一家 CRM 平台**找到 Towards AI**」
- 「客户提到了这个功能」——对谁提到？必须补全受事

检查方法：文章里每个关系动词前后都必须能回答「谁对谁做了这事」。编辑阶段用 grep 扫 `找 / 联系 / 接触 / 问 / 提到 / 批评 / 指出`，逐一确认施事和受事都在句子里出现。

**英文名词化 → 中文动词化**

定义：英文里常见的 "its own X call / the Y-ification of X / the processing of Z" 这类名词化结构，直译成中文读起来生硬。中文更自然的表达是换成动宾结构。

典型错例：

- 英文 "each tool can have its own LLM call" → 直译「每个 Tool 有自己的 LLM 调用」。改法：「每个 Tool 可以自己调用 LLM」
- 英文 "the generation of marketing content" → 直译「营销内容的生成」。改法：「生成营销内容」
- 英文 "the validation of results" → 直译「结果的验证」。改法：「校验结果」

触发特征：`X 的 Y 调用` / `X 的 Y 生成` / `X 的 Y 检测` / `Y 化的 X`。编辑阶段 grep 扫 `的.{1,8}调用 / 的.{1,8}生成 / 的.{1,8}处理 / 的.{1,8}验证`，凡是能改成动宾的都改。

**英文修饰前置 → 中文自然语序还原**

定义：英文里 "the X they built on top of Y" 这类后置定语 / 介词短语，直译成中文会变成把整段动作硬塞进定语，读起来不自然。中文倾向于把动作 / 状语前置，定语保持简短。另外，英文里 "X that the generalist AI doesn't have" 这种关系从句直译成中文会漏掉「是…所没有的」的句式骨架，变成倒装。

典型错例：

- 英文 "the automations they built on top of your product" → 直译「他们建在你产品之上的自动化」（把"建在你产品之上"硬塞进定语）。改法：「他们在你产品之上搭建的自动化」（把"在你产品之上"作前置状语）
- 英文 "specific logic that the generalist AI doesn't have" → 直译「specific logic 通用 AI 没有」（漏了"是…所没有的"的句式骨架）。改法：「specific logic 是通用 AI 所没有的」
- 英文 "build on top of your product" → 直译「在你产品之上建东西」。中文工程语境下"建"单字偏轻，改用「搭建 / 开发」：「在你产品之上搭建东西」

判断方法：每次写完一个有英文影子的句子，把它念一遍，问自己——这是不是中国人自然会用的语序？如果定语过长、动词单字、主谓宾被截断成定语，就调。

中文动词选词偏好：英文 "build" 在产品 / 工程语境里译中文，单字「建」偏轻，「搭建 / 开发」更自然。同理 "make / create" 在抽象内容生产里译「生成 / 制作 / 写出」更准。

**被动 / 容器隐喻 → 主动拥有关系**

定义：把抽象实体放在位置容器里（「X 留在 Y 里」「X 保存在 Y 中」），读者要多花一步才能 parse 出真正的拥有关系。中文更自然的是让真正的持有者做主语。

典型错例：

- 「全局上下文留在那个唯一的 Agent 里，它同时是 Planner 和决策者」——主语是"上下文"，行动者是"留在"，Agent 变成容器。改法：「唯一的 Agent 拥有全局上下文，这个 Agent 同时是 Planner 和决策者」——Agent 做主语，关系直接

判断边界：如果 Y 真的是一个**位置/存储**（工作目录、内存、磁盘），「X 保留在 Y 里」是合理的。如果 Y 是一个**实体角色**（Agent、Reviewer、Writer、系统），让 Y 做主语用「Y 拥有 / 持有 X」更好。

**弱代词 → 具体名词重复**

定义：跨句或句末的 `它 / 这个 / 那个 / 他 / 她` 作为唯一指代，中文读者跟不上。

典型错例：

- 「教别人怎么搭它」——「它」指代前面的"系统"，但读到句末时读者已经回溯一次。改法：「教别人怎么搭建这套系统」
- 「两个 Agent 放同一个项目，它能同时看懂两边的代码」——「它」指代模糊（是"项目"还是"放法"）。改法：把主语换成具体的东西（「这样的目录布局让 AI 辅助工具能同时看懂两边的代码」）

判断规则：段落内第一次指代可以用代词，但跨段、章节收尾句、定义句、总结句必须用具体名词。特别是句末的"搭它 / 做它 / 改它"这种含混收尾，一律换成具体名词。

**预告-展开结构（「只有一个，…」式）**

定义：前半句做悬念引导（「只有一个」「只有一句话」「原因只有一个」「答案很简单」），后半句才给真正内容。形式上制造节奏感，实际上多数是 filler。

典型错例：

- 「他们想回答的问题只有一个，AI 工程师的判断力到底应该放在哪里」——前半句"只有一个"是纯预告，没有信息量。改法：「他们想回答的是 AI 工程师的判断力到底应该放在哪里」

判断边界：如果前半句本身有独立信息量（比如「他们问过十几个客户，答案出奇地一致——X」），可以保留。纯节奏性的预告（「只有一个 / 只有一句」），删掉。

**拟人化 / 诗意动词用于工程语境**

定义：把文艺动词用在工程技术描述里，会把读者从技术上下文拉到散文上下文。

典型错例：

- 「判断力应该**长在**哪里」——"长"是植物隐喻，工程文里不用。改法：「判断力应该**放在**哪里」或「**体现在**哪里」
- 「产品**生根**在用户工作流里」——"生根"是植物动词。改法：「产品**嵌入**用户工作流」或「**贴着**用户工作流做」
- 「让飞轮**转起来**」——"飞轮"是商业金句式隐喻，过度使用

触发词：`长在 / 生根 / 扎根 / 落脚 / 落地（当做动作而非结果时） / 开花结果 / 飞轮转起来 / 土壤`。

判断边界：如果引用原作者的原话并且原话就是这么说的，可以保留（属于原文金句）。作者自己写时避免。

**引入新概念要给一个迷你说明**

定义：第一次出现某个专名、自造术语或比喻时，要让读者在当下就能猜到它是什么，而不是再读三段才能 infer。

典型错例：

- 「Louis 开场先画了一条**自主性滑杆**」——"滑杆"是比喻，读者不知道它是条什么线。改法：「Louis 开场先画了一条**关于系统自主性的滑杆**」
- 「Paul 采用了**评估器-优化器循环**」——直接给术语名，读者猜不到它干什么。改法：「Paul 采用了**评估器-优化器循环（写一轮、审一轮、改一轮的反复打磨流程）**」

检查方法（通读级，不是 grep 级）：通读时问每一个英文术语和比喻「如果读者不认识这个词，文章在这里或临近位置有没有给出说明」。如果要往下读三段才能拼出意思，前面必须加一句。

边界：业界通用且 AI 工程师熟知的术语（RAG / F1 / Pydantic / few-shot）不用解释；作者自造术语或比喻（滑杆 / 拳头产品 / 水坝）必须解释。

**段落间的逻辑衔接词不能省**

定义：上段和下段之间有延续、对比、转折、补充四种关系，字面上必须通过衔接词表达出来。省掉衔接词的时候，读者的脑子要自己补连接，读起来断。

典型错例：

- 上段讲了正例（Agent 合适的场景），下段开头「反例很具体」——缺「也」。改法：「反例**也**很具体」
- 上段讲 Agent 的优点，下段开头「这种克制带来一条反直觉的工程约束」——缺「但」。改法：「**但**这种克制带来一条反直觉的工程约束」

检查方法（通读级）：每段首句读一遍，问自己「这段和上段是什么关系？是**延续 / 对比 / 转折 / 补充**四种中的哪一种？这个关系在字面上表达了吗？」

衔接词参考：延续（还 / 同样 / 进一步）、对比（也 / 相反 / 反过来）、转折（但 / 不过 / 虽然）、补充（此外 / 另一个）。

**列举推论类**（用于多个独立观点的有序串联，常见于"主轴拆推论"段落）：首先 / 因此 / 同时 / 最后 / 接下来 / 进一步。

**语气转折类**（用于轻微转折、强调反预期、不破坏正向推进的回马枪）：其实 / 反而 / 但实际上 / 相反 / 事实上（注意：「事实上」属于强调副词，慎用，详见 §6.x 强调副词稀释信任）。

新增反例 / 改写对照（来自 IlqJqcl8ONE session）：

- 「主轴是这样的。模型在变强。两个推论:你越想约束模型,代价越大;Agent 跑得越久,核查必须前移。三层进阶就是把这两条推论分别落到 ...」 — 三个独立推论挤一段，缺列举标记。改写：「**首先**,模型在变强,Agent 能跑越来越长的任务。**因此**,你越想约束模型,代价越大;**同时**,Agent 跑得越久,核查必须前移。**最后**,需要将以上两条推论分别落到 ...」
- 「你也一样。Claude 比你更擅长把它挖出来。」 — 上一句「用户讲不清自己要什么」，下一句直接「你也一样」，缺语气转折标记。改写：「**其实**,你也一样。」
- 「信息密度高的 HTML spec 让后续迭代次数少很多」 — 缺「反而」表达「直觉认为费 token，实际相反」。改写：「信息密度高的 HTML spec **反而**让后续迭代次数少很多」
- 「这一条乍看像小技巧,实际是一个反复出现的工程瓶颈。」 — 缺「但」标转折。改写：「这一条乍看像小技巧,**但实际上**是一个反复出现的工程瓶颈。」
- 「烂 prompt 长这样,`make it better`」 — 上一段讲好 prompt，下一段是反例，缺「相反」。改写：「**相反**,烂 prompt 长这样,`make it better`」
- 「这条工作流明确不推荐 Sonnet。Sonnet 的视觉模型不够强。」 — 两句间是因果关系，缺「因为」。改写：「这个方法明确不推荐 Sonnet,**因为** Sonnet 的视觉模型不够强。」

为什么写稿时容易漏：列举推论类和语气转折类的衔接词在中文里常被当成"可以省略的副词"。但它们承担的不是修饰，是逻辑骨架——把它们删掉，读者不知道几条事实之间是顺序、并列还是因果。

判断方法（通读级）：写完每段，逐句读相邻两句，问"这两句的逻辑关系是什么？字面上有没有标出来？"。没标出来又不能从语序自然 infer，就补一个衔接词。

grep 不容易精确触发（否定型规则），靠通读。如果想做兜底扫描，找连续句号但缺衔接词开头的段落：把每段拆成句子列表，第二句开头是名词或主语而非衔接词时，人工核对一次。

**冗余「那个叫 X 的 Y」翻译腔**

定义：英文 "that X thing called Y" 或 "the X called Y" 直译，用「那个叫 / 所谓的 / 一种被称为」把术语包起来。中文里术语本身可以直接做定语，不用套这个框架。

典型错例：

- 「那个叫「Context Rot」的天花板」——改法：「「Context Rot」的天花板」
- 「所谓的 MCP 协议」——改法：「MCP 协议」

触发词扫描：`那个叫 / 那种叫 / 所谓的 / 一种被称为 / 一个叫做`

边界：如果这个概念在业界有多个名字，说明哪个是你采用的叫法时可以用「业界一般叫做 X / 原作者把它叫作 Y」——这时候「叫」有信息量。纯粹装饰性的「那个叫」一律删。

**抽象名词堆 → 问句或动作陈述**

定义：`X 的真正门槛 / Y 的核心挑战 / Z 的关键在于 / W 的本质是` 这种把判断或问题包成名词短语的写法，读者要 unpack 两层才能 get 到。中文里更直接的表达是问句或动作陈述。

典型错例：

- 「上多 Agent 的真正门槛也随之清晰」——三重抽象（"上多 Agent" + "真正门槛" + "随之清晰"）。改法：「什么时候应该上 Multi-Agent 也就明确起来」
- 「CTR 预估的核心挑战是数据稀疏」——改法：「CTR 预估最难的是数据稀疏」或「数据稀疏是 CTR 预估最难的地方」

触发词扫描：`的真正门槛 / 的核心挑战 / 的关键在于 / 的本质是 / 随之清晰 / 一目了然 / 跃然纸上`

改法方向：
- 「X 的真正门槛是 Y」→「什么时候才该上 X」或「Y 什么时候会让 X 失效」
- 「随之清晰 / 一目了然」→「也就明确 / 也就清楚」

**路径 / 旅程隐喻用作动作**

定义：`走这条路 / 走不通 / 绕开 / 踩过这条路 / 下山 / 转弯`，把方法选择描述成旅程。在工程文里换成白描动词更准。

典型错例：

- 「Paul 没走这条路」——改法：「Paul 没使用这个方法」
- 「这条技术路线走不通」——改法：「这个方法行不通」
- 「绕开了这个坑」——改法：「避开了这个问题」

触发词扫描：`走这条路 / 走不通 / 绕开 / 踩过 / 下山 / 转弯 / 撞上（表巧合/邂逅时）`

边界：原讲者原话里用了这个隐喻（加引号引用时），保留。作者自己写时不用。

**口语形容词 → 工程描述词**

定义：描述系统行为时用 `很吵 / 很糊 / 很乱 / 很死板 / 很硬 / 很油` 这种口语化、拟人化的形容词。生动但不精准，工程文里换成对应的工程词。

典型错例：

- 「这个 loop 会变得**很吵**」——"吵"是拟人。改法：「这个 loop **噪音很大**」
- 「这个模型输出**很糊**」——改法：「这个模型**精度很低**」或「这个模型**结果模糊**」
- 「架构**很乱**」——改法：「架构**结构混乱**」
- 「代码写得**很死板**」——改法：「代码**僵化**」或「代码**缺乏灵活性**」

触发词扫描：`很吵 / 很糊 / 很乱 / 很死板 / 很硬 / 很油 / 很脆`

边界：描述**人物行为**可以口语化（"他直接说"、"他一口气讲完"），描述**系统 / 流程 / 指标**不行。区别在于读者期待哪种词——对人，期待生动；对系统，期待精准。

**度量 / 触感形容词（轻 / 重 / 大 / 小 / 硬 / 软 / 宽 / 窄）表抽象义、或评价定义判断时改直说**

定义：用"轻 / 重 / 小 / 大 / 硬 / 软"这类描述物理量度或触感的形容词去表达抽象判断（简单 / 复杂 / 重要 / 不重要 / 影响小 / 影响大 / 严格 / 宽泛），读者要先 parse 出比喻义。当上下文里它实际表达的是"简单 / 不简单"时，直接写"简单 / 不简单"更清楚——"轻"在中文里可指重量、可指重要性、可指难度，歧义大。**尤其当这类词用来修饰一个"定义 / 说法 / 判断"本身时（"X 讲的 taste 更硬一些"），它有双重问题：既要读者 parse 比喻义，又把一个本该客观的定义变成了作者下场的主观比较。这种情况直接删掉形容词、用"X 认为 / X 指的是"把定义还原出来。**

典型错例 → 改写：

- 「Vision Banana 这个名字很轻，但它指向的变化不轻。」（这里"轻"实指"简单 / 不起眼"，但读者第一反应可能理解成"分量轻 / 不重要"） → 「Vision Banana 这个名字很简单，但它指向的变化不简单。」
- 「这个改动很小」（影响小还是改动量小？） → 按实义写「这个改动影响很小」或「这个改动量很小」
- 「Fadell 讲的 taste 更硬一些，指的是在数据不足时仍然能做取舍。」（用"硬 / 软"评价一个**定义**，"硬"既歧义又是作者主观比较，且"软 / 硬"根本不是定义的合法形容词） → 「Fadell 认为 taste 指的是在数据不足时仍然能做取舍。」（删度量形容词，用"X 认为"直接给定义）
- 「WebMCP 的定义很窄。它让网站把能力定义成结构化工具。」（"宽 / 窄"是描述物理宽度的词，不能形容"定义"；作者实际想说的是"范围很专一 / 只做一件具体的事"） → 「WebMCP 只做一件很具体的事。它让网站把能力定义成结构化工具。」（不补"范围窄"这层意思，直接改成动作陈述"只做一件具体的事"，下一句正好接着说这件事是什么）
- 「梯度下降能把信息压得很紧」（"紧"是触感词，表压缩效果时读者会先 parse 物理紧绷感，实义是信息密度高） → 「梯度下降能把信息压得密度很高」（直接写出工程实义"密度高"，不用触感比喻）

判断方法：看到"轻 / 重 / 大 / 小 / 硬 / 软"修饰抽象对象（名字 / 判断 / 变化 / 改动 / 问题 / 定义 / 说法）时，问一句"我实际想说的是简单 / 复杂 / 重要 / 严格 / 影响大小哪一个？"，把那个意思直接写出来；如果它修饰的是一个定义 / 判断本身，多半连那个意思都不必补，直接删掉形容词、用"X 认为 / X 指的是"给定义。

边界：物理量度真实存在时（模型体积小、延迟低、文件大、手感硬）保留；作者刻意用"轻 / 重"做修辞对仗且上下文无歧义时可保留一次。

grep 触发词：`更硬一些` `更软一些` `(定义|说法|判断|理解|观点|范围)更?[硬软轻重宽窄]` `定义很[宽窄硬软轻重]` `压得很[紧密]`

**评价式名词结构 → 动作式主谓宾结构**

定义：以被评价对象做主语、把真正的施事"S"变成定语的句式。中文读起来主语是抽象概念，而动词是"非常+形容词"的软表达，力度被稀释。

典型错例：

- 「Marc 描述的场景非常具体」——主语是"场景"，动词是"非常具体"。改法：「Marc 描述了一个具体场景」，主语是 Marc，动作是"描述"
- 「他提出的论点很有意思」——改法：「他提出了一个有意思的论点」
- 「作者写的段落挺精彩」——改法：「作者写出了一段精彩的段落」

判断方法：每一句里问"谁是主语？谁在做动作？"。如果主语是抽象概念（场景/细节/论点/段落）而动词是"非常/很/挺+形容词"，就转回成主谓宾结构。

触发词扫描：`描述的.{2,10}非常 / 说的.{2,10}很 / 写的.{2,10}挺 / 提出的.{2,10}很 / 构造的.{2,10}很`

**概念引入要用完整主谓宾，不要用省略 / 口头表达**

定义：引入一个新名词 / 新角色 / 新职能时用"出口叫 X"、"这就是 X"、"结果就成了 X" 这类省略式的口头表达，读起来轻浮、没重量。

典型错例：

- 「于是出口叫 Builder」——"X 的出口叫 Y"是口头省略。改法：「于是被称为 Builder 的新角色诞生了」
- 「然后就是一个 Agent Harness」——改法：「接下来出现的是一种叫做 Agent Harness 的运行框架」

改法方向：用完整结构"被称为 X 的 Y 诞生了" / "出现了一种叫做 X 的 Y" / "X 定义为 Y" 来引入概念。

**段内句间的逻辑连接词要显式（模式 15 的段内版本）**

定义：这是第六之二节里「段落间逻辑衔接词不能省」的段内版本。模式 15 针对**段落之间的首句**，这条针对**同一段里相邻句子**。

典型错例：

- 「这个合伙人已经搭起了一整套 AI 系统来处理他日常工作的所有事情，这套系统成了他工作里的合作伙伴。」——两句之间是递进关系，缺衔接词
- 改法：「但是这个合伙人已经搭起了一整套 AI 系统来处理他日常工作的所有事情，并且这套系统已经成了他工作里的合作伙伴。」

检查方法：每段内每两个相邻句之间问"这两句是什么关系（延续 / 对比 / 转折 / 递进）？连接词写出来了吗？"

关于时间衔接的扩展：「10 到 20 年后我们回头看」这种表达，模糊可能指向过去也可能指向未来。改法：加"再过"或"到 20XX 年"让时间指向明确。

**同一概念在文章里用词前后要一致**

定义：前段用"场景"指代某个画面，后段换成"画面"或"细节"指代同一个东西。读者要在脑子里反复确认"这几个词是不是同一个东西"，认知负担上升。

典型错例：第 13 行「这个画面里还有一个更反直觉的细节」，第 15 行「Marc 用这个画面引出他的核心论点」——"画面 / 场景 / 细节"三个词混用却指同一个东西。

改法：全文锁定一个词。第 13 行改成「还有一个反直觉的场景」，第 15 行改成「Marc 用这个场景引出他的核心论点」。

检查方法（通读级，不是 grep 级）：文章里作为概念锚点反复出现的词（场景 / 画面 / 观察 / 论点 / 细节 / 画像 / 现象 / 时刻），在终稿阶段统计每个词的出现次数，只应活一个。辅助 grep：`场景.*画面 / 画面.*场景` 在同一篇文章里不应共存。

**中文行业术语精度**

定义：用通用词 / 口语词替代行业内约定俗成的术语。

典型对照表：

- 软件行业："造" → "开发"。"更多产品会被造出来" 应为 "被开发出来"。类似：`代码被造 / 软件被造 / 系统被造` 都要改
- 借鉴 vs 抄袭："抄" → "借鉴"。`这套体系最值得抄的是它的分层逻辑` 应为 `最值得借鉴的`。"抄" 在中文里偏负面 / 偏口语，工程文里换 "借鉴 / 参考"。例外：自嘲式 "抄作业" 可保留
- 改进 vs 修改："改" → "改进"（在描述持续优化、迭代时）。`KP 一直在改 Deep Research Agent` 应为 `在改进 Deep Research Agent`。边界：小修小补的 bugfix / refactor 用 "改" 没问题；和 "优化 / 迭代 / 完善" 在同一段并列时用 "改进"
- 解决方案的全称："解" → "解决方案"（单字 "解" 指方案时）。`绕开 context bloat 的一个反复出现的解` 应为 `一个反复出现的解决方案`。单字 "解" 在数学语境保留（"方程的解"），但在工程方案语境里要写全
- 演讲场景动名："讲的人 / 讲的两个人" → "演讲的人 / 演讲的两个人"。开场介绍讲者时，"讲的两个人是 X 和 Y" 偏口语，"演讲的两个人是 X 和 Y" 更书面
- 薪酬体系："报酬" → "薪酬"。"报酬数据 / 报酬曲线" 应为 "薪酬数据 / 薪酬曲线"。"报酬" 多用于一次性劳务
- 工程动作："搭" → "搭建"（正式语境）、"做出" → "开发出来 / 打造出来"、"跑" 描述代码正常，描述项目应为 "推进 / 运行"
- 工程推广："推到 X" → "应用到 X / 应用在 X"。`把模型能力推到具体产品和工作流` 应为 `把模型能力应用在具体产品和工作流`。"推到" 偏 push 口语，"应用到 / 应用在" 在描述把能力 / 工具放进某个场景时更书面。边界：`把代码推到 GitHub` 这种 git 语境的 "推到" 保留，只针对 "推到产品 / 推到流程 / 推到 X 中" 这种抽象推广。
- 人事："岗位 / 职位" 要看上下文选准一个
- 数据 / 仓库术语："ledger" → "明细"或"流水"，不是"账本"。"账本"是会计场景，进入开发/数据流/仓库语境（Git repo、写操作日志、事件流）时用"明细"。类似：`写进账本 / 回到账本 / 账本末尾` 都应该换成"明细"。同一篇里同一个 ledger 概念要统一成一个词
- 软件发布："发货" → "发布"（在软件/产品迭代语境）。"发货"是物流/电商术语，软件功能上线/版本发布一律用"发布"。边界：`发货 + 物流 / 快递 / 收货` 语境保留"发货"，其余换"发布"
- 流程/需求删减："杀掉" → "砍掉"。"砍掉流程 / 砍掉功能 / 砍掉需求"是项目管理标准用词，"杀掉"偏口语且语气过重。边界：刻意口语化收尾（段尾强力度）可保留"杀掉"，正式叙述一律"砍掉"
- 职业成长："长大" → "成长"（职业发展语境）。"长大"是年龄/身体自然成长，描述工程师能力提升、人才培养用"成长"。`工程师怎么长大 / 新人长大的路径` 都应改为"成长"
- 心灵鸡汤惯用语：固定说法是「心灵鸡汤」，不是「心态鸡汤」。grep 触发：`心态鸡汤`
- 系统/架构的基本单元："构件" → "组件"（component 的行业译法）。"构件"偏土木/建筑术语，软件/Agent/系统语境一律用"组件"。`六个构件 / 核心构件 / 基础构件` 均改为"组件"
- 「build」按语境三档选词：①软件/系统的开发动作 → 「开发」（"开发 agentic system / 开发 agent / 开发没有消失"）；②文章收尾、核心意象、含义丰富的金句 → 直接保留英文「build」（"工程师没有停止 build"，保留英文有分量感）；③真实建筑/实体构造 → 「建造」（工程文章里极少见）。不要把所有 build 都译成「建造」。grep 触发：`建造.{0,6}(系统|agent|工程师|产品|工具|能力|流水线)` 命中即检查是否该用「开发」或「build」
- 人物身份介绍："创建者" → "创始人"（founder）。"创建者"偏技术项目（"仓库的创建者"），公司/产品的创始人用"创始人"。边界：开源工具/工具库的维护者可用"创建者"；有融资/公司背景的产品一律"创始人"
- 人物与产品的关系描述：「X 与 Y 关系很深 / X 和 Y 渊源很深」是模糊关系词，当实际关系是明确身份时（创始人/核心作者/负责人）直接陈述：「X 是 Y 的创始人」比「X 与 Y 关系很深」信息量高一倍。grep 触发：`与.{1,10}关系很深` `和.{1,10}渊源`

改法：写完扫一遍这些通用动词 / 名词，看有没有对应的行业专有说法。

判断边界：口语化是这套 skill 的特色，不是所有"造"都要改成"开发"——但在描述**产品开发 / 软件工程 / 薪酬体系**这种明确行业语境时，用行业内固定说法。

触发词扫描：`被造 / 制造.{0,3}软件 / 报酬.{0,3}数据 / 报酬.{0,3}曲线 / 值得抄 / 一直在改[^进良善] / 一个.{0,3}解[。，] / 讲的两个人 / 讲的人是 / 推到[^GitHub仓远端]{0,8}产品 / 推到.{0,8}流程 / 能力推到`

**同一概念在文章里同义词要择一**

定义：上面"作为概念锚点的词只活一个"主要针对叙事词（场景 / 画面 / 现象）。这条针对**意义重叠的描述性近义词**——同一篇文章里描述同一个事物用了不同中文译法或近义形容，读者得反复确认这是不是同一个东西。

常见易混对偶：

- **完成度 / 完整度 / 完整性** → 任选其一，全文统一。描述一个产品做出来"有多完整"时，"完成度"是更书面的中文表达
- **体感 / 感受 / 体验** → 任选其一
- **信号 / 迹象 / 苗头** → 任选其一
- **出乎预期 / 反直觉 / 意外** → 任选其一（注意"反直觉"是作者评价词，写公众号正文里慎用，详见第六章「作者视角主观评价词类」）
- **判断 / 结论 / 论断** → 任选其一
- **模式 / 套路 / 做法 / 玩法** → 选准用词，"玩法"偏口语

终稿操作：把每个候选词 grep 一遍，统计次数；如果两个近义词都出现，选定一个，把另一个全部换掉。多数派优先。

反例：

- 同一篇文章前段写"这个**完整度**是 harness 给的"，后段又写"两个版本**完成度**差不多"——同一概念两词混用
- 同一篇文章写了三处"反直觉"、一处"出乎预期"、一处"意外"——读者不知道作者是不是在指同一种感觉

触发词扫描：`完整度.*完成度 / 完成度.*完整度 / 反直觉.*出乎预期 / 出乎预期.*反直觉`（同一文档内两词共存即触发）

**英文数字习语 / 比喻习语要本地化**

定义：英文里约定俗成的数字习语或比喻习语直译成中文后读者看不懂。

典型错例：

- 「high nines」直译成「高 9」—— 中文读者看不懂。本地化为「90% 大几」/「99% 以上」
- 「first order of business」直译成「第一等大事」—— 应为「眼下最要紧的事」
- 「low-hanging fruit」直译成「低垂的果实」—— 业界有人用但非必需，可改「容易摘到的果实」或「显而易见的机会」

改法：翻译时对数字习语和固定习语多走一步，查一下中文对应的口语说法。

检查方法：回译自检时特别关注数字表达和固定短语。触发词扫描：`高 9 / 低垂的果实 / 第一等大事`

**引用同义冗余择一**

定义：同一段里连续引用多个说同一件事的短语，给读者的信息量是重复的。

典型错例：「底下回复都在说『我以前那家公司是 8 倍膨胀』『你这还太宽容了』」——两句是同义的。

改法：择一留下，一般选更具体 / 更有画面感的那个。这里保留「8 倍膨胀」更具体，删掉「你这还太宽容了」。

**主观评判补充句要删**

定义：客观陈述之后作者加一句主观价值判断。

典型错例：

- 「花 200 美元订 premium。**这点钱不多。**」——"这点钱不多"是作者的价值判断，对不同收入读者意义不同
- 「这个岗位年薪 50 万。**非常高。**」——"非常高"对不同行业的读者不同

改法：删掉这句。如果是原作者/讲者说的那句，在正文里明确标注是讲者原话再保留。

**段尾「可以借鉴的」桥接句换 Tips: 标签**

定义：写到段落最后想给读者一个"能拿走什么"时，常用「可以借鉴的判断:X」「可借鉴的是 X」「值得记下来的是 X」「值得带走的判断是 X」「最值得抄走的一条是 X」这种「作者评价 + 桥接」组合。这种句式有两个问题：作者下场做评价（违反 §六「作者视角主观评价词类」），同时把段落变成"事实段 + 评价桥 + 干货"的三段式，桥句本身没信息量。

中文公众号里更紧的写法是直接用 `Tips:` 这种结构标签，跳过桥句，直接给干货。读者扫到 `Tips:` 就知道接下来是可以拿走的东西，不需要作者亲自指认。

典型错例 → 改写：

- 「可以借鉴的判断:每次想跟 Claude 描述前端 UI 问题之前,先问自己一句。这个问题截图比文字更省力吗?」 → 「**Tips:** 每次想跟 Claude 描述前端 UI 问题之前,先问自己一句。这个问题截图比文字更省力吗?」
- 「可借鉴的是分层。底层 contract 用 data-attribute 暴露状态,中层用 schema...」 → 「**Tips:** 底层 contract 用 data-attribute 暴露状态,中层用 schema...」
- 「值得记下来的是,Skill 治理已经是一个工种。」 → 「**Tips:** Skill 治理已经是一个工种。」
- 「最值得抄走的一条是,把核查从日志变成证据。」 → 这一句要分两步处理:先把"抄走"换成中性表达，再把整条桥句改成 `Tips: 把核查从日志变成证据包`

判断方法：写完段落最后一句时，问自己——这一句是"事实"还是"评价 + 桥接"？是后者就换 `Tips:`。如果连 Tips 标签都没必要（段尾干货已经自然落在前一句的事实里），直接删桥句，什么都不补。

边界：`Tips:` 标签密度别太高。一篇 2000 字稿子最多 2-3 个 `Tips:`，再多读者会觉得在读 cheat sheet。如果一段段都想加 `Tips:`，回去检查段落结构——多半是把太多干货塞到一段了。

grep 触发词：`可以借鉴的判断` `可借鉴的是` `值得记下来的是` `值得带走的` `值得抄走` `最值得.{0,5}的一条`

**冗余默认属性限定词**

定义：给本来就默认具备某个属性的名词再加一个限定词，是废字。公开人物的公开发言默认是"公开的"，作者写的文章默认是"他写的"，巨头产品默认是"它旗下的"。

典型错例：

- 「Mitchell Hashimoto 的**公开**判断」——他作为前创始人在博客写的东西默认是公开的。改法：「Mitchell Hashimoto 的判断」
- 「他在 Twitter 上**公开**发的一条推」——Twitter 发推默认公开。改法：「他在 Twitter 上发的一条推」
- 「Google **旗下的** YouTube」——熟知关系不需要限定。改法：「Google 的 YouTube」或「YouTube」
- 「作者**亲自**写的」——作者署名的文章默认是本人写的，加"亲自"反而暗示"可能不是"。改法：「作者写的」

判断方法：把限定词抽掉，信息是否丢失？没丢就是默认属性冗余，删。

触发词扫描：`公开(判断|发言|说过|表示) / 旗下的 / 亲自(写|做|主持) / 本人(写|发|说)`

**时间推进衔接词用副词，不用定语**

定义：把"接过 / 站起 / 拿起话筒"这种动作做成人物前面的定语，再用"的"连到人物，是英文分词短语（participial phrase）直译。中文自然写法是把动作从定语里抽出来，用时间副词做段落衔接。

典型错例：

- 「**接过来的** Hugo 把视角从加速切到工作单元」——"接过来的 Hugo"是英文 "Taking over, Hugo..." 的分词结构直译。改法：「**接下来** Hugo 把视角从加速切到工作单元」
- 「**站起身的** Marc 说」→「Marc 站起身说」或「**接着** Marc 说」
- 「**拿起话筒的** 主持人问」→「**这时** 主持人拿起话筒问」

触发词扫描：`[来去起下住上]的 [A-Z]` / `的 Hugo / 的 Marc / 的 Madison`

**"那个 X 词 / 那个 X 东西"尾缀冗余**

定义：句尾用"那个 X 词""那种 X 做法"作为代词收尾，X 本身不承载新信息，属于装饰性回指。

典型错例：

- 「`continue` 大概是他们团队现在最常敲**的那个词**」——"最常敲"已足够指明对象。改法：「`continue` 大概是他们团队现在**最常敲**」
- 「这是他提出**的那种做法**」——"他提出"已经具体。改法：「这就是他提出的」或干脆删掉
- 「最反直觉**的那一条**」——可改成「最反直觉的一条」或去掉"那"

触发词扫描：`的那个[词句句子做法说法情况东西] / 的那一[条种个]`

**"让 X 快 / 让 X 好"里形容词要动词化**

定义：在"让 X Y"的使役结构里，Y 是形容词时，中文读起来像静态描述；改成"变 Y / 成为 Y"之后才是动作。

典型错例：

- 「要让这一切真的**快**」——"快"是状态形容词。改法：「要让这一切真的**变快**」
- 「这样做能让代码**整洁**」→「这样做能让代码**变整洁**」
- 「这个改动让循环**更稳**」→「这个改动让循环**变得更稳**」

判断边界：如果 Y 本身就是动词（"让它跑起来""让它停下"），不用变。只针对形容词 Y。

触发词扫描：`让[^，。]{1,15}真的[快好稳准稳严紧松] / 让[^，。]{1,15}更[快好稳准严紧松复杂简单困难容易]`

补充说明：「让 X 更 Y」里 Y 是形容词、且表达程度加深时，必须补「变得」：「让这个问题更复杂」→「让这个问题变得更复杂」。触发信号是"更"后面直接跟多字形容词（复杂/简单/困难/麻烦），单字形容词已在上表覆盖。

**"不再有 X" 要补谓语**

定义："不再有 X"后面省略了谓语（X 在做什么），读者要自己补。加一个动词让陈述完整。

典型错例：

- 「External Validation **不再有人**」——不再有人什么？改法：「External Validation **不再有人参与**」
- 「系统里不再有人**在管**这件事」→对。
- 「前端页面不再有**手工录入**」→对（录入是名词也能成谓语）。

判断方法：每次写完"不再有 X"，问自己"X 在做什么/承担什么"，把动作补出来。

触发词扫描：`不再有[人团队产品]。 / 不再有[人团队产品]，`

**规范性判断要显式用"应该 / 应当"**

定义：陈述规范（事情应该怎么做）时如果只写"是 X 的地方"，中文读者会先读成事实描述；加"应该"才能把规范语气传出来。

典型错例：

- 「这个前置队列才是**人类介入的地方**」——读起来像描述事实"事实上人类就在这里介入"。改法：「这个前置队列才是**人类应该介入的地方**」
- 「这才是产品经理**做决策的场合**」→「这才是产品经理**应该做决策的场合**」

判断边界：如果上下文明确是描述（"今天在座的工程师大多在这里介入"），保留无"应该"版本。只在作者是在给规范建议时加"应该"。

**"由 X Y"显式归因**

定义：涉及"谁主导/谁负责"的关系时，"X Y"（动宾直连）容易被读成对等参与，加"由"才能明确主导关系。

典型错例：

- 「Namespace 今年 3 月宣布累计融资 2300 万美元，**NEA 领投**」——"NEA 领投"字面上像 NEA 自己做了个动作。改法：「Namespace 今年 3 月宣布累计融资 2300 万美元，**由 NEA 领投**」
- 「这次会议**张三主持**」→「这次会议**由张三主持**」
- 「产品**王五牵头**」→「产品**由王五牵头**」

判断边界：如果本来就是清楚的施事关系（"张三说"、"王五做了 X"），不用加。只在被动/主导关系需要明确时用。

**文艺/美术性名词比喻在工程文里换工程名词**

定义：规则「拟人化 / 诗意动词用于工程语境」处理了动词侧（长在 / 生根 / 扎根）；这条处理名词侧。「底色」「脉络」「骨架」「纹理」「色调」这些美术/文艺性名词比喻，在工程判断或商业判断里会把严肃陈述软化。

典型错例：

- 「也是人类评审崩不住这个论断**的底色**」——"底色"是美术词，讲的是背景色调。改法：「论断**的本质原因**」或「论断**成立的前提**」
- 「产品设计**的纹理**」→「产品设计**的细节层次**」
- 「团队的**底色**是工程文化」——作为团队/组织描述可以保留（常见表达），不算工程判断
- 「技术栈**的骨架**是 Rust」→「技术栈**的主干**是 Rust」

判断边界：描述**人、团队、文化**时这些美术词可以保留（"这家公司的底色""这位作者的纹理"属常用中文修辞）；描述**系统、指标、论断、判断**时必须换工程名词。

触发词扫描：`的底色 / 的脉络 / 的骨架 / 的纹理 / 的色调`

**作者对自己叙事行为的元评述要删**

定义：规则「判断前后的元评价」处理了作者对**内容品质**的评价（"这是最值得抄的一句"）；这条处理作者对**自己叙事行为**的元评述（"这层关系先摆出来""接下来进入第二个点""下面再展开"）。

典型错例：

- 「所以这对搭档同台，本身也是领投方合伙人和被投公司 CEO 的联合发声，**这层关系先摆出来**」——"这层关系先摆出来"是作者宣告自己要做披露，宣告动作本身没有信息量。改法：保留前半句披露，删掉"这层关系先摆出来"
- 「**先说一句**，下面两段有点技术细节」→ 删
- 「**话又说回来**，我们聊回正题」→ 删（除非用于真正的话题回切）

判断方法：作者说的是"内容"还是"我即将要做的叙事动作"？是后者的一律删。

触发词扫描：`这层关系先摆出来 / 先说一句 / 这点先放这里 / 下面展开说 / 接下来进入 / 这部分先过`

### 6.2.x 主语漂浮 / 主谓宾在一处

一句话的主语应能被读者**第一遍就读出来**，不要让主语藏在长名词短语里、要读到句末才反推。

**反例**：Retro Forge 那个 demo，这两个 agent 谈出来的 contract 总共 27 条。
（主语「contract」被「这两个 agent 谈出来的」修饰成长名词短语，读者读完才反推「27 条」对应什么）

**改写**：回到 Retro Forge 演示的那个 demo，两个 agent 总共谈出来 27 条 contract。
（主语「两个 agent」 + 谓语「谈出来」 + 宾语「27 条 contract」，标准主谓宾）

检查方法：把句子简化成「主-谓-宾」骨架，如果骨架里没有合法主语、或主语是修饰过的长名词短语，重写。

更隐蔽的失误形式：用并列从句堆叠遮蔽主语。

**反例**：custom sub-agents 已经是基础能力，你的 evaluator 和 QA 角色，给它一个严厉 system prompt 加详细 rubric 即可。
（"你的 evaluator 和 QA 角色"作为话题悬空，"给它"的主语是谁？读者得反推）

**改写**：custom sub-agents 已经是一个基础能力了，你只需要定义一个严格的 system prompt 加详细 rubric，即是一个 evaluator 和 QA 的角色。

更隐蔽的失误形式三：分句间主语切换缺承接词。

当一个长句里前半交代了动作主体，后半切换为被动接尾结构（「跑出来」「做出来」「写出来」「算出来」「弄出来」），需要在切换点用「结果」「最终」「这一来」「于是」等连接词显式承接，否则后半句的主语会"悬空"——理论上可以从前文推出来，但读者第一遍读会卡顿。

**反例**：「第一版用最简单的 prompt 加最便宜的模型跑出来烂得离谱。」（"跑出来"前缺承接，主语像漂浮在两个分句之间。读者第一遍会问"什么跑出来？"）

**改写**：「第一版用最简单的 prompt 加最便宜的模型，结果跑出来烂得离谱。」（"结果"承接前半的方法描述，把"跑出来"明确指向上一节的结果）

**反例**：「他拿了三家公司的财报花了一周时间分析得出来一个结论。」（"分析得出来"前缺承接，且整句没有清晰的句段断点）

**改写**：「他拿了三家公司的财报，花了一周时间分析，**最终**得出一个结论。」

扫描提示（很难精确 grep，扫描提示）：检查含「跑出来」「做出来」「写出来」「算出来」「弄出来」的句子，若前半是工具或方法描述、紧接被动结尾，多半缺一个连接词。`grep -nE "[^结于这]?跑出来|做出来|写出来|算出来|弄出来" article.md` 命中后人工核查。

### 6.2.x 省略指代要补回来

把口语化的省略指代（"前沿一动"、"老板一拍"、"市场一变"）显式补回主语，避免读者反推。

**反例**：前沿一动，他们就跑个简化版看看够不够用。
**改写**：前沿模型一改动，他们就跑个简化版看看够不够用。

**反例**：风口一来，所有人都开始抄。
**改写**：风口一来到，所有人都开始抄。/ 一旦风口出现，所有人都开始抄。

grep 触发词：`一动` / `一变` / `一拍` / `一来` / `一开`（后接逗号或句号且前面省略主语时命中）

### 6.2.x 类比 / 对照要补全参照对象

类比省了本体、对照省了分界点，读者补不上，句子就悬空。英文原文常把参照对象留在上下文里（"it's like 1997"、"before and after"），直译进中文丢了锚，读者不知道"像什么"、"以什么为界"。

**反例**：今天的 AI 像 1997 年。（像 1997 年的什么？）
**改写**：今天的 AI 像 1997 年的互联网。

**反例**：写代码这件事出现了之前和之后。（以什么为分界？）
**改写**：写代码这件事，从此分成 AI 之前和 AI 之后。

判断方法：每写一个「像 X」「X 之前和之后」「和 X 一样」，问一句——读者知道"像的是什么 / 以什么为界 / 和什么比"吗？参照对象只在原文上下文里、没写进这一句的，补出来。

grep 触发词：`像 [0-9]` / `之前和之后` / `前和.{0,2}后`（命中后人工看参照对象是否在同句出现）

### 6.2.x 多重定语顺序中文化

定义:中文多重定语的自然顺序是 **时间 > 主体 > 性质 > 数量 > 类别 > 中心词**。英文里"adjective + noun"直译过来时,这个顺序经常被打乱。

**典型错例 → 改写**:

- "昨天的人类能力"(时间和主体被并列做定语,语义模糊) → "昨天人类的能力"(先成对"人类的能力",再用"昨天"修饰整体)
- "全公司的所有员工" / "我团队的每个成员" → "公司里所有员工" / "我团队里每个成员"(数量词 / 量词靠近中心词)
- "项目的关键的几个时间点" → "项目里几个关键的时间点"

判断方法:写完一个含两个以上定语的名词短语,问自己——"这两个定语谁修饰谁?"。是中心词被两个独立定语修饰(并列),还是其中一个先和中心词成对、再被另一个整体修饰(嵌套)?嵌套结构是中文更自然的表达。

### 6.2.x 数量 / 范围副词应放在所修饰短语最前

定义:表数量、范围、比例的修饰词("绝大部分"、"大多数"、"所有"、"一半"、"几乎所有"、"少量"),在中文里倾向于放在所修饰短语的**最前面**,而不是嵌套在中间。这是英文 "the majority of X" / "most of your X" 直译过来的常见错误。

**典型错例 → 改写**:

- "你日常工作的绝大部分" → "你绝大部分的日常工作"
- "团队成员中的大多数" → "团队里大多数成员"
- "用户中的一半" → "一半用户"
- "我们时间的 80%" → "我们 80% 的时间"

判断方法:看到"X 的 Y 的 数量词"或"X 中的 数量词"这种结构,把数量词挪到 X 之前,看读起来是不是更顺。多数情况下都更顺。

grep 触发词:`的绝大部分` `的大多数` `的一半` `中的大多数` `中的一半` `的 [0-9]+ ?%`

### 6.2.x 表语连接动词的英文翻译腔

定义:英文里 "the X is to do Y" / "the X is to say Y" / "his job is to..." 这种把动词原型挂在表语后做"作用"说明的句式,直译成中文是"X 的工作是说 Y" / "X 的角色是去做 Y",读起来生硬。中文更自然的写法是"X 的工作是 Y" / "X 的工作变成 Y" / "X 做的是 Y" / "X 在于 Y"。

**典型错例 → 改写**:

- "人类的工作是说『OK 我们手上有这堆能力,怎么用?』" → "人类的工作变成『OK 现在我们手上有这堆能力,我应该怎么用?』"(用"变成"承接;原文 Dan 的语境是状态转变,不是恒定职责)
- "他的角色是去做协调" → "他的角色是协调" / "他做的是协调"
- "这一步的目的是要确认 X" → "这一步的目的是确认 X" / "这一步是为了确认 X"

判断方法:看到"是说 / 是去做 / 是要做"组合,问自己——这里"是"后面跟动词结构是不是中文最自然的连接?多数情况下不是,可以删掉"说 / 去 / 要"或换成"变成 / 在于 / 做的是"。

grep 触发词:`的工作是说` `的角色是去` `的目的是要` `的任务是去`

### 6.2.x 抽象名词改动词短语

抽象名词作宾语或表语时，往往让句子变虚。能换成动词短语的尽量换。

**反例**：你需要这种对模型的同理心。
**改写**：你需要站在模型的视角去思考。

**反例**：团队需要更强的同理心。
**改写**：团队需要更愿意从对方角度想问题。

**反例**：这一招的本质是对约束的尊重。
**改写**：这一招的本质是不去突破已知约束。

判断方法：句尾是抽象名词（同理心、责任感、判断力、敏感度、同情心、敬畏心）时，问自己「这种能力具体表现为什么动作？」用动作描述替代名词。

### 6.2.x 副词修饰范围错位

定义：中文里副词的修饰范围由位置决定。把状态副词放到触发动词前面，会让修饰错位——读者读到副词时它的 anchor 还没出现，下意识把副词挂到错的动词上。

**反例**：「原本通过的用例突然零零散散开始失败。」（"零零散散"想修饰"失败"的状态，但放在"开始"前，第一遍读会让人以为它在修饰"开始"这个触发动作）

**改写**：「原本可以通过的用例突然开始零零散散跑不通了。」（"零零散散"挪到它实际修饰的"跑不通"前面；同时"开始失败"换成更自然的"跑不通了"，"开始失败"作为工程语境措辞偏书面）

**反例**：「服务**忽然慢慢地变卡**。」（"慢慢地"在"变卡"前是对的，但和"忽然"并置语义打架——一个在说骤变，一个在说渐变。读者要停一下消化）

**改写**：「服务**忽然变卡，而且越来越卡**。」或「**服务慢慢变卡了**。」（看作者实际意思取一个）

判断方法：找到每个状态/方式副词，问"它实际修饰哪个动词？"，再问"这个动词和副词之间是否被其他动词隔开？"。被隔开就把副词挪到 anchor 旁边，必要时把触发动词提到副词之前。

边界：当多个副词形成累积时间感（"先慢慢地，然后突然"），允许在前文铺垫，但每个副词的 anchor 都要在它之后紧邻。

触发词扫描（这一类很难精确 grep，扫描提示而非触发词）：
- 「开始 + 状态副词」组合：`开始[一-鿿]{0,3}地?[一-鿿]{0,3}`，命中后人工核查"开始"和后面动词之间是否插了状态副词
- 「突然 + 渐变副词」组合：`突然[一-鿿]{0,3}地?(慢慢|渐渐|逐渐|不断)`，命中后看是否需要拆成两句

### 6.2.x 冗余的叙述动作要砍掉

定义：把"讲者说出某件事"这个动作本身写得很具体（在台上把数字念给观众、走上台抛出一个判断、停下来强调一句、举起话筒说），但读者只关心被说的内容,不关心说的动作。这类"叙述动作"是转述腔的一种,把信息密度稀释了。

**改法:把动作压成最简单的"说 / 讲 / 提到 / 告诉",或者直接让内容出场。**

典型错例 → 改写:

- ✗ 「Phil Hetzel 在台上把一个数字念给观众:一个 trace 可以超过 1GB」 → ✓ 「Phil Hetzel 分享了几个数字:一个 trace 可以超过 1GB」(去掉"在台上""念给观众"的舞台动作)
- ✗ 「他走上讲台,抛出一个判断」 → ✓ 「他的判断是」
- ✗ 「Marc 停下来,一字一句地强调」 → ✓ 「Marc 强调」
- ✗ 「主持人举起话筒问了一个问题」 → ✓ 「主持人问」

为什么写稿时容易写出来:转录里讲者确实在台上做了这些动作,直接搬进文章会让作者觉得"有现场感"。但读者不在现场,这些动作对他理解内容零贡献,只是把句子拉长。

边界:当动作本身承载信息(比如"他停下来"是因为现场出了状况、"他举手让大家投票"是一个真实的互动事件),保留。判断:这个动作如果删掉,读者对内容的理解会不会变?不会就砍。

grep 触发词:`在台上(把|抛|念|讲)` `走上(台|讲台)` `念给.{0,4}(观众|大家)` `举起话筒` `一字一句` `停下来(说|强调|讲)`

### 6.2.x 元叙述命名包装句要砍掉,直接给讲者的主张

定义:作者先描述讲者"给某东西起名 / 命名 / 归类 / 提出一个说法 / 打个比方"这个**叙述动作**,再给真正的内容。常见句式:「他给这个动作起了一个很直接的说法,X」「他把它归到一种 X」「他用了一个说法叫 X」「他提出了一个概念,X」。这个包装动作本身几乎没有信息量,而且经常再叠一层评价词(「很直接」「很容易懂」「很形象」),双重稀释信息密度。本质是转述腔的一种——作者在转播"讲者命名 / 表达了什么"这个动作,而不是直接写主张。这种句式在转述访谈 / 演讲时高频出现,是用户反复指出的别扭表达。

改法:砍掉命名 / 归类 / 起名的包装,让讲者的主张、建议、定义直接出场。

典型错例 → 改写:

- ✗「他给这个动作起了一个很直接的说法,别向机器做认知投降。」 → ✓「他建议我们不要向机器做认知投降。」(删"起了一个很直接的说法",把内容还原成一句建议)
- ✗「Fadell 把它归到一种 opinion-based decision。」 → ✓「Fadell 称这是一种 opinion-based decision。」(归类动作"把它归到一种"换成最短的"称这是")
- ✗「他用了一个很容易懂的类比。AI 生成的东西像快时尚……」 → ✓「他打了个比方。AI 生成的东西像快时尚……」(删评价词"很容易懂";类比 / 比方确需引入词时,用最短的"打了个比方"即可,不要再加评价)
- ✗「这期访谈里最直接的 AI 判断,落在代码上。Fadell 说,Agent 可以写出能跑的代码……」 → ✓「在代码上,Fadell 说得很直接。Agent 可以写出能跑的代码……」(同时砍掉"这期访谈里"指向来源场合的元叙述、"最直接的判断"作者排序评价、"Fadell 说"转述腔三层包装)

边界:当被命名的术语是关键概念、需要明确归属给讲者时,保留一个**最短的引入动词**("称 / 叫 / 把它叫作"),但去掉前面的评价词和"起了一个说法"这类冗余包装。引入概念用完整主谓宾的规则见本文§「概念引入要用完整主谓宾」——那条管"出口叫 X"这类口头省略(太轻),这条管"起了个说法叫 X"这类元叙述膨胀(太重),两条互补,目标都是让概念用一个干净的主谓宾出场。

判断方法:看到"起了 / 起名 / 命名 / 归到一种 / 用了一个说法 / 提出一个概念 / 打了个比方"加评价词("很直接 / 很容易懂 / 很形象")的组合,问一句——这个叙述动作删掉、让内容直接出场,信息有没有丢?没丢就砍。

**特别警告(去转述腔最容易走偏的方向):** 把"X 说"改成"X 说得很直接 / 说得很清楚 / 说得很狠"**不算**去掉转述腔——它只是给转述动词"说"叠了一个方式评价词,问题反而更重(转述动词 + 评价形容词双重包装)。当"X 说"后面的内容其实是作者要传达的**判断**时,正确方向是把转述动词直接换成"X 的判断是 / X 认为":

- ✗「在代码上,Fadell 说得很直接。Agent 可以写出能跑的代码……」(把"Fadell 说"改成"说得很直接",叠了评价词,没解决转述腔) → ✓「在代码上,Fadell 的判断是,Agent 可以写出能跑的代码……」

这条是真实踩过的坑:第一轮把"Fadell 说"误改成"Fadell 说得很直接",第二轮才纠正成"Fadell 的判断是"。去转述腔只有两个正确出口——要么直接陈述内容(连"X 认为"都省),要么换成"X 的判断是 / X 认为";给"说"加修饰词是假动作。

grep 触发词:`起了一个.{0,6}的(说法|叫法|名字|类比|比方)` `把它归到一种` `用了一个.{0,6}的(说法|类比|比方|概念)` `提出了一个.{0,4}概念,` `给.{0,6}起了` `说得很(直接|清楚|狠|透|明白)` `讲得很(直接|清楚|透)`

### 6.2.x 空间方位隐喻替代转折连词

定义:用"另一边 / 另一头 / 这一头 / 这边"这类空间方位词来表达对比 / 转折关系,代替显式转折连词"但 / 不过"。常见于"A 抓的是 X,B 另一边看到的是 Y"这种英文 "on the other hand" 的直译。问题:方位隐喻让转折关系含糊,读者要先把"另一边"在脑子里翻译成"相反地",不如直接用转折词显式标出对比。

典型错例 → 改写:

- ✗「BlackBerry 抓住的是最忠诚的邮件用户……Apple 另一边看到的是那 98% 没用 BlackBerry 的普通用户。」 → ✓「……但 Apple 看到的是那 98% 没用 BlackBerry 的普通用户。」(方位词"另一边"换成转折连词"但")

边界:作者刻意用"一边……一边……""一头……一头……"做对仗排比、且上下文无歧义时,可保留。只针对用"另一边 / 另一头"含糊代替"但"的情况。

grep 触发词:`另一边(看|是|想|认为|觉得)` `另一头(看|是)` `这一头`

### 6.2.x 否定判断不要用"很不像 / 不太像"软化,直接"不是"

定义:做一个否定判断时,用"X 很不像 Y / 不太像 Y / 不太算 Y"这种程度化的相似性否定,把一个干脆的判断("不是 Y")软化成"在相似度上离 Y 比较远"。读者要多 parse 一层,语气也含糊。当作者实际要表达的是"不是 Y"时,直接写"不是"。

典型错例 → 改写:

- ✗「Fadell 对营销的看法很不像「发布前包装」。」 → ✓「Fadell 对营销的看法不是「发布前包装」。」
- ✗「这套做法不太像传统瀑布流。」(如果意思就是"不是") → ✓「这套做法不是传统瀑布流。」

边界:当作者真的要表达"相似但不完全等于"(刻意保留一点相似度)时,"不太像"是准确的,保留。只针对实际意思是干脆否定、却用"很不像"软化的情况。

grep 触发词:`很不像` `不太像` `不太算是` `不太是`

### 6.2.x 形容词与抽象名词的搭配(配价)要成立

定义:形容词和它修饰的抽象名词之间有固定搭配。用错搭配(把形容 A 类名词的词用到 B 类名词上),读者能感到别扭却常说不出哪错。最高发的是用"紧 / 要紧 / 快 / 重"这类形容"事情 / 进度 / 物理量"的词,去修饰"提醒 / 判断 / 定义 / 观点"这类名词。

典型错例 → 改写:

- ✗「这层提醒放到 AI 时代更要紧。」("要紧"形容事情紧迫,不能形容"提醒"的分量;"放到 X 时代"还套了容器框架) → ✓「这个提醒在 AI 时代更重要。」("提醒"配"重要";"放到…里"换成"在…")
- ✗「他的判断很快。」(判断配"准确 / 果断",不配"快") → ✓「他判断得很果断。」或「他的判断很准。」

判断方法:写完"X + 形容词"后,把名词和形容词单独读一遍,问"中文里这个名词通常用这个形容词修饰吗?"。提醒 / 启示 → 重要、深刻;判断 / 结论 → 准确、果断;定义 → 清楚、严格。配不上就换。顺带:"把 X 放到 Y 时代 / Y 场景里更 Z"的容器框架,多数能简化成"X 在 Y 更 Z"。

grep 触发词(通读级为主):`提醒.{0,4}要紧` `判断.{0,4}很快` `放到.{0,6}时代更` `放到.{0,6}场景`

### 6.2.x 祈使 / 使役名词化要还原成主谓句

定义:用"让 X 做 Y 的 Z"这种使役结构当名词短语主语(让他跳槽的转折、让她改主意的原因、让团队加速的关键),读起来绕,而且"让"字带一点翻译腔(英文 "what made him..." 的直译)。中文更自然的是把它还原成"X 做 Y 的 Z"或"X 之所以做 Y,是因为"。

典型错例 → 改写:

- ✗ 「让他从乙方咨询切到 Braintrust 的转折,是反复观察到客户的同一类问题」 → ✓ 「他从乙方咨询跳槽到 Braintrust 的契机,是反复观察到客户的同一类问题」(去掉"让...的"使役框架,主语直接是"他")
- ✗ 「让她改变主意的原因很简单」 → ✓ 「她改变主意的原因很简单」
- ✗ 「让整个团队提速的关键,是那套自动化基建」 → ✓ 「整个团队能提速,靠的是那套自动化基建」

判断方法:看到"让 X ……的 + 名词"(让……的转折 / 原因 / 关键 / 契机),把"让"去掉,让 X 直接做主语。

反向边界(来自 ngBraLDqzdI session):当"让……的"后面**省略了中心名词**(原因 / 关键 / 契机),只剩"让 X 的,是 Y"时,"的,是"读起来悬空。这种情况要么按上面去掉"让"让 X 直接做主语,要么把省略的中心名词补回来。错例:「让他离开的,是 FAIR 和 Meta 的重心变化」→「让他离开的**原因**是 FAIR 和 Meta 的重心变化」(补回中心名词"原因",去掉悬空的逗号)。

grep 触发词:`让[^,。]{2,15}的(转折|原因|关键|契机|理由|起点|动力)` `让[^,。]{2,15}的[,,]是`

### 6.2.x 开头段的"先否定后断句 / 存现句"连接生硬

定义:开头一两段最容易写出三种生硬连接,grep 抓不到,只能段级精修时逐句读。这三种都集中在文章前两段——读者在第一段卡住就退出,所以这里的连接质量比正文更要紧。

**一、"先否定一个窄断言 + 句号 + 短句补正面动作"。** 英文 "didn't just X. It also Y" 直译进中文会变成"没有只 X。它做了 Y",两句之间断成两口气,转折关系要读者自己接。改成"不仅 X,还 / 并且 / 也 Y"的递进连贯句。

- ✗ 「模型没有只改文字。它顺手做了一个搜索框,带模糊动画,还把无障碍支持一起做好了。」
- ✓ 「模型不仅改了文字,它还顺手做了一个搜索框,并且增加了模糊动画,还把无障碍支持也一起做好了。」

**二、存现句"这件事发生在 X 身上 / 这事出现在 Y 那里"。** 用"发生在……身上"把一个实践硬塞进存现框架,生硬且像翻译腔。直接判断它是什么。

- ✗ 「这件事发生在 Postman 一位 Staff Engineer 身上。」
- ✓ 「这是 Postman 一位 Staff Engineer 的一个实践。」

**三、对比项主语不一致还硬并到一个"是 X,不是 Y"里。** "他看到的是 X,不是 Y"——当 Y 其实在描述另一个主体(AI / 产品 / 模型)的能力状态,而不是"他看到的东西"时,两个并列项主语不一致,读者会卡。拆成各自主语明确的两句。

- ✗ 「他看到的是前端工程本身的边界变化,不是一个玩具 demo。」(「边界变化」的主体是前端工程,「玩具 demo」的主体其实是 AI 的产物,并到一个宾语里主语就乱了)
- ✓ 「他看到前端工程本身的边界变化,AI 不再只能生成玩具 demo 了。」(前半主语「他/前端工程」,后半主语「AI」,各自成句)

判断方法:把开头两段每一句单独抽出来读。出现"没有只…。它…""发生在…身上""看到的是 X,不是 Y"三种结构之一,就回到这条改。路径隐喻式连接("再往前走""这条路")另见 §6.2.x 路径 / 旅程隐喻用作动作,中性陈述里换"再进一步""这个方案"。

### 6.2.x 核心概念锚点要名词化,全文统一

定义:一个概念如果在全文反复作为主题出现(被定义、被分类、被反复指代),它应该是一个**名词锚点**,而不是动词。用动词(「卡住」「掉链子」「跑偏」)反复做主语 / 标题 / 分类对象时,读者每次都要在脑子里把动词转成名词才能继续,而且动词做主语容易主语漂浮。改法:把核心动词名词化(「卡住」→「卡点」),全文统一用这个名词。

错例 → 改写(同一篇文章里「卡住」反复出现):

- H2 标题:「用户卡住,就是产品失败的入口」→「用户的卡点就是问题的入口」(「X 就是 Y」判断句的 X 要用名词短语;「用户卡住」是主谓动词结构做主语,读起来主语漂浮)
- 动作句:「先把『卡住』拆开」→「先拆解卡点」(顺带删掉反复出现的讲者主语,见 narration §6.3.x)
- 分类句:「Verbeek 把卡住分成两类」→「卡点分为两类」

判断方法:找出文章反复出现的核心概念,如果它是动词,问一句「这个概念能不能用一个名词指代?」。能就名词化,然后全文统一——标题、主语、分类对象、指代,都用这个名词。

边界:只针对**反复作为主题锚点**的核心概念。一次性出现的动词(「他卡在某一步」)保持动词没问题,不用强行名词化。

### 6.2.x 受事方的状态用被动句显式

定义:当句子主语是动作的**受事方**(被产品、系统、AI、流程作用的一方),用「被 X」把受事关系显式标出来,避免读者第一眼把它读成施事。中文里「卡住了」「拦下了」「打断了」这类动词,主语既能是施事也能是受事,靠语境判断;当主语明确是被作用的一方时,加「被」更准。

错例 → 改写:

- 「一个用户重复提出同一个请求,可能卡住了」→「……可能被卡住了」(用户是被产品 / AI 卡住的一方,不是主动卡)
- 「用户明确说失败,或者放弃一个本来会继续的会话,也可能卡住了」→「……也可能被卡住了」

判断方法:问一句「这个主语是动作的发出者还是承受者?」。是承受者就用被动「被 X」。

边界:当主语确实是施事(「他卡住了对方的提案」),或动词本身不产生施事 / 受事歧义时,不用强行加「被」。

### 6.2.x 指代量词按场景:泛指一类用「这类 / 这种」,单个实例才用「这个」

定义:回指前文提到的对象时,「这个」指**单个具体实例**,「这类 / 这种」指**一类现象 / 一种模式**。当前文讲的是一种反复出现的现象、一类可归纳的情况时,回指要用「这类 / 这种」;用「这个」会把一类现象错指成孤例,读者会以为只在说刚才那一个例子。

错例 → 改写:

- 「更有价值的信号出现在会话的转折点……这个转折点里藏着修法」 → 「……这类转折点里藏着修复方法」(转折点是一类可反复捕捉的现象,不是单个实例,用「这类」;顺带把口语简写「修法」写全为「修复方法」)

判断方法:回指时问一句「前文说的是一个具体实例,还是一类现象?」。一类现象用「这类 / 这种」,单个实例用「这个」。中文量词丰富(条 / 类 / 种 / 个 / 处 / 道),按对象性质选,别一律「这个」。

边界:确实只指刚才那一个具体对象时,「这个」是对的,不要为了书面感硬换「这类」。

### 6.2.x 评价式导语句要删,直接上内容

定义:作者在真正的内容出场前,先用一句话给它定性 / 评价(「比外界想象的更具体」「也很直接」「很有意思」「这个故事很精彩」「值得说一句」),为后文制造预期。这句导语有两个问题:作者下场替读者评价(违反 §六「作者视角主观评价词类」),同时把"评价"和"内容"拆成两句 / 两段,稀释了信息密度。和 §「主观评判补充句要删」是镜像——那条删**句尾 / 段尾**的评价,这条删**句首 / 段首 / H2 下第一句**的评价导语。

典型错例 → 改写(来自 ngBraLDqzdI session):

- ✗「LeCun 对离开 Meta 的解释,比外界想象的更具体。」(段末) + 下一段开头「他 2013 年底加入 Facebook,2014 年开始组建 FAIR。」 → ✓ 删掉评价句,下一段直接写「LeCun 2013 年底加入 Facebook,2014 年开始组建 FAIR。」(评价导语整句删,事实直接出场;原来下一句的主语"他"并回 LeCun)
- ✗「他给年轻研究者的建议也很直接。如果你读博士,不要把全部精力放在当代 LLM 上。」 → ✓「他给年轻研究者的建议是不要把全部精力放在当代 LLM 上。」(删"也很直接"评价,用"是"把"建议"和"不要 X"合并成一句完整陈述,顺带消掉预告-展开的两句拆分)
- ✗「这件事背后的判断很直接。Agent 已经开始替人使用网页,网页就不能只为人类眼睛和鼠标设计。」 → ✓「这件事背后的判断是,Agent 已经开始替人使用网页,网页就不能只为人类眼睛和鼠标设计。」(用户反复指出的高频坑:"判断 / 建议 / 解释 + 很直接 / 也很直接"是给后文内容预先定调的评价副词,删掉副词、用"判断是 / 建议是"把导语和内容合并成一句。"很直接"对读者零信息量——后文内容直不直接,读者看完自己会感觉到)

判断方法:每个 H2 下的第一句、每个段落的第一句,问一句——这句是在"给后文内容下评价 / 定性",还是"内容本身"?是前者(尤其含「比…更 X」「也很 X」「更具体 / 更直接 / 更有意思 / 更克制」这类定性)就删,让内容直接出场;能和下一句合并成完整主谓宾就合并。

边界:如果这句定性带了**具体信息差**(不是空泛的"更具体",而是"比上一版多了三条限制"),保留信息、只删评价词。

grep 触发词:`比外界.{0,6}(更|想象)` `也很直接` `也很具体` `更具体。` `更直接。` `更有意思。` `挺(犀利|锋利|狠)。` `判断很(直接|清楚|简单)` `(判断|建议|解释|说法)很直接`(评价词单独成句结尾时高发)

同批顺带的改写(同一 ngBraLDqzdI session,各归已有规则,登记备查):

- **H2 标题冗余主语**:「他离开 Meta 是因为方向变了」→「离开 Meta 是因为方向变了」。标题本身已是话题语境,人称主语"他 / 它 / 这个"可省,省掉更紧。判断:H2 以"他 / 它 / 这个"开头,抽掉后语义不丢就删。
- **存现 + 命名啰嗦**(归本文 §冗余「那个叫 X 的 Y」翻译腔):「访谈里还有一条和 AMI Labs 不同的线,叫 Project Tapestry」→「访谈中还提到 AMI Labs 另一条线 Project Tapestry」。"还有一条…的线,叫 Y"是存现句套命名结构,改成动词"提到"直接带出。
- **缺定语 + 含混指代**(归本文 §弱代词 → 具体名词 + §施事 / 受事完整性):「他说,安全风险被高估,开源能启动整个 AI 产业,后来这件事发生了」→「他认为大模型的安全风险被高估了,而开源能启动整个 AI 产业,这促成了 Llama 的开源」。"安全风险"补定语"大模型的";句末"后来这件事发生了"指代含混,补成具体所指"这促成了 Llama 的开源"。

### 6.2.x 提宾作话题主语导致主谓宾颠倒(高发,标题尤甚)

定义:把动词的**宾语**提到句首当话题 / 主语,谓语动词就丢了宾语,读起来颠倒、不完整。常见于"X 不该再让 Y 做 Z"句式——本想说"Y 不该再 [对 X] 做 Z"(X 是 Z 的受事),却写成"X 不该再让 Y 做 Z",把 X 顶到主语位,动词"做"悬在那里没有宾语,读者要回头把 X 接到句末才读得通。这是用户反复指出的高发错误,标题里因为追求"把关键词前置做主语"更容易踩。

典型错例 → 改写:

- ✗「网页不该再让 Agent 连蒙带猜,Google Chrome 给出了新解法」(把"连蒙带猜"的宾语"网页"提到句首当主语,"连蒙带猜"丢了宾语,主谓宾颠倒) → ✓「不该再让 Agent 连蒙带猜网页,Google Chrome 给出了新解法」(宾语"网页"放回动词"连蒙带猜"后面,动词与宾语相邻;标题省略主语没问题)
- ✗「这套数据,团队再也不用手动清洗了」 → ✓「团队再也不用手动清洗这套数据了」(宾语"这套数据"归位到"清洗"后)
- ✗「这个页面,Agent 现在能直接调用」 → ✓「Agent 现在能直接调用这个页面」(确需强调"这个页面"做话题时,在它后面补回指代"它":「这个页面,Agent 现在能直接调用它」,让动词有宾语)

判断方法:写完一句,先找主要动词,问一句"这个动词的宾语在哪里?"。如果宾语被提到了句首当主语 / 话题、而动词后面空着,就把宾语挪回动词后面;确实要把宾语前置做话题时(强调它),必须在动词后补一个回指代词("它 / 这个"),不能让动词裸着。

为什么写稿(尤其拟标题)时容易写出来:作者想把最重要的关键词(网页 / 这套数据 / 这个页面)放到最显眼的句首位置,顺手把它当成了主语,但它在语义上其实是后面动词的宾语。前置和提宾是两件事——前置做话题可以,但动词的宾语位不能因此空着。

grep 触发词(很难精确,扫描提示):标题与开头段里出现"<名词>,?<某主体>不?(再)?(该|能|要|用)让?<某主体><及物动词>[,。]"且及物动词后紧跟逗号 / 句号(宾语缺位)时,人工核查动词的宾语是不是被顶到了句首。

### 6.2.x 动作链白描的机械感:补因果 / 被动 / 回指 / 收尾

定义:把一连串短动作句直接拼接("A 把 B 怎样,C 都白做。D 消耗了 E,最后仍然 F"),句与句之间不加因果连接词、受事不标被动、名词不回指前文、收尾用单字截断,读起来像机械直译——信息没错,但每一跳都硬。这类句子 grep 抓不到,只能段级精修时逐句读,问"这一串动作之间的逻辑、受事、指代、收尾,有没有交代清楚"。

修复四个动作(按需用,不必全上):

1. **补因果连接词**:动作之间是因果关系时,加"导致 / 于是 / 结果",别只用"把"硬接。
2. **受事用被动标出**:被动作作用的一方做主语时,用"被"显式标(详见本文 §受事方的状态用被动句显式)。
3. **名词补回指定语**:泛指名词("内容""页面")回扣前文具体步骤,加一个定语锚回去("刚截屏的内容")。
4. **收尾补完整**:单字 / 半句收尾("点错""白做")补成完整动宾("点错了地方")(详见 narration §单字结尾收句不顺口)。

典型错例 → 改写:

- ✗「如果页面顶部刚好加载了一条广告,把内容往下推,前面这些推理都可能白做。Agent 消耗了一堆 token,最后仍然点错。」 → ✓「如果页面顶部刚好加载了一条广告,**导致**刚截屏**的内容被**往下推,前面**的**这些推理都可能白做。Agent 消耗了一堆 token,最后仍然点错**了地方**。」(① "把内容往下推"→"导致刚截屏的内容被往下推":补因果"导致"+ 受事被动"被"+ 回指定语"刚截屏的"回扣上一句的截屏步骤;② 收尾"点错"→"点错了地方"补完整)

判断方法:段级精修时,遇到由三四个短动作句直接拼起来、读着"硬"的句子,逐项过这四问——动作之间是不是因果(缺"导致"?)、有没有受事被当主语(缺"被"?)、泛指名词指的是前文哪一步(缺回指定语?)、收尾是不是单字截断(缺宾语 / 补语?)。命中几条补几条,不必四条全上,目标是让动作链读起来有逻辑黏连,不是机械罗列。

边界:刻意用短促动作句营造紧迫节奏时(如"他冲进来,抓起电话,拨号"),保留断裂感是对的。只针对说明 / 论证语境里因缺连接而显得机械的句子。

### 6.2.x 名词作谓语要补动词

定义:中文名词不能直接用在谓语位置，需要搭配动词。常见名词如「复利」「债」「损耗」直接用在「会/能」之后，读起来缺一个动词支撑。

典型错例 → 改写:

- ✗「好奇心会复利」→ ✓「好奇心会产生复利」("复利"是名词，补"产生"作动词)
- ✗「好奇心的价值在这种持续试探里会复利」→ ✓「……会产生复利」(同上)
- ✗「这条路会变成债」→ ✓「这条路会留下技术债」("债"作名词，补出完整宾语)

高发位置：章节标题、段落首句。标题里"会 X""能 X"后接名词时必查。

grep 触发词：`会[复债耗损]` `能[复债耗损]` + 人工判断"X"是否是名词。

### 6.2.x 形容词谓语改"是/不是"名词判断

定义:用"X 并不新 / X 不轻 / X 不小"这类「主语 + 否定副词 + 形容词」做判断时，中文读者要先把形容词反推成名词概念。改成「X 并不是新 Y / X 不是轻量的」，用「是/不是」直接给名词判断，更快抵达意思。

典型错例 → 改写:

- ✗「这些词并不新」(读者要自己推"不新=老词=已有说法") → ✓「这些并不是新词」(直接给名词判断)
- ✗「这个问题不小」→ ✓「这是一个大问题」
- ✗「这份责任不轻」→ ✓「这份责任很重」(正向形容词比双重否定更直接)

判断方法:看到「并不 + 形容词」或「不 + 形容词」作谓语时，问"能不能改成'是/不是 + 名词短语'?"。能改的改，尤其当形容词的对立名词在中文里有固定表达时("新→新词""小→小问题")。

grep 触发词：`并不[新旧轻重大小简复]` `不算[新旧轻重大小]`

### 6.2.x 长宾语前置导致"把"字句绕口

定义:用「把 + 长名词短语 + 动词」结构时，如果名词短语很长（超过 6 字），读者的注意力陷在宾语里，主要动词来得很晚。中文更自然的方式是把动词提前，让主谓关系第一时间成立，再跟宾语。

典型错例 → 改写:

- ✗「AI 把一个人的产出上限抬高以后」→ ✓「AI 抬高一个人的产出上限后」(动词"抬高"提前，主谓宾直接成立)
- ✗「他把整个团队的注意力吸引到这件事上」→ ✓「他吸引了整个团队的注意力」

判断方法:看到「把 + 超过 6 字的名词短语 + 动词」，试着删掉"把"、把动词提前，读是否更顺。多数情况更顺。

边界:当"把"字句有强调受事的语用功能（"他把这件事搞定了"，强调已完成），保留。只针对长宾语前置后动词显得姗姗来迟的情况。

grep 触发词：`把[一-鿿]{6,}[动了过]`（命中后人工判断是否绕口）
