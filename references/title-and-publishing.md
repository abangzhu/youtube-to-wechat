# 标题与发布准备

读取时机：文章定稿后拟标题、加章节标题、写来源注记、准备发布。

## 八、标题策略

**核心结构**：具体主体 + 可验证张力 + 来源锚点（可选）。三个元素中至少两个。

- **具体主体**：读者一眼就能在脑子里锚定的对象。Mission / Anthropic 增长部门 / MCP / Pydantic 创始人 这种，不是「Agent」「方法论」这种抽象概念。
- **可验证张力**：暗示一个判断或悬念，但文章正文真的要给出答案。"如何打造"、"什么才是稀缺的"、"一人跑赢整支团队"、"要退出历史舞台了" 都属于这一类。
- **来源锚点**：「From Lenny's Podcast」「MCP 联合创始人谈」这种出处标签，给判断加权威。非必需，但在访谈类内容里很好用。

  **中文机构名前缀格式**：`红杉：` `YC：` `OpenAI：` 这种"机构名 + 冒号"作为标题前缀，表示内容来源于该平台/节目，不等于"该机构投资/声明/背书"。比「Sequoia Capital 节目里的 X」更简洁，比「Sequoia 投资的 X」更安全。**只有能从文章正文或公开资料确认投资关系时，才能用「X 投资的」**；来源仅是节目、访谈或演讲时，用「红杉：」这种前缀格式。

**长度**：15–25 字最舒服，最长不要超过 30 字。

**样本标题参照：**

- 「Mission 是如何打造 Multi-Agent 系统的」
- 「专业技能平权后什么才是稀缺的？From Lenny's Podcast」
- 「Anthropic 增长部门如何一人跑赢整支市场团队」
- 「MCP 要退出历史舞台了？MCP 联合创始人谈 MCP 的未来」
- 「罗马军团式的公司过期了，YC 提出了一种全新的公司形态」

**问题句 vs 陈述句：** 两种都能用。问题句必须真问真答，文章里要明确给出答案，不能光问不答。演讲类、访谈类内容问题句常常更好，因为描述性标题预设结论，问题句留给读者自己确认。

### 8.x 从“如实概括”到“制造认知缺口”——我们最常见的标题短板

定义：我们的内容质量常不输头部账号，但标题习惯“如实概括内容”——把结论写进标题、用读者不认识的裸人名、复盘已经发生的事。结果是读者扫一眼就拿到了结论，没有点击动机。更好的做法是在**文章兑现得起的前提下**，制造一个“认知缺口”驱动点击。这一节是出标题时的思维总纲，先于下面的具体句式模板。

**核心诊断**（同一篇文章的两个标题，大 V 版效果显著更好）：

| | 标题 | 诊断 |
|---|---|---|
| 短板写法 | Boris 和 Cat 复盘 Claude Code 过去一年：从写代码到接管工作流 | ① 裸人名（圈外读者不认识 Boris/Cat）② 复盘**过去** ③ 冒号后**把结论说完了**，读者不必点进来 |
| 目标写法 | Claude Code 之父一周年回顾：下一年的形态，一定跟现在完全不同 | ① 封号（“之父”自带权威）② 指向**未来** ③ 给了强断言但**没说怎么个不同法**，留了缺口——而文章里真能填上，所以不是标题党 |

**六条转换规律（短板写法 → 目标写法）：**

1. **认知缺口 > 信息完整（头号规律）**。别把结论写进冒号后面，留一个“怎么做到 / 为什么 / 是什么”的缺口。`从写代码到接管工作流`（结论说完）→ `下一年一定跟现在完全不同`（留“怎么个不同法”）。判断：把标题给没读过文章的人，他是“已经知道结论了”还是“想点进来确认”？前者就是剧透。
2. **身份封号 > 裸人名**。`Boris 和 Cat` → `Claude Code 之父`；`Sutton` → `强化学习之父 Sutton`。圈外读者不认识真名，但认识称号背后的权威。封号必须真实（确是该领域公认源头 / 创始人 / 代表），不能为冲击力硬封。
3. **前瞻断言 > 过程复盘**。读者更想知道未来。`过去一年从 X 到 Y` → `下一年的形态会完全不同`。**边界见分界线**：只在原素材真讲了未来判断时用。
4. **硬数字 / 规格前置**。`300 个 Agent` `400 Tok/s` `四大核心阶段`——可验证的具体物，制造画面 + 信息密度承诺。数字必须来自正文，不为冲击力放大。
5. **受众点名 / 第一人称场景**。`非程序员们` `我桌面 AI 伴侣`——让目标读者对号入座，比泛泛“适合做 X”更勾人。
6. **强结论词承载真实判断**。`绝配` `法典` `一定完全不同` `从根本上无法`——有态度有重量，但口语真实，不是“震惊体”。这条直接连分界线：强词的合法性来自正文真有这个判断。

**大牛署名是钩子，但角色要标准**（规律 2 的延伸）：当论文 / 项目署名里出现业界公认的大牛（何凯明、谢赛宁、LeCun、Karpathy 这类圈外也认得的名字），即使他们不是一作 / 项目主创，把名字放进标题或开头也是强爆款元素——大牛本身就是「封号」。但必须先回原始资料核实其真实角色（Leadership Sponsor / Advisor / 共同作者 / 主创），标题和正文用「署名里有 X」「X 作为 Y 参与」这类措辞，**不能写成「X 的论文」「X 团队」让读者误以为是主创**（违反分界线的「真实身份」约束）。本 case：何凯明、谢赛宁是 Vision Banana 论文的 Leadership Sponsors，项目主创是 Valentin Gabeur 等，标题写「何凯明、谢赛宁署名：……」而非「何凯明谢赛宁的论文」，正文与来源行另注明二人的 Leadership Sponsors 身份。

**留白的标记手法**（规律 1 的具体招式）：用引号包裹一个悬念占位，明示“答案存在，但要点进来看”。样本：`强化学习之父 Sutton：LLM 从根本上无法做出科学发现，原因出在缺失了“这一步”`。“这一步” / “这个东西” / “这一点” 是可复用的留白槽——前提是文章真的给出了那一步。

**四个样本标题的手法速查：**

- `AI 变了，要用 300 个 Agent 解救非程序员们`——数字（300）+ 情绪动词（解救）+ 受众点名（非程序员们）
- `这个 400 Tok/s 的开源模型，是我桌面 AI 伴侣的绝配`——硬规格前置 + 第一人称场景 + 强结论词（绝配）+ “这个”设悬念
- `强化学习之父 Sutton：LLM 从根本上无法做出科学发现，原因出在缺失了“这一步”`——封号 + 反共识断言 + 引号留白
- `Anthropic 发布「AI 原生创业公司」手册：涵盖全流程四大核心阶段，一人公司法典来了`——机构背书 + 事件动词（发布）+ 量化结构（四大阶段）+ 重量级封号（法典）

#### 分界线：钩子可以强，但承诺必须兑现

> 标题制造的每一个缺口 / 断言 / 数字 / 封号，正文必须有对应的答案 / 论证 / 出处 / 真实身份。

- ✓ 合规：`缺失了“这一步”`——文章真给了那一步；`一定完全不同`——文章真论证了为什么；`300 个 Agent`——正文真有这个量级
- ✗ 越界（回到 §八末尾「避免」清单）：`你绝对想不到`（纯悬念不揭晓）、`改变一切 / 颠覆认知`（夸大到与内容不符）、原素材没讲未来却硬写 `下一年会颠覆`（承诺文章没有的东西）、为冲击力硬封不实头衔

**最易滑坡的是规律 3（前瞻）和规律 6（强词）**：原素材只复盘了过去，你硬写“下一年会 X”，就从“留缺口”滑成“编缺口”。出这类标题前回原文确认：这个未来判断 / 这个强断言，讲者或文章真的给了吗？

#### 出标题前自检清单

- [ ] 标题里的人名，圈外读者认识吗？不认识就换成封号 / 身份标签（规律 2）
- [ ] 冒号后面是不是把结论说完了？说完了就改成留缺口（规律 1）
- [ ] 有没有可前置的硬数字 / 规格 / 量化结构？有就提到前面（规律 4）
- [ ] 能不能点名受益人群或给第一人称场景？（规律 5）
- [ ] 是复盘过去还是指向未来？未来更勾人——但确认原素材真讲了未来（规律 3 + 分界线）
- [ ] 标题里每个缺口 / 断言 / 数字 / 封号，正文都兑现得起吗？兑现不起就是标题党，降级（分界线）

### 8.x 旧框架过期 + 新形态登场（双段式陈述句）

定义：把标题写成两个分句，前一句宣告"某种为人熟知的旧组织 / 旧做法 / 旧框架 X 已经过期"，后一句宣告"某个具体主体 Y 提出了一种新形态 / 新替代 / 新做法"。一前一后，旧体系的死和新体系的生在同一行里完成切换，读者点进来时已经被锚住要看「Y 提的新东西到底什么样」。

为什么有效：

1. **前半句自带反常识**——"X 过期了"立刻挑战读者对组织 / 行业默认假设的信念，制造点击动机
2. **后半句给具体主体 Y**——主体必须是具体的人 / 公司 / 团队，不是抽象概念。读者通过 Y 的可信度决定要不要相信这个判断
3. **不靠 A 变贵 B 变便宜的对比口号**——这个结构是"旧死 + 新生"的递进，而不是"两件事互相对立"，避免把复杂论点压成廉价对比

适用情境：原素材里讲者明确要替换某种旧组织 / 旧框架 / 旧方法，且在文章里给出了一种可识别的替代形态。如果原素材只是抱怨旧的、没给替代品，不用这个结构。

样本：

- 「罗马军团式的公司过期了，YC 提出了一种全新的公司形态」（具体主体是 YC，旧框架是罗马军团式公司）
- 「面试驱动的招聘走到头了，xx 提出了一种新的工程师评估流程」（假设样本，不要套用未真实出现的内容）

避免变体：

- "X 过期了，未来是 Y" / "X 是过去式，Y 是未来" — 太口号，没有具体主体
- "X 死了，Y 万岁" — 戏剧化过头
- "X 不行了，应该 Y" — 教训口吻，读者抗拒

### 8.x 教学问句式（陈述句形态）：「X 是如何 Y 的」

定义：把标题写成「<具体主体> 是如何 <关键动词短语> 的」「<具体主体> 怎么 <关键动词短语>」「<具体主体> 内部 <关键名词> 的方法」这种教学问句结构。表面是问句结构，本质是陈述句——读者一看就知道文章会回答"X 怎么做的"。

为什么有效：

1. **自带教学口吻**——"是如何 Y 的"承诺读者会拿到一份可复用的做法，比"X 把 Y 换成了 Z"这种反潮流陈述更适合实操干货
2. **主体必须具体**——X 是具体的公司 / 团队 / 个人，不是"AI 时代""所有团队"这种空泛主体
3. **关键动词必须有信息量**——"使用 X""管理 X""搭建 X""治理 X""跑 X 到规模"，不能是"做事""工作"这种空动词

适用情境：

- 内部经验分享、工作方式 walkthrough（"Anthropic 是如何使用 Claude Code 的"、"Stripe 怎么管理 Feature Flag"）
- 框架介绍、方法论拆解（"Linear 是如何决定优先级的"）
- 实操干货、流水线 / 工具链类（"Notion 怎么做内部知识库"）

样本：

- 「Anthropic 是如何使用 Claude Code 的」（本 case：实操 walkthrough，三层进阶 + 具体设置组合）
- 「Stripe 怎么把支付错误率压到 0.01%」
- 「Vercel 内部 Code Review 的方法」

和反潮流陈述式的取舍：

| 标题类型 | 适合稿件 | 钩子尖度 | 教学含金量 |
|---|---|---|---|
| 反潮流陈述式（"X 把 Y 换成了 Z" / "X 工程师当面唱衰 Y"） | 有冲突感、有反常识判断的 | 高 | 中 |
| 教学问句式（"X 是如何 Y 的" / "X 怎么 Y"） | 实操 walkthrough、方法论 | 中 | 高 |
| 双段式（"旧 X 过期了，Y 提出 Z"） | 同时有反潮流 + 替代方案 | 高 | 高 |

判断方法：稿件的核心是"反某个共识"还是"分享一套做法"？反共识用陈述式，分享做法用教学问句式。两者都有时考虑双段式。

避免变体：

- "X 是怎么做 Y 的（揭秘）"——加"揭秘"会变成标题党
- "X 是如何用 Y 的"（动词太弱）—— "用"是空动词，要换成"使用 / 治理 / 管理 / 跑 / 搭建"等更具体的动词
- "X 是如何成功的"——"成功"是抽象结果词，没信息量，要换成具体动作
- "如何 Y"（缺主体）—— X 是必填项，没主体读者点不进来

### 8.x 祈使式 H2 章节标题

定义：把章节小标题写成祈使句或对仗短句，常用三种结构。「X，不要 Y」「先 X，再 Y」「X 是 A，Y 是 B」。这种标题自带行动指令或权衡对比，读者扫到时能直接拿走一个判断。

样本：

- 「烧 token，不要烧人头」（X，不要 Y）
- 「MVP 阶段先写 CLAUDE.md，再写第一行生产代码」（先 X，再 Y）
- 「业务 context 是资产，软件是消耗品」（X 是 A，Y 是 B）
- 「没记下来的事，对你的 AI 就等于没发生过」（结果型陈述）
- 「拆成多个 prompt 还有一个隐藏好处」（带钩子）

为什么有效：

1. **指令感强**——「不要 Y」「先 X」直接把读者带到行动层，不是停留在描述层
2. **对仗短**——「X 是 A，Y 是 B」用六到十个字完成两个判断的并置，节奏紧
3. **可独立成立**——这一类标题脱离上下文也读得通，读者扫到一个就能记住一个

适用情境：素材里这一节确实有强烈的"该做什么 / 不该做什么"对比，或者一个核心判断可以压缩成对仗的两个分句。如果材料只是描述事实没有立场判断，不要硬套这个结构。

避免误用：

- 「X，不要 Y」用在没有 trade-off 的场景里——比如「读懂代码，不要复制代码」是无意义的对立
- 「先 X，再 Y」用在没有时序意义的场景里——比如「先理解，再表达」太抽象
- 把任何并列两件事都套成「X 是 A，Y 是 B」——用于真正能形成资产 vs 消耗、原因 vs 结果这类二分时才有力

### 8.x H2 要压成可复用判断,不是素材目录

定义:章节标题不只是标出这一节讲什么,还要把这一节的可复用经验压成一句可带走的判断。读者只扫 H2 时,应该已经能带走方法、权衡或反常识结论。

优先使用三类标题:

- **事故后的机制判断**:「让老实干活,比撒谎更省事」「每次失败都是 Harness 的 bug」
- **反直觉结果**:「删掉 95% 的 Skill 更准」「少给上下文,输出反而稳定」
- **行动指令**:「强制它,别指示它」「先拿证据,再看代码」

避免把 H2 写成素材目录:

- ✗「内部工具 case」
- ✗「WorkOS CLI」
- ✗「三条原则」
- ✗「复盘 agent」

改写方法:

1. 先问这一节读者具体带走什么
2. 把答案写成带动词的短句
3. 具体工具名、人物名、产品名下放到正文第一句

示范:

- 「内部工具 case」→「他把 Agent 当成会撒谎的初级工程师」
- 「测试证明」→「让老实干活,比撒谎更省事」
- 「三条原则」→「强制它,别指示它」

判断方法:把所有 H2 抽出来连读。如果这一串像目录,回去改标题;如果这一串像一组可以复用的工程判断,合格。

**避免：**

- 动词与名词配价错误，例如「打造时刻 / 创造机会」——"时刻/时机"是到来的，配「迎来 / 开启 / 触发 / 将有」；"阶段/浪潮"配「进入 / 开启」；只有"产品/系统/体系"才配「打造 / 构建 / 搭建」。改：「打造 AI 记忆的 ChatGPT 时刻」→「AI 记忆将有自己的 ChatGPT 时刻」
- 制造悬念但文章没有揭晓，例如「你绝对想不到」
- 夸大重要性，例如「改变一切」「颠覆认知」
- 泛泛描述，例如「X 的现状与未来」
- 把复杂观点压成口号的对比框架，例如「A 变贵了，B 变便宜了」
- 纯抽象判断做标题，例如「AI 可观测性不是一个独立品类」——缺少具体主体，读者点不进来
- 标题里带可省略的限定词或主观未来时，例如「**一位** DeepMind 工程师当面唱衰 MCP」「模型自动 fallback **是 KP 想要的未来**」「**X 觉得 / 我觉得** Y 是 Z」「Y **可能成为** 下一个范式」。改法：去掉「一位」直接 `DeepMind 工程师当面唱衰 MCP`；去掉「是 X 想要的未来」改成方案命名 `将模型自动 fallback 做成 Harness 内置能力`；去掉「我觉得」改成事实陈述。理由：标题是文章的承诺凭据，每一个字都该承担信息量。「一位」「是 X 想要的」「可能成为」这种限定词或主观未来时，没有它读者也能正确理解，加上反而稀释钩子。
- **依赖正文解释才能看懂的标题**。具体两类:
  - **非周知缩写**:PVP、HRT、ELT、ETL、MoE、CRDT 这种圈外人不立刻认识的缩写。判断标准:把标题给一个不读 AI / 工程圈日常 newsletter 的中文读者看,他能不能立刻 infer 出意思?不能,就换大白话。例如「PVP 解饱和」→「让模型互相打游戏」。
  - **专业领域隐喻**:「解饱和」(化学)、「击穿」(物理)、「跌停」(金融)、「爬坡」(自行车 / 跑步)、「破防」(游戏)、「拉满」(游戏)。这些词在圈内是术语,在公众号读者那里要么需要正文解释、要么造成误读。改法:换成动作描述。
  - 周知缩写不在此列:RAG、API、SDK、LLM、PR、CLI、Agent 这些已被多数中文读者接受,可以裸用。
  - 判断分界:这个词需要走进文章前两段才能让读者懂吗?是 → 换。读者一眼看完标题就懂吗?是 → 留。

grep 触发词(章节标题层):章节标题里出现纯英文缩写 ≥3 字母 + 没有立刻给中文等价词、出现「解 X」「击 X」「跌 X」「爬 X」「破 X」「拉 X」这类专业领域动词,人工通读判断该缩写 / 隐喻是否周知。

### 8.x 标题词精度：能力下放表述 + 非货币性价值形容词

**能力下放句：「AI 给了 X 做 Y 的能力」优于「AI 让 X 能做 Y」**

`AI 让谁都能做出来` 是使役结构，把能力转让写成"催化动作"，读者感受的是 AI 在驱动；`AI 给了谁都能做出来的能力` 把"能力"显式做名词，读者感受的是"一件具体的东西被转移了"。后者更有画面感，也让标题的主张更具体可验证。

- ✗「AI 让任何团队都能做出来」→ ✓「AI 给了任何团队做出来的能力」
- ✗「AI 让非程序员也能写代码」→ ✓「AI 给了非程序员写代码的能力」

触发条件：标题里出现「AI 让 X 能/都能 Y」结构，试着改写成「AI 给了 X 做 Y 的能力」，看是否更具体。

**非货币性价值用「宝贵」，不用「贵」**

`更贵了` 在中文里优先触发价格联想（这东西涨价了）；当描述判断力、品味、技能、稀缺程度这类非货币性价值时，`宝贵` 准确传达"珍贵、不可替代"的含义，没有价格联想的干扰。

- ✗「taste 反而更贵了」→ ✓「taste 反而更宝贵了」
- ✗「判断力变贵了」→ ✓「判断力变得更宝贵了」

判断方法：标题里出现「更贵」「变贵」，问一句——这里的"贵"是说价格，还是说珍贵/稀缺程度？是后者就换「宝贵」。

**H2 标题词汇要映射段落核心内容，不取局部词**

定义：H2 标题里的主名词/动词应该是段落里反复出现、力度最强的核心词，而不是段落里一个顺手抓来的局部词。局部词放进标题会让读者扫完标题后对段落主旨判断错误。

典型错例 → 改写：

- ✗「下一轮优势会回到系统」（段落核心词是"竞争"和"基础设施"，"优势"和"系统"是局部词）→ ✓「下一轮竞争会回到基础设施」
- ✗「模型只建议，平台来决定」（单字谓语力度不足，且段落核心词是"建议/决策分离"这条原则）→ ✓「模型只提供建议，平台来做决定」

判断方法：写完 H2 后，读段落首句和末句，找出段落里频率最高/被重复强调的名词和动词，和标题对比——标题里的核心词是它们吗？不是就换。

同时检查单字谓语：H2 标题里的谓语动词是单字（建议/决定/执行）时，尝试扩展成动宾短语（提供建议/做决定/完成执行），力度更完整。

**H2 标题要直白完整，避免意象化句式**

定义：H2 标题使用隐喻、意象或"因果诗意"句式时，读者需要停下来 parse 比喻义，扫读时无法一眼理解。标题应让没有看过正文的人一遍读懂"这节讲什么"。

典型错例 → 改写：

- 「模型不说失败，幻觉就有了入口」（"有了入口"是意象比喻）→「模型不声明失败就会引发幻觉」（直白完整句）
- 「错误在黑暗中蔓延」（散文意象）→「没有错误日志，Agent 会悄悄出错」
- 「网页已经开始反过来误导 Agent」（"反过来"是多余方向副词）→「网站已经开始误导 Agent」

**H2 标题精简：删多余副词和方向词**

H2 里出现「反过来/反而/倒是/居然/甚至」等方向副词时，先尝试直接删掉，看语义是否仍然完整。动词本身往往已包含方向信息，副词是冗余。

反例：「网页已经开始**反过来**误导 Agent」→ 删"反过来"：「网站已经开始误导 Agent」

判断方法：把每个 H2 标题念给没看过原文的人，他能不能一句话说出"这节在讲什么具体问题"？能——合格；需要追问"什么叫入口""反过来是什么意思"——改直白。

### 8.x 标题施受关系核查：『A 服务 B』还是『B 服务 A』

定义：标题里出现"A 服务 B""A 支撑 B""A 撑住 B"这类及物动词时，A、B 谁是施事、谁是受事必须核对清楚。方向写反不会让句子读不通，却会让读者理解成完全相反的意思——这类错误 grep 抓不到，语法也通顺，只有对照正文论证方向才能发现。

典型错例 → 改写（来自 9DHZLw5653E session）：

- 「2000 万行代码能服务 Agent」——字面读成"代码库为 Agent 提供服务"，但正文实际讲的是"Agent 能不能扛住 / 驾驭 2000 万行规模的代码库"，受益方向反了 → 「Agent 扛得住 2000 万行代码库」

判断方法：写完含"服务 / 支撑 / 驾驭 / 扛住"这类及物动词的标题，把 A、B 的角色互换读一遍，问自己"正文说的到底是谁在服务谁"，选真正符合正文论证方向的那个词序。

### 8.x 标题里『V + 先 + 动词』读起来悬空，改成定语名词短语

定义：标题写成"X 先 V1"（比如"Honk 先从维护工具长出来"）时，"先"字暗示后面还有一个对应的"后来 V2"，但如果标题本身不给出这个后续动作，"先"就悬在那里，读者会觉得这句话没说完。改法是把整句压成一个带定语的名词短语，让"V1"这个过程降级为修饰语，主体本身做核心词。

典型错例 → 改写（来自 9DHZLw5653E session）：

- 「Honk 先从维护工具长出来」（"先"没有呼应的"后"，读起来卡）→ 「从维护工具长出来的 Honk」（去掉"先"，把"从维护工具长出来"整体降级为定语，Honk 做句子的核心名词）

判断方法：标题里出现"先"字，检查是否有对应的"再 / 后来 / 接着"呼应。没有呼应就删"先"，把动词短语转成定语修饰主体，或改成完整的两段式陈述。

### 8.x 『从 A 走向 B』标题里 A、B 不同类时要补评估动词

定义：标题写"X 会从 A 走向 B"这种句式时，如果 A、B 不是同一类可以相互转化的东西（比如"PR 数"和"用户价值"），字面上会被读成"A 本身变成了 B"，但实际讲的是"评估维度从盯着 A 转向盯着 B"。这时要在 A、B 前补一个共同的动词（看 / 盯 / 衡量 / 抓），显式说明这是同一个动作在两个对象间切换，不是 A 变成了 B。

典型错例 → 改写（来自 9DHZLw5653E session）：

- 「ROI 会从 PR 数走向用户价值」（"PR 数"不会变成"用户价值"，字面有歧义）→ 「ROI 会从看 PR 数走向看用户价值」（补"看"，显式两者是同一种评估动作切换的对象，不是互相转化）

判断方法：写完"从 A 走向 / 到 B"的判断句，问自己 A 能不能真的变成 B？不能就在 A、B 前各补一个共同动词（看 / 盯 / 衡量 / 抓）。

### 8.x H2 标题里「名词，动词短语」的松散并列要收成完整动宾句

定义：H2 标题写成"X 一词，触发 Y 模式"这种"名词 + 逗号 + 动词短语"的松散并列结构时，前半是孤立的名词短语，后半才是真正的判断，中间靠逗号硬接，读起来像两个半句拼起来，不像一句可以独立成立的判断。改法是把名词收进动宾结构里做宾语，让标题变成一个完整、紧凑的动作句。

典型错例 → 改写：

- 「workflow 一词，触发编排模式」（"workflow 一词"是孤立名词短语，靠逗号接后半句）→「通过 workflow 触发编排模式」（"workflow"收进"通过...触发..."的动宾框架，一句话读完）

判断方法：H2 里如果开头是一个名词短语 + 逗号，试着把这个名词塞进"通过 / 用 / 靠 X + 动词 + Y"的框架里重写，看是否比原来的两段式更紧凑。

### 8.x H2 标题避免「比 X 更 Y」式主观比较，换成客观分类描述

定义：H2 标题用"这 N 种 / 这套方法比 X 更值得记 / 更好用"这类比较句式时，"更值得记"是作者自己的价值判断，不是内容本身的事实描述,读者看完标题得到的是作者的评价而不是这一节讲了什么。换成客观的"...的 N 种 / 类 X"描述式标题，信息量更直接，也不需要读者认同作者的评价才能理解标题。

典型错例 → 改写：

- 「六种模式比 API 更值得记」（"比 API 更值得记"是作者自己的比较评价）→「Subagents 的 6 种工作模式」（去掉比较评价，直接给出这一节的内容分类）

这条和正文里"主观评判补充句要删"（见 [self-review.md](self-review.md) 段级精修必查清单）同源：标题层面的"比 X 更 Y"和正文里"...更值得记，因为 Z"这种评价 + 理由从句，都是作者在替读者下判断，不如直接给客观事实或分类。

判断方法：H2 或正文判断句里出现"比 X 更 Y"，问自己这个"更 Y"是不是作者自己的主观评价（更值得记 / 更好 / 更重要）？是就删掉比较，换成客观描述这一节的内容是什么。

## 九、结尾策略

样本里有两种常见的收尾套路，对应不同的文章类型：

**第一种：启发清单型**。适合方法论密集、可迁移动作多的文章。最后一段不是 Markdown 列表，而是用自然段落把 3–5 条读者能直接拿走的建议说清楚。Mission 篇以及「值得抄作业的三件事」这种收尾都是这种。写的时候每条建议要带一个具体动作或场景，不要只写抽象原则。

**第二种：收束画面型**。适合偏观点、偏视野的文章。关键动作是"抽象爬到顶之后降回一个具体画面"——如果文章前面在讨论尼采、戴尔菲神谕、AI 模拟论这种抽象度很高的内容，结尾不要再加一条抽象结论，而是让镜头切回一个具体的人、一个动作、一个场景。这种收尾放弃了"替读者得出结论"的姿态，用画面替代判断。

Max Schoening 那篇的"去你所在的城市或森林走一走……对于身边大多数东西，只要给你 6 到 9 个月，你都可以弄清楚怎么从零做出来"是一个样本。另一个样本是在一段讨论虚无和永恒回归的文章里，作者把结尾落在一位朋友照顾小狗、在日常里找到安静的画面上。抽象的问题不会被这句画面"回答"，但它让读者在自己的情绪里自然找到落点。

**如果原材料能对应具体的实操场景**，在结尾（或接近结尾处）自然做一次桥接，把抽象判断落到一个具体岗位、具体决策、具体团队日常的动作上。不硬凑。示范：

- 「这个判断对正在做企业流程自动化的团队有直接参考意义。」
- 「很多产品团队习惯把流程逻辑写进代码和状态机，等到模型升级就得重写一轮。」
- 「做企业数据 Agent 的团队，至少要有一个通用 Query Tool 兜底。」

不是每篇都要有。桥接的价值在于把判断具象化到读者能辨识的场景里，不是做「海外 vs 国内」「别人 vs 我们」「成熟市场 vs 本土」这种地域/阵营二分。

**地域二分是桥接最常见的翻车姿势**。出稿时如果下笔写出「国内团队」「中国 AI 圈」「海外做法」「外国团队」这类词，停下来问一句：这个实操经验是否真的只适用于某一地域，还是本来就是通用的？样本材料大多数时候讲的是通用工程判断（「Tool Description 要写三层」「错误要返回给 Agent 自修复」），它们不是进口商品，不需要「搬回」某个地域。把地域限定词拿掉，让经验自己站住。

只有当材料里确实讨论了明显的中外差异（监管、供应链、特定市场数据、本地产品生态），才保留地域指向，且要写得具体：不是「国内团队更倾向于」，而是「做跨境支付合规的团队在中国境内执行时……」。

**结尾禁区：**

- 以「总结一下」「综上所述」「让我们一起」开头的段落
- 空洞的升华，比如「这是属于每一个从业者的时代」
- 纯口号，比如「让我们一起迎接 AI 时代」
- 廉价反问，比如「你准备好了吗？」

**事件总结口吻不可用——要用概念回顾口吻**

定义：「回头看,这场 X 讲的是 Y」「回顾一下这次分享」「这场 workshop 的核心是」「这次访谈给我的启发是」——以"那场分享 / 那次会议 / 那个采访"作为回顾对象，本质是"作者站出来替读者总结一场事件"。这种口吻有两个问题：把读者从概念拉回到来源场合（破坏前文沉浸），且作者下场做总结违反 §六之三的"判断前后元评价"。

更好的做法是**概念回顾口吻**：回顾的对象不是事件，是文章主题（X 的方法 / X 的做法 / X 的三个阶段）。读者读完已经在概念层，结尾停留在概念层比拉回事件层更自然。

典型错例 → 改写：

- 「回头看,这场 31 分钟的 workshop 讲的是 Anthropic 内部已经稳定下来的一套工作方式」 → 「回顾一下 Anthropic 如何使用 Claude Code 的方法」
- 「这场分享的核心是 X」 → 「X 的核心是」 或 直接删铺垫，给具体内容
- 「这次访谈给我的启发是 X」 → 「最值得带走的判断是 X」（注意 §段尾"可以借鉴的"桥接句换 Tips: 标签 — 桥接句若太多直接换 Tips:）
- 「回顾这次会议」 → 「回顾这套做法」 / 「回顾这套方法的三个阶段」

判断方法：写完结尾段，读第一句。问"这一句指向的是来源场合（演讲 / 文章 / 访谈），还是文章主题（X 的方法 / X 的做法）？"。指向来源场合就改。

边界：当文章本身就是关于"那场会议 / 那次发布"的事件评论（比如 OpenAI DevDay 的现场观察、苹果发布会的吐槽），可以保留事件总结口吻，因为事件本身就是文章主体。当文章是从来源里提炼出可复用经验的工作方式型稿件，必须用概念回顾口吻。

grep 触发词：`回头看[^一-鿿]{0,5}这场` `回头看,这[场次]` `这场.{0,5}讲的是` `这次分享.{0,5}核心` `这场.{0,5}核心是` `这场.{0,5}给我的启发` `回顾.{0,5}这[场次]`

**结尾的行动指引必须是读者现在能用的动作,不是演讲现场感的转录**

定义:很多文章在结尾会给读者一个"接下来可以做什么"的指引(去看文档、去试产品、去读源代码)。这条原则没问题——读者读完想行动是好事。但写稿时容易把演讲现场对台下听众有效的指引,1:1 转录到文章里,变成对读者完全无用的细节。

为什么写稿时容易翻车:转录原文里讲者经常给现场听众具体动作("GitHub 链接在幻灯片里"、"中场来 DeepMind 展台找我"、"扫这个二维码加微信群"、"现场前 50 名免费"、"会后我会留下"、"Q&A 时间问我")。这些指引在台下有效,因为听众身处现场。直译进文章,读者既看不到幻灯片,也不在大会展台,这些指引变成纯噪音。

典型错例 → 改写:

- 「Kaggle 的四款产品已经上线、大部分开源,**GitHub 链接在演讲幻灯片里**。两位讲者也都说了,**中场会在 DeepMind 展台**。」 → 删除括号内的两条现场指引,改成读者能直接行动的版本:「这四款产品都已经在 Kaggle 上线,大部分开源。」
- 「想了解更多的话,**Q&A 环节有人问了类似问题**,他给的答案是……」 → 删「Q&A 环节」框架,直接给答案
- 「演讲最后,**他鼓励大家中场来展台**继续聊」 → 整句删除
- 「想知道具体配置的,**幻灯片在 SlideShare 上**」 → 改成「这套配置已经在 SlideShare 上,搜 X 标题就能找到」(具体到读者能搜的关键词)

判断方法:文章结尾的每一条行动指引,问一句——"这条指引,读者现在打开手机/电脑,能不能照着做?"

- 能 → 留(例如"在 Kaggle 上搜 X 就能找到")
- 不能 → 改成能照做的形式,或删除

合格行动指引的形态:

- **可搜索关键词**:「在 Kaggle 上搜 Benchmarks」「Hugging Face 搜 Y」
- **可访问 URL**:「项目地址 github.com/X/Y」
- **可下载 / 可使用的资源**:「这套 prompt 模板他放在 GitHub 上了,可以直接抄」
- **明确的下一步动作**:「想试的话,装 X 包跑 Y 命令就能跑通」

不合格行动指引(全是现场感转录,删):

- 「PPT 在大会官网上 / 幻灯片里」(读者不知道大会官网是什么)
- 「展台 / 会后 / 中场可以找他聊」(读者不在现场)
- 「Q&A 环节有更多细节」(读者听不到 Q&A)
- 「现场举手过半的人都做过 X」(读者不在现场,无法 verify)
- 「他在台上还演示了 Y」(读者没看到演示)

grep 触发词:`幻灯片里` `PPT 上` `展台` `Q&A.{0,3}问到` `中场` `会后` `现场.{0,5}(举手|笑|鼓掌|看到)` `演讲幻灯片` `大会官网`

## 十、高阶手法（可选）

以下是几条从"作者自己的高表现个人写作"里提炼出来的手法。它们不是必须，也不适合每一篇文章。当素材允许时用一两条，可以让文章的气质明显和千篇一律的 AI 生成稿拉开距离。

### 10.1 两件事 + 合流结尾

有些文章不必只讲一件事。把两件看似无关的事并列放进一篇里，各自推进几段，末尾让它们在一个判断或画面处悄悄合流。读者不会被你提前告知合流点，他们是在结尾自己感觉到两条线合在一起的。

为什么可用：比"论点 + 支撑证据"的线性结构更给读者参与感。他们不是被灌输结论，而是自己在脑子里把这两条线接起来了。

什么时候用：当访谈 / 演讲 / 原材料里有一个核心论点，同时你手头还有一件相关但出身不同的事（一个项目、一个书摘、一段个人经历），两者合起来能让观点更立体。纯信息密度型文章（比如纯行业新闻）不用这个结构。

一个示范骨架：
- A 线：素材里某位主讲人的一个反常识判断
- B 线：一个能把这个判断落地的案例，或一个来自另一个领域的对照
- 合流：在倒数第二段让 A 线的判断和 B 线的事实在一个具体动作上交汇，不点破

### 10.2 用另一个领域的框架重讲

把眼前的具体事件用另一个领域的解释框架重新描述一遍。比如把"引体向上"重新描述成"和时空曲率的短暂对抗"，把"给 AI 装手指"重新描述成哲学家说的"视觉是主动的探索性的"。

这不是比喻。比喻是「A 像 B」，这种手法是「用 B 的解释框架去重新映射 A 的事实」，结果是让日常动作获得意外的重量。

什么时候用：当你在写的事件本身容易被读者当成"又一条产品新闻"或"又一次模型更新"时，用一个不相关领域的框架把它重讲一次，能让事件显得陌生又合理。这是让文章具备"智识杠杆感"最容易学的一招。

训练方法：每写到一件具体的事，先问一遍"有没有另一个领域的框架可以把这件事重新讲一遍"。物理、生物、哲学、建筑、运动、音乐都可以是候选。不强求，合适的时候用。

### 10.3 自嘲切换器

如果文章的典故密度或理论密度高，中间要穿插一两句口语化的自嘲做节奏调节。比如正经讨论了半页哲学之后突然写一句「装杯完毕，说说第二件」，或者在展示完一个技术成果之后写「说好的我来，怎么还抢活呢」。

为什么重要：没有这个调节器，严肃段落之间缺少松弛点，读者会觉得作者在端着。自嘲不是破坏严肃，是保护严肃——让读者知道作者自己也意识到刚才那段有点端，愿意自己先笑一下。

操作建议：每当发现自己连续写了两段以上理论或典故，就检查一下要不要插一句松的。自嘲的句式不重要，字眼要俏皮但不油滑，节奏要对。这条没法机械套，得靠感觉。

**自嘲也可以做整篇文章的标题反转钩**：当一篇文章是对前文（自己的上一篇、对一个观点的初步表态）的续写或修正时，标题可以用自嘲承认前文的不足，暗示反转即将发生。这是延续型写作里非常有效的钩。

样本：「fine 是我肤浅了」（承认上一篇拉踩某派别的判断过度，暗示这一篇要给一个更深的视角）。

适用情境只在续写型文章里——上一篇文章读者刚读完、还在记忆里时，这一篇标题用自嘲钩。一次性独立文章不要硬套自嘲标题，会让读者困惑"作者在自嘲什么"。

### 10.4 重复 + 递进的段落节奏

定义：连续 3-5 个段落用同一个词或短句开头，每段做一层递进。这是中文修辞里最有力的工具之一，但只在文章里有一个核心概念需要从多个层面拆开讲时用。

样本（来自用户的卡夫卡《审判》分析）：

```
必要性破坏了判断和思考，执行和适应就可以了。

必要性消解了责任，反正是必要的，也就没有责任主体，每个人都是机器中的零件。

必要性摧毁了行动，人被必要性奴役，不再具备自由行动、改变世界的意愿和能力。

必要性最厉害的地方，是慢慢让你也按照它的逻辑理解自己。
```

四段连用「必要性 + 动词」开头，每段把"必要性"对人的侵蚀往内推一层——破坏判断 → 消解责任 → 摧毁行动 → 内化逻辑。最后一段用「必要性最厉害的地方」做收尾，递进闭环。

为什么有效：

1. **节奏感强**——重复开头给读者一个稳定的句式预期，每段是一个独立单元，读起来像鼓点
2. **递进意识被强迫**——每段必须比上一段更深一层，不能只是同义重复，写的时候会逼自己把概念拆透
3. **核心概念被锚定**——一个词重复 4 次，会成为整段的视觉和语义锚点，读完读者一定会记住

什么时候用：

- 当一个核心概念（比如"必要性""可读性""沉默"）从多个角度才能讲清楚时
- 段落之间是清楚的递进关系（程度加深 / 范围扩大 / 内化加深），不是并列罗列
- 一篇文章里只用一次。用两次会让读者感觉作者在炫技

不要乱用：

- 没有递进关系的并列，用项目符号或正常段落即可，不要硬套这个结构
- 用在工程描述里会显得过于文学化（"性能优化破坏了 X，性能优化消解了 Y"——读者会觉得作者在发力过猛）
- 不要用在文章开头的钩子段落里，开头需要的是具体场景或硬数字，不是节奏型修辞

### 10.5 不要硬抄的东西

有一种风格看起来特别有辨识度，就是**把古文、哲学家名字、物理定律直接混排进正文，不加引号不加解释**。这个看似是技巧，其实不是。它是长期阅读带出来的底层结构，作者能这么写是因为他真的读过，读过的东西在写的时候会自己浮上来。

如果你不是正在消化同类素材、也没有这些人物作为日常思维背景，硬抄这种密度会反效果。读者会觉得"这人好像很有文化但我没跟上"，而且容易翻车——引用错、语境错、轻重错。

规则：**不要为了显风格而增加典故密度**。典故出现的触发条件应该是"这个判断本来就可以由这句话更精准地表达"，不是"我希望这篇文章看起来更有文人气"。如果素材本身不带这种密度，不加。

## 十一、来源交代格式

文章最后必须注明来源。格式固定：

```
内容来源，[渠道 / 频道 / 平台]「[原标题]」，[类型] 链接 [URL]
```

样本里的实际用法：

- 「内容来源，AI Engineer 频道「The Multi-Agent Architecture That Actually Ships — Luke Alvoeiro, Factory」，YouTube 链接 https://...」
- 「内容来源，Lenny's Podcast「Why cultivating agency matters more than cultivating skills in the AI era | Max Schoening (Notion)」，YouTube 链接 https://...」
- 「内容来源，GRITCULT「Marketing is dead. Long live The Distribution Engineer.」，原文链接 https://...」

如果主讲人身份对读者有帮助，可以在来源行后补一句 "演讲者为 X（Y 公司，身份）"。
