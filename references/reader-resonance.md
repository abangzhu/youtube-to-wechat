# 读者共鸣设计

三条正向规则，补充现有禁区类规则的空白。适用于公众号 + 技术/产品知识分享场景。

---

## 规则 1：价值翻译——技术解释必须落地到读者处境

每次介绍一个技术概念、工具功能或研究结论，必须在**同一段落内**回答「对读者意味着什么」。纯描述性的技术内容不完整。

**判断标准**：读完这段话，读者能不能用一句话说出「所以我以后可以/应该……」？不能就补。

```
❌ 弱版：
Context Caching 会在连续对话中复用相同 token，降低 API 成本。

✓ 强版：
Context Caching 会在连续对话中复用相同 token，降低 API 成本。
对于每天高频调用的独立开发者，同等使用量下月账单可能缩减一半。
```

```
❌ 弱版：
该框架支持多智能体并行编排。

✓ 强版：
该框架支持多智能体并行编排——以前需要串行等待的步骤可以同时跑，
对需要处理大批量文档的工作流来说，速度差距在 3-5 倍。
```

**常见缺口**：工具对比类文章逐条列出功能差异，但不说「哪类人用哪个更合适」；趋势分析列出变化，但不说「对正在做 X 的人意味着什么」。

## 规则 2：写给一个具体的人，不是一类人群

**在 Step 4 分析阶段**，明确写出本文的目标读者快照，并在写稿时以这个人为唯一听众。

### 快照格式

> 「[具体身份] + [具体处境/当下正在解决的问题]」

### 适合本号的典型读者快照

- 「正在开发或刚上线第一个 AI 产品、需要判断某个工具/方法论是否值得用的独立开发者」
- 「在大公司里做产品或工程、想理解 AI 能力边界、准备在团队内推动落地的人」
- 「对 AI 技术感兴趣但不写代码、需要用中文把概念讲清楚再决定要不要学的人」

### 如何用快照指导写稿

快照确定后，每逢遇到「要不要展开解释这个细节」「要不要举这个例子」，就问：**这个具体的人需要吗？对他有用吗？**

不需要在正文里写出快照，它是写作指南针，不是文章要素。

**反面信号**：如果一篇文章试图同时服务「技术小白」和「有经验的工程师」，两类读者最终都会觉得读了一半不是给自己写的。选一个。

## 规则 3：分享触发点——至少一个「这说的就是我」的时刻

公众号的「在看」和「分享」主要由读者的身份认同感驱动：**这篇文章说出了我想说但没说清楚的话，分享它能表达我是谁。**

每篇文章应包含至少一个触发这种感受的时刻。

### 三种有效触发方式

**① 点名一个具体困境**

说出目标读者有但通常不好意思承认的处境。

```
「你有没有遇到过这种情况：prompt 调了三遍，效果还是差不多，
最后只能靠加大量 few-shot 硬撑。」
```

**② 一个反直觉结论**

让读者产生「我之前的认知是错的，幸好看到这篇」的感受。

```
「大多数人把 RAG 做慢的原因不是检索慢，而是 rerank 没做。」
「更长的 context window 不一定带来更好的理解，有时候反而更差。」
```

**③ 说出一个读者有但没说出口的判断**

文章替读者把一个模糊的感觉变成清晰的判断，读者的反应是「对，我就是这么觉得的」。

```
「市面上 90% 的 AI 写作工具本质上是在做同一件事：
把 GPT-4 的输出套一个好看的界面，再收你 20 美元一个月。」
```

### 注意事项

- 触发点不需要是全文最重要的论点，它的作用是制造共鸣感，不是承担论证责任。
- 不要为了制造共鸣而夸大或歪曲。触发点必须是真实的、有具体依据的。
- 一篇文章一到两个就够，多了会显得在讨好读者。

## 规则 4：认知颠覆型开头——挑战读者已有的假设

适用于技术/产品知识分享类文章，尤其是「某个领域正在发生变化」的主题。

技术读者的自我认知是「我懂这个领域」。他们分享文章的动机不是「获得新信息」，而是「我以为我懂，但这篇更新了我的认知」——被更新的感受比获得信息的感受更强，更容易触发分享。

### 结构公式

> **[读者持有的默认假设]** → **[具体证据说明假设失效]** → **[新的理解框架]**

### 从高表现文章提取的正例

**DeepMind 文章**：
- 假设（隐含）：「工程能力够就能做好 Agent」
- 颠覆证据：「模型不弱，代码能力也够，卡住人的是思维方式没跟着切换」
- 新框架：「过去工程师是交通控制员，做 Agent 要变成调度员」

**CI/CD 文章**：
- 假设（隐含）：「加 Runner、优化 pipeline 可以解决速度问题」
- 颠覆证据：「PR 是原来的 4 倍，人类不可能再评审得完每一个 PR」
- 新框架：「这套流程为人类节奏设计，扛不住 Agent 节奏，问题在架构层」

**Playbook 文章**：
- 假设（普遍）：「AI 改变了执行成本」
- 颠覆（深一层）：「42% 的创业公司死于做了没人要的东西，这个比例只会上升」
- 新框架：「瓶颈不再是你能做什么，而是你选择做什么」

### 与已有规则的关系

- **与「反直觉数字优先」的差异**：数字是证据形式，认知颠覆是整体框架。数字可以是颠覆证据的一种，但认知颠覆的核心是先命名假设，再提供否定证据。
- **与「分享触发点」的差异**：分享触发点讲的是共鸣（「这说的就是我的处境」），认知颠覆讲的是被更新（「我以为我懂，但这一层我没想到」）。两者可以同时存在于一篇文章中：开头用认知颠覆建立张力，中间或结尾用分享触发点产生共鸣。

### 注意事项

- 假设必须是真实存在的，不能是稻草人。读者看到那个假设要能认出「对，我确实是这么想的」。
- 颠覆证据要具体，不能只是「其实没那么简单」——这是空颠覆。具体意味着有数字、有案例、有来源。
- 不是每篇都适合这个结构。信息梳理类、工具评测类文章可以直接进入内容；认知颠覆型开头最适合「范式正在切换」「大家普遍认为 X，但实际上 Y」类主题。

---

## 四条规则的协作关系

| 规则 | 解决什么问题 | 在哪个 Step 主要起作用 |
|---|---|---|
| 价值翻译 | 防止技术段落自说自话 | Step 5 写稿、Step 6 段级精修时逐段检查 |
| 具体读者 | 防止内容试图服务所有人 | Step 4 分析阶段确定，写稿时持续参照 |
| 分享触发点 | 防止文章有信息量但缺共鸣感 | Step 5 写稿、Step 6 整稿自审时检查是否存在 |
| 认知颠覆型开头 | 防止开头没有张力、读者没有继续读下去的动力 | Step 4 判断是否适用；Step 5 写开头时执行 |
